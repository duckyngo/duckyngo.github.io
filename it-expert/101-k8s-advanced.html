📚 Thân Thế & Lai Lịch

Đại Sư bắt đầu với raw YAML — hàng nghìn dòng copy-paste giữa environments. Rồi chàng ta gặp Helm: "Template YAML? Values per environment? Dependency management? Take my money!" Chàng ta viết Helm charts cho mọi thứ, thậm chí chart cho... deploying other charts.

Khi Kustomize xuất hiện (built vào kubectl), chàng ta có dilemma: Helm hay Kustomize? Câu trả lời sau 5 năm: CẢ HAI. Helm cho third-party packages, Kustomize cho internal applications. "Đừng dogmatic," chàng ta nói.

Rồi Operators: custom controllers mở rộng Kubernetes API. CRDs cho custom resources. Operator pattern cho complex stateful applications. "Kubernetes cho phép bạn extend nó theo cách bạn muốn — đó là sức mạnh và cũng là nỗi đau," chàng ta thở dài khi debug operator thứ 50.

📜 Bí Kíp

1. Helm — Package Manager for K8s: helm install/upgrade/rollback cho release management. values.yaml cho configuration. Chart dependencies cho modular architecture. Helm hooks cho pre/post install/upgrade. Chart testing (helm test). Chart repositories cho distribution. Tip: helm template cho dry-run rendering — review YAML trước apply.
2. Kustomize — Overlay-Based Customization: base/ + overlays/dev + overlays/prod. Patches cho targeted changes. ConfigMap/Secret generators. Namespace transformations. Strategic merge patches vs JSON patches. Built vào kubectl (kubectl apply -k). Tốt cho: internal apps, simple environment differences. No templating language to learn.
3. Operators — Extend Kubernetes: Custom Resource Definitions (CRDs) define new resource types. Controller watches CRDs, reconciles desired vs actual state. Operator SDK (Go, Ansible, Helm) cho development. Operator Hub cho community operators. Use cases: database management (PostgreSQL Operator), certificate management (cert-manager), backup management.
4. Helm Chart Best Practices: Use semantic versioning cho charts. values.yaml documented với comments. README cho usage instructions. Chart.yaml với dependencies pinned. templates/_helpers.tpl cho reusable template functions. Lint: helm lint trước publish. Test: helm test sau install.
5. CRD Design: Validation schema trong CRD spec. Status subresource cho controller updates. Versioning (v1alpha1 → v1beta1 → v1). Conversion webhooks cho version migration. Finalizers cho cleanup logic. Printer columns cho kubectl get output. Đừng over-design — start simple, iterate.

📖 Chuyện Kể Đêm Khuya

📦 Chuyện Helm Upgrade Phá Hủy PVC

helm upgrade release-name chart/ với change trong StatefulSet volume claim template. Helm delete và recreate StatefulSet — PVCs bị orphaned. New PVCs created empty. Database data: gone. 4 giờ downtime. DBA khóc.

Root cause: Helm re-creates StatefulSet khi volumeClaimTemplate changes. PVCs preserved nhưng detached. Fix: backup data trước upgrade. KHÔNG thay đổi volumeClaimTemplate trên existing StatefulSets. Manual migration nếu cần change storage.

💡 Bài học: StatefulSet volumeClaimTemplates IMMUTABLE sau creation. Helm upgrade có thể recreate StatefulSet = data loss risk. ALWAYS backup stateful data trước Helm upgrades. Test upgrades trên staging. Review helm diff trước production upgrade.

🔄 Chuyện Operator Infinite Reconciliation

Custom operator cho database management. Bug trong reconciliation logic: update status → trigger watch → reconcile → update status → trigger watch... Infinite loop. API server flooded with requests. etcd storage growing. Other controllers slow down. Cluster-wide impact.

Fix: add generation tracking (skip reconcile if generation unchanged). Rate limiting trong controller. Resource version comparison. Proper status update logic.

💡 Bài học: Operator reconciliation loops là common bug. Track resource generation/version. Rate limit reconciliation. Test operator under load. Watch for API server throttling. controller-runtime library handles many edge cases — use it.

📋 Chuyện Helm Values Override War

Team A set values.yaml trong git. Team B override qua helm upgrade --set. CI/CD pipeline dùng values file. Manual override via --set takes precedence → CI/CD và git out of sync. 'Why is production different from what git says?' — vì ai đó helm upgrade --set 3 tháng trước.

💡 Bài học: NEVER use helm upgrade --set cho production. All values trong version-controlled values files. GitOps (ArgoCD) cho source of truth. helm get values cho current values. Audit Helm releases regularly cho drift.

⚠️ Những Sai Lầm Kinh Điển

1. helm install without reviewing values: Default values có thể expose services publicly, use weak passwords. ALWAYS review values.yaml. helm show values chart/ trước install.
2. Not version-pinning chart dependencies: Chart.yaml: dependency version '^1.0.0' → pulls 1.9.0 → breaking changes. Pin exact versions.
3. CRDs không có validation: Invalid CRD resources accepted → controller crash. Schema validation bắt buộc cho production CRDs.
4. Too many Helm releases: helm list returns 500 releases. Cleanup old releases. Release history limit (--history-max).
5. Operator without RBAC: Operator with cluster-admin = security risk. Minimum required RBAC for operator service account.
