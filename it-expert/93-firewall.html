📚 Thân Thế & Lai Lịch

Tường Lửa Đại Sư bắt đầu sự nghiệp với iptables trên Red Hat 7. Một chain, một rule, DROP all. Giờ chàng ta quản lý firewall policies cho 500 servers với hàng nghìn rules, và vẫn nhớ cú pháp iptables -A INPUT -p tcp --dport 22 -j ACCEPT bằng cơ bắp.

Chàng ta đã chứng kiến evolution: ipchains → iptables → nftables. Mỗi lần đổi, chàng ta phải relearn syntax. "Tại sao không giữ iptables?" — vì nftables nhanh hơn, unified framework, và không cần nhớ 4 tables riêng biệt.

Kỷ niệm đau thương nhất: chạy iptables -F (flush all rules) trên production server qua SSH... quên rằng default policy là DROP. Mất SSH access. Phải drive 2 giờ đến datacenter để console vào server.

📜 Bí Kíp

1. Default DENY — Nguyên Tắc Vàng: Default policy: DROP tất cả. Chỉ ACCEPT những gì cần thiết. Whitelist approach, không blacklist. iptables -P INPUT DROP, iptables -P FORWARD DROP, iptables -P OUTPUT ACCEPT (hoặc DROP nếu paranoid). NHƯNG: LUÔN thêm rule allow SSH TRƯỚC khi set default DROP.
2. Stateful Inspection: ESTABLISHED,RELATED connections tự động allow. Chỉ cần rule cho NEW connections. -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT phải là rule đầu tiên. Giảm số rules cần evaluate = better performance.
3. nftables — Tương Lai Của Linux Firewall: Unified framework thay thế iptables, ip6tables, arptables, ebtables. Syntax nhất quán hơn. Atomic rule replacement (không downtime). Sets và maps cho efficient matching. Backward compatible qua iptables-nft translation layer. Migrate dần, đừng big-bang.
4. NAT — Network Address Translation: SNAT/MASQUERADE cho outbound traffic. DNAT/port forwarding cho inbound. MASQUERADE cho dynamic IP (home/VPN), SNAT cho static IP (server). NAT traversal phức tạp cho VoIP, gaming. conntrack table size tuning cho high-connection servers.
5. Rule Ordering Matters: Rules evaluated top-to-bottom, first match wins. Most specific rules first, generic rules last. Frequently matched rules near top cho performance. Log rules trước drop rules cho debugging. Comment rules (iptables -m comment) cho maintainability.
6. Firewall Management At Scale: firewalld cho RHEL/CentOS (zones, services). UFW cho Ubuntu (simplified interface). Ansible/Terraform cho managed firewalls. Centralized policy management. Change management process — mọi thay đổi phải reviewed. Audit log cho compliance.

📖 Chuyện Kể Đêm Khuya

🔒 Chuyện Firewall Rule Lock Out Admin

Admin SSH vào production server, chạy: iptables -P INPUT DROP rồi iptables -F. Default policy DROP + flush all rules = NO traffic allowed. Bao gồm SSH session hiện tại. Connection freeze. Admin mất access. 500 users affected. Phải physical console access — datacenter ở 50km.

From that day: LUÔN thêm SSH allow rule TRƯỚC set default DROP. Hoặc dùng at command: at now + 5 minutes <<< 'iptables -P INPUT ACCEPT' — tự động rollback nếu mất access.

💡 Bài học: KHÔNG BAO GIỜ flush rules khi default policy là DROP qua remote session. Script rollback: cron job restore rules sau 5 phút nếu không cancelled. Test firewall changes trên staging trước. Có out-of-band management (IPMI/iLO) cho emergency.

💥 Chuyện iptables Flush Disaster

Team triển khai firewall mới cho cluster 20 servers. Ansible playbook: step 1 flush old rules, step 2 apply new rules. Network glitch giữa step 1 và step 2 → Ansible timeout. Kết quả: 20 servers với FLUSHED firewall rules = hoàn toàn OPEN. Mọi port exposed to internet. 15 phút trước khi team phát hiện và manually apply rules.

Fix: atomic firewall update — iptables-restore thay vì individual commands. nftables: nft -f ruleset.nft cho atomic replacement. Ansible handler chạy restore nếu apply fails.

💡 Bài học: Firewall updates phải atomic — all or nothing. iptables-restore < rules.v4 cho atomic apply. nftables native atomic replacement. ĐỪNG BAO GIỜ flush-then-apply — race condition window = security hole.

🛡️ Chuyện DDoS Attack Và Firewall Rate Limiting

Web server nhận 100K requests/giây từ botnet. iptables rules cho HTTP: ACCEPT. Server overwhelmed, legitimate users can't access. Team panic: 'Block tất cả IPs!' — nhưng botnet có 50K unique IPs, rotate liên tục.

Đại Sư implement: hashlimit module cho rate limiting per source IP (30 req/s max), GeoIP blocking cho countries không có customers, SYN cookies cho SYN flood protection, connlimit cho max connections per IP. Kết hợp với Cloudflare DDoS protection phía trước. Attack mitigated.

💡 Bài học: Firewall alone không đủ cho DDoS. Rate limiting (iptables hashlimit/recent) cho basic protection. CDN/DDoS protection (Cloudflare, AWS Shield) cho serious attacks. SYN cookies, connection limits, GeoIP filtering là layers of defense.

⚠️ Những Sai Lầm Kinh Điển

1. iptables -F qua SSH: Flush rules khi default DROP = lock yourself out. ALWAYS ensure SSH rule exists before flushing.
2. ACCEPT ALL default policy: 'Tôi sẽ thêm DROP rules sau.' Rồi quên. Default DENY, whitelist approach.
3. Không log dropped packets: Không biết traffic bị block là security hay misconfiguration. LOG trước DROP cho troubleshooting.
4. Rules không có comments: 500 rules, không ai biết rule nào cho mục đích gì. Comment EVERY rule với date và purpose.
5. Firewall rules không version controlled: Manual changes = drift, inconsistency. Store rules trong git, apply qua automation.
6. NAT table quá tải: conntrack table full = new connections dropped. Monitor conntrack count, tăng nf_conntrack_max cho high-connection servers.
