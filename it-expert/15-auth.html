📜 Thân Thế & Lai Lịch

Trần Bảo Vệ từng là developer bình thường cho đến khi ứng dụng ông xây bị hack. Attacker exploit JWT stored trong localStorage, XSS → token stolen → toàn bộ user accounts compromised. 50,000 users ảnh hưởng. Báo chí đưa tin. Startup gần chết. Từ ngày đó, ông thề sẽ học bảo mật đến nơi đến chốn. Ông đọc RFC 6749 (OAuth 2.0) 5 lần, RFC 7519 (JWT) 3 lần, nghiên cứu mọi attack vector từ XSS, CSRF đến session hijacking, token confusion. Ông trở thành 'Người Gác Cổng' — security reviewer cho mọi PR liên quan đến auth. Triết lý: 'Đừng tự implement auth nếu không phải expert. Dùng libraries, dùng services (Auth0, Clerk, Supabase Auth). Sai ở auth = game over.'

🗝️ Bí Kíp

Bí Kíp #1: OAuth 2.0 + OIDC — Hiểu đúng để dùng đúng

OAuth 2.0 = AUTHORIZATION (cấp quyền truy cập resource). OIDC (OpenID Connect) = AUTHENTICATION (xác minh danh tính) built on top of OAuth 2.0. Authorization Code Flow + PKCE = standard cho SPAs và mobile apps (thay thế Implicit Flow đã deprecated). Access Token: ngắn hạn (15-60 phút), dùng để access API. Refresh Token: dài hạn, dùng để lấy access token mới. ID Token: JWT chứa user info, KHÔNG dùng để access API. Đừng nhầm lẫn 3 loại token này.

Bí Kíp #2: JWT — Biết dùng, biết hạn chế

JWT = Header.Payload.Signature (base64). Stateless: server không cần lưu session. Nhưng: KHÔNG THỂ revoke JWT trước expiry (trừ khi maintain blocklist = stateful lại). Đặt expiry ngắn (15-30 phút) + refresh token rotation. ĐỪNG lưu JWT trong localStorage (XSS vulnerable). Lưu trong httpOnly, Secure, SameSite cookie — browser auto-send, JavaScript KHÔNG access được. Payload: ĐỪNG lưu sensitive data (PII, passwords) — JWT chỉ encoded, KHÔNG encrypted.

Bí Kíp #3: Session Management — Cái cũ nhưng vẫn tốt

Server-side sessions: session ID trong cookie, data trong server (Redis/DB). Ưu: dễ revoke (xóa session), dễ manage (track active sessions), data không expose cho client. Nhược: stateful (cần shared session store cho multiple servers). Session fixation prevention: regenerate session ID sau login. Session timeout: absolute (tối đa 24h) + idle (15-30 phút no activity). HttpOnly + Secure + SameSite=Strict cho session cookie.

Bí Kíp #4: RBAC vs ABAC — Ai được làm gì?

RBAC (Role-Based Access Control): User → Role → Permissions. Simple, well-understood. Role: admin, editor, viewer. Permission: create, read, update, delete. Nhược: role explosion khi business phức tạp. ABAC (Attribute-Based Access Control): policy based on attributes (user department, resource owner, time of day, IP address). Flexible nhưng complex. Recommendation: start with RBAC, evolve to ABAC khi RBAC không đủ. Tools: Casbin, Open Policy Agent (OPA), Cerbos.

Bí Kíp #5: Multi-Factor Authentication (MFA)

Something you know (password) + something you have (phone/hardware key) + something you are (biometric). TOTP (Time-based One-Time Password): Google Authenticator, Authy — simple, no network needed. WebAuthn/Passkeys: hardware security keys, biometric — phishing-resistant, best security. SMS OTP: better than nothing but vulnerable to SIM swapping — avoid for high-security. Implement MFA cho admin accounts MANDATORY, offer for all users.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: JWT trong localStorage — Mở Cửa Cho Hacker

SPA lưu JWT access token trong localStorage. XSS vulnerability trong third-party chat widget → attacker inject script → localStorage.getItem('token') → token stolen → attacker access API as victim. 5,000 accounts compromised trong 6 giờ trước khi team phát hiện. Attacker dùng stolen tokens để đổi email → reset password → full account takeover. Thiệt hại: user data leaked, GDPR violation notification required, trust destroyed. Fix: chuyển JWT sang httpOnly cookie (JavaScript không access được → XSS không steal được). Thêm CSRF protection (SameSite cookie + CSRF token cho non-GET requests). Implement token rotation + revocation. Audit tất cả third-party scripts.

💡 Bài học: NEVER store tokens trong localStorage/sessionStorage — XSS = game over. httpOnly cookies là nơi an toàn nhất cho tokens trong browser. Nếu dùng localStorage (mobile WebView edge case), phải eliminate MỌII XSS vectors: CSP strict, sanitize user input, audit third-party scripts.

📖 Chuyện 2: Infinite Token Expiry — Vé Vĩnh Viễn Cho Hacker

API dùng JWT, developer set expiry 'cho tiện': expiresIn: '365d'. User login 1 lần/năm, không bị logout, UX tuyệt vời! Cho đến khi employee quit company — JWT vẫn valid 11 tháng nữa. Employee truy cập internal API, download customer data. Worst case: JWT không thể revoke (stateless). Team phải implement token blocklist trong Redis — basically biến JWT thành stateful session. Tại sao không dùng sessions từ đầu? Fix: access token expiry 15 phút, refresh token 7 ngày + rotation (mỗi lần dùng refresh token → issue new refresh token, invalidate old one). Employee offboarding: revoke tất cả refresh tokens.

💡 Bài học: Token expiry phải ngắn cho access tokens (15-60 phút). Refresh tokens dài hơn nhưng phải ROTATABLE + REVOCABLE. Employee offboarding checklist: revoke tokens, deactivate accounts, rotate shared secrets. Nếu bạn cần revoke JWT instantly → bạn cần token blocklist hoặc nên dùng sessions.

📖 Chuyện 3: OAuth Misconfiguration — Redirect URI Open Redirect

OAuth flow: user login → redirect back to app. Developer set redirect_uri validation loose: chỉ check domain, không check full path. Attacker craft URL: redirect_uri=https://legit-app.com/callback/../../../evil-site.com. Authorization server redirect token đến evil site. Account takeover. Fix: EXACT match redirect_uri (không wildcard, không path traversal). Register tất cả valid redirect URIs trong OAuth provider. Validate state parameter để prevent CSRF trong OAuth flow.

💡 Bài học: OAuth redirect_uri validation phải EXACT MATCH. Không dùng wildcard domains, không allow path traversal, không allow subdomain matching rộng. State parameter MANDATORY cho CSRF protection. PKCE MANDATORY cho public clients (SPAs, mobile apps).

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Tự implement crypto/auth

Rolling your own authentication/encryption = rolling your own parachute. Dùng proven libraries: bcrypt/argon2 cho password hashing, established JWT libraries, Auth0/Clerk/Supabase cho hosted auth.

❌ Sai Lầm #2: Password requirements phức tạp

'Minimum 12 chars, 1 uppercase, 1 lowercase, 1 number, 1 special character, no dictionary words' → users viết trên sticky note. NIST recommendation: minimum 8 chars, check against breached password list, no complexity requirements. Dùng passkeys khi possible.

❌ Sai Lầm #3: Không rate limit login endpoint

Brute force attack: try 10,000 passwords/second. Rate limit: 5 attempts → lockout 15 phút. Account lockout → captcha → progressive delay. Implement fail2ban hoặc application-level rate limiting.

❌ Sai Lầm #4: Expose error details trong auth responses

'Email not found' vs 'Wrong password' → attacker biết email nào tồn tại (account enumeration). Response chung: 'Invalid email or password' cho mọi failure case.

❌ Sai Lầm #5: Không invalidate sessions khi password change

User đổi password (vì nghi bị hack) nhưng old sessions vẫn active → hacker vẫn access được. Password change PHẢI invalidate tất cả existing sessions/tokens.
