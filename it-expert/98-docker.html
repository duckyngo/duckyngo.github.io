📚 Thân Thế & Lai Lịch

Đại Sư nhớ thời "works on my machine" là câu trả lời cho mọi bug. Dev nói: "Máy tôi chạy được!" QA nói: "Máy tôi không!" Ops nói: "Production cũng không!" Vòng xoáy đổ lỗi bất tận.

Docker xuất hiện như vị cứu tinh: đóng gói application + dependencies vào container. "Works on my machine? Great, we'll ship your machine." Đại Sư bắt đầu containerize mọi thứ — web apps, databases, monitoring, thậm chí cả... desktop environment (đừng hỏi tại sao).

Sau 8 năm Docker, chàng ta đã thấy đủ thứ: 2GB images vì dùng ubuntu base cho Python script, secrets hardcode trong Dockerfile, và docker-compose down -v xóa sạch database production. Chàng ta giờ dạy Docker như dạy võ: đẹp và nguy hiểm.

📜 Bí Kíp

1. Dockerfile Best Practices: Multi-stage builds: build stage (compile) + runtime stage (minimal image). FROM alpine/slim/distroless cho small images. COPY thay vì ADD (trừ khi cần extract tar). RUN instructions combine với && cho fewer layers. .dockerignore cho node_modules, .git, build artifacts. Non-root USER bắt buộc. HEALTHCHECK cho container health monitoring.
2. Image Optimization: Minimal base image: alpine (~5MB) vs ubuntu (~77MB). Layer caching: put rarely-changing layers first (dependencies trước source code). Multi-stage: build image 1GB → runtime image 50MB. Distroless images cho minimal attack surface. Pin versions: python:3.11-slim, không phải python:latest. Scan images cho vulnerabilities (trivy, grype).
3. Docker Compose — Local Development: services, volumes, networks trong một file. Environment variables cho configuration. depends_on với health checks cho startup order. docker compose watch (2.22+) cho hot reload. Profiles cho optional services. Override files (docker-compose.override.yml) cho dev customization. ĐỪNG dùng Compose cho production — dùng Kubernetes hoặc ECS.
4. Networking: Bridge network (default): containers communicate by name. Host network: container dùng host network stack (performance). Overlay network: multi-host networking (Swarm). Custom networks cho isolation. Published ports (-p) expose to host. KHÔNG expose database ports to host trừ khi development.
5. Volumes — Persistent Data: Named volumes cho data persistence (docker volume create). Bind mounts cho development (source code). tmpfs mounts cho sensitive/temporary data. Volume drivers cho remote storage. BACKUP volumes regularly. docker-compose down -v XÓA TẤT CẢ VOLUMES — cẩn thận!
6. Resource Limits: --memory và --cpus cho resource limits. Prevent container từ eating all host resources. --restart=unless-stopped cho auto-restart. ulimit settings cho file descriptors. PID limits cho fork bomb protection. Log rotation: --log-opt max-size=10m --log-opt max-file=3.

📖 Chuyện Kể Đêm Khuya

📦 Chuyện 2GB Image Từ Wrong Base

Developer build Python Flask app: FROM ubuntu:latest → apt-get install python3 → pip install requirements → COPY app. Image size: 2.1GB. Pull time trên CI/CD: 5 phút. Deployment time tăng 300%. Docker registry storage full nhanh chóng.

Đại Sư refactor Dockerfile: FROM python:3.11-slim. Multi-stage build. Remove unnecessary packages. Image size: 85MB. 96% reduction. Pull time: 15 giây. CI/CD pipeline giảm 4 phút per build.

💡 Bài học: Base image matters enormously. python:slim (~120MB) vs ubuntu + python (~800MB). Multi-stage builds separate build-time và runtime dependencies. Alpine even smaller nhưng musl libc có thể gây compatibility issues. Regularly audit image sizes — docker images --format table.

💾 Chuyện docker-compose down -v Xóa Data

Developer chạy docker-compose down -v để 'clean restart' local environment. -v flag xóa tất cả volumes — bao gồm PostgreSQL data volume. Database chứa 6 tháng test data, seed data scripts, và custom migrations. Tất cả mất.

'Nhưng em chỉ muốn restart containers!' — docker-compose restart là đủ. down -v là nuclear option. Team thêm README warning và alias docker-compose-safe='docker compose down' (không có -v).

💡 Bài học: docker compose down -v là DESTRUCTIVE operation. Hiểu sự khác biệt: restart (restart containers), down (stop + remove containers/networks), down -v (stop + remove EVERYTHING including volumes). Label critical volumes. Backup trước destructive operations.

🔐 Chuyện Secret Trong Dockerfile

Dockerfile: ENV DATABASE_URL=postgres://admin:SuperSecret123@db:5432/production. Image push to Docker Hub public repository. Credentials exposed. Bất kỳ ai pull image có thể docker inspect và thấy password.

Worse: mọi layer trong image history đều giữ secrets. Ngay cả khi developer 'fix' bằng cách remove ENV trong layer sau, docker history vẫn show. Phải rebuild image hoàn toàn.

💡 Bài học: KHÔNG BAO GIỜ put secrets trong Dockerfile (ENV, ARG, COPY). Dùng runtime environment variables, Docker secrets, hoặc mount secret files. docker history shows ALL layers. Multi-stage builds: secrets trong build stage không appear trong final stage (nếu đúng). Scan images cho leaked secrets.

⚠️ Những Sai Lầm Kinh Điển

1. Dùng latest tag: latest có thể thay đổi bất cứ lúc nào. Pin specific version: nginx:1.25-alpine. Reproducible builds require pinned versions.
2. Run as root trong container: Default Docker runs as root. Compromised container = root access. USER directive cho non-root. Rootless Docker cho extra security.
3. Không set resource limits: Container eat all host RAM/CPU → other containers starved. --memory, --cpus limits. Docker Compose: deploy.resources.limits.
4. Không cleanup unused resources: docker system df cho disk usage. docker system prune cho cleanup. Dangling images, stopped containers, unused volumes accumulate.
5. COPY . . quá sớm trong Dockerfile: Source code change → rebuild ALL subsequent layers. COPY requirements.txt first → pip install → COPY . → tận dụng layer caching.
