📚 Thân Thế & Lai Lịch

Đại Sư từng implement service-to-service security bằng tay: mỗi service cần TLS certificate, mỗi connection cần authentication, mỗi request cần authorization. 50 microservices × 50 connections = 2500 configurations. "There must be a better way," chàng ta than thở.

Istio ra đời: "Install service mesh, get mTLS, observability, and traffic management for free!" Chàng ta install Istio. Pod count double (sidecars). Memory usage +40%. "Free" có giá riêng của nó. Nhưng: mTLS everywhere, distributed tracing automatic, canary deployments simple. Trade-off đáng giá cho complex microservice architectures.

Rồi Cilium xuất hiện: eBPF-based, sidecar-less service mesh. Performance overhead minimal. "Best of both worlds," chàng ta nghĩ. Và đúng vậy — cho đến khi debug eBPF programs lúc 2 giờ sáng.

📜 Bí Kíp

1. Sidecar vs Sidecarless: Sidecar (Istio, Linkerd): Envoy proxy per pod. Pros: transparent, well-understood. Cons: resource overhead, latency. Sidecarless (Cilium): eBPF in kernel. Pros: lower overhead, no extra containers. Cons: newer, kernel version requirements. Ambient mesh (Istio): per-node proxy instead of per-pod. Middle ground.
2. mTLS — Mutual TLS: Encrypt all service-to-service traffic. Mutual authentication: both sides verify identity. Automatic certificate rotation. Zero-code changes in applications. PERMISSIVE mode for migration (accept both TLS and plaintext). STRICT mode for production. Service identity based on SPIFFE.
3. Traffic Management: Canary deployments: route 5% traffic to new version. A/B testing: route by headers. Circuit breaking: stop sending to failing services. Retry policies: automatic retries with backoff. Timeout policies: prevent slow services from blocking others. Fault injection: test resilience.
4. Observability — Golden Signals: Latency: p50/p95/p99 per service pair. Traffic: requests per second. Errors: error rate per service. Saturation: resource utilization. Distributed tracing: automatic span injection. Service graph: visualize dependencies. Metrics exported to Prometheus/Grafana.
5. Choosing a Service Mesh: Istio: feature-rich, complex, large community. Linkerd: simple, lightweight, Rust data plane. Cilium: eBPF-based, sidecarless, fastest. Consul Connect: HashiCorp ecosystem integration. Question first: do you NEED a service mesh? Under 10 services → probably not.

📖 Chuyện Kể Đêm Khuya

🕸️ Chuyện Istio Sidecar Double Pod Count

500-pod cluster enable Istio. Pod count: 500 app + 500 Envoy sidecar = 1000 pods. Memory per sidecar: 100-200MB. Total sidecar memory: 50-100GB. Node count increase 40%. Cost increase: $4,000/tháng. DevOps team: 'We added observability nhưng now we can not afford it.'

Optimization: tune sidecar resource limits (reduce from 200MB to 80MB). Exclude non-critical services from mesh. Enable Istio ambient mode (per-node instead of per-pod). Result: overhead reduced 60%.

💡 Bài học: Service mesh resource overhead is significant. Calculate overhead TRƯỚC deployment. Tune sidecar resources. Exclude services that don't need mesh. Consider sidecarless options (Cilium, Istio ambient). Start with subset of services, expand gradually.

🔐 Chuyện mTLS Cứu Mạng Từ Insider Threat

Network engineer compromise internal credentials. Attempt lateral movement trong Kubernetes cluster. Without mesh: service-to-service traffic plaintext, easy to intercept and replay. WITH mTLS: every connection requires valid SPIFFE identity certificate. Attacker cannot spoof service identity without compromising service account.

mTLS blocked: unauthorized service-to-service calls, traffic interception (encrypted), service impersonation. Security team detect anomalous connection attempts qua mesh telemetry.

💡 Bài học: mTLS is THE reason to adopt service mesh for security-conscious organizations. Zero-trust network: don't trust internal network. Service identity > network identity. mTLS + authorization policies = defense in depth. Even without full mesh features, mTLS alone justifies adoption.

🔀 Chuyện Canary Deployment Bắt Bug

New version of payment service deployed as canary (5% traffic). Mesh dashboard shows: canary version error rate 15% vs stable 0.1%. Automatic rollback trigger. Zero customer impact. Without mesh: bug would hit 100% traffic, require manual detection and rollback.

Post-mortem: database schema change incompatible with new code. Caught by mesh-level error rate monitoring trước khi widespread impact.

💡 Bài học: Canary deployments qua service mesh = safety net cho production deploys. Automatic rollback based on error rates. Traffic shifting gives confidence. Percentage-based rollout: 5% → 25% → 50% → 100%. Monitor golden signals at each step.

⚠️ Những Sai Lầm Kinh Điển

1. Deploy mesh cho 5 microservices: Overkill. mTLS library hoặc simple mutual TLS sufficient. Mesh overhead ≠ justify cho small architectures.
2. Enable all features at once: mTLS + traffic management + policies + tracing simultaneously. Start with mTLS, add features incrementally.
3. Không tune sidecar resources: Default Envoy sidecar resources generous. Tune dựa trên actual traffic. Most sidecars need less than defaults.
4. Ignore control plane HA: Istiod single instance = mesh SPOF. Multiple istiod replicas + PDB cho production.
5. Debugging mesh issues: Problem in app or mesh? Bypass mesh (curl directly) to isolate. istioctl analyze cho configuration issues. Envoy access logs cho traffic debugging.
