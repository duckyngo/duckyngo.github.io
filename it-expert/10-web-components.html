Trương Vi Mô — Bậc Thầy Web Components & Micro-frontend

"Chia để trị — nhưng chia sai thì chỉ có trị loạn."

📜 Thân Thế & Lai Lịch

Trương Vi Mô lớn lên trong một công ty lớn nơi mà 15 teams cùng contribute vào một monolithic frontend. Mỗi lần deploy là một cuộc phiêu lưu: Team A merge code → break Team B's feature → rollback → Team C đang chờ deploy → deadlock. Ông mơ về một thế giới nơi mỗi team có thể deploy độc lập, dùng framework riêng, không ai block ai. Ông khám phá Web Components và Micro-frontends — rồi phát hiện rằng independence đi kèm với complexity. Style leaking, shared state, routing conflicts, performance overhead. Sau nhiều năm thử nghiệm, ông ngộ ra: 'Micro-frontend không phải giải pháp kỹ thuật — nó là giải pháp tổ chức. Nếu team bạn nhỏ, bạn không cần nó.'

🗝️ Bí Kíp

Bí Kíp #1: Shadow DOM — Encapsulation thực sự

Shadow DOM tạo isolated DOM tree: CSS bên trong không leak ra ngoài, CSS bên ngoài không affect bên trong. element.attachShadow({mode: 'open'}). Slots cho content projection: <slot name='header'>. Cẩn thận: Shadow DOM KHÔNG inheritable cho tất cả CSS properties — font-family, color inherit qua shadow boundary, nhưng class selectors không. Styling Shadow DOM từ bên ngoài: CSS custom properties (CSS variables) penetrate shadow boundary, ::part() selector cho exposed parts.

Bí Kíp #2: Custom Elements — HTML tags của riêng bạn

customElements.define('my-button', MyButton) — tạo HTML tag tái sử dụng. Lifecycle: connectedCallback (attached to DOM), disconnectedCallback (removed), attributeChangedCallback (attribute updated). Phải có hyphen trong tên (my-button, not mybutton). Framework-agnostic: Custom Elements hoạt động trong React, Vue, Angular, vanilla. Lit library giúp viết Custom Elements dễ dàng hơn với declarative templates và reactive properties.

Bí Kíp #3: Micro-frontend Patterns

1) Build-time integration: shared component library, npm packages. Simplest nhưng coupled deploy. 2) Server-side integration: SSI, ESI, server compose HTML fragments. 3) Runtime integration: Module Federation (Webpack 5), import maps, iframe (old school). 4) Edge-side integration: edge workers compose at CDN level. Chọn based on: team autonomy needs, performance budget, complexity tolerance. Rule: nếu < 3 teams, bạn KHÔNG cần micro-frontends.

Bí Kíp #4: Module Federation — Webpack 5's Game Changer

Module Federation cho phép nhiều Webpack builds share code at runtime. Host app load remote modules từ separate deployments. Shared dependencies (React, lodash) loaded một lần. Versioning: host và remote có thể dùng different versions, Federation auto-resolve. Vite Federation plugin cho Vite ecosystem. Cẩn thận: shared singleton (React) phải cùng version, khác version = cryptic runtime errors.

Bí Kíp #5: Khi nào KHÔNG cần Micro-frontends

Micro-frontends thêm complexity: separate builds, deployment coordination, shared state management, consistent UX across teams. Chi phí justify khi: > 5 teams cùng code frontend, deploy frequency cần independent, teams cần technology freedom. KHÔNG justify khi: team nhỏ (< 10 devs), single product team, startup đang tìm product-market fit. Monolith with good architecture > premature micro-frontends.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Style Leaking — Khi CSS Không Biết Ranh Giới

3 teams, 3 micro-frontends trên cùng trang. Team A dùng global CSS class .button → màu xanh. Team B cũng có .button → màu đỏ. Khi cả hai mount cùng page: buttons có khi xanh, có khi đỏ, phụ thuộc vào CSS load order. Monday deployment: Team A deploy trước → tất cả buttons xanh. Wednesday: Team B deploy → buttons random xanh đỏ. Fix đầu tiên: CSS Modules (class name hashing). Fix tốt hơn: Shadow DOM encapsulation. Fix tốt nhất: Design System shared — cả 3 teams dùng cùng button component từ shared library, không tự viết.

💡 Bài học: CSS isolation là BẮT BUỘC trong micro-frontend. Options: CSS Modules, Shadow DOM, CSS-in-JS scoping, Tailwind prefix. Tốt nhất: shared Design System Library — consistent UI + no duplication + single source of truth cho styles.

📖 Chuyện 2: Micro-frontend Chaos — Frankenstein App

Công ty 'hiện đại hóa' bằng micro-frontends. Team A dùng React 18, Team B dùng Vue 3, Team C dùng Angular 15. Mỗi team tự do chọn tech stack — sounds great on paper. Reality: page load 3 frameworks → 2MB+ JavaScript runtime overhead. Shared header/footer phải render trong cả 3 frameworks → 3 implementations → 3 lần maintain. User experience inconsistent: React team dùng Material UI, Vue team dùng Vuetify, Angular team dùng PrimeNG. App trông như Frankenstein — mỗi section một style. Lesson learned: 'Technology freedom' != 'Technology anarchy'. Team tạo Architecture Decision Records (ADRs): max 2 frameworks, shared Design System bắt buộc, performance budget per micro-frontend.

💡 Bài học: Technology freedom trong micro-frontend cần guardrails. Shared: Design System, auth module, analytics, performance budget. Independent: business logic, deployment, internal architecture. Limit framework diversity — mỗi framework thêm = bundle size tăng + DX complexity tăng.

📖 Chuyện 3: Module Federation Version Hell

Host app dùng React 18.2. Remote micro-frontend vô tình upgrade lên React 18.3 (minor version). Module Federation share React as singleton. Deploy remote → host app crash: 'Invalid hook call. Hooks can only be called inside of the body of a function component.' Nguyên nhân: 2 instances React co-exist khi version mismatch. Debug mất 8 giờ vì error message misleading (looks like a hook error, actually a dependency resolution error). Fix: pin shared dependency versions, add CI check cho version compatibility trước khi deploy remote.

💡 Bài học: Shared dependencies trong Module Federation PHẢI version-controlled nghiêm ngặt. Pin versions, automated compatibility checks trong CI, semver ranges cần test kỹ. Singleton dependencies (React, Vue) KHÔNG được có version mismatch — crash guaranteed.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Micro-frontend cho team 3 người

3 devs không cần infrastructure overhead của micro-frontends. Monolith with good folder structure, lazy loading, và CI/CD đã đủ. Micro-frontend là organizational pattern, không phải technical flex.

❌ Sai Lầm #2: Quên shared dependencies gây duplicate

React loaded 3 lần từ 3 micro-frontends = 3 x 40KB = 120KB wasted. Configure shared dependencies properly — chỉ load một lần, share across all micro-frontends.

❌ Sai Lầm #3: iframe cho micro-frontend

iframe hoạt động cho isolation nhưng: không share routing, không responsive layout, SEO nightmare, performance overhead, accessibility broken. Dùng trong last resort, không phải default choice.

❌ Sai Lầm #4: Custom Elements không progressive enhance

Web Components mà không hoạt động khi JavaScript disabled = bad. Implement fallback content inside custom element tags. Graceful degradation for critical functionality.

❌ Sai Lầm #5: Over-engineering shared state

Micro-frontends communicate qua: Custom Events (simple), shared state library (medium), event bus (complex). Đừng dùng Redux store shared giữa micro-frontends — coupling cao, independence mất. Prefer loose coupling: events + contracts.
