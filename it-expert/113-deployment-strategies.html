Chuyên Gia Blue/Green, Canary & Rolling Deployments
"Deployment strategy giống như cách bạn nhảy xuống hồ: canary là nhúng chân trước, rolling là đi từng bước, blue/green là nhảy thẳng — nhưng có bể bơi phụ nếu nước lạnh."
🔵 Blue/Green 🐤 Canary 🔄 Rolling Update 🏁 Feature Flags ⏮️ Rollback 🎯 A/B Testing 📊 Progressive Delivery

📚 Thân Thế & Lai Lịch

Chiến Lược Gia nhớ thời "deploy Friday 5pm, pray Saturday, fix Sunday." Deployments là events — stressful, risky, manual. "Deploy window" là 2 giờ sáng. "Rollback plan" là "fix forward and hope."

Rồi chàng ta học blue/green: two identical environments, switch traffic. Canary: route 5% traffic to new version. Rolling: replace instances gradually. Feature flags: deploy code but control behavior. Deployments trở thành non-events — boring, predictable, automated. "Boring deployments are good deployments," chàng ta thường nói.

📜 Bí Kíp

1. Blue/Green Deployment: Two identical environments: Blue (current) and Green (new). Deploy to Green, test, switch traffic. Instant rollback: switch back to Blue. Pros: simple rollback, full testing before switch. Cons: double infrastructure cost during deployment. Suitable for: databases difficult (both must point to same DB).
2. Canary Deployment: Route small percentage (1-5%) traffic to new version. Monitor error rates, latency. Gradually increase: 5% → 25% → 50% → 100%. Automated rollback if metrics degrade. Pros: minimize blast radius, real user testing. Cons: complex routing, must handle two versions simultaneously.
3. Rolling Update: Replace instances gradually: remove old, add new. Default Kubernetes strategy. maxSurge: how many extra pods. maxUnavailable: how many can be down. Pros: no extra infrastructure, simple. Cons: mixed versions during rollout, slower rollback (rollout undo).
4. Feature Flags: Deploy code but control behavior via configuration. Toggle features per user, percentage, environment. A/B testing for product decisions. Gradual rollout without deployment. Kill switch cho problematic features. Tools: LaunchDarkly, Unleash, Flagsmith. Decouple deployment from release.
5. Progressive Delivery: Combination: canary + automated analysis + auto-promote/rollback. Argo Rollouts, Flagger, Istio cho Kubernetes. Analysis: Prometheus metrics, custom checks. Automated decision: promote or rollback based on SLOs. The goal: zero-touch, safe deployments.

📖 Chuyện Kể Đêm Khuya

🐤 Chuyện Canary Bắt Critical Bug

New payment service version deployed as canary (5% traffic). 10 phút sau, canary metrics show: error rate 8% (vs 0.1% stable). Automated rollback triggered. Total users affected: ~200 (5% of 4000 concurrent). Without canary: 4000 users affected, manual detection time ~30 phút.

Root cause: database connection pool size wrong in new config. Bug only manifests under load (not caught in staging with low traffic). Canary with real production traffic caught it.

💡 Bài học: Canary deployments catch bugs that staging misses. Production traffic = real conditions. Automated rollback based on metrics = fast recovery. 5% canary = 95% protection. Define clear success/failure metrics TRƯỚC deployment.

🔵 Chuyện Blue/Green Switch Nightmare

Blue/green deployment cho stateful application. Green environment deployed, tested, switch DNS to Green. Users on Blue still have active sessions. Sessions lost. Shopping carts empty. Users furious. Rollback to Blue: users who started on Green lose THEIR sessions.

Root cause: session state stored in-memory on application servers. Blue/Green switch = session loss. Fix: externalize sessions to Redis. Both Blue and Green share session store. Switch is seamless.

💡 Bài học: Blue/Green requires stateless applications hoặc shared state store. Session management critical: Redis, database-backed sessions. Database migrations must be forward AND backward compatible (both versions run simultaneously briefly). Test switch procedure including rollback.

🏁 Chuyện Feature Flag Cứu Launch

Product launch ngày thứ Hai. Friday deployment: new feature behind feature flag (disabled). Weekend testing: enable flag cho internal users. Monday 9am: enable flag cho 10% users. Monitor. 12pm: 50% users. Issue detected: performance regression cho specific user segment. Disable flag immediately. Fix code. Re-enable. Smooth launch. No deployment needed cho any of these changes.

💡 Bài học: Feature flags decouple deployment from release. Deploy anytime, release when ready. Instant kill switch cho problematic features. Gradual rollout cho risk mitigation. A/B testing cho product decisions. Feature flag lifecycle management: clean up old flags — tech debt.

⚠️ Những Sai Lầm Kinh Điển

1. No rollback plan: Deploy and hope. ALWAYS have tested rollback procedure. Automated rollback cho canary/progressive delivery.
2. Deploying Friday afternoon: Weekend = reduced team. Issues harder to handle. Deploy Monday-Thursday, early in the day, full team available.
3. Feature flags never cleaned up: 100 feature flags, nobody knows which are still needed. Flag lifecycle management. Remove after 100% rollout stable.
4. Testing in production without guardrails: Canary without monitoring = just deploying. Automated analysis and rollback based on metrics required.
5. Big bang releases: 6 months of changes in one deployment. Small, frequent deployments = lower risk per deployment. Continuous delivery.
