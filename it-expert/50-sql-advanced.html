📜 Bí Kíp

Bí Kíp #1: CTEs — Viết SQL Như Viết Văn

WITH active_users AS (...), user_orders AS (...), order_summary AS (...) SELECT * FROM order_summary. CTEs biến query 100 dòng thành chuỗi steps dễ đọc. Mỗi CTE = 1 concept. Debug từng CTE riêng biệt. Đặt tên CTE có ý nghĩa. CTE không phải performance tool — nó là readability tool.

Bí Kíp #2: Window Functions — Superpower Của SQL

ROW_NUMBER(): đánh số thứ tự. RANK()/DENSE_RANK(): xếp hạng. LAG()/LEAD(): lấy row trước/sau. SUM() OVER(ORDER BY date): running total. SUM() OVER(PARTITION BY customer_id): sum per customer. Không cần self-join, không cần subquery. Window functions = elegant + fast.

Bí Kíp #3: Recursive CTEs — Traversal Trees Và Hierarchies

WITH RECURSIVE org_tree AS (SELECT id, name, manager_id, 1 as level FROM employees WHERE manager_id IS NULL UNION ALL SELECT e.id, e.name, e.manager_id, t.level + 1 FROM employees e JOIN org_tree t ON e.manager_id = t.id) SELECT * FROM org_tree. Traverse org charts, category trees, BOM (Bill of Materials). Nhớ set max recursion depth!

Bí Kíp #4: Query Plan — Đọc Được Query Plan Là Đọc Được Tương Lai

EXPLAIN ANALYZE: xem actual execution. Key metrics: Seq Scan vs Index Scan, Nested Loop vs Hash Join, Sort method (in-memory vs disk), actual rows vs estimated rows. Nếu estimated rows sai nhiều → statistics outdated → ANALYZE table. Nếu Seq Scan trên bảng lớn → thiếu index.

Bí Kíp #5: LATERAL JOIN — Secret Weapon

SELECT c.*, latest_order.* FROM customers c, LATERAL (SELECT * FROM orders o WHERE o.customer_id = c.id ORDER BY created_at DESC LIMIT 1) latest_order. Lấy "top N per group" elegantly. Thay thế correlated subquery trong SELECT clause. Performance thường tốt hơn window function approach cho "get latest per group".

Bí Kíp #6: GROUPING SETS, CUBE, ROLLUP — Aggregation Nâng Cao

GROUP BY ROLLUP(country, city): tạo subtotals (country level) và grand total. GROUP BY CUBE(country, city): mọi combination. GROUP BY GROUPING SETS((country, city), (country), ()): custom combinations. Thay vì UNION ALL 3 queries aggregation khác nhau = 1 scan duy nhất.

Bí Kíp #7: Optimization Patterns — Viết SQL Nhanh

Sargable predicates: WHERE created_at >= '2024-01-01' (index-friendly) thay vì WHERE YEAR(created_at) = 2024 (function = no index). Avoid SELECT DISTINCT khi EXISTS đủ. EXISTS nhanh hơn IN cho large subqueries. Avoid OR — rewrite thành UNION ALL khi có thể.

📖 Chuyện Kể Đêm Khuya

🚀 Chuyện 1: Window Function Cứu 100x Performance

Query: "Tìm customers có order amount tăng liên tục 3 tháng gần nhất." Anh junior viết: 3 self-JOINs trên bảng orders (50M rows). Self-join 50M × 50M × 50M. Query chạy... timeout sau 30 phút.

Lão viết lại bằng window function: LAG(monthly_amount, 1) OVER (PARTITION BY customer_id ORDER BY month) — so sánh tháng này với tháng trước. 1 pass duy nhất qua data. Query time: 18 giây. Từ timeout xuống 18 giây. Anh junior nhìn lão như nhìn phù thủy.

💡 Bài học: Self-JOIN cho "compare with previous row" = quadratic complexity. Window functions (LAG, LEAD, running totals) = linear complexity. Master window functions = level up SQL skills 10x.
🔄 Chuyện 2: Self-Join Nightmare — N^2 Không Tha Ai

Requirement: "Tìm pairs of products thường được mua cùng nhau." Anh dev viết: SELECT a.product_id, b.product_id FROM order_items a JOIN order_items b ON a.order_id = b.order_id AND a.product_id < b.product_id. Order_items: 100M rows. Self-join 100M × 100M = hàng tỷ tỷ comparisons. Database crash.

Lão refactor: pre-aggregate thành basket (array of products per order), rồi dùng UNNEST + CROSS JOIN chỉ trên orders có > 1 item. Giảm từ 100M self-join xuống 5M baskets. Kết hợp bloom filter cho popular products. Query time: 45 giây, acceptable.

💡 Bài học: Self-JOIN trên large tables = dangerous. Tìm cách reduce dataset trước JOIN. Pre-aggregate, filter early, use window functions hoặc array operations thay thế khi có thể.
📊 Chuyện 3: Query Plan Tells All — Câu Chuyện Của 1000x Speedup

Dashboard query chạy 45 giây. Team đã "optimize" bằng cách thêm index nhưng không giúp. Lão gõ EXPLAIN ANALYZE và thấy: estimated rows = 100, actual rows = 5,000,000. Statistics hoàn toàn sai → planner chọn Nested Loop Join thay vì Hash Join.

ANALYZE table_name; — update statistics. Chạy lại: planner chọn Hash Join. Query time: 45ms. Từ 45 giây xuống 45ms = 1000x speedup, chỉ bằng 1 lệnh ANALYZE. Team: "Sao không ai nói cho mình biết lệnh ANALYZE?" Lão: "Vì các bạn không đọc query plan."

💡 Bài học: LUÔN đọc EXPLAIN ANALYZE trước khi optimize. Check estimated vs actual rows. Nếu lệch nhiều → ANALYZE để update statistics. Statistics outdated = query planner make wrong decisions = slow queries.

⚠️ Những Sai Lầm Kinh Điển

❌ SELECT DISTINCT thay vì fix JOINs

Query trả duplicates → thêm DISTINCT → "fix rồi!" Thực ra: JOINs tạo ra cartesian product vì thiếu condition. Fix root cause, không dùng DISTINCT che giấu.

❌ Function trên indexed column trong WHERE

WHERE UPPER(email) = 'TEST@ABC.COM' — index on email vô dụng. Dùng WHERE email = lower('TEST@ABC.COM') hoặc functional index.

❌ NOT IN với NULL values

WHERE id NOT IN (SELECT id FROM other_table) — nếu subquery trả NULL, toàn bộ NOT IN = FALSE. Dùng NOT EXISTS thay thế, luôn luôn safe hơn.

❌ ORDER BY trong subquery

SELECT * FROM (SELECT * FROM t ORDER BY date) sub WHERE rownum <= 10 — ORDER BY trong subquery bị optimizer bỏ qua (trừ khi có LIMIT). Đặt ORDER BY ở outer query.

❌ Quá nhiều CTEs nested

CTE A dùng CTE B dùng CTE C dùng CTE D... 8 levels deep. Khó debug, khó optimize. Flatten khi có thể, materialize intermediate results cho complex pipelines.
