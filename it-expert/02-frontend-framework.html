📜 Thân Thế & Lai Lịch

Nguyễn Khung Cứng bắt đầu sự nghiệp vào thời jQuery thống trị thiên hạ. Khi Angular.js (phiên bản 1) ra đời, ông là một trong những người đầu tiên nhảy vào, hào hứng với two-way data binding như trẻ con có đồ chơi mới. Rồi Angular 2 ra đời — và breaking changes đã khiến ông mất 3 tháng migrate, tóc rụng thêm 30%.

Trải qua "Cuộc Chiến Framework Vĩ Đại" khi React, Vue, và Angular tranh giành thiên hạ, ông ngộ ra rằng framework nào cũng có thời — quan trọng là hiểu nguyên lý cốt lõi: component model, reactivity, và state management. Ông tự phong mình là "Đại Hiệp Không Phe" — sẵn sàng dùng bất cứ framework nào phù hợp với dự án.

Câu nói nổi tiếng của ông trên Stack Overflow: "Framework tốt nhất là framework mà team bạn biết dùng. Framework tệ nhất là framework bạn chọn vì Medium article nói nó 'revolutionary'."

🗝️ Bí Kíp

Bí Kíp #1: React — Triết lý "UI = f(state)"

React không phải framework, nó là library. Triết lý cốt lõi: UI là hàm của state. Khi state thay đổi, React tính toán virtual DOM diff và chỉ update phần DOM thực sự thay đổi. React Hooks (useState, useEffect, useMemo, useCallback) là paradigm shift — từ class-based sang functional. Nhưng cẩn thận: useEffect không phải lifecycle method — nó là "synchronization mechanism". Nắm vững dependency array, cleanup function, và strict mode double-render behavior. React Server Components (RSC) là tương lai — component chạy trên server, zero JS gửi về client.

Bí Kíp #2: Vue — Sự cân bằng hoàn hảo

Vue 3 với Composition API + <script setup> là developer experience tốt nhất hiện tại. Reactivity system dựa trên Proxy — tự động track dependencies, không cần dependency array như React. ref() cho primitives, reactive() cho objects, computed() cho derived state. Single File Components (.vue) gói gọn template + logic + style — có người ghét, có người yêu, nhưng ai dùng rồi đều quen. Vue DevTools là công cụ debug tốt nhất trong các framework.

Bí Kíp #3: Angular — Opinionated Enterprise Beast

Angular là framework "batteries included" — routing, forms, HTTP client, i18n, testing, tất cả built-in. TypeScript là bắt buộc (không phải optional). Dependency Injection, RxJS cho reactive programming, NgModules cho code organization. Angular Signals (v16+) đang dần thay thế Zone.js — hiệu năng tốt hơn, fine-grained reactivity. Dùng Angular khi team lớn, project phức tạp, cần consistency. Đừng dùng khi team 2 người làm landing page.

Bí Kíp #4: Svelte — Compiler, không phải Runtime

Svelte là compiler, không ship runtime framework lên browser. Code let count = 0 tự động reactive — không cần useState hay ref(). Bundle size nhỏ hơn đáng kể. $: cho reactive declarations — compiler tự tính dependency graph. Nhưng ecosystem nhỏ hơn React/Vue, ít thư viện third-party hơn. Svelte 5 với Runes ($state, $derived, $effect) đang thay đổi cách tiếp cận — từ compiler magic sang explicit primitives.

Bí Kíp #5: Solid.js — React Concepts, No Virtual DOM

Solid.js dùng JSX giống React nhưng KHÔNG có virtual DOM. Components chỉ chạy MỘT LẦN — không re-render. Reactivity system fine-grained dựa trên signals (createSignal, createEffect, createMemo). Performance benchmark gần native DOM manipulation. Nếu bạn thích React API nhưng muốn performance tốt hơn, Solid là lựa chọn. Cẩn thận: destructuring props sẽ mất reactivity — đây là "gotcha" số 1 cho người từ React sang.

Bí Kíp #6: Virtual DOM vs No Virtual DOM — Cuộc tranh luận vĩnh cửu

Virtual DOM (React, Vue) tạo bản sao của DOM trong memory, diff khi state thay đổi, rồi batch update DOM thật. Overhead? Có. Nhưng nó đơn giản hóa mental model — developer chỉ cần nghĩ "render lại toàn bộ". No virtual DOM (Svelte, Solid) update trực tiếp DOM — nhanh hơn nhưng compiler/reactivity system phải thông minh hơn. Thực tế: với app thông thường, sự khác biệt performance là không đáng kể. Chọn based on DX, ecosystem, và team experience.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Cuộc Chiến Framework — Battle Royale tại Meeting Room

Năm 2020, công ty quyết định build sản phẩm mới. Meeting room biến thành chiến trường. Phe React: "React có ecosystem lớn nhất, tuyển dụng dễ nhất!" Phe Vue: "Vue dễ học, developer happy, productivity cao!" Phe Angular: "Enterprise-grade, TypeScript built-in, Google đứng sau!"

CTO — người không biết JavaScript — nhìn benchmark performance trên một blog và tuyên bố: "Chúng ta dùng Svelte." Cả team im lặng. Không ai biết Svelte. CTO đã đọc một bài "Svelte is the future" trên Medium và bị convince hoàn toàn.

6 tháng sau, team gồm 8 React developers đang viết Svelte, Google mỗi dòng code, Stack Overflow hầu như không có câu trả lời cho edge cases. Cuối cùng, product launch trễ 4 tháng. Không phải vì Svelte tệ — mà vì team không có experience. Sản phẩm cuối cùng chạy tốt, nhưng cái giá là tóc và deadline.

💡 Bài học: Framework tốt nhất là framework team bạn BIẾT DÙNG. Thử nghiệm công nghệ mới ở side project, không phải ở sản phẩm có deadline. Nếu muốn adopt framework mới, cho team 2-3 tháng học trước, build internal tool nhỏ trước khi all-in.

📖 Chuyện 2: Angular to React Migration — Cuộc Di Cư Đau Thương

Team quyết định migrate từ Angular sang React vì "Angular quá verbose" và "React ecosystem phong phú hơn". Kế hoạch ban đầu: 3 tháng. Thực tế: 14 tháng.

Tuần 1: "Dễ lắm, chỉ cần convert component thôi." Tuần 4: "Ủa, Angular DI system không có equivalent trong React?" Tháng 2: "RxJS observable patterns... làm sao convert sang hooks?" Tháng 4: "Angular Forms module làm được nhiều thứ mà React form library nào cũng thiếu." Tháng 8: "Chúng ta đang maintain HAI codebase song song."

Kết quả cuối cùng: app chạy được, nhưng 40% code là "wrapper" để mimic Angular patterns trong React. Team mới join nhìn vào code: "Cái này là React hay Angular viết trong React?"

💡 Bài học: Migration framework là quyết định BUSINESS, không phải quyết định KỸ THUẬT. Tính toán chi phí thực tế: thời gian migrate, feature freeze, regression bugs, team learning curve. Thường thì "strangler fig pattern" (dần dần thay thế) tốt hơn "big bang rewrite". Và đôi khi, đáp án đúng là KHÔNG migrate.

📖 Chuyện 3: useEffect Infinite Loop — Vòng Lặp Vô Tận

Junior dev viết useEffect fetch data, set state, nhưng quên dependency array. Component mount → fetch → setState → re-render → useEffect chạy lại → fetch → setState → re-render... API server nhận 10,000 requests trong 30 giây. CloudWatch alarm kêu inh ỏi. AWS bill tháng đó tăng gấp 3.

Nghĩ mà xem: một dòng code thiếu [] mà gây ra cả một sự cố production. Senior ngồi review code, thở dài: "Cái dependency array mà cả team React core còn thừa nhận là khó hiểu, thì junior miss là bình thường."

💡 Bài học: useEffect là API bị hiểu sai nhiều nhất của React. Nó KHÔNG phải lifecycle method. Nó là tool để SYNC component với external system. Rule: nếu effect không interact với browser API, server, hoặc subscription — bạn có thể không cần useEffect. Dùng ESLint plugin react-hooks/exhaustive-deps — nó sẽ cứu bạn.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Chọn framework theo hype, không theo nhu cầu

"Twitter nói framework X tốt nhất" — nhưng Twitter không biết team bạn gồm 2 junior, deadline 3 tháng, và product chỉ là CRUD app. Landing page? Astro/vanilla. CRUD app? Bất kỳ framework nào team biết. Real-time dashboard? Framework có strong reactivity. Đừng dùng búa tạ cho đinh ghim.

❌ Sai Lầm #2: Không tách biệt business logic khỏi framework

Khi toàn bộ business logic nằm trong useEffect, useState, hay Angular service — bạn bị lock-in. Viết business logic thuần (plain functions/classes), rồi "wrap" trong framework layer. Khi migrate framework, chỉ cần viết lại wrapper.

❌ Sai Lầm #3: Over-component — chia nhỏ quá mức

Một button không cần là component riêng nếu nó chỉ dùng 1 lần. "Component cho mọi thứ" dẫn đến prop drilling, file explosion, và developer mở 20 file để hiểu 1 feature. Rule of thumb: extract component khi nó được REUSE hoặc khi logic quá phức tạp.

❌ Sai Lầm #4: Không hiểu rendering behavior

React re-render khi state/props thay đổi — KỂ CẢ children. React.memo(), useMemo(), useCallback() là optimization tools, nhưng dùng sai còn tệ hơn không dùng (vì compare cost). Profile trước khi optimize — đừng premature optimization.

❌ Sai Lầm #5: Dùng global state cho mọi thứ

Form input value KHÔNG cần trong Redux store. Modal open/close state KHÔNG cần trong Vuex. Phân biệt: server state (React Query/SWR), global UI state (theme, auth), và local component state. 80% state nên là local.

❌ Sai Lầm #6: Bỏ qua accessibility khi dùng custom components

Framework UI libraries đẹp nhưng nhiều cái quên ARIA attributes. Custom dropdown không navigate được bằng keyboard. Modal không trap focus. Dùng headless UI libraries (Radix, Headless UI) — chúng handle accessibility, bạn handle styling.
