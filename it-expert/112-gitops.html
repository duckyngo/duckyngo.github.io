📚 Thân Thế & Lai Lịch

GitOps Đạo Sĩ tin rằng Git là chân lý duy nhất. "Nếu nó không trong Git, nó không tồn tại. Nếu nó trong Git, nó PHẢI tồn tại trên cluster." Đây là triết lý GitOps: Git repository là declarative source of truth cho infrastructure và application state.

Chàng ta phát hiện ArgoCD và giác ngộ: controller continuously reconciles cluster state với Git state. Drift? Auto-sync sửa. Muốn rollback? Git revert. Muốn audit? Git log. PR review = deployment review. "CI/CD without GitOps is imperative chaos," chàng ta tuyên bố.

Nhưng GitOps cũng có challenges: sync loops khi controller và application fight over state, secrets management (đừng commit secrets vào git!), và "git revert deploys old version" — đôi khi đó là bug, không phải feature.

📜 Bí Kíp

1. GitOps Principles: Declarative: desired state described declaratively. Versioned: Git as source of truth, immutable, auditable. Automated: approved changes auto-applied. Reconciled: controllers ensure actual = desired state. Pull-based: controllers pull from Git (not push-based CI/CD).
2. ArgoCD — Popular GitOps Tool: Kubernetes-native. Web UI cho visualization. Multi-cluster support. Application CRD. Sync policies: auto-sync, self-heal. Hooks cho pre/post sync. RBAC cho team access. SSO integration. ApplicationSet cho templated applications.
3. Flux — CNCF GitOps: Kubernetes-native (set of controllers). Source controllers: Git, Helm, OCI. Kustomize controller cho overlays. Helm controller cho Helm releases. Notification controller cho alerts. Image automation: auto-update image tags in Git. Multi-tenancy support.
4. App of Apps Pattern: Root application manages child applications. Single entry point cho cluster configuration. Hierarchical: platform team manages root, app teams manage children. ApplicationSet cho dynamic application generation. Monorepo hoặc multi-repo — both supported.
5. Sync Strategies: Auto-sync: Git change → auto-deploy. Manual sync: require button click. Self-heal: detect và fix drift. Prune: remove resources not in Git. Sync windows: restrict sync times. Progressive delivery: Argo Rollouts cho canary/blue-green via GitOps.
6. Secrets in GitOps: ĐỪNG commit plain secrets to Git. Sealed Secrets: encrypt secrets, commit ciphertext. External Secrets Operator: reference external secret store. SOPS: encrypt specific values in YAML. Vault integration cho dynamic secrets.

📖 Chuyện Kể Đêm Khuya

🔄 Chuyện ArgoCD Sync Loop

ArgoCD auto-sync enabled. Operator modifies resource status field. ArgoCD detects 'drift' (status differs from Git). ArgoCD re-syncs, overwriting status. Operator re-modifies status. ArgoCD re-syncs. Infinite loop. API server overwhelmed. Cluster performance degraded.

Fix: configure ArgoCD ignoreDifferences cho status fields. ResourceExclusions cho operator-managed resources. Understand what ArgoCD should manage vs what operators manage. Not everything in cluster should be ArgoCD-managed.

💡 Bài học: ArgoCD + operators = potential sync conflicts. ignoreDifferences cho dynamic fields (status, metadata.generation). ResourceExclusions cho resources managed by other controllers. Carefully scope what ArgoCD manages. Test sync behavior with operators before production.

⏮️ Chuyện Git Revert Deploy Old Version

Production bug detected. Engineer git revert last commit. ArgoCD auto-sync deploy reverted version. But: reverted version has database schema incompatible with current data. Application crash. Rollback makes things worse.

Root cause: git revert reverses application code but database schema already migrated forward. Fix: deploy forward (hotfix) instead of rollback. Database migrations are one-way. GitOps rollback = application rollback ≠ full system rollback.

💡 Bài học: Git revert = code rollback, NOT full system rollback. Database migrations, external state not reversible via git revert. Rollback strategy must account for stateful dependencies. Forward-fix often better than rollback cho database-dependent applications. Test rollback scenarios in staging.

🏗️ Chuyện App of Apps Bootstrap

Team implement App of Apps pattern: root Application → infra apps (cert-manager, ingress, monitoring) → business apps. Bootstrap: apply root Application → ArgoCD creates children → children deploy in order. Beautiful.

Problem: cert-manager not ready khi Ingress needs certificates. Monitoring not ready khi apps need ServiceMonitors. Ordering dependencies in GitOps = complex. Fix: sync waves (annotations), health checks, retry policies. Pre-sync hooks cho prerequisites.

💡 Bài học: App of Apps = powerful but dependency ordering complex. Sync waves (argocd.argoproj.io/sync-wave) cho ordering. Health checks ensure prerequisite ready before dependent deploys. Pre-sync hooks cho initialization. Test bootstrap process from scratch regularly.

⚠️ Những Sai Lầm Kinh Điển

1. Auto-sync without approval: Every git push auto-deploys to production. Require PR review + approval. Environment promotion gates.
2. Secrets in Git: Plain text secrets committed. Use Sealed Secrets, SOPS, External Secrets Operator. NEVER plain secrets in Git.
3. Monorepo without structure: All apps in one repo without clear boundaries. Directory structure per team/app. CODEOWNERS cho review gates.
4. No drift alerting: Drift exists but nobody knows. Alerts on sync failures and drift detection. Dashboard monitoring.
5. GitOps for everything: Not everything fits GitOps. Database migrations, one-time scripts, manual approvals — use appropriate tools. GitOps cho desired state management.
