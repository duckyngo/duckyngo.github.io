📜 Thân Thế & Lai Lịch

Lý Vi Xử Lý bắt đầu sự nghiệp bằng cách lập trình vi điều khiển 8051 bằng Assembly — mỗi instruction đếm từng clock cycle. Khi C compiler cho microcontroller ra đời, ông cảm thấy như được giải phóng. 'Viết C mà compile ra code gần bằng hand-written Assembly — đây là tương lai!' Ông đã làm việc với mọi kiến trúc: AVR (Arduino), ARM Cortex-M (STM32), Xtensa (ESP32), RISC-V, và cả PIC. Mỗi project là một bài toán resource constraint: 2KB RAM cho firmware xử lý cảm biến, 4MHz clock cho real-time control, 32KB flash cho toàn bộ program. Motto: 'Trong embedded, bạn không optimize cho performance — bạn optimize cho EXISTENCE. Code phải FIT trong hardware constraints, không phải ngược lại.'

🗝️ Bí Kíp

Bí Kíp #1: Memory Management — Khi Heap Là Xa Xỉ

Embedded thường KHÔNG dùng dynamic memory allocation (malloc/free). Lý do: fragmentation trong limited RAM = system crash. Stack allocation cho local variables, static allocation cho global state. Fixed-size buffer pools thay vì malloc. Memory-mapped I/O: registers access trực tiếp memory addresses. Stack overflow: embedded stack thường 1-4KB — recursive functions = danger. Monitor stack usage với stack painting technique (fill stack với pattern, check how far it's consumed).

Bí Kíp #2: Bare Metal vs RTOS

Bare metal: code chạy trực tiếp trên hardware, no OS overhead. Simple state machine hoặc super loop architecture. Tốt cho: simple tasks, resource-constrained devices, predictable timing. RTOS (FreeRTOS, Zephyr): task scheduling, inter-task communication, timers, mutexes. Tốt cho: multi-task applications, complex state management, team development. ESP-IDF dùng FreeRTOS by default. STM32CubeIDE support FreeRTOS integration. Tradeoff: RTOS thêm 5-10KB flash, small RAM overhead, nhưng giảm complexity cho multi-task apps.

Bí Kíp #3: Sensor Integration — I2C, SPI, UART

I2C: 2-wire (SDA, SCL), multiple devices trên cùng bus, moderate speed (100-400KHz). Dùng cho: sensors (temperature, humidity, accelerometer), EEPROM. SPI: 4-wire (MOSI, MISO, SCK, CS), faster than I2C (MHz range), point-to-point. Dùng cho: displays, SD cards, high-speed ADC. UART: asynchronous, point-to-point, simple. Dùng cho: GPS modules, debug console, Bluetooth modules. ADC: analog sensor → digital value. Tips: filter noise (moving average, Kalman), calibrate sensors, handle sensor failure gracefully.

Bí Kíp #4: ESP32 — IoT Powerhouse

Dual-core 240MHz, WiFi + Bluetooth, 520KB SRAM, deep sleep mode (10uA). ESP-IDF: official SDK (FreeRTOS based). Arduino core: simpler API, compatible libraries. WiFi: station mode (connect to AP), AP mode (create hotspot), mesh networking. MQTT: lightweight pub/sub protocol cho IoT → cloud communication. OTA (Over-the-Air) updates: update firmware remotely — essential cho deployed devices. Power management: deep sleep + wake on GPIO/timer cho battery-powered devices.

Bí Kíp #5: Rust cho Embedded — Safety Without Runtime

Rust no_std: compile without standard library, bare metal compatible. Memory safety tại compile time — no buffer overflow, no null pointer dereference. Embedded Rust ecosystem: embedded-hal (hardware abstraction), cortex-m (ARM Cortex-M support), embassy (async embedded framework). Tradeoff: Rust embedded ecosystem nhỏ hơn C, learning curve cao, some HAL libraries immature. Dùng cho: safety-critical embedded systems, new projects by Rust-proficient teams.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Stack Overflow trên 2KB RAM

Firmware cho IoT sensor: ESP8266 (80KB RAM nhưng chỉ 4KB stack cho main task). Developer viết recursive JSON parser cho MQTT message. JSON nested 5 levels deep → recursive calls → stack overflow → device reset loop (watchdog timer restart → parse → crash → restart...). Device deployed ở remote location. Restart loop = device offline. No OTA update possible (WiFi connect code runs AFTER crash point). Team phải drive 200km đến location, flash firmware qua USB cable. Fix: iterative JSON parser (no recursion), static buffer allocation, stack usage monitoring. Add: OTA recovery mode (connect WiFi FIRST, check for update BEFORE main logic).

💡 Bài học: Embedded rule: NO RECURSION trên constrained devices. Iterative algorithms always. Monitor stack usage (stack painting, compiler stack analysis). Implement failsafe boot: if main firmware crash N times → boot into recovery mode (minimal firmware that accepts OTA update). ALWAYS have physical access plan for deployed devices.

📖 Chuyện 2: Off-by-one Bricked 1000 Devices

Firmware update for production IoT devices. OTA update system works fine. New firmware: bug trong flash memory write routine — off-by-one error overwrite bootloader sector. Devices receive update → write new firmware → overwrite bootloader → device brick (cannot boot anymore). 1000 devices deployed across 50 locations. All bricked simultaneously (batch OTA). Recall: ship 1000 devices back, JTAG flash new bootloader + firmware. Cost: $50,000 (shipping + labor + downtime). Customer trust: destroyed. Fix: write-protected bootloader sector (hardware lock), dual-bank firmware (A/B update scheme — if new firmware fails, boot from old bank), staged rollout (update 1% → verify → 10% → verify → 100%).

💡 Bài học: Firmware update = HIGHEST RISK operation in embedded. Protect: 1) Write-protect bootloader, 2) A/B firmware banks (rollback possible), 3) Staged rollout (never 100% at once), 4) Verify firmware integrity (CRC/signature) before committing, 5) Factory reset option that doesn't require bootloader. One bad update = thousands of bricks.

📖 Chuyện 3: Power Consumption Debug — Battery 1 Ngày Thay Vì 1 Năm

IoT sensor thiết kế chạy pin CR2032 (225mAh) suốt 1 năm. Prototype: pin hết sau 1 ngày. Power profiling: device KHÔNG enter deep sleep. Nguyên nhân: I2C sensor chip không power down → pull-up resistors draw 1mA constantly. WiFi modem stay connected → 80mA average. Fix: 1) Power gate sensors (MOSFET switch, turn off khi không đọc), 2) Deep sleep giữa readings (wake every 15 minutes, read sensor, send data, sleep), 3) WiFi connect only when data ready (không maintain connection). Power consumption: 80mA → 20uA average. Battery life: 1 day → 14 months.

💡 Bài học: Power optimization là CRITICAL cho battery devices. Measure ACTUAL current (oscilloscope + shunt resistor, or dedicated power analyzer). Every peripheral: power down khi không dùng. Deep sleep modes: research MCU sleep modes (light sleep, deep sleep, hibernate). WiFi/Bluetooth: connect → transmit → disconnect (don't stay connected). Target: uA range average current for year-long battery life.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Dùng floating point trên MCU không có FPU

ATmega328 (Arduino Uno): no FPU. Float operations = software emulation = 10-100x slower than integer. Dùng fixed-point arithmetic (multiply by 1000, work in integers). Hoặc chọn MCU có FPU (Cortex-M4F, ESP32).

❌ Sai Lầm #2: Không handle watchdog timer

Watchdog timer: reset device nếu firmware hang. Nhưng: dài initialization → watchdog reset trước khi firmware ready. Feed watchdog trong init sequence. Và: infinite loop bug → watchdog reset → boot → same bug → reset loop. Log crash reason để debug.

❌ Sai Lầm #3: Debug bằng printf trên production

printf trên UART: mỗi character ~10us ở 115200 baud. 100 chars = 1ms. In vòng lặp 1KHz → printf take 100% CPU time. Remove debug prints cho production build. Hoặc: conditional compilation (#ifdef DEBUG).

❌ Sai Lầm #4: Không handle sensor failure

Sensor disconnect, wire break, I2C bus hang → firmware freeze waiting for response. Implement: timeout cho mọi I/O operation, sensor health check, fallback values, I2C bus recovery (clock stretching reset).

❌ Sai Lầm #5: Magic numbers trong register config

REG = 0x3F; — 6 tháng sau không ai biết 0x3F nghĩa gì. Dùng named constants, bitfield structs, hoặc comment từng bit. Readability quan trọng hơn cleverness.
