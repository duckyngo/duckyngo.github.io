📜 Bí Kíp Framework

1

PyTorch — Vua của Research & Production (2026)

Năm 2026, PyTorch chiếm ~80% market. Lý do: eager execution (debug bằng print!), Pythonic API, ecosystem mạnh (HuggingFace, Lightning). PyTorch 2.x với torch.compile() tăng speed 30-200% không đổi code. Dùng torch.compile(model) cho production. PyTorch Lightning cho training loop boilerplate. torchrun cho distributed training.

2

JAX — Tương lai của High-Performance ML

JAX = NumPy + autograd + XLA compiler. jit: compile Python → optimized XLA. vmap: vectorize any function. pmap: parallelize across devices. JAX code nhanh hơn PyTorch 2-5x cho certain workloads (nhất là TPU). Nhưng: functional programming style, khó debug, ecosystem nhỏ hơn. Dùng JAX khi: cần maximum performance, TPU training, hoặc research cần custom gradients.

3

CUDA Management — Nghệ thuật quản lý GPU

GPU memory là tài nguyên quý nhất. Tricks: (1) torch.cuda.empty_cache() giải phóng cached memory. (2) Mixed precision training (torch.cuda.amp) giảm memory 50%, tăng speed 2x. (3) Gradient accumulation: simulate large batch trên small GPU. (4) torch.utils.checkpoint: trade compute for memory. (5) Monitor memory: nvidia-smi hoặc torch.cuda.memory_summary().

4

Distributed Training — Scale lên nhiều GPU

DataParallel: đơn giản nhưng bottleneck ở GPU 0 (gradient aggregation). DistributedDataParallel (DDP): tốt hơn, each GPU có copy riêng, all-reduce gradients. Dùng torchrun --nproc_per_node=4. FSDP (Fully Sharded Data Parallel): shard model across GPUs, train model lớn hơn GPU memory. DeepSpeed: ZeRO optimization stages 1-3, offload to CPU/NVMe.

5

Mixed Precision Training — Nhanh hơn, nhẹ hơn

FP16 thay vì FP32: memory giảm 50%, speed tăng 2-3x trên GPU có Tensor Cores. PyTorch: torch.cuda.amp.autocast() + GradScaler. Cẩn thận: một số operations cần FP32 (loss, softmax, layer norm) — autocast handle tự động. BFloat16: ít underflow hơn FP16, supported trên A100+. Năm 2026: FP8 training đang trở thành standard trên H100/B100.

6

Profiling & Optimization

PyTorch Profiler: torch.profiler.profile() + TensorBoard plugin. Tìm bottleneck: CPU-GPU transfer, data loading, kernel execution. Data loading: num_workers=4*num_gpus, pin_memory=True, persistent_workers=True. Pre-fetch data để GPU không idle. Profile trước khi optimize — đừng đoán bottleneck ở đâu.

📖 Chuyện Kể Đêm Khuya

💀 Chuyện 1: CUDA out of memory — nỗi ám ảnh muôn đời

RuntimeError: CUDA out of memory. Tried to allocate 2.00 GiB — dòng chữ này xuất hiện trong giấc mơ của anh Framework. Một lần, anh đang train model với batch_size=64 trên V100 16GB. Out of memory. Giảm xuống 32. Vẫn OOM. 16. Vẫn OOM. 8. Vẫn OOM.

Cuối cùng phát hiện: code có bug — append output vào list mỗi step mà không detach → gradient graph tích lũy → memory leak vô hạn. Fix: loss.item() thay vì loss, output.detach() khi log.

💡 Bài học: CUDA OOM thường không phải do model lớn, mà do memory leak. Top causes: (1) Không detach tensors khi logging, (2) Tích lũy gradients không clear, (3) Không dùng torch.no_grad() khi inference. Dùng torch.cuda.memory_summary() để debug.
😱 Chuyện 2: TF1 → TF2 migration — 6 tháng ác mộng

Năm 2020, công ty anh Framework có 200,000 dòng code TensorFlow 1.x. Sếp quyết định: "Migrate sang TF2!" Estimate: 2 tháng. Thực tế: 6 tháng.

tf.Session biến mất. tf.placeholder biến mất. tf.contrib biến mất hoàn toàn. Mỗi ngày là một surprise: function quen thuộc bỗng deprecated, API thay đổi signature, model behavior khác subtle. Cuối cùng anh viết tool tự động convert — tool cũng bug.

Từ kinh nghiệm đó, anh có rule: "Chọn framework có backward compatibility. Hoặc chọn PyTorch."

💡 Bài học: Framework migration là project lớn hơn bạn nghĩ. Luôn: (1) Abstract framework-specific code behind interfaces, (2) Có comprehensive test suite TRƯỚC khi migrate, (3) Migrate incrementally, không big-bang. Và cân nhắc: có đáng migrate không, hay viết code mới tốt hơn?
🚀 Chuyện 3: torch.compile() — "tốn 1 phút compile, tiết kiệm 1 ngày train"

Anh Framework đang train ViT model, mỗi epoch mất 45 phút trên 4 A100. Tổng 100 epochs = 75 giờ. Một ngày, anh thêm MỘT DÒNG: model = torch.compile(model).

Epoch đầu tiên: 60 phút (compile overhead). Từ epoch 2: 18 phút/epoch. Tổng: 30 giờ thay vì 75 giờ. Tiết kiệm $900 cloud bill. Anh gửi email toàn công ty: "Nếu bạn chưa dùng torch.compile, bạn đang đốt tiền."

💡 Bài học: torch.compile() trong PyTorch 2.x là free performance. Hầu hết models tăng 30-200% speed với zero code change. Cẩn thận: dynamic shapes và custom ops có thể gây graph breaks. Dùng torch._dynamo.config.suppress_errors = True để fallback gracefully.

⚠️ Những Sai Lầm Kinh Điển

❌ Không dùng mixed precision: Train FP32 trên GPU có Tensor Cores = lãng phí 50% memory và 2-3x speed. Dùng AMP.
❌ DataLoader với num_workers=0: GPU idle chờ CPU load data. Set num_workers=4*num_gpus, pin_memory=True.
❌ DataParallel thay vì DDP: DataParallel bottleneck ở GPU 0. Luôn dùng DistributedDataParallel cho multi-GPU.
❌ Không detach khi logging: losses.append(loss) giữ entire computation graph → memory leak. Dùng loss.item().
❌ Quên torch.no_grad() khi inference: Lãng phí memory cho gradient computation không cần thiết. Luôn wrap inference trong with torch.no_grad():.
❌ Hard-code device: .cuda() thay vì .to(device). Code không chạy được trên CPU hoặc multi-GPU.
❌ Không profile trước khi optimize: Tưởng bottleneck ở model computation, thực tế ở data loading. Profile first, optimize second.
