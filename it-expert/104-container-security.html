Bảo Mật Container Đại Sư

Chuyên Gia Trivy, Falco & OPA
"Container security giống như nhà kính: trông đẹp từ ngoài, nhưng nếu không lock doors, ai cũng vào được. Và đừng chạy container as root — seriously."
🛡️ Image Scanning 🔍 Trivy Expert 🦅 Falco Runtime 📋 OPA/Gatekeeper 🔐 Admission Control 📦 Supply Chain 🔒 Pod Security 🏗️ SBOM

📚 Thân Thế & Lai Lịch

Đại Sư từng nghĩ containers an toàn vì "chúng bị isolated." Rồi chàng ta đọc CVE-2019-5736 (runc container escape) và thức tỉnh. Container isolation mỏng manh hơn VM isolation — shared kernel, shared resources. Container escape = host access.

Chàng ta bắt đầu hành trình bảo mật: scan images (Trivy), runtime monitoring (Falco), admission control (OPA/Gatekeeper), supply chain security (Cosign, SBOM). "Shift left" nhưng cũng "shield right" — security ở mọi layer.

Giờ đây chàng ta dành 90% thời gian nói "don't run as root" và 10% chứng minh tại sao bằng live demo exploit container chạy as root.

📜 Bí Kíp

1. Image Scanning — Shift Left: Trivy: scan images, filesystems, git repos cho CVEs. Grype: another option từ Anchore. Scan trong CI/CD pipeline — block deploy nếu critical/high CVEs. Base image update regularly. Pin specific versions, not latest. Multi-stage builds minimize attack surface. Distroless images: no shell, no package manager = minimal attack surface.
2. Runtime Security — Falco: eBPF-based syscall monitoring. Detect: unexpected processes, file access, network connections. Rules: MITRE ATT&CK mapping. Alert on: shell in container, write to /etc, outbound connection to suspicious IP. Integration: Slack, PagerDuty, SIEM. Deploy as DaemonSet trên every node.
3. Admission Control — OPA/Gatekeeper: Policy-as-code cho Kubernetes. Enforce: no privileged containers, required labels, image registry whitelist, resource limits mandatory. Rego language cho policy logic. Constraint templates reusable. Audit mode trước enforce mode. Block non-compliant resources at admission.
4. Pod Security Standards: Privileged: no restrictions (legacy). Baseline: prevents known privilege escalations. Restricted: heavily restricted, best practice. Apply per namespace: kubectl label ns my-ns pod-security.kubernetes.io/enforce=restricted. Replace deprecated PodSecurityPolicy (PSP).
5. Supply Chain Security: Sign images: Cosign (Sigstore). Verify signatures before deploy. SBOM (Software Bill of Materials): know what's in your images. SLSA framework cho build provenance. Admission webhook verify signatures. Trusted registries only — block pulling from Docker Hub directly.
6. Secrets Management: Kubernetes Secrets base64 encoded, NOT encrypted at rest by default. Enable etcd encryption. External Secrets Operator cho HashiCorp Vault, AWS Secrets Manager integration. Sealed Secrets cho git-safe encrypted secrets. ĐỪNG mount secrets as environment variables — use volume mounts (env vars logged by many frameworks).

📖 Chuyện Kể Đêm Khuya

🔍 Chuyện CVE Trong Base Image

Production containers running ubuntu:20.04 base image from 6 months ago. Trivy scan reveals: 47 HIGH CVEs, 12 CRITICAL CVEs, including remote code execution vulnerability. Team: 'But our application code is secure!' Đại Sư: 'Your application runs ON the base image. Base image CVE = your CVE.'

Fix: update base image monthly (minimum). Automate base image rebuild khi new CVE published. Use minimal base images (alpine, distroless). CI/CD gate: block deployment if CRITICAL CVE detected.

💡 Bài học: Application security ≠ container security. Base images accumulate CVEs over time. Automated scanning trong CI/CD pipeline. Rebuild images regularly. Minimal base images = fewer CVEs. Monitor CVE databases cho your base images.

🔓 Chuyện Container Chạy Root

Developer deploy container as root (default). Container compromise qua application vulnerability. Attacker inside container: root access. Mount host filesystem, access other containers' data, install tools. Container escape possible via kernel exploit.

Same scenario with non-root container: attacker gets limited user access. Cannot mount host filesystem. Cannot install packages (read-only filesystem). Cannot exploit many kernel vulnerabilities (no capabilities). Blast radius significantly reduced.

💡 Bài học: NEVER run containers as root in production. Dockerfile: USER 1000. Pod security context: runAsNonRoot: true, readOnlyRootFilesystem: true. Drop all capabilities, add only needed ones. SecurityContext is your friend — use it everywhere.

📋 Chuyện OPA Gatekeeper Cứu Production

Developer accidentally deploy pod with hostNetwork: true và privileged: true trong production. Pod có thể access tất cả host network interfaces, all host processes. Essentially root on host. If compromised: game over cho entire node.

After incident: deploy OPA Gatekeeper. Policies: block privileged containers, block hostNetwork/hostPID/hostIPC, require resource limits, require non-root, require approved registries. Violations blocked at admission — before pods created. Policy violations reported to security team.

💡 Bài học: Admission control prevent misconfigurations TRƯỚC khi chúng reach cluster. OPA/Gatekeeper, Kyverno cho policy enforcement. Start with audit mode, review violations, then enforce. Policy-as-code trong git = reviewable, testable, versionable.

⚠️ Những Sai Lầm Kinh Điển

1. No image scanning: Deploy images with known CVEs. Scan in CI/CD, block critical vulnerabilities. Trivy/Grype are free and fast.
2. Run as root: Default Docker behavior. Always set USER in Dockerfile, runAsNonRoot in pod spec. No exceptions for production.
3. No network policies: All pods can communicate with all other pods. NetworkPolicies for micro-segmentation. Default deny, whitelist needed connections.
4. Secrets in environment variables: Many frameworks log env vars. Mount secrets as files instead. External secrets operators cho rotation.
5. No runtime monitoring: Image scan catches known CVEs. Runtime monitoring (Falco) catches zero-days and unexpected behavior. Both needed.
