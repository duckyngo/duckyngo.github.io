📜 Bí Kíp Đặc Vụ

1

Agent Architectures — ReAct, Plan-and-Execute, ReWOO

ReAct: Reason → Act → Observe → loop. Simple, effective. Plan-and-Execute: plan all steps first → execute sequentially. Better for complex multi-step tasks. ReWOO: plan with tool placeholders → execute tools in batch → reason on results. Fewer LLM calls. Chọn: simple tasks → ReAct. Complex tasks → Plan-and-Execute. Cost-sensitive → ReWOO. Năm 2026: Claude Agent SDK và LangGraph agents là state-of-art.

2

Tool Use — Cho agent sức mạnh

Tools: functions agent có thể gọi (search, calculator, database query, API calls). Design: (1) Clear descriptions: LLM chọn tool dựa trên description, mơ hồ → chọn sai. (2) Typed parameters: schema rõ ràng. (3) Error handling: tool fail gracefully, return error message, agent retry. (4) Sandbox: tool thực thi trong sandbox, không có access tùy ý. Giới hạn tool permissions theo principle of least privilege.

3

Multi-Agent Coordination

CrewAI: role-based agents, sequential/parallel tasks. Easy to setup. AutoGen (Microsoft): conversational agents, flexible topologies. Claude Agent SDK: tool-use focused, reliable function calling. Multi-agent patterns: (1) Supervisor: one agent delegates to specialists. (2) Debate: agents argue, converge on answer. (3) Pipeline: agent output → next agent input. Key: clear role definitions, explicit communication protocol, shared context.

4

Guardrails — Kiểm soát AI agent

Agents can go wrong: infinite loops, harmful actions, cost explosion. Guardrails: (1) Max iterations: limit loop count (10-20). (2) Max tokens/cost: budget per task. (3) Action approval: human-in-the-loop cho destructive actions (delete, send email). (4) Output validation: check output format, content safety. (5) Timeout: kill agent sau N minutes. (6) Sandboxing: agent không có direct access to production systems.

5

Khi nào dùng Agent vs Chain vs Simple prompt

Simple prompt: deterministic task, no external data needed. Chain/RAG: need external data, fixed steps. Agent: dynamic decision-making, tool selection, multi-step reasoning. Agent overhead: more tokens, more latency, more failure modes. Don't use agent for tasks a chain can handle. "Agent is powerful. Agent is expensive. Agent is unpredictable. Use responsibly."

📖 Chuyện Kể Đêm Khuya

🔄 Chuyện 1: Agent enters infinite loop — $200 bill

Agent task: "Research competitor pricing and create report." Agent: search web → get results → "not enough info" → search again → get similar results → "need more detail" → search again... 847 iterations. $200 in API calls. Report: 3 bullet points, same info as iteration 1.

Root cause: no stopping criteria, no max iterations. Agent kept searching vì "there might be more information." Fix: max_iterations=15, early stopping when new info < threshold, explicit "conclude with available info" instruction.

💡 Bài học: LUÔN set max_iterations và cost limits cho agents. Agents không biết khi nào "đủ" — chúng optimize cho completeness, không phải efficiency. Add explicit stopping criteria: "After 3 searches, summarize what you have." Budget limits là safety net.
🤼 Chuyện 2: Agents cãi nhau vô tận

Multi-agent debate system: Agent Reviewer critique Agent Writer's output. Writer revise. Reviewer critique revision. Writer revise again. After 45 iterations: output worse than original because agents kept making unnecessary changes to satisfy each other.

Agent Writer: "I changed paragraph 3 per your feedback." Reviewer: "Good, but now paragraph 1 is inconsistent." Writer changes paragraph 1. Reviewer: "Now paragraph 3 is inconsistent again." Deadlock.

💡 Bài học: Multi-agent debate cần: (1) Max rounds (2-3 revisions enough), (2) Convergence criteria (stop when changes minimal), (3) Final arbiter (supervisor agent makes final decision). Agents arguing ≠ agents improving. Sometimes first draft + one revision is optimal.
🛡️ Chuyện 3: Agent gửi email production — không ai approve

Agent task: "Draft and send follow-up emails to customers." Agent interpreted "send" literally — sent 500 emails trước khi anyone noticed. 200 emails had hallucinated product features. 50 emails had wrong customer names. 3 emails promised discounts that don't exist.

Aftermath: 2 weeks damage control. New rule: ALL external actions require human approval. Agent drafts, human sends.

💡 Bài học: Agents with real-world actions (send email, update database, make API calls) MUST have human-in-the-loop cho destructive/external actions. Sandbox first: let agent draft/plan → human review → human approve execution. "Trust but verify" → "Don't trust, always verify."

⚠️ Những Sai Lầm Kinh Điển

❌ Không set max iterations: Agent loop vô hạn = cost vô hạn. Luôn giới hạn.
❌ Agent cho mọi bài toán: Agent = complexity + cost + unpredictability. Dùng chain/prompt khi đủ.
❌ Không sandbox tools: Agent có quyền execute code, write files → security risk. Sandbox everything.
❌ Multi-agent khi single agent đủ: 3 agents argue khi 1 agent + good prompt đủ. More agents ≠ better output.
❌ Không log agent decisions: Agent made wrong decision? Tại sao? Cần trace mọi reasoning step.
❌ External actions mà không có approval: Agent gửi email, delete data, make purchases mà không có human review = disaster.
