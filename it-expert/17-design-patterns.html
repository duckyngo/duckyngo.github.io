🗝️ Bí Kíp

Bí Kíp #1: CQRS — Tách đọc và ghi

Command Query Responsibility Segregation: write model (optimized for consistency, validation, business rules) tách biệt read model (optimized for queries, denormalized, fast). Write: command → validate → persist → publish event. Read: query → read from optimized read store (có thể là materialized view, Elasticsearch, Redis cache). Khi nào dùng: read/write patterns rất khác nhau, cần scale read independently, complex domain logic. Khi nào KHÔNG: CRUD app đơn giản — CQRS thêm complexity không cần thiết.

Bí Kíp #2: Event Sourcing — Lưu sự kiện, không lưu trạng thái

Thay vì lưu current state trong DB, lưu SEQUENCE of events. Account balance = replay all transactions. Ưu: full audit trail, time-travel debugging, replay events to rebuild state. Nhược: eventual consistency, complex querying (cần projection), event schema evolution, replay performance. Dùng cho: financial systems, audit-critical domains, systems cần undo/redo. ĐỪNG dùng cho CRUD apps — overkill guaranteed.

Bí Kíp #3: Saga Pattern — Distributed Transactions

Microservices: no distributed ACID transactions. Saga: sequence of local transactions, mỗi step publish event trigger next step. Compensation: nếu step N fail → execute compensation cho steps N-1, N-2, ..., 1. Choreography: services react to events (simple, decoupled). Orchestration: central coordinator manage flow (visible, easier to debug). Dùng orchestration cho complex flows (order processing), choreography cho simple flows (notification chain).

Bí Kíp #4: Circuit Breaker — Ngắt mạch khi lỗi

Khi downstream service fails, tiếp tục gọi = waste resources + cascade failure. Circuit Breaker: CLOSED (normal, requests pass through) → OPEN (failures exceed threshold, requests fail fast without calling downstream) → HALF-OPEN (test requests to check if downstream recovered). Libraries: Resilience4j (Java), Polly (.NET), opossum (Node.js). Configure: failure threshold, timeout, reset timer. Kết hợp với retry (exponential backoff) và fallback (cached/default response).

Bí Kíp #5: Outbox Pattern — Reliable Event Publishing

Vấn đề: save to DB + publish event. DB save success nhưng event publish fail → inconsistent state. Outbox: save domain data + event vào SAME database transaction (outbox table). Separate process poll outbox table → publish events → mark as published. Guarantees: at-least-once delivery (idempotent consumers handle duplicates). Alternative: Change Data Capture (CDC) tools (Debezium) capture DB changes → publish as events. No application code change needed.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Event Sourcing Replay Nightmare

Banking system dùng event sourcing. 2 năm production, 500 triệu events. Cần rebuild read model (vì schema change). Replay 500 triệu events... mất 3 NGÀY. Trong 3 ngày đó, read model outdated → reports sai → executive decisions based on stale data. Team phải maintain 2 read models parallel: old (serving current queries) + new (being rebuilt). Operational complexity kinh hoàng. Fix: implement snapshots — save aggregate state tại checkpoint (every 1000 events). Replay từ snapshot thay vì từ event đầu tiên. Rebuild time: 3 ngày → 4 giờ. Lesson: event sourcing without snapshots = time bomb at scale.

💡 Bài học: Event sourcing tại scale cần: 1) Snapshots cho aggregate state, 2) Projection rebuild strategy (parallel, blue-green), 3) Event schema versioning (upcasters), 4) Archival strategy cho old events. Plan cho scale DAY 1, không phải khi đã có 500 triệu events.

📖 Chuyện 2: Circuit Breaker Saved Production

Payment service call fraud detection service. Fraud service deploy buggy version → response time tăng từ 100ms lên 30 giây. Mỗi payment request blocked 30 giây chờ fraud check. Thread pool exhausted → payment service crash → ALL payments fail. After implementing circuit breaker: fraud service slow → circuit breaker OPEN after 5 failures → payment requests skip fraud check (fallback: allow transaction + flag for manual review) → payment service stays healthy. Fraud service team fix bug, circuit breaker HALF-OPEN → test request succeeds → CLOSED → normal operation resumes. Total impact with circuit breaker: 30 seconds of degraded service (manual fraud review). Without circuit breaker: 45 minutes of COMPLETE payment outage.

💡 Bài học: Circuit breaker = insurance against downstream failures. Every external service call should have: circuit breaker (fail fast), timeout (don't wait forever), retry with backoff (transient errors), fallback (degraded but functional). Design fallback strategy per dependency: cache, default value, manual queue, graceful error message.

📖 Chuyện 3: Saga Compensation Gone Wrong

E-commerce order flow: Reserve inventory → Charge payment → Create shipment. Saga orchestrator handles flow. Payment charged but shipment creation fails (warehouse system down). Compensation: refund payment + unreserve inventory. Bug: refund compensation sends FULL refund amount, but payment service applies 2% processing fee → refund = original - 2%. Customer paid $100, refunded $98. 10,000 orders affected. Total: $200,000 discrepancy. Accounting team panics. Fix: compensation logic must account for REAL business rules (processing fees, tax, discounts). Test compensation flows as thoroughly as happy path. Saga testing should include: step N fails → verify compensation for steps 1 to N-1.

💡 Bài học: Saga compensation logic is AS IMPORTANT as forward logic. Test scenarios: every step fails (N failure scenarios cho N steps). Compensation must be idempotent (re-run safe). Consider: partial failures, real-world business rules (fees, taxes), timing (refund processing time). Saga testing checklist should be part of Definition of Done.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Pattern cho mọi thứ

AbstractSingletonProxyFactoryBean — khi mọi class cần pattern. Patterns có cost (complexity, indirection). Apply khi solving SPECIFIC problem, not prophylactically.

❌ Sai Lầm #2: CQRS + Event Sourcing cho CRUD app

Blog website không cần event sourcing. Todo app không cần CQRS. Assess complexity budget: nếu team 3 người, startup phase, simple domain → KISS (Keep It Simple, Stupid).

❌ Sai Lầm #3: Saga without idempotency

Network retry → saga step execute twice → double charge, double shipment. MỌII saga step PHẢI idempotent. Idempotency key + deduplication at each step.

❌ Sai Lầm #4: Circuit breaker timeout quá ngắn

Timeout 100ms cho service thường respond 80ms → circuit breaker always open → service 'down' khi thực ra chỉ slow. Set timeout based on P99 latency + buffer. Monitor và adjust.

❌ Sai Lầm #5: Event schema không versioned

Event sourcing: event schema changes → old events unreadable. Version every event, write upcasters (v1 → v2 transformers). Never delete old event schemas.
