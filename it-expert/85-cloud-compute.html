📚 Thân Thế & Lai Lịch

Đại Sư bắt đầu với việc quản lý một rack server duy nhất. Chàng ta biết chính xác mỗi server có bao nhiêu RAM, bao nhiêu cores, và tốc độ quạt bao nhiêu RPM. Khi chuyển sang cloud, chàng ta choáng ngợp: EC2 có hơn 750 instance types. "Ai cần 750 loại xe?" chàng ta hỏi.

Sau khi chọn sai instance type 47 lần (compute-optimized cho memory-heavy workload, memory-optimized cho CPU-heavy workload, và một lần legendary chọn GPU instance để chạy WordPress), chàng ta đã master nghệ thuật right-sizing.

Ngày nay, chàng ta còn phải deal với GPU shortage cho AI workloads — NVIDIA A100 wait list 6 tháng, H100 thì "call for pricing" (nghĩa là bạn không đủ tiền). Chàng ta biết mọi trick: spot instances, reserved capacity, và "chạy lúc 3 giờ sáng khi GPU rẻ hơn".

📜 Bí Kíp

1. Instance Family — Hiểu Alphabet Soup: T (burstable, dev/test), M (general purpose), C (compute-optimized), R (memory-optimized), P/G (GPU), I (storage-optimized), X (memory-intensive). Graviton (ARM) instances: 20-40% better price-performance cho nhiều workloads. Benchmark trước khi chọn.
2. Spot Instances — 90% Savings, 100% Risk Management: Spot giá rẻ hơn On-Demand tới 90%. Nhưng có thể bị terminate với 2-minute warning. Use cases: batch processing, CI/CD, stateless web servers (behind ASG), big data processing. KHÔNG dùng cho: databases, stateful services, single-instance workloads.
3. GPU Computing — AI/ML Infrastructure: NVIDIA T4 cho inference (giá rẻ, hiệu quả). A10G cho mixed training/inference. A100/H100 cho large-scale training. Multi-GPU instances cho distributed training. Spot GPU instances cho training (checkpoint frequently!). Consider cloud-specific AI chips: AWS Inferentia, Google TPU.
4. Auto Scaling — Scale Right, Not Just Scale Out: Target tracking policies (CPU 70%, request count). Step scaling cho predictable patterns. Scheduled scaling cho known traffic patterns (e.g., business hours). Predictive scaling dùng ML. Cool-down periods để tránh flapping. Scale IN cũng quan trọng — đừng để instance chạy idle.
5. Container Compute — ECS vs EKS vs Fargate: ECS: AWS-native, simpler. EKS: Kubernetes standard, portable. Fargate: serverless containers, no node management. ECS + Fargate cho simplicity. EKS + managed node groups cho control. Fargate cho teams không muốn manage nodes. Karpenter cho EKS auto-scaling tối ưu.
6. HPC — High Performance Computing: Placement groups (cluster) cho low-latency networking. EFA (Elastic Fabric Adapter) cho MPI workloads. FSx for Lustre cho high-throughput parallel storage. AWS ParallelCluster, Azure CycleCloud cho HPC cluster management. Spot Instances cho burst HPC — có thể tiết kiệm millions.

📖 Chuyện Kể Đêm Khuya

💀 Chuyện Spot Instance Bị Terminate Giữa Training

ML team chạy training job trên p3.16xlarge spot instance (8 x V100 GPUs, ~$7/hour spot vs $25/hour on-demand). Training dự kiến 48 giờ. Checkpoint mỗi 12 giờ "vì checkpoint tốn thời gian."

Giờ thứ 11: spot instance terminated. 11 giờ training mất trắng vì checkpoint cuối cùng là... chưa có. Re-run từ đầu. Giờ thứ 23: terminated AGAIN. Team chuyển sang on-demand, tốn gấp 3.5x nhưng ít nhất hoàn thành được.

💡 Bài học: Spot cho training = PHẢI checkpoint frequently (mỗi 30-60 phút). Dùng Spot Instance interruption notice (2 phút) để save checkpoint emergency. Mix on-demand (1 instance guaranteed) + spot (additional capacity). Training frameworks hỗ trợ resume from checkpoint — USE IT.

🤦 Chuyện Chọn Sai Instance Type 6 Tháng

Team chạy PostgreSQL trên c5.4xlarge (16 vCPUs, 32GB RAM) — compute-optimized. Database chạy chậm, queries timeout. Team scale up lên c5.9xlarge. Vẫn chậm. c5.18xlarge. Hơi khá hơn nhưng tốn $2,000/tháng.

Đại Sư vào analyze: CPU usage 15%, memory usage 95%. Database cần RAM cho buffer pool, không cần CPU. Chuyển sang r5.4xlarge (16 vCPUs, 128GB RAM): performance tăng 5x, chi phí giảm 30%. Problem solved trong 5 phút analysis.

💡 Bài học: Monitor actual resource usage trước khi scale. CPU-bound workload → C series. Memory-bound → R series. Storage-bound → I series. Right instance family > bigger instance size. 5 phút analysis có thể tiết kiệm hàng nghìn đô la mỗi tháng.

🎮 Chuyện GPU Shortage Năm 2024

Startup AI cần 64 A100 GPUs cho training. AWS: "p4d.24xlarge available in us-east-1... trong 4 tháng." Azure: "6 tháng wait list." GCP: "Contact sales." Startup có funding $5M và 12 tháng runway. 4-6 tháng chờ GPU = death sentence.

Đại Sư suggest: (1) Lambda Labs cloud — smaller provider nhưng có GPU inventory, (2) CoreWeave — GPU-specialized cloud, (3) Optimize model để cần ít GPU hơn, (4) Mixed precision training giảm memory, (5) Gradient checkpointing. Cuối cùng: 32 GPUs trên CoreWeave + optimization = same result với 50% less GPUs.

💡 Bài học: GPU shortage là real. Đừng depend hoàn toàn vào Big 3 cho GPU. Smaller specialized providers (Lambda, CoreWeave, RunPod) thường có availability tốt hơn. Optimize model trước khi throw more GPUs. Reserved capacity cho long-term GPU needs.

⚠️ Những Sai Lầm Kinh Điển

1. Default instance type cho mọi workload: t3.medium cho mọi thứ — web server, database, batch job. Mỗi workload cần instance type khác nhau.
2. Không dùng ARM/Graviton: 20-40% cheaper, better performance cho nhiều workloads. Test compatibility và migrate — ROI cao.
3. Over-provisioning "cho an toàn": 80% instances trên cloud over-provisioned. Monitor và right-size continuously. Savings tiềm năng: 30-50%.
4. Spot instance cho stateful workload: Database trên spot = data loss khi terminate. Spot chỉ cho stateless, checkpoint-able workloads.
5. Quên turn off dev/test instances: Dev instances chạy 24/7 nhưng chỉ dùng 8h/ngày = 67% waste. Auto-stop scripts, scheduling, hoặc dùng burstable instances.
