📜 Bí Kíp

Bí Kíp #1: OAuth 2.0 & OpenID Connect — Đừng Tự Viết Auth

OAuth 2.0 cho authorization (access tokens), OIDC cho authentication (ID tokens). Flows: Authorization Code + PKCE cho web/mobile (LUÔN dùng PKCE), Client Credentials cho machine-to-machine, Device Authorization cho IoT. ĐỪNG dùng Implicit flow — deprecated vì security reasons. Access token nên short-lived (15-60 phút), refresh token longer (hours-days) với rotation. JWT claims: iss, sub, aud, exp phải validate đầy đủ. Và nhớ: OAuth KHÔNG phải authentication protocol — OIDC mới là.

Bí Kíp #2: SSO & Identity Federation

Single Sign-On cho phép user login một lần, access nhiều applications. SAML 2.0 cho enterprise (XML-based, mature), OIDC cho modern apps (JSON-based, developer-friendly). Identity federation: kết nối multiple identity providers (corporate AD, Google, GitHub) qua standard protocols. Keycloak là lựa chọn open-source tuyệt vời: hỗ trợ SAML + OIDC, user federation (LDAP/AD), social login, admin console. Key: SCIM (System for Cross-domain Identity Management) để auto-provision/deprovision users khi onboard/offboard.

Bí Kíp #3: MFA & FIDO2 — Vượt Qua Password

MFA factors: something you know (password), something you have (phone/key), something you are (biometrics). SMS OTP là yếu nhất (SIM swapping, SS7 attacks) — dùng TOTP (Google Authenticator) hoặc tốt hơn là FIDO2/WebAuthn. FIDO2 phishing-resistant vì bound to origin — fake site không thể replay credential. Hardware keys (YubiKey) cho high-security accounts. Passwordless flow: FIDO2 passkey + biometric = no password needed, phishing-proof, user-friendly. Đây là tương lai.

Bí Kíp #4: RBAC vs ABAC — Chọn Đúng Model

RBAC (Role-Based Access Control): user -> role -> permissions. Đơn giản, phổ biến, nhưng role explosion khi phức tạp. ABAC (Attribute-Based Access Control): decisions dựa trên attributes (user department, resource sensitivity, time, location). Flexible hơn nhưng complex hơn. ReBAC (Relationship-Based Access Control): permissions dựa trên relationships (Google Zanzibar model, dùng bởi Google Docs). Thực tế: bắt đầu với RBAC, evolve sang ABAC/ReBAC khi cần. Tools: OPA (Open Policy Agent), Casbin, SpiceDB (ReBAC).

Bí Kíp #5: Keycloak — Swiss Army Knife của IAM

Keycloak cung cấp: SSO (SAML + OIDC), user management, social login, identity brokering, MFA, admin console, account management portal. Deploy: container-based, HA với database cluster (PostgreSQL recommended). Realms để multi-tenancy. Client adapters cho Java, Node.js, Python. Customization qua themes (login page branding), SPI extensions (custom authenticators), event listeners (audit logging). Performance: cache sessions trong Infinispan, database connection pooling. Và backup realm export regularly!

Bí Kíp #6: Least Privilege & Access Reviews

Principle of Least Privilege: user chỉ có quyền TỐI THIỂU cần thiết. Implement: start with zero access, add permissions as needed (whitelist approach). Regular access reviews: quarterly review ai có quyền gì, revoke unused permissions. Just-In-Time (JIT) access: temporary elevated permissions cho specific tasks, auto-revoke after time window. Break-glass procedures: emergency access với full audit trail. Và phải có offboarding checklist — revoke ALL access within hours khi employee rời công ty.

📖 Chuyện Kể Đêm Khuya

🔥 Chuyện 1: SSO Config Sai — Mọi Người Đều Là Admin

Thứ Ba bình thường, deploy Keycloak config update để thêm role mapping cho team mới. Typo trong mapper configuration: thay vì map group "platform-admins" sang role "admin", nó map group "authenticated" (tức LÀ MỌI USER ĐÃ LOGIN) sang role "admin".

30 phút sau, intern phát hiện mình có nút "Delete Production Database" trong admin panel và — may mắn — báo ngay cho manager thay vì tò mò bấm thử. Rollback config, audit log review: 2 users đã vô tình access admin pages nhưng không thay đổi gì. Close call.

Từ đó: mọi IAM config change phải qua peer review, có staging environment test trước, và automated tests verify role assignments.

💡 Bài học: IAM configuration changes là HIGH-RISK changes. Treat chúng như infrastructure code — version controlled, peer reviewed, tested in staging. Một dòng config sai có thể biến cleaning staff thành super admin.

🔥 Chuyện 2: MFA Bypass Qua Account Recovery

Company tự hào 100% MFA adoption. Pentest phát hiện: account recovery flow cho phép reset password qua email, sau reset login KHÔNG yêu cầu MFA. Attacker chỉ cần compromise email -> reset password -> login without MFA -> full access.

Worse: "security questions" cho recovery là "Tên thú cưng?" và "Trường cấp 3?" — thông tin public trên Facebook. MFA enforcement bypass hoàn toàn chỉ bằng social engineering + password reset flow.

💡 Bài học: MFA phải enforce ở MỌI login path, kể cả recovery flows. Security questions là theater, không phải security. Account recovery nên yêu cầu multiple verification steps và re-enroll MFA device.

🔥 Chuyện 3: SCIM Auto-Deprovision Cứu Ngày

Engineer bất mãn bị sa thải, HR process offboarding trong hệ thống HR. SCIM integration tự động: disable user trong Keycloak -> revoke tất cả OAuth tokens -> remove từ mọi groups -> notify security team. Tất cả trong vòng 5 phút từ khi HR click "Terminate".

30 phút sau, cựu employee thử login — blocked. Thử dùng saved API tokens — revoked. Thử access Git repos — removed. Không có cửa nào mở. Trước khi có SCIM automation, offboarding mất 3-5 ngày và thường miss vài systems.

💡 Bài học: Automated provisioning/deprovisioning qua SCIM không phải nice-to-have — đó là critical security control. Manual offboarding luôn có gaps. Minutes matter khi có disgruntled employee.

🔥 Chuyện 4: Token Lifetime Quá Dài

Access token lifetime: 24 giờ. Refresh token: 30 ngày, không rotation. User logout khỏi app nhưng tokens vẫn valid. Attacker steal token từ browser localStorage (XSS) và dùng suốt 30 ngày mà không bị detect vì "user vẫn active".

Fix: access token 15 phút, refresh token 8 giờ với rotation (mỗi lần dùng refresh token, issue new refresh token, invalidate old one). Token stored trong httpOnly cookie (không phải localStorage). Và implement token revocation endpoint + blacklist cho compromised tokens.

💡 Bài học: Token lifetime phải SHORT. Refresh token rotation detect token theft (cùng refresh token dùng 2 lần = compromised, revoke toàn bộ session). Và NEVER store tokens trong localStorage.

⚠️ Những Sai Lầm Kinh Điển

❌ Sai Lầm #1: Tự viết authentication system

Password hashing, session management, token generation — tự viết = tự tạo vulnerabilities. Dùng proven solutions: Keycloak, Auth0, Firebase Auth, AWS Cognito. Custom auth chỉ khi bạn là security expert.

❌ Sai Lầm #2: SMS-based MFA cho accounts quan trọng

SIM swapping, SS7 interception, social engineering carrier. SMS OTP là MFA yếu nhất. Dùng TOTP hoặc FIDO2 cho admin accounts. SMS chỉ acceptable cho low-risk accounts.

❌ Sai Lầm #3: Shared admin accounts

"root" password chia sẻ giữa 10 người. Khi có incident, ai đã login? Không biết. Mỗi admin phải có individual account với MFA. Shared accounts = no accountability.

❌ Sai Lầm #4: Role explosion — 500 roles, không ai hiểu

Mỗi request tạo role mới vì "không có role phù hợp". Kết quả: 500 roles, 80% overlap. Regular role review, consolidation, và proper RBAC hierarchy là bắt buộc.

❌ Sai Lầm #5: Không revoke access khi offboarding

Employee nghỉ việc 6 tháng vẫn access được production. Automated deprovisioning + regular access audit + offboarding checklist. Access phải revoke trong GIỜ, không phải tuần.
