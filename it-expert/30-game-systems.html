📜 Thân Thế & Lai Lịch

Hoàng Hệ Thống bắt đầu bằng việc implement physics engine từ scratch cho game đại học — collision detection dùng bounding boxes, gravity bằng Euler integration. Game chạy 'tạm ổn' cho đến khi player di chuyển nhanh → xuyên qua tường (tunneling). Từ đó ông hiểu rằng physics simulation là bài toán toán học nghiêm túc. Ông chuyển sang multiplayer networking — nơi mà latency, packet loss, và cheating là kẻ thù. Implement prediction, reconciliation, lag compensation. Ông từng debug desync issue mất 2 tuần — root cause: floating point determinism khác nhau giữa Intel và AMD CPU. Motto: 'Game systems là ảo thuật — tạo illusion hoàn hảo cho player. Physics không cần đúng 100%, chỉ cần FEEL đúng. AI không cần thông minh, chỉ cần SEEM thông minh.'

🗝️ Bí Kíp

Bí Kíp #1: Physics Engine Fundamentals

Collision detection: broad phase (AABB, spatial hashing — quickly eliminate non-colliding pairs) → narrow phase (GJK, SAT — precise collision check). Collision response: impulse-based (realistic but unstable at rest) → constraint-based (stable stacking, joints). Integration: Euler (simple, inaccurate), Verlet (stable for constraints, Cloth/particles), RK4 (accurate, expensive). Substeps: physics tick multiple times per frame cho accuracy. Continuous collision detection (CCD): prevent tunneling (fast objects passing through thin walls).

Bí Kíp #2: AI & Pathfinding

Navigation mesh (NavMesh): pre-computed walkable area, A* pathfinding trên NavMesh. Behavior Trees: modular, composable AI logic (Selector, Sequence, Decorator nodes). State Machines: simple AI states (Idle → Chase → Attack → Flee). Steering Behaviors: seek, flee, wander, flock — combine for emergent behavior. GOAP (Goal-Oriented Action Planning): AI plans sequence of actions to achieve goals. Perception: AI 'see' through raycasts, 'hear' through sound propagation radius. Tip: AI không cần thông minh — cần unpredictable và entertaining.

Bí Kíp #3: Multiplayer Netcode — The Hard Problem

Architecture: Authoritative server (server validates all actions, prevent cheating) vs Peer-to-peer (no server, faster but cheat-vulnerable). Prediction: client predict outcome of actions BEFORE server confirms (responsive feel). Reconciliation: server correct client prediction if wrong (smooth correction, not teleport). Lag compensation: server rewind time to when player fired (fair hit detection despite latency). Delta compression: send only changes, not full state. Protocol: UDP cho real-time (no head-of-line blocking), reliable UDP (ENet, GameNetworkingSockets) cho ordered reliable messages.

Bí Kíp #4: Lag Compensation & Interpolation

Entity interpolation: render entities at past position (add artificial delay, smooth movement). Extrapolation/prediction: guess future position (fast but can be wrong, causing correction snaps). Lag compensation (rewind): when player shoots → server rewind world state to player's perceived time → check if shot hit → apply result at current time. Tick rate: server simulation rate (64/128 tick for competitive FPS). Client send rate: how often client sends input. Netcode settings massively affect game feel — VALORANT và CS2 invest enormously in netcode quality.

Bí Kíp #5: Optimization Patterns cho Games

Object pooling: reuse instead of create/destroy (bullets, particles, enemies). Spatial partitioning: quadtree (2D), octree (3D), spatial hashing — only check nearby entities for collision/AI/rendering. Level of Detail (LOD): reduce complexity for distant objects (geometry, AI, physics). Culling: frustum culling (don't render outside camera view), occlusion culling (don't render behind walls). Fixed-point math cho deterministic simulation (networking, replay).

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Physics Glitch Goes Viral

Racing game: khi 2 xe chạm nhau tại góc chính xác → physics solver explode → xe bay lên trời xoay 720° → land perfectly → continue racing. Video posted YouTube → 10 triệu views → game sales TĂNG 500% vì mọi người muốn recreate glitch. Team dev meeting: 'Fix hay giữ?' Marketing: 'GIỮGIỮGIỮ!' Physics programmer: 'Nhưng... Newton...' Marketing wins. Team add achievement: 'Space Program' — perform 5 aerial spins from physics glitch. Root cause: collision resolver apply impossible impulse at contact point with extreme penetration depth. Fix (optional setting): clamp impulse magnitude, limit angular velocity. Default: unclamped (giữ glitch).

💡 Bài học: Physics glitches trong games thường entertaining — player discovery, speedrun techniques, viral moments. Evaluate: is the glitch harmful (game-breaking, unfair advantage) or fun (emergent gameplay, shareable moments)? Fix game-breaking glitches, embrace fun ones. Test extensively: physics edge cases (high speed, extreme angles, stacking, corner cases).

📖 Chuyện 2: Desynced Multiplayer — Player A Thấy Khác Player B

Real-time strategy game: 2 players thấy game state KHÁC NHAU. Player A: quân lính đi phải. Player B (cùng game): quân lính đi trái. Cả hai 'correct' trên client của mình nhưng khác nhau trên server. Root cause: floating point determinism. Player A dùng Intel CPU, Player B dùng AMD CPU. Floating point operations cho kết quả khác nhau ở digit thứ 15 sau dấu phẩy. Sau 10,000 simulation steps, tiny differences accumulate → completely different game states. Fix options: 1) Fixed-point math (integer-based, deterministic across all platforms), 2) Server-authoritative simulation (client only renders), 3) Periodic state sync (detect desync, correct). Lockstep networking (used by RTS): all clients simulate identically, send only inputs — requires PERFECT determinism.

💡 Bài học: Determinism là MANDATORY cho lockstep networking (RTS games). Floating point = NOT deterministic across platforms/compilers. Solutions: fixed-point math, soft-float implementation, hoặc switch to server-authoritative model. Test determinism: replay same inputs on different machines → compare final state hash. If hashes differ → determinism bug.

📖 Chuyện 3: AI Pathfinding Meltdown — 1000 Enemies, 2 FPS

Tower defense game: wave 50 spawns 1000 enemies. All enemies calculate path using A* simultaneously. NavMesh A* cho mỗi enemy: 0.5ms. 1000 enemies × 0.5ms = 500ms PER FRAME. Game: 2 FPS. Fix 1: Distribute pathfinding across frames (100 enemies per frame, cycle through all in 10 frames). Fix 2: Shared path — enemies with same destination SHARE path result (cache). Fix 3: Flow field — single field-based pathfinding, ALL entities follow field vectors. O(grid_size) thay vì O(entity_count × grid_size). After optimization: 1000 enemies, 60fps. Flow field cho tower defense = game changer.

💡 Bài học: Pathfinding cho large entity counts: individual A* doesn't scale. Solutions: 1) Hierarchical pathfinding (A* on coarse grid, detailed on local), 2) Flow fields (single solve, all entities follow), 3) Path caching (same destination = same path), 4) Distribute across frames (not all entities need path every frame). Profile early khi entity count nears target.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Physics timestep = render timestep

Variable physics = different behavior at different framerates. 30fps: objects fall slower than 60fps. FIXED timestep cho physics (physics simulate same regardless of framerate). Interpolate cho smooth rendering.

❌ Sai Lầm #2: Client-side authority cho multiplayer

Client says 'I have 999 HP' → server believes → cheater wins. Server authoritative: client sends INPUT, server validates và computes result. Client NEVER decides game outcome.

❌ Sai Lầm #3: A* pathfinding mỗi frame

AI recalculate path every frame = waste. Recalculate when: destination changes, path blocked, periodic check (every 0.5s). Cache paths, invalidate on world change.

❌ Sai Lầm #4: Ignoring network jitter

Average latency 50ms nhưng jitter: 20ms, 80ms, 30ms, 200ms, 40ms. Design cho worst case, not average. Jitter buffer cho smooth experience. Test with simulated bad network (clumpy, netem).

❌ Sai Lầm #5: Deterministic simulation mà dùng float

Float arithmetic = platform-dependent results. Same operation, different CPU/compiler = different bit-level result. Over many frames = completely different game state. Fixed-point for determinism.
