📜 Bí Kíp Pipeline

1

Chọn ML Platform phù hợp

MLflow: lightweight, open-source, tốt cho experiment tracking + model registry. Start here. Kubeflow: full platform trên K8s, tốt cho team lớn, complex pipelines. Steep learning curve. Vertex AI (GCP): managed, tốt nếu đã dùng GCP. SageMaker (AWS): mature, nhiều features, tốt nếu đã dùng AWS. Rule: team < 5 → MLflow. Team 5-20 + K8s → Kubeflow. Enterprise + cloud commitment → Vertex/SageMaker.

2

Reproducibility — Tái tạo kết quả

5 pillars: (1) Code: Git commit hash. (2) Data: data version (DVC, Delta Lake). (3) Environment: Docker image hash. (4) Config: hyperparameters logged (MLflow, W&B). (5) Random seeds: set everywhere. MLflow log: mlflow.log_param(), mlflow.log_metric(), mlflow.log_artifact(). Mỗi experiment phải reproducible — "nếu không tái tạo được, đó là magic, không phải science."

3

Model Registry — Quản lý model lifecycle

MLflow Model Registry: stage models qua None → Staging → Production → Archived. Mỗi model version: linked to experiment run, có metrics, có artifacts. Model card: document model purpose, limitations, performance, bias. CI/CD for ML: khi model mới push → auto test → compare vs production model → promote nếu tốt hơn. Never deploy model mà không có rollback plan.

4

Pipeline Design Patterns

DAG-based: mỗi step là node, dependencies = edges. Idempotent steps: re-run safe. Caching: skip steps nếu input unchanged. Feature pipeline: data ingestion → cleaning → feature engineering → store. Training pipeline: load features → train → evaluate → register. Inference pipeline: load model → preprocess → predict → postprocess. Separate concerns: data engineer owns feature pipeline, ML engineer owns training, MLOps owns serving.

5

Data Versioning — DVC và alternatives

DVC: Git for data. Track large files, create data pipelines, version datasets. Store data in S3/GCS, track hashes in Git. Delta Lake: versioned data lake, time travel, ACID transactions. LakeFS: Git-like for data lakes. Minimum: version training data + code + config cho mọi experiment. "Nhưng tôi đã dùng data nào train model v1.3?" → Nếu không trả lời được → chưa đủ versioning.

📖 Chuyện Kể Đêm Khuya

🔄 Chuyện 1: Unreproducible training results

Data scientist: "Model v2 tốt hơn v1 10%!" Sếp: "Tuyệt, deploy!" 1 tuần sau, retrain với cùng code: performance giảm 5%. Tại sao? (1) Random seed khác, (2) Data thay đổi (new records added), (3) Library version update (PyTorch patch), (4) GPU khác (V100 vs A100 → floating point differences).

Anh Ống Dẫn setup: Docker image pinned, data versioned (DVC), seeds logged, hardware specified. Từ đó: mọi experiment reproducible within 0.1%.

💡 Bài học: Reproducibility cần đầu tư upfront nhưng tiết kiệm vô số giờ debug. Pin EVERYTHING: code (git), data (DVC), environment (Docker), config (MLflow), seeds. "Works on my machine" là câu nói tệ nhất trong ML.
⏰ Chuyện 2: Pipeline breaks at 3am

Training pipeline chạy hàng đêm: data refresh → retrain → evaluate → auto-deploy nếu metrics tốt hơn. Một đêm: data source thay đổi schema (thêm column). Pipeline crash ở preprocessing step. Auto-retry 3 lần. 3 lần fail. PagerDuty alert lúc 3:17 AM.

Anh Ống Dẫn thức dậy, fix schema, pipeline chạy lại. Hôm sau: thêm schema validation step, data quality checks, và graceful degradation (nếu retrain fail, giữ model cũ).

💡 Bài học: ML pipelines cần defensive programming: (1) Schema validation cho data inputs, (2) Data quality checks (missing values, distributions), (3) Graceful degradation (keep old model if new one fails), (4) Alerting with context (không chỉ "failed" mà "failed because schema changed"). Production ML = production engineering.
📦 Chuyện 3: Model registry cứu mạng

Deploy model v5, production metrics giảm 20%. Panic. Nhờ model registry, rollback về v4 trong 5 phút. Investigate v5: data drift — training data không representative cho current traffic.

Nếu không có model registry: "v5 overwrite v4, code đã merge, không ai nhớ config v4 là gì." Disaster recovery time: 5 phút vs 5 ngày.

💡 Bài học: Model registry là insurance policy. Mỗi model version: linked to code, data, config, metrics. Rollback plan cho mọi deployment. A/B testing hoặc canary deployment cho model releases. Never yolo-deploy model mới lên 100% traffic.

⚠️ Những Sai Lầm Kinh Điển

❌ Jupyter notebook làm production pipeline: Notebooks tốt cho exploration, tệ cho production. Convert sang .py scripts + proper pipeline.
❌ Không version data: Code versioned, model versioned, data... "ở đâu đó trên S3." Không reproducible.
❌ Auto-deploy mà không có guardrails: Model mới auto-deploy → metrics crash → no rollback. Cần evaluation gate + canary deployment.
❌ Over-engineer pipeline từ đầu: Team 2 người mà setup Kubeflow cluster → 3 tháng chỉ để maintain infra. Start với MLflow + scripts.
❌ Không monitor model in production: Deploy xong rồi quên. Data drift, concept drift → model degrade dần mà không biết.
