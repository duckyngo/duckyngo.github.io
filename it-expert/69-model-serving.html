📜 Bí Kíp Serving

1

Chọn Serving Framework

Triton (NVIDIA): multi-model, multi-framework, GPU optimized. Best cho high-throughput production. vLLM: LLM-specific, PagedAttention, continuous batching. Best cho LLM serving. TGI (HuggingFace): LLM serving, easy setup, HF ecosystem. Ollama: local LLM, easiest setup, good cho development/personal. TorchServe: PyTorch native, good cho custom models. Decision: LLM → vLLM/TGI. Vision/custom → Triton. Local dev → Ollama.

2

Quantization — Nhỏ hơn, nhanh hơn

FP32 → FP16: 2x smaller, ~same quality. FP16 → INT8: 2x smaller, < 1% quality loss. INT8 → INT4: 2x smaller, 1-3% quality loss. Tools: GPTQ (post-training, good quality), AWQ (activation-aware, better than GPTQ), GGUF (llama.cpp format, CPU-friendly). LLM 70B: FP16 = 140GB, INT4 = 35GB → fit in single GPU 48GB. Luôn evaluate quality sau quantization trên your specific tasks.

3

Batching & Throughput Optimization

Dynamic batching: group incoming requests, process together. Triton: dynamic_batching { max_queue_delay_microseconds: 100 }. Continuous batching (vLLM): không đợi batch đầy, add/remove requests dynamically. 2-3x throughput vs static batching. PagedAttention (vLLM): virtual memory cho KV cache, giảm waste 60-80%. Throughput optimization: batch_size tối đa mà GPU memory cho phép.

4

GPU Utilization — Mỗi % đều đáng tiền

GPU idle = tiền đốt. Monitor: nvidia-smi, nvitop. Target: > 80% GPU utilization. Techniques: (1) Right-size batch. (2) Model parallelism cho large models. (3) Multi-model serving (share GPU). (4) Auto-scaling: scale to zero khi không có traffic. Cloud GPU: ~$2-4/hr cho A100. 24/7 = $1,500-3,000/month. Auto-scaling có thể tiết kiệm 60-80%.

5

Latency Optimization cho LLMs

Time to First Token (TTFT): user wait time trước khi thấy response. Tokens per Second (TPS): generation speed. Optimize TTFT: prefix caching (cache system prompt), speculative decoding. Optimize TPS: quantization, tensor parallelism, Flash Attention. Streaming: gửi tokens ngay khi generate → perceived latency giảm đáng kể. Target: TTFT < 500ms, TPS > 30 cho interactive apps.

📖 Chuyện Kể Đêm Khuya

💀 Chuyện 1: Model OOM in production

Deploy Llama 13B trên GPU A10 24GB. Test với 1 request: OK. Load test 10 concurrent: OK. Production launch: 50 concurrent users → CUDA out of memory. Service crash. Auto-restart. Crash again. Loop.

Root cause: KV cache grows linearly với concurrent requests × context length. 50 requests × 4096 tokens = massive KV cache. Fix: (1) Limit max concurrent, (2) vLLM PagedAttention, (3) Shorter max_tokens, (4) Upgrade GPU. Từ đó anh luôn load test với 2x expected peak traffic.

💡 Bài học: LLM memory = model weights + KV cache (grows with requests). Test với realistic concurrent load, không chỉ single request. Set max concurrent requests limit. Monitor GPU memory continuously. PagedAttention (vLLM) giảm KV cache waste 60-80%.
🥶 Chuyện 2: Cold start giết trải nghiệm

LLM serving với auto-scaling (scale to zero khi không có traffic). Tiết kiệm 80% cost. Nhưng: user đầu tiên sau idle period → load model (45 giây) → TTFT 47 giây. User đã rời page.

Fix: minimum 1 replica luôn warm. Cost tăng nhưng UX acceptable. Hoặc: model preloading (load khi predict traffic spike), hoặc smaller quantized model cho instant response + quality model cho detailed response.

💡 Bài học: Cold start là trade-off: cost vs latency. Solutions: (1) Min replicas > 0, (2) Predictive scaling (dựa trên traffic patterns), (3) Model caching in memory, (4) Smaller fallback model. Users wait 2 seconds max — 45 seconds là unacceptable cho interactive apps.
💰 Chuyện 3: Quantization cứu budget

Serve Llama 70B FP16: cần 4 × A100 80GB = ~$12,000/month. Company budget: $3,000/month. "Giảm model size?" "Không, boss muốn 70B." Anh Tốc Độ dùng AWQ INT4: fit trong 1 × A100 80GB = $3,000/month. Quality drop: 1.2% trên benchmarks, users không nhận thấy.

💡 Bài học: Quantization là free performance/cost optimization cho LLMs. INT4 giảm 4x memory với minimal quality loss. Luôn evaluate quantized model trên YOUR tasks — benchmark loss ≠ task-specific loss. AWQ > GPTQ cho most cases năm 2026.

⚠️ Những Sai Lầm Kinh Điển

❌ Không load test trước deploy: Single request OK ≠ 100 concurrent OK. Load test với realistic traffic patterns.
❌ Serve FP32 khi FP16 đủ: 2x memory, 2x slower, ~0% quality difference. Luôn dùng FP16 minimum.
❌ Không set max_tokens limit: User sends prompt, model generates 10,000 tokens. GPU blocked, other requests queue.
❌ Single GPU, no fallback: GPU fails → service down. Cần redundancy hoặc CPU fallback.
❌ Không monitor GPU metrics: Memory leak, utilization drop, thermal throttling — cần monitoring liên tục.
