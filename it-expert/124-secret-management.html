📚 Thân Thế & Lai Lịch

Phạm Văn Khóa Kín lớn lên với niềm đam mê giấu đồ — từ bé đã giấu kẹo trong tủ sách, giấu nhật ký dưới gạch, giấu điểm kém trong cặp sách. Tài năng "giấu" này đạt đỉnh cao khi anh bước vào ngành IT và phát hiện ra rằng 90% developer đang commit password vào Git repo công khai.

Kiếp nạn thứ nhất: "Đại Họa .env" — khi anh phát hiện file .env với database credentials nằm trong Docker image được push lên Docker Hub public. Kiếp nạn thứ hai: "Sự Kiện AWS Key" — một junior dev commit AWS access key vào GitHub, bot scan phát hiện trong 30 giây, và hóa đơn AWS sáng hôm sau là $47,000 tiền crypto mining. Kiếp nạn thứ ba: "Khai Ngộ Vault" — khi anh nhận ra rằng static secrets là dead secrets, và dynamic secrets mới là con đường chính đạo.

Ngày nay, Khóa Kín được biết đến với quy tắc bất di bất dịch: "Nếu secret có thể bị đọc bởi con người, thì nó sẽ bị lộ bởi con người."

📜 Bí Kíp

Bí Kíp #1: HashiCorp Vault — Ngân Hàng Bí Mật

Vault là central secret management platform — mọi secret đều qua Vault. Key features: dynamic secrets (tạo DB credentials on-demand, tự hủy sau TTL), encryption as a service (Transit engine), PKI certificate management. Architecture: Vault server sealed by default, cần unseal keys để mở. Dùng auto-unseal với AWS KMS hoặc Azure Key Vault. Backend storage nên dùng Consul hoặc Raft (integrated). Audit logging bắt buộc — mọi access đều phải được ghi lại.

Bí Kíp #2: Dynamic Secrets — Bí Mật Tự Hủy

Static secrets (cố định, chia sẻ, không expire) là nguồn gốc mọi tai họa. Dynamic secrets được tạo on-demand, unique per session, tự expire. Ví dụ: Vault Database engine tạo PostgreSQL user/password khi app request, TTL 1 giờ, tự revoke sau đó. Nếu credential bị leak, damage window chỉ là 1 giờ thay vì vĩnh viễn. Setup: enable database engine, configure connection, create role với creation/revocation SQL statements.

Bí Kíp #3: Secret Rotation — Xoay Vòng Liên Tục

Secret không rotate = ticking time bomb. Rotation strategy: automated rotation cho database passwords (mỗi 24-72 giờ), API keys (mỗi 90 ngày), TLS certificates (trước khi expire). Graceful rotation: hỗ trợ 2 versions active đồng thời (old + new) để tránh downtime. AWS Secrets Manager có built-in rotation với Lambda functions. Đừng quên rotate cả secrets của secret management system — inception level security.

Bí Kíp #4: SOPS — Encrypt Config Files

Mozilla SOPS (Secrets OPerationS) encrypt giá trị trong YAML/JSON/ENV files nhưng giữ nguyên keys — nên bạn vẫn có thể diff và review trong Git. Encrypt bằng AWS KMS, GCP KMS, Azure Key Vault, hoặc PGP. Perfect cho Kubernetes secrets, Terraform tfvars, Ansible vault. Workflow: sops --encrypt secrets.yaml > secrets.enc.yaml, commit encrypted file, decrypt at deploy time. Kết hợp với .sops.yaml config để auto-detect encryption rules per path.

Bí Kíp #5: Supply Chain Security & SBOM

SBOM (Software Bill of Materials) là "danh sách thành phần" của software — giống như bảng thành phần trên hộp thực phẩm. Format: SPDX hoặc CycloneDX. Generate SBOM trong CI/CD với syft hoặc trivy. Khi CVE mới xuất hiện, query SBOM để biết ngay service nào bị ảnh hưởng. SLSA (Supply-chain Levels for Software Artifacts) framework đảm bảo integrity từ source code đến deployment. Sigstore/cosign để sign container images.

Bí Kíp #6: Git Secret Scanning

Phòng tuyến đầu tiên: ngăn secrets vào Git repo. Pre-commit hooks với detect-secrets, gitleaks, hoặc trufflehog. GitHub có built-in secret scanning cho public repos. Nhưng nếu secret đã bị commit — git filter-branch hoặc BFG Repo-Cleaner để rewrite history, rồi rotate secret ngay lập tức. Nhớ: nếu secret đã push lên remote, hãy coi nó đã bị compromised — rewrite history chỉ là cleanup, không phải fix.

📖 Chuyện Kể Đêm Khuya

🔥 Chuyện 1: AWS Key Trên GitHub — $47,000 Một Đêm

Junior developer push code lên GitHub repo public, trong đó có file config.py chứa AWS access key và secret key. Trong vòng 30 giây — đúng rồi, 30 GIÂY — bot tự động đã scan, phát hiện, và bắt đầu spin up EC2 instances ở mọi region để mine cryptocurrency.

Sáng hôm sau, AWS billing alert: $47,000. 200+ EC2 instances đang chạy. Team scramble để revoke key, terminate instances, contact AWS support xin giảm bill. AWS thông cảm giảm còn $3,000 nhưng kèm theo "đây là lần cuối".

Từ đó, pre-commit hook với gitleaks trở thành mandatory. Mọi repo đều có .gitignore template. Và junior dev có thêm một bài học trị giá $3,000 — rẻ hơn nhiều bài học khác trong ngành.

💡 Bài học: Prevention > Detection > Response. Pre-commit hooks là $0, nhưng ngăn được incident $47,000. Và KHÔNG BAO GIỜ dùng long-lived credentials cho development — dùng AWS SSO / temporary credentials.

🔥 Chuyện 2: Secret Sprawl — 300 Services, 3000 Secrets, 0 Inventory

Công ty startup grow từ 5 lên 300 microservices trong 3 năm. Mỗi service có secrets riêng, lưu ở đủ nơi: Kubernetes secrets, .env files, AWS Parameter Store, hardcoded trong code, trong CI/CD variables, trong Confluence pages (?!). Tổng cộng ước tính 3000+ secrets, không ai biết chính xác bao nhiêu.

Khi cần rotate database master password, phải hỏi từng team: "Service của bạn có dùng DB này không?" — mất 2 tuần mới "khá chắc" là đã tìm hết. Migrate toàn bộ sang Vault mất 6 tháng, nhưng sau đó: centralized audit log, automated rotation, dynamic secrets, và — quan trọng nhất — một chỗ duy nhất để trả lời "secret này ai đang dùng?"

💡 Bài học: Centralize secret management TỪ ĐẦU. Khi đã có 300 services thì migration là ác mộng. Vault/Secrets Manager không phải luxury — đó là hygiene, giống như version control vậy.

🔥 Chuyện 3: Dynamic Secrets Cứu Cả Đêm Thứ Sáu

Thứ Sáu 11 giờ đêm, nhận alert: database credentials bị leak qua debug endpoint mà ai đó quên tắt. Panic mode — cho đến khi nhớ ra: credentials này là dynamic secret từ Vault, TTL 2 giờ, đã expire từ 45 phút trước.

Leaked credentials hoàn toàn vô giá trị. Không cần rotate, không cần incident response phức tạp, không cần gọi team dậy nửa đêm. Fix debug endpoint, tạo Jira ticket, đi ngủ. Dynamic secrets đã biến một potential P1 incident thành một minor bug fix.

💡 Bài học: Dynamic secrets với short TTL là insurance policy tốt nhất. Khi (không phải "nếu") credentials bị leak, damage window càng nhỏ càng tốt. 2 giờ tốt hơn vĩnh viễn.

⚠️ Những Sai Lầm Kinh Điển

❌ Sai Lầm #1: Commit secrets vào Git — dù là private repo

Private repo hôm nay có thể thành public ngày mai (acquisition, open-source). Git history giữ mọi thứ vĩnh viễn. Dùng secret management tool, không phải Git.

❌ Sai Lầm #2: Dùng cùng một secret cho mọi environment

Production password = staging password = dev password? Khi dev environment bị breach, production cũng xong. Mỗi environment phải có secrets riêng biệt hoàn toàn.

❌ Sai Lầm #3: Không có secret rotation policy

Database password "đặt từ ngày thành lập công ty" = 5 năm, 50 người đã biết (kể cả 30 người đã nghỉ việc). Rotate secrets định kỳ và ngay lập tức khi có người rời team.

❌ Sai Lầm #4: Kubernetes Secrets không encrypted at rest

K8s Secrets mặc định chỉ base64 encode — KHÔNG phải encryption. Enable encryption at rest cho etcd. Hoặc tốt hơn: dùng External Secrets Operator kết hợp với Vault.

❌ Sai Lầm #5: Shared service accounts

"Dùng chung service account cho tiện" — khi có incident, không biết service nào gây ra. Mỗi service phải có identity riêng, credentials riêng, audit trail riêng.

❌ Sai Lầm #6: Không audit secret access

Secret được read 10,000 lần/phút từ IP lạ? Không biết vì không có audit log. Enable audit logging cho mọi secret access — đây là detection layer quan trọng nhất.

❌ Sai Lầm #7: Backup secrets không encrypted

Vault data backup để trên S3 bucket không encrypted, public access. Tất cả secrets vừa bị leak qua backup. Encrypt backups, restrict access, test restore regularly.
