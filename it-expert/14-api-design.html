📜 Thân Thế & Lai Lịch

Hoàng API bắt đầu khi REST còn được viết hoa thành 'REST' một cách trang trọng. Ông từng design API cho ngân hàng — nơi mà một endpoint sai format có thể gây ra chuyển tiền nhầm. Kinh nghiệm đó dạy ông: API design là giao kèo (contract) — phá vỡ giao kèo là phá vỡ niềm tin. Ông sống qua cuộc chiến REST vs SOAP, rồi REST vs GraphQL, rồi GraphQL vs gRPC. Mỗi cuộc chiến, ông đều đứng giữa và nói: 'Đúng tool cho đúng job.' Ông đặc biệt ghét developer nói 'REST is dead' hoặc 'GraphQL solves everything' — vì cả hai đều sai. Motto: 'API là sản phẩm. Treat it like one — documentation, versioning, backward compatibility, developer experience.'

🗝️ Bí Kíp

Bí Kíp #1: REST Done Right — Không chỉ là CRUD

REST = Representational State Transfer, không phải 'dùng HTTP methods cho CRUD'. Resources (nouns, not verbs): /users không phải /getUsers. HTTP methods: GET (read), POST (create), PUT (replace), PATCH (partial update), DELETE (remove). Status codes: 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 429 Too Many Requests, 500 Internal Server Error. Pagination: cursor-based (scalable) > offset-based (simple). HATEOAS: response chứa links đến related actions — client không hardcode URLs.

Bí Kíp #2: GraphQL — Khi Client Cần Linh Hoạt

Client request ĐÚNG data cần — không thừa (over-fetching), không thiếu (under-fetching). Single endpoint, strongly typed schema. Introspection: client khám phá API schema tự động. Subscriptions cho real-time data. Nhược: complexity cao (resolver, N+1 queries, caching khó), learning curve, tooling requirement. DataLoader pattern giải quyết N+1. Persisted queries cho security (client chỉ gửi query hash, không gửi full query). Dùng cho: mobile apps cần flexible data fetching, BFF (Backend for Frontend), complex data relationships.

Bí Kíp #3: gRPC — Performance cho Microservices

Protocol Buffers (protobuf): binary serialization, 10x nhỏ hơn JSON, typed schema. HTTP/2: multiplexing, server push, header compression. Streaming: unary, server streaming, client streaming, bidirectional streaming. Code generation: .proto file → client/server code in any language. Dùng cho: inter-service communication (microservices), mobile apps (bandwidth sensitive), real-time streaming. KHÔNG dùng cho: browser (cần gRPC-web proxy), public APIs (tooling/debugging khó hơn REST/GraphQL).

Bí Kíp #4: tRPC — End-to-End Type Safety

TypeScript fullstack: define API on server, call type-safely from client — ZERO code generation, ZERO schema duplication. Server: define router with procedures. Client: import type, call like function. Autocompletion, compile-time error checking. Perfect cho: Next.js/Nuxt fullstack apps, TypeScript monorepos. Limitation: TypeScript only (server + client), không phù hợp cho public APIs (cần language-agnostic protocol).

Bí Kíp #5: API Versioning — Khi thay đổi là unavoidable

URL versioning: /api/v1/users — simple, clear, cacheable. Header versioning: Accept: application/vnd.api.v2+json — URL clean nhưng ẩn version. Query param: /api/users?version=2 — easy nhưng messy. Best practice: URL versioning cho simplicity + cacheability. KHÔNG breaking changes nếu possible: add fields (don't remove), deprecate (don't delete), new endpoint (don't modify). sunset header cho deprecation timeline.

Bí Kíp #6: API Documentation — Swagger/OpenAPI

OpenAPI Specification: describe API bằng YAML/JSON → auto-generate documentation (Swagger UI, ReDoc), client SDKs, mock servers, tests. Design-first: write spec → implement → validate against spec. Code-first: annotate code → generate spec. Tools: Swagger Editor, Stoplight Studio, Redocly. Documentation là FEATURE — API without docs = API nobody uses. Include: examples, error responses, rate limits, authentication guide.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: REST Endpoint Explosion — 200 Endpoints Cho 1 App

Startup build mobile app, backend team create REST API. Mỗi screen cần data khác nhau → endpoint riêng. Home screen: /api/home-feed. Profile with posts: /api/user-profile-with-posts. Profile with photos: /api/user-profile-with-photos. User with friends and mutual friends: /api/user-friends-mutual. 6 tháng: 200 endpoints, hầu hết chỉ serve 1 screen. Mobile team thay đổi UI → backend phải tạo endpoint mới. Backend team thành 'endpoint factory' cho mobile team. API documentation outdated vì thay đổi quá nhanh. Team evaluate GraphQL: single endpoint, mobile client request đúng fields cần. Migration 3 tháng. Kết quả: 200 endpoints → 1 GraphQL endpoint + clear schema. Mobile team tự quyết data cần, backend team focus vào resolvers và performance.

💡 Bài học: Khi client needs drive API shape quá nhiều → GraphQL shines. Khi API simple, CRUD-based, nhiều consumers → REST stable hơn. BFF (Backend for Frontend) pattern: thin API layer customize response cho specific client, giữ core API clean.

📖 Chuyện 2: GraphQL N+1 Query — Performance Sinkhole

GraphQL API, query users with posts. Resolver: load users → for each user, load posts (separate DB query). 100 users = 1 + 100 = 101 queries. Trang load 5 giây. DBA gọi: 'Cái gì đang blast database 100 queries/request?' Fix: DataLoader — batch multiple getUserPosts(userId) calls into single getUserPostsBatch(userIds) query. 101 queries → 2 queries. Response time: 5s → 200ms. DataLoader là MUST-HAVE cho bất kỳ GraphQL server nào — đừng deploy production GraphQL mà không có DataLoader.

💡 Bài học: GraphQL's flexibility = N+1 trap. LUÔN implement DataLoader (hoặc equivalent batching) cho mọi resolver fetch data. Monitor query complexity — dùng query cost analysis, depth limiting, và timeout để prevent abuse. Automated performance testing cho common queries.

📖 Chuyện 3: API Breaking Change Without Versioning

API v1 response: {{'name': 'Nguyễn Văn A'}}. Team quyết định split thành first/last name: {{'firstName': 'A', 'lastName': 'Nguyễn Văn'}}. Deploy trực tiếp lên production, không version, không deprecation warning. Kết quả: 3 mobile apps, 2 partner integrations, 1 webhook consumer — tất cả crash cùng lúc. Mobile app display 'undefined undefined' thay vì tên user. Partner gọi: 'API các bạn broken.' Hotfix: rollback, thêm cả name (old) + firstName/lastName (new), deprecation header cho name field, 6-month sunset period. Tổng thiệt hại: 4 giờ downtime, 3 angry partner emails, 1 App Store review 'app hiển thị tên sai'.

💡 Bài học: API backward compatibility là SACRED. Quy tắc: 1) Add fields, don't remove 2) Deprecate before removing 3) Version khi breaking changes unavoidable 4) Communicate timeline to consumers 5) Automated contract tests (Pact) catch breaking changes trước khi deploy.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Verbs in REST URLs

/api/getUsers, /api/createUser, /api/deleteUser — HTTP method đã express verb rồi. Dùng: GET /api/users, POST /api/users, DELETE /api/users/:id.

❌ Sai Lầm #2: Inconsistent response format

Endpoint A trả {{'data': [...]}}, endpoint B trả [...], endpoint C trả {{'users': [...]}}. Standardize: {{'data': ..., 'meta': ..., 'errors': ...}} cho mọi endpoint.

❌ Sai Lầm #3: Không rate limit

Không rate limit = một client gửi 10,000 requests/second → service crash. Implement rate limiting: per-client, per-endpoint. Return 429 Too Many Requests + Retry-After header.

❌ Sai Lầm #4: Expose internal IDs

Auto-increment database IDs → attacker enumerate: /users/1, /users/2, /users/3... Dùng UUIDs hoặc hashids cho external-facing IDs.

❌ Sai Lầm #5: Không handle partial failures

API A calls API B, API C. B succeeds, C fails. What happens to B's changes? Implement: saga pattern, compensation transactions, or idempotent retry. ĐỪNG leave system in inconsistent state.
