📚 Thân Thế & Lai Lịch

Hồ Dữ Liệu Tiên Nhân, tên thật là Hồ Chứa Minh (đúng vậy, họ Hồ, tên Chứa, destiny đã quyết định), lớn lên bên Hồ Tây. Lão thích ngắm hồ và suy nghĩ: "Hồ này chứa bao nhiêu nước? Nước từ đâu đến? Đi đâu?"

Vào nghề, lão bắt đầu với data warehouse truyền thống. Nhưng khi phải lưu logs, images, sensor data, social media feeds — structured, semi-structured, unstructured — lão nhận ra warehouse không đủ. Data lake trên S3/HDFS sinh ra để giải quyết. Nhưng cũng sinh ra vấn đề mới: data swamp.

Rồi Delta Lake, Iceberg, Hudi xuất hiện — mang ACID transactions, schema enforcement, time travel vào data lake. Lão gọi đó là "lakehouse revolution": tốt nhất của cả hai thế giới. Lão hiện là tín đồ cuồng nhiệt của lakehouse architecture.

📜 Bí Kíp

Bí Kíp #1: Lakehouse = Data Lake + Data Warehouse

Data trên object storage (S3, GCS, ADLS) — rẻ, scalable. Table format (Delta/Iceberg/Hudi) thêm: ACID transactions, schema enforcement, time travel, audit log. Query engine (Spark, Trino, Presto) đọc trực tiếp. Không cần copy data vào warehouse riêng. One copy of data, multiple use cases.

Bí Kíp #2: Delta Lake — Databricks Ecosystem

ACID transactions trên Parquet files. Transaction log (_delta_log) track mọi thay đổi. Time travel: SELECT * FROM table VERSION AS OF 5. Schema enforcement mặc định (reject data không match schema). Schema evolution: mergeSchema = true. MERGE INTO cho upserts. Tích hợp sâu với Spark và Databricks.

Bí Kíp #3: Apache Iceberg — Vendor Neutral

Catalog-based metadata (không phụ thuộc vào directory structure). Hidden partitioning: partition by month(timestamp) mà query không cần biết partition structure. Snapshot isolation. Schema evolution không cần rewrite data. Hoạt động với Spark, Flink, Trino, Dremio, Snowflake. Community lớn, đang trở thành standard.

Bí Kíp #4: Compaction — Gom Files Nhỏ Thành Files Lớn

Streaming writes tạo nhiều small files (mỗi micro-batch = 1 file). 1 triệu files 1MB → query chậm kinh hoàng (file open overhead). Compaction gom lại: 1000 files → 1 file. Delta Lake: OPTIMIZE table. Iceberg: rewrite_data_files. Schedule compaction daily hoặc khi small files > threshold.

Bí Kíp #5: Medallion Architecture — Bronze, Silver, Gold

Bronze: raw data, as-is from source, append-only. Silver: cleaned, deduplicated, conformed, business entities. Gold: aggregated, business-level, ready for consumption. Mỗi layer có quality guarantees tăng dần. Downstream chỉ đọc Silver/Gold, không bao giờ Bronze.

Bí Kíp #6: Time Travel — Undo Mistakes

Ai đó chạy DELETE FROM customers WHERE active = false nhưng quên WHERE clause? Time travel: SELECT * FROM customers VERSION AS OF '2024-01-15' → restore. Delta Lake giữ history theo retention (default 30 ngày). Iceberg snapshot expiration configurable. Time travel = insurance policy miễn phí.

📖 Chuyện Kể Đêm Khuya

🏚️ Chuyện 1: Data Lake Thành Data Swamp

Company xây data lake trên S3. Ban đầu: structured folders, documented schemas. 2 năm sau: 50 teams dump data vào. Folders: s3://lake/team-a/data-2024/export_final_v2_REAL_USE_THIS.csv. Không schema registry. Không data catalog. Không access control.

Khi analyst cần tìm customer data: 15 folders có "customer" trong tên, 8 có data format khác nhau, 3 đã outdated nhưng không ai biết cái nào. "Data lake" trở thành "data swamp" — mọi thứ được dump vào nhưng không ai lấy ra được gì hữu ích.

Lão phải: implement Iceberg tables (schema enforcement), set up data catalog (AWS Glue), naming conventions + folder structure standards, access control per team. Mất 6 tháng "drain the swamp".

💡 Bài học: Data lake CẦN governance từ ngày đầu: schema enforcement, naming conventions, data catalog, access control, data ownership. Không governance = data swamp guaranteed.
📁 Chuyện 2: Small Files Problem — Khi 1 Triệu Files Giết Chết Performance

Kafka → Spark Structured Streaming → Delta Lake. Trigger: every 1 minute. Mỗi trigger tạo 1 Parquet file per partition. 100 partitions × 1440 minutes/day = 144,000 files/ngày. Sau 1 tuần: 1 triệu files. Query scan 1 ngày data = open 144,000 files = 10 phút cho simple SELECT.

Lão setup: OPTIMIZE Delta Lake chạy mỗi 6 giờ (gom small files), Z-ORDER BY common filter columns, tăng trigger interval lên 5 phút (giảm files 5x). Query time giảm từ 10 phút xuống 15 giây. Storage costs cũng giảm vì compression hiệu quả hơn trên large files.

💡 Bài học: Streaming writes + object storage = small files problem. PHẢI có compaction strategy. Delta OPTIMIZE, Iceberg rewrite_data_files, Hudi compaction. Schedule regular compaction. Target file size: 128MB-1GB.
🔄 Chuyện 3: Schema Evolution — Khi Source Thay Đổi Không Báo

Source API thêm 3 fields mới, bỏ 2 fields cũ, đổi type 1 field (string → integer). Không ai thông báo. Bronze layer ingest vẫn OK (append raw JSON). Nhưng Silver layer Spark job crash: schema mismatch, field không tồn tại, type incompatible.

Lão implement: schema evolution trong Iceberg (thêm columns tự động, rename/drop cần explicit), schema validation layer giữa Bronze→Silver (detect changes, alert, không tự động apply). Và "data contract" với source teams: mọi schema change phải notify 1 tuần trước.

💡 Bài học: Schema WILL change. Design for it: schema evolution support (Iceberg/Delta), validation between layers, data contracts với source teams, schema registry cho tracking. Defensive programming trong data pipelines.

⚠️ Những Sai Lầm Kinh Điển

❌ Dump everything, organize nothing

"Lưu hết vào S3, mai mốt tính" → mai mốt không bao giờ đến. Governance từ ngày 1: folder structure, naming, catalog.

❌ Không compaction

1 triệu small files = query chết. Compaction là maintenance bắt buộc, không phải optional. Schedule nó như schedule backup.

❌ Ignore time travel retention

Delta Lake default 30 ngày history. Vacuum xóa old files. Nếu vacuum mà retention quá ngắn → mất ability to time travel. Set retention phù hợp.

❌ No access control

Mọi team đọc/ghi mọi data. Team marketing vô tình overwrite customer PII data. Fine-grained access control per table/column.

❌ Skip Bronze layer

"Transform luôn khi ingest cho nhanh" → source data thay đổi, cần reprocess từ raw → không có raw data. LUÔN giữ Bronze (raw) layer.
