📚 Thân Thế & Lai Lịch

Đại Sư từng manually scale deployments: kubectl scale --replicas=10 lúc 8 giờ sáng, kubectl scale --replicas=2 lúc 10 giờ tối. Mỗi ngày. "Tôi là human autoscaler," chàng ta tự trào.

Khi HPA (Horizontal Pod Autoscaler) ra đời, chàng ta cấu hình: scale on CPU > 70%. Works! Cho đến khi traffic spike nhanh hơn pod startup time. Users thấy latency spike 30 giây trước khi HPA react. "Reactive scaling has a delay," chàng ta giác ngộ.

Rồi KEDA (Kubernetes Event-Driven Autoscaling): scale based on queue length, HTTP requests, cron schedule. Scale-to-zero cho cost savings. "Now we are talking!" Nhưng scale-to-zero + cold start = tradeoff. Cuộc đời autoscaling là chuỗi tradeoffs vô tận.

📜 Bí Kíp

1. HPA — Horizontal Pod Autoscaler: Scale pods based on metrics: CPU, memory, custom metrics. Target utilization: 70% CPU common. minReplicas, maxReplicas setting. Scaling behavior: scaleUp/scaleDown policies (stabilization window, percentage). Use request-based metrics thay vì utilization cho accurate scaling. Multiple metrics supported (CPU AND custom).
2. VPA — Vertical Pod Autoscaler: Automatically adjust pod resource requests/limits. Modes: Off (recommend only), Auto (apply), Initial (set at creation). KHÔNG dùng VPA + HPA trên same metric (conflict). VPA recreates pods khi adjusting = brief downtime. Good for: right-sizing workloads bạn không biết optimal resources.
3. KEDA — Event-Driven Scaling: Scale based on event sources: Kafka lag, SQS queue length, Prometheus metrics, cron schedule. ScaledObject CRD cho configuration. Scale-to-zero capability (HPA không support). 60+ scalers available. Perfect for: batch processing, event-driven microservices, scheduled workloads.
4. Cluster Autoscaler: Scale nodes based on pending pods (insufficient resources). Scale down nodes with low utilization. Cloud provider integration (ASG cho AWS, VMSS cho Azure). Settings: scan-interval, scale-down-delay. Karpenter (AWS): faster, more flexible alternative — provisions RIGHT instance type cho pending pods.
5. Custom Metrics — Beyond CPU: Prometheus Adapter cho Prometheus metrics → K8s custom metrics API. External metrics: SQS queue depth, Datadog metrics. Business metrics: requests per second, queue length. Custom metrics cho accurate scaling theo actual workload, không chỉ CPU.
6. Scaling Best Practices: Start with HPA on CPU/memory. Add custom metrics khi CPU doesn't correlate với load. minReplicas ≥ 2 cho HA. maxReplicas = cost guardrail. PodDisruptionBudgets cho scale-down safety. Pre-scaling cho predicted traffic (scheduled scaling). Test scaling behavior under load.

📖 Chuyện Kể Đêm Khuya

📈 Chuyện HPA Flapping

HPA configured: min 2, max 20, target CPU 50%. Traffic oscillating → HPA scale up → CPU drops → scale down → CPU rises → scale up → repeat. Pod count ping-ponging: 2 → 8 → 3 → 7 → 4 → 9. Container images pulled/removed constantly. Node resources wasted.

Fix: stabilization window (scaleDown: stabilizationWindowSeconds: 300). Scale down slower than scale up. Target CPU 70% thay vì 50% (less sensitive). Result: stable pod count, smooth scaling.

💡 Bài học: HPA flapping = aggressive scaling configuration. Stabilization windows smooth oscillations. Scale up fast, scale down slow. Higher target utilization = less sensitive. Behavior policies trong HPA v2 cho fine control.

⏰ Chuyện Cluster Autoscaler Quá Chậm

Flash sale: traffic spike 10x trong 30 giây. HPA create pending pods. Cluster Autoscaler detect pending pods → request new nodes from cloud provider. Node provisioning: 3-5 phút (AWS ASG). Pods scheduled after 5 phút. Users experience 5 phút degraded service.

Fix: Karpenter (provisions in 60 giây). Over-provision: keep warm nodes (pause pods với low priority). Predictive scaling: schedule scale-up 15 phút trước expected spike. Cloud provider: warm pools cho faster instance launch.

💡 Bài học: Cluster autoscaler has minutes-level latency. For spiky workloads: over-provision với warm nodes. Karpenter cho faster provisioning. Predictive/scheduled scaling cho known events. Combine HPA + Cluster Autoscaler + prediction cho comprehensive scaling.

💰 Chuyện Scale-to-Zero Tiết Kiệm 80%

Development cluster 50 microservices, mỗi cái minimum 2 replicas. Cost: $3,000/tháng. Usage ngoài giờ làm việc: gần 0. Đại Sư deploy KEDA với cron scaler: scale-to-zero from 8pm to 8am, weekends. Business hours: normal scaling. Result: 14 giờ/ngày × 5 ngày = 70 giờ scaled, 98 giờ/tuần at zero. Cost giảm xuống $600/tháng. Tiết kiệm $2,400/tháng.

💡 Bài học: Scale-to-zero cho non-production environments = massive savings. KEDA cron scaler cho scheduled scaling. Cold start tradeoff acceptable cho dev/staging. Production: keep minReplicas ≥ 1. FinOps quick win: schedule non-prod scaling.

⚠️ Những Sai Lầm Kinh Điển

1. HPA và VPA cùng metric: HPA scale pods, VPA resize pods — trên cùng CPU metric → conflict. Dùng HPA cho CPU, VPA chỉ cho memory (Off mode recommend).
2. maxReplicas quá cao: maxReplicas: 1000 → runaway scaling → exhaust cluster resources → cost explosion. Set realistic max based on capacity và budget.
3. Không test scaling behavior: Assumption: HPA sẽ handle Black Friday. Reality: scaling too slow, too aggressive, or wrong direction. Load test scaling before production events.
4. Ignore scale-down: Focus on scale-up, ignore scale-down delay. Pods chạy idle 30 phút trước scale-down = waste. Tune scaleDown policies.
5. Custom metrics unstable: Custom metric source (Prometheus) down → HPA can't scale → stuck at current replicas. Fallback metrics, metric reliability monitoring.
