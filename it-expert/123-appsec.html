📚 Thân Thế & Lai Lịch

Nguyễn Thị Lỗ Hổng sinh ra trong một gia đình mà bố là sysadmin và mẹ là QA tester — một sự kết hợp DNA hoàn hảo để tạo ra người phụ nữ không tin tưởng bất kỳ input nào từ người dùng. Năm 12 tuổi, cô đã hack được form đăng ký câu lạc bộ tin học của trường bằng cách nhập ' OR 1=1 -- vào ô tên — và từ đó, số phận cô đã được định đoạt.

Trải qua ba kiếp nạn: Kiếp đầu là "Thời Đại XSS Hoành Hành" — khi mà alert(1) xuất hiện trên mọi website và cô phải giải thích cho dev rằng innerHTML = userInput không phải là ý hay. Kiếp hai là "Cuộc Chiến Dependency" — khi mà một package npm bé xíu bị compromise và cả hệ thống sụp đổ như domino. Kiếp ba là "Khai Ngộ Shift-Left" — khi cô nhận ra rằng tìm bug ở production tốn gấp 100 lần so với tìm ở design phase.

Ngày nay, Lỗ Hổng nổi tiếng với triết lý: "Trust nothing. Validate everything. Và luôn encode output." Cô có thể nhìn một đoạn code và đánh hơi ra lỗ hổng bảo mật như chó nghiệp vụ đánh hơi thuốc nổ.

📜 Bí Kíp

Bí Kíp #1: OWASP Top 10 — Bảng Xếp Hạng Tội Phạm

OWASP Top 10 là danh sách 10 lỗ hổng bảo mật phổ biến nhất trong ứng dụng web. Phiên bản 2021 bao gồm: Broken Access Control (#1), Cryptographic Failures (#2), Injection (#3), Insecure Design (#4), Security Misconfiguration (#5). Đây không chỉ là checklist — đây là bản đồ chiến trường. Mỗi mục đều có CWE mapping, ví dụ thực tế, và cách phòng chống. Nếu team của bạn chưa nắm vững OWASP Top 10, thì giống như lính ra trận mà không biết địch ở đâu.

Bí Kíp #2: SQL Injection — Kẻ Thù Không Bao Giờ Chết

SQL Injection tồn tại từ năm 1998 nhưng vẫn xuất hiện hàng ngày. Quy tắc vàng: LUÔN dùng parameterized queries / prepared statements. Đừng bao giờ nối chuỗi SQL bằng tay. Dù dùng ORM cũng phải cẩn thận — raw queries trong ORM vẫn có thể bị inject. Kiểm tra cả stored procedures — chúng không miễn nhiễm. Và nhớ: SQLi không chỉ xảy ra qua form — headers, cookies, và URL parameters đều là vector tấn công.

Bí Kíp #3: XSS — Cross-Site Scripting Ba Đầu Rắn

XSS có 3 loại: Stored (lưu trong DB, nguy hiểm nhất), Reflected (qua URL), và DOM-based (xảy ra phía client). Phòng chống: encode output theo context (HTML, JavaScript, URL, CSS khác nhau). Dùng Content-Security-Policy header để chặn inline scripts. Framework hiện đại như React auto-escape by default — nhưng dangerouslySetInnerHTML là cái tên đã cảnh báo bạn rồi đó. Và đừng quên: Markdown renderers cũng có thể bị XSS nếu không sanitize.

Bí Kíp #4: SAST — Quét Mã Nguồn Tĩnh

SAST (Static Application Security Testing) phân tích source code mà không cần chạy ứng dụng. Tools phổ biến: SonarQube (đa ngôn ngữ), Semgrep (pattern matching mạnh mẽ, custom rules dễ viết), CodeQL (semantic analysis sâu). Tích hợp SAST vào CI/CD pipeline — mỗi PR phải pass security scan trước khi merge. Quan trọng: tune rules để giảm false positives — nếu dev bị alert fatigue, họ sẽ ignore tất cả, kể cả true positives.

Bí Kíp #5: DAST — Tấn Công Chính Mình Trước

DAST (Dynamic Application Security Testing) test ứng dụng đang chạy từ bên ngoài — giống như hacker thật sự. OWASP ZAP là tool miễn phí tuyệt vời, Burp Suite là chuẩn enterprise. Chạy DAST trong staging environment, không phải production. Kết hợp với authenticated scanning để test các trang cần đăng nhập. DAST tìm được những gì SAST bỏ sót — ví dụ security misconfigurations, missing headers, exposed endpoints.

Bí Kíp #6: SCA — Kiểm Soát Supply Chain

SCA (Software Composition Analysis) quét dependencies để tìm CVE đã biết. Snyk, Dependabot, Trivy đều là lựa chọn tốt. Nhưng SCA không chỉ là tìm CVE — còn phải kiểm tra license compliance, abandoned packages, và typosquatting. Lock file (package-lock.json, go.sum) là thiêng liêng — đừng commit mà không review changes. Và luôn nhớ: dependency của dependency cũng có thể là threat vector.

Bí Kíp #7: CSRF — Cross-Site Request Forgery

CSRF lợi dụng session cookie của user để thực hiện request trái phép. Phòng chống bằng CSRF tokens (synchronizer token pattern hoặc double-submit cookie). Dùng SameSite=Lax hoặc SameSite=Strict cho cookies. API dùng Bearer token thay vì cookie thì miễn nhiễm CSRF. Và luôn kiểm tra Origin / Referer header cho critical operations.

📖 Chuyện Kể Đêm Khuya

🔥 Chuyện 1: SQL Injection Giữa Đêm Production

Nửa đêm, alert dồn dập: database đang bị dump toàn bộ. Kiểm tra access log thấy request với payload UNION SELECT * FROM users-- đang kéo hết dữ liệu. Truy nguồn thì ra... một endpoint search được viết tạm bởi intern 6 tháng trước, dùng string concatenation để build SQL query. Endpoint này bypass mọi middleware vì "chỉ là search thôi mà".

Ba tiếng sáng đó, team phải: block IP, patch endpoint, rotate tất cả credentials, notify affected users, và viết incident report dài 15 trang. Intern thì đã nghỉ việc từ lâu. Dòng code tạm đó tồn tại qua 47 lần deploy mà không ai review.

💡 Bài học: Không có code nào là "tạm". Mọi code đều phải qua security review. Và parameterized queries là bắt buộc — không có ngoại lệ, không có "nhưng mà endpoint này đơn giản lắm".

🔥 Chuyện 2: SCA Phát Hiện CVE Nghiêm Trọng

Sáng thứ Hai, Snyk scan báo critical CVE trong log4j — đúng vào tuần Log4Shell bùng nổ (CVE-2021-44228). Team có 200+ microservices, 80% dùng Java. Câu hỏi: service nào dùng log4j phiên bản nào? Không có SBOM (Software Bill of Materials), không ai biết chính xác.

Mất 3 ngày quét toàn bộ registry, tìm mọi dependency tree, phát hiện 47 services bị ảnh hưởng — kể cả 12 services mà "không ai nghĩ là dùng log4j" vì nó là transitive dependency. Từ đó, SBOM generation trở thành bắt buộc trong CI/CD pipeline.

💡 Bài học: SBOM không phải nice-to-have — đó là critical requirement. Khi CVE zero-day xuất hiện, câu hỏi đầu tiên luôn là "chỗ nào dùng thư viện này?" và bạn cần câu trả lời trong PHÚT, không phải ngày.

🔥 Chuyện 3: CSP Header Cứu Cả Hệ Thống

Một ngày đẹp trời, bug bounty hunter báo Stored XSS trên trang profile — user có thể inject script qua field "Bio". Nhưng khi thử exploit, script không chạy. Tại sao? Vì 6 tháng trước, team đã triển khai Content-Security-Policy header nghiêm ngặt: script-src 'self', không cho inline scripts.

XSS payload nằm trong DB, render ra HTML, nhưng browser refuse to execute vì CSP violation. Defense in depth đã cứu ngày hôm đó. Tất nhiên, team vẫn fix XSS bug ngay — CSP là lớp phòng thủ cuối cùng, không phải cái cớ để lười sanitize input.

💡 Bài học: Defense in depth nghĩa là nhiều lớp bảo vệ chồng lên nhau. CSP, input validation, output encoding — mỗi lớp có thể fail, nhưng cùng lúc tất cả fail thì hiếm hơn nhiều. Đầu tư vào security headers — chúng là free insurance.

⚠️ Những Sai Lầm Kinh Điển

❌ Sai Lầm #1: "Frontend validation là đủ rồi"

Frontend validation chỉ để UX tốt hơn — attacker bypass nó trong 5 giây bằng curl hoặc Burp Suite. Mọi validation phải lặp lại ở backend. Mọi input. Mọi endpoint. Không ngoại lệ.

❌ Sai Lầm #2: Lưu password bằng MD5 hoặc SHA1

MD5 và SHA1 là hash functions, không phải password hashing algorithms. Dùng bcrypt, scrypt, hoặc Argon2id — chúng được thiết kế chậm intentionally để chống brute force. Và luôn dùng unique salt per password.

❌ Sai Lầm #3: Trả về error message chi tiết cho user

"Invalid password for user admin" — congratulations, bạn vừa confirm username tồn tại. Stack trace trong API response? Bạn vừa leak technology stack. Error messages cho user phải generic, chi tiết chỉ nên log phía server.

❌ Sai Lầm #4: Hardcode secrets trong source code

API keys, database passwords, JWT secrets trong source code = tặng quà cho hacker. Dùng environment variables, secret management tools (Vault, AWS Secrets Manager). Và scan git history — vì git log nhớ tất cả.

❌ Sai Lầm #5: Bỏ qua security headers

Thiếu X-Content-Type-Options: nosniff, X-Frame-Options, Strict-Transport-Security, Content-Security-Policy là để cửa mở cho XSS, clickjacking, và protocol downgrade attacks.

❌ Sai Lầm #6: "Endpoint nội bộ không cần auth"

Internal network không phải trusted zone. Lateral movement là kỹ thuật cơ bản của attacker. Mọi endpoint, dù internal, đều cần authentication và authorization. Zero trust không phải buzzword.

❌ Sai Lầm #7: Ignore dependency vulnerabilities

"Không có thời gian update dependencies" — cho đến khi bạn có thời gian xử lý security incident. Dependencies chiếm 80-90% codebase. Automate dependency updates với Dependabot/Renovate và SCA scanning trong CI.
