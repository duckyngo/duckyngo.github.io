Chuyên Gia Managed DB, Cache & Kubernetes Services
"Managed services giống như thuê người giúp việc: bạn trả tiền để không phải lau nhà, nhưng đôi khi họ xếp đồ sai chỗ và bạn không tìm ra."
🗄️ RDS/Aurora 🌍 CosmosDB/DynamoDB ⚡ ElastiCache/Redis ⚓ Managed K8s 📨 Managed Kafka 🔍 Elasticsearch Service 🗃️ Managed MongoDB 📊 Managed Analytics

📚 Thân Thế & Lai Lịch

Đại Sư từng là DBA — Database Administrator, người canh giữ MySQL clusters lúc 3 giờ sáng, người chạy VACUUM FULL trên PostgreSQL lúc nửa đêm, người debug replication lag khi mọi người ngủ ngon. Chàng ta biết mùi của server room, tiếng quạt của Dell PowerEdge, và cảm giác hoảng loạn khi disk usage chạm 95%.

Khi RDS ra đời, chàng ta vui mừng: "Automated backups? Auto failover? Không cần thức đêm patch nữa?" Chàng ta chuyển tất cả databases sang RDS và ngủ ngon lần đầu tiên trong 5 năm.

Cho đến khi RDS storage full lúc midnight, automated scaling không kịp, và application timeout hàng loạt. Chàng ta nhận ra: managed không có nghĩa là unmanaged — nó nghĩa là "managed DIFFERENTLY." Bạn vẫn cần hiểu internals, chỉ là ở level khác.

📜 Bí Kíp

1. RDS vs Self-Managed — Khi Nào Chọn Gì: RDS: automated backups, patching, failover, read replicas. Tốt cho: hầu hết workloads. Self-managed EC2: cần OS-level access, custom extensions, specific version. Aurora: MySQL/PostgreSQL compatible, 5x throughput, auto-scaling storage. Aurora Serverless v2 cho variable workloads.
2. DynamoDB — NoSQL Managed: Single-digit millisecond latency at any scale. On-demand capacity cho unpredictable traffic. Provisioned capacity + auto-scaling cho predictable. Global Tables cho multi-region. DynamoDB Streams cho event-driven. Partition key design QUAN TRỌNG — hot partition kills performance. KHÔNG dùng scan operations.
3. ElastiCache — Redis/Memcached Managed: Redis cho: caching + data structures + pub/sub + persistence. Memcached cho: simple caching, multi-threaded. Cluster mode enabled cho sharding. Failover cho HA. Cache-aside pattern phổ biến nhất. Cache invalidation strategy: TTL + explicit invalidation. Monitor cache hit ratio — dưới 80% cần review.
4. Managed Kubernetes — EKS/AKS/GKE: Control plane managed bởi cloud provider. Worker nodes: managed node groups hoặc Fargate/Autopilot. Upgrades: plan carefully — test trên staging trước. Add-ons: CoreDNS, kube-proxy, VPC CNI cần maintain. Cost: control plane fee + worker node compute + networking.
5. Managed Kafka — MSK/Event Hubs: Amazon MSK: Apache Kafka managed. Không cần manage ZooKeeper (KRaft mode). Serverless option cho variable throughput. Azure Event Hubs: Kafka-compatible managed service. GCP Pub/Sub: different model nhưng similar use cases. Evaluate managed vs self-managed dựa trên team expertise và scale.
6. When NOT to Use Managed Services: Khi bạn cần version/config cloud provider không support. Khi pricing model không fit workload. Khi vendor lock-in là dealbreaker. Khi team có deep expertise và muốn full control. Calculate TCO: managed service cost vs engineering time to self-manage.

📖 Chuyện Kể Đêm Khuya

🗄️ Chuyện RDS Storage Full Lúc Nửa Đêm

Friday night, 11:30 PM. RDS PostgreSQL instance storage đạt 100%. Automated storage scaling enabled nhưng đang ở maximum allowed storage (16TB). Application bắt đầu throw errors. On-call engineer nhận PagerDuty alert, login AWS console, thấy storage full. Không thể tăng max storage khi instance đang stressed.

Root cause: một developer chạy migration script tạo bảng temporary 2TB, quên cleanup. Bảng temporary + WAL logs + backups = storage explosion. Fix: delete temp table, VACUUM FULL, tăng storage max. Mất 3 giờ. 200 customers affected.

Prevention: storage monitoring alerts ở 70%, 80%, 90%. Query audit cho large operations. Automated cleanup scripts. RDS Performance Insights để identify resource-heavy queries.

💡 Bài học: Storage monitoring với alerts ở nhiều thresholds. Review large queries/operations trước khi chạy production. Auto-scaling storage có upper limit — biết limit là gì. Cleanup policies cho temporary data. VACUUM scheduling cho PostgreSQL.

⚡ Chuyện Managed Service Upgrade Breaks Everything

ElastiCache Redis upgrade từ 6.x lên 7.x. AWS console nói 'seamless upgrade, minimal downtime.' Team schedule upgrade cho Saturday morning. Upgrade starts, nodes begin restarting. 15 phút later: application errors spike. Redis commands returning unexpected errors — một số commands deprecated trong Redis 7.

Application code dùng HMSET (deprecated, replaced by HSET). ElastiCache 6.x tolerate HMSET, ElastiCache 7.x strict mode reject nó. Rolling back Redis version? AWS không support downgrade. Team phải hotfix application code Saturday afternoon.

💡 Bài học: LUÔN test upgrades trên staging environment TRƯỚC production. Read release notes và breaking changes. Deprecation warnings trong logs = action items, không phải decorations. Have rollback plan — nhưng biết rằng managed service downgrades thường không possible.

🌍 Chuyện CosmosDB Request Units Bất Ngờ

Team dùng Azure CosmosDB với provisioned 400 RU/s (minimum). Application launch day: traffic spike. CosmosDB throttle requests — 429 Too Many Requests. Users see errors. Team tăng RU lên 4000 — cost tăng 10x. Traffic normalize sau 2 giờ, nhưng RU vẫn ở 4000 vì ai cũng sợ giảm xuống lại throttle.

Đại Sư suggest: Autoscale mode — tự động scale RU từ 400 đến 4000 dựa trên traffic. Chi phí giảm 60% so với fixed 4000 RU. Hoặc serverless mode cho unpredictable traffic.

💡 Bài học: Hiểu pricing model của managed service TRƯỚC khi dùng. CosmosDB RU, DynamoDB RCU/WCU — capacity-based pricing cần right-sizing. Autoscale options cho variable workloads. Monitor consumed capacity vs provisioned capacity.

⚠️ Những Sai Lầm Kinh Điển

1. Treating managed service as black box: Managed không có nghĩa là ignore. Vẫn cần hiểu performance characteristics, limits, failure modes. Read documentation — seriously.
2. Không test failover: RDS Multi-AZ failover: 60-120 seconds downtime. Application phải handle reconnection. Test failover — đừng đợi production incident để discover issues.
3. Bỏ qua connection pooling: Application mở 500 connections tới RDS — hit max connections limit. Connection pooler (PgBouncer, ProxySQL, RDS Proxy) là must-have cho production.
4. Wrong database cho use case: Dùng RDS PostgreSQL cho time-series data (DynamoDB hoặc Timestream better). Dùng DynamoDB cho complex queries (RDS better). Chọn database theo access patterns.
5. Không encrypt data: Managed services hỗ trợ encryption at rest — ENABLE IT. KMS keys cho encryption. In-transit encryption (TLS) bắt buộc. Compliance requirement cho hầu hết industries.
