📚 Thân Thế & Lai Lịch

Đại Sư từng quản lý 200 servers, thức đêm patch kernels, và sống trong nỗi sợ disk full. Một ngày, chàng ta đọc bài viết "Serverless: The Future of Cloud Computing" và như được khai sáng. "Không cần quản lý servers nữa? Sign me up!"

Chàng ta chuyển mọi thứ sang Lambda. Web app? Lambda + API Gateway. Batch job? Lambda. Image processing? Lambda. Cron job? Lambda + CloudWatch Events. Chàng ta trở thành "Lambda Man" trong team — mọi vấn đề đều có thể giải quyết bằng Lambda.

Cho đến khi bill tháng đó lên $8,000 — cao hơn 3 lần so với EC2 trước đây. Chàng ta giác ngộ: serverless tuyệt vời cho đúng use case, nhưng không phải silver bullet. Sau đó chàng ta dành 10 năm tiếp theo để giúp người khác tránh sai lầm tương tự.

📜 Bí Kíp

1. Event-Driven Architecture — Tư Duy Đúng: Serverless hoạt động tốt nhất với event-driven patterns. API request → Lambda → DynamoDB. S3 upload → Lambda → image processing. SQS message → Lambda → notification. Nghĩ theo events, không phải processes. Mỗi function làm MỘT việc, làm tốt.
2. Cold Start — Kẻ Thù Số 1: Node.js/Python: 100-500ms. Java/C#: 1-10 giây. VPC: thêm 1-5 giây. Giảm thiểu: keep package small, avoid VPC khi không cần, Provisioned Concurrency cho latency-critical. SnapStart (Java) giảm cold start đáng kể. Hoặc dùng Cloud Run/Fargate cho container-based serverless.
3. Cost Model — Hiểu Trước Khi Build: Lambda pricing: requests + duration + memory. $0.20 per 1M requests + $0.0000166667 per GB-second. Cheap khi traffic thấp/sporadic. EXPENSIVE khi traffic cao, sustained. Break-even point: ~10M requests/tháng thường cheaper trên containers.
4. Step Functions — Orchestration Cho Complex Workflows: Khi logic cần nhiều steps, error handling, retries, parallel execution. Visual workflow editor. Standard Workflows cho long-running (up to 1 year). Express Workflows cho high-volume, short-duration. Đừng chain Lambdas bằng code — dùng Step Functions.
5. Observability — Khó Hơn Bạn Nghĩ: Distributed tracing quan trọng khi có 50 Lambda functions. X-Ray cho AWS, Application Insights cho Azure. Structured logging (JSON) để CloudWatch Insights query được. Custom metrics cho business logic. Powertools for Lambda simplifies everything.
6. Testing — Local vs Cloud: Serverless functions khó test locally vì phụ thuộc cloud services. SAM Local cho AWS, Azure Functions Core Tools. Nhưng không bao giờ 100% giống production. Integration tests trên cloud environment quan trọng hơn unit tests cho serverless.

📖 Chuyện Kể Đêm Khuya

💸 Chuyện Serverless Đắt Hơn EC2

Một team chuyển REST API từ 2 EC2 t3.medium ($60/tháng) sang Lambda + API Gateway. Ban đầu tuyệt vời: chi phí $5/tháng vì traffic thấp. Team manager khoe với CTO: "Tiết kiệm 90%!"

6 tháng sau, app có traction. Traffic tăng 100x. Lambda bill: $2,000/tháng. API Gateway bill: $1,500/tháng. CloudWatch Logs: $300/tháng. Tổng: $3,800/tháng cho workload mà 2 EC2 instances ($60) handle được thoải mái.

Đại Sư analyze: API Gateway per-request pricing killed them. Lambda running at 100% concurrency = always-on = expensive. Solution: move high-traffic API sang ALB + Fargate, keep event-driven async functions trên Lambda.

💡 Bài học: Serverless cost-effective cho sporadic/unpredictable traffic. Cho sustained high traffic, containers hoặc VMs thường rẻ hơn. Calculate break-even point TRƯỚC khi migrate. Hybrid approach: serverless cho glue logic, containers cho core services.

⏰ Chuyện 15-Minute Timeout

Team build video processing pipeline trên Lambda. Upload video → Lambda transcode → S3. Works great cho video dưới 5 phút. Một ngày, client upload video 2 giờ. Lambda hit 15-minute timeout. Function terminated mid-processing. Output file corrupted.

Team retry tự động. Lambda hit timeout lại. Retry. Timeout. 10 retries = 10 corrupted files + $50 in Lambda costs. DLQ (Dead Letter Queue) đầy messages. On-call engineer nhận 10 PagerDuty alerts.

Fix: chuyển video processing sang ECS Fargate task (no timeout limit), dùng Lambda chỉ để trigger Fargate task. Step Functions orchestrate workflow với error handling.

💡 Bài học: Lambda max timeout = 15 phút. Cho long-running jobs: ECS Fargate, Step Functions, hoặc EC2. Lambda tốt cho glue logic và short tasks. Đừng force-fit long processes vào Lambda.

🔄 Chuyện Lambda Recursive Loop

Developer viết Lambda trigger: S3 upload → Lambda process file → save result back to SAME S3 bucket. Lambda chạy → save result → trigger Lambda → save result → trigger Lambda... Infinite loop.

1000 concurrent executions in seconds. AWS account throttled. Bill chạy lên $500 trong 10 phút trước khi engineer nhận alert và disable trigger. May mắn là Lambda có concurrency limit mặc định 1000.

💡 Bài học: NEVER trigger Lambda từ output mà Lambda writes to. Dùng separate buckets/prefixes cho input và output. Set Lambda reserved concurrency limit. Enable Lambda recursive loop detection (AWS tính năng mới).

⚠️ Những Sai Lầm Kinh Điển

1. "Serverless cho mọi thứ": Serverless tốt cho event-driven, glue logic, APIs traffic thấp. Không tốt cho: long-running processes, high-throughput sustained workloads, WebSocket heavy apps.
2. Mega-function: Một Lambda function 5000 dòng code xử lý 20 use cases. Single Responsibility Principle áp dụng cho serverless. Mỗi function = 1 job.
3. Bỏ qua cold start trong production: Demo thì smooth vì function đã warm. Production lúc 3am, function cold, user đầu tiên chờ 10 giây. Plan cho cold starts hoặc dùng Provisioned Concurrency.
4. Không set concurrency limits: Traffic spike → thousands of Lambda invocations → overwhelm downstream database. Set reserved concurrency per function. Use SQS to buffer.
5. Logging everything: CloudWatch Logs pricing by ingestion. Lambda logs mỗi request = $$$ khi traffic cao. Log selectively, use log levels, set retention policies.
6. Ignoring API Gateway costs: API Gateway REST API: $3.50/million requests. Cho high-traffic API, ALB ($0.008/LCU-hour) thường rẻ hơn nhiều. HTTP API ($1.00/million) là alternative rẻ hơn.
