📚 Thân Thế & Lai Lịch

Vào nghề tech, lão trở thành "người gác cổng" A/B testing. Không ai được ship feature mà không qua tay lão. PM nào kêu "Nhìn mà, metrics tăng rõ ràng!" là lão lắc đầu: "Sample size bao nhiêu? Statistical power? Multiple testing correction? Novelty effect?"

📜 Bí Kíp

Bí Kíp #1: p-value — Hiểu Đúng Rồi Mới Dùng

p-value = xác suất thấy kết quả extreme như vậy (hoặc hơn) NẾU null hypothesis đúng. p = 0.03 KHÔNG có nghĩa "xác suất 97% rằng treatment có effect". p-value không cho biết effect size (lớn hay nhỏ), chỉ cho biết unlikely hay không. Effect có thể statistically significant nhưng practically insignificant.

Bí Kíp #2: Sample Size Calculation — Trước Khi Test, Tính Trước

Cần bao nhiêu users? Phụ thuộc: baseline conversion rate, minimum detectable effect (MDE), significance level (alpha, thường 0.05), statistical power (1-beta, thường 0.8). MDE 1% absolute change cần nhiều users hơn MDE 10%. Tính trước bằng power analysis. Không tính = chạy test vô nghĩa.

Bí Kíp #3: A/B Test Setup — Randomization Là Tất Cả

Random assignment: mỗi user có equal probability vào group A hoặc B. Stratified randomization cho balanced groups (country, device type). Hash-based bucketing: hash(user_id) % 100 = stable assignment. Không random = confounded results = wrong conclusions.

Bí Kíp #4: Confidence Intervals — Hữu Ích Hơn p-value

CI cho biết range of plausible values cho true effect. "Conversion rate tăng 2.3%, 95% CI [0.5%, 4.1%]" → có ý nghĩa hơn "p = 0.02". CI chứa 0 = not significant. Narrow CI = precise estimate. Wide CI = cần more data. Always report CI alongside p-value.

Bí Kíp #5: Multiple Testing Problem — Càng Test Nhiều, Càng Dễ Sai

Test 20 metrics cùng lúc, alpha = 0.05 mỗi metric. Xác suất ít nhất 1 false positive: 1 - 0.95^20 = 64%! Bonferroni correction: alpha/n = 0.05/20 = 0.0025 mỗi metric. Hoặc Benjamini-Hochberg FDR control. Hoặc: chọn 1-2 primary metrics, còn lại là secondary/exploratory.

Bí Kíp #6: Bayesian A/B Testing — Alternative Approach

Frequentist: "reject or fail to reject H0". Bayesian: "probability that B is better than A = 95%". Dễ hiểu hơn cho stakeholders. Có thể stop early khi evidence đủ mạnh. No fixed sample size required. Nhưng: prior specification matters, computationally heavier. Tools: PyMC, Stan, VWO Bayesian engine.

📖 Chuyện Kể Đêm Khuya

🎪 Chuyện 1: A/B Test Với Wrong Sample Size

PM muốn test new checkout flow. Current conversion: 3%. Expected improvement: 0.1% (3.0% → 3.1%). Cần sample size: ~1.5 triệu users per group (3 triệu total). Nhưng PM nói: "2 tuần là đủ." 2 tuần = 50,000 users per group.

Kết quả: conversion 3.2% vs 3.0%. PM hét: "Tăng 0.2%! Ship ngay!" Lão check: p = 0.38. Completely not significant. Statistical power: 8% (cần 80%). Test này vô nghĩa — nếu có effect thật, test không đủ power để detect. PM buồn nhưng lão giải thích: "Chạy test ngắn với effect nhỏ giống như tìm kim trong đại dương bằng cần câu."

💡 Bài học: LUÔN calculate sample size TRƯỚC KHI chạy test. Nếu cần quá nhiều users → increase MDE (accept only larger effects) hoặc improve conversion funnel trước. Under-powered tests = waste of time.
🔗 Chuyện 2: Correlation vs Causation — Câu Chuyện Kinh Điển

Data team phát hiện: users dùng feature "Dark Mode" có retention cao hơn 40%. PM kết luận: "Dark Mode tăng retention! Push mặc định cho tất cả users!"

Lão hỏi: "Hay là users engaged hơn thì explore settings nhiều hơn → tìm thấy Dark Mode? Correlation: Dark Mode usage ↔ retention. Nhưng causation có thể ngược: high retention → discover Dark Mode, không phải Dark Mode → high retention."

Lão chạy A/B test: force Dark Mode cho random 50% users mới. Kết quả: retention KHÔNG KHÁC BIỆT. Dark Mode không cause retention. Engaged users chỉ tình cờ dùng Dark Mode nhiều hơn.

💡 Bài học: Correlation ≠ Causation. Observational data chỉ cho correlation. Muốn prove causation → cần experiment (A/B test) hoặc quasi-experimental methods (instrumental variables, regression discontinuity). ĐỪNG make causal claims từ correlational data.
👀 Chuyện 3: Peeking Problem — Nhìn Trộm Kết Quả Sớm

A/B test planned 4 tuần. Sau ngày 3, PM mở dashboard: "Wow, treatment tốt hơn 5%, p = 0.04! Ship!" Lão phản đối nhưng bị overrule. Ship. 2 tuần sau: metric trở về baseline. "False positive" — lão nói.

Vấn đề: peeking at results repeatedly inflates false positive rate. Nếu check mỗi ngày trong 30 ngày với alpha 0.05, false positive rate lên ~15-25%. Giải pháp: group sequential analysis (planned interim looks), hoặc Bayesian (okay to peek), hoặc kỷ luật: chỉ check khi sample size đạt target.

💡 Bài học: "Peeking" at A/B test results inflates false positive rate. Solutions: pre-commit to sample size, use sequential testing methods, hoặc Bayesian framework. ĐỪNG stop early khi "looks good" — trừ khi dùng proper sequential method.

⚠️ Những Sai Lầm Kinh Điển

❌ "p < 0.05 = truth"

p-value depends on sample size. With enough data, even trivial differences become "significant". Report effect size + CI, not just p-value.

❌ No guardrail metrics

Test tăng conversion nhưng decrease revenue per user? Tăng clicks nhưng increase bounce rate? Set guardrail metrics: thứ phải KHÔNG giảm.

❌ Test too many variants

A/B/C/D/E/F testing = split traffic 6 ways. Each variant gets 1/6 traffic. Need 6x longer. And multiple comparison penalty. A/B hoặc max A/B/C.

❌ Novelty effect / Primacy effect

New design looks "better" first week vì users curious. Wait > 2 weeks, compare same cohort. Filter out returning users cho cleaner signal.

❌ Wrong randomization unit

Randomize by page view (same user sees different variants on different visits) = contaminated experiment. Randomize by user ID = consistent experience.
