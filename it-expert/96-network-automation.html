📚 Thân Thế & Lai Lịch

Tự Động Hóa Mạng Sư bắt đầu với 200 switches cần update VLAN. SSH vào từng switch, gõ commands, copy-paste config. Mất 3 ngày. Typo ở switch thứ 47 gây loop. Mất thêm 1 ngày debug. "Phải có cách tốt hơn," chàng ta nghĩ.

Chàng ta học Python, phát hiện Paramiko, rồi Netmiko (Paramiko cho network devices), rồi NAPALM (vendor-agnostic), rồi Ansible (automation framework). Workflow mới: viết playbook 30 phút, test trên lab 1 giờ, deploy 200 switches trong 15 phút. Đời đẹp hơn.

Nhưng automation cũng mang đến rủi ro mới: một playbook sai có thể phá hủy 200 switches trong 15 phút thay vì 1 switch trong 3 ngày. "With great automation comes great responsibility," chàng ta thích nói vậy.

📜 Bí Kíp

1. Ansible for Network Automation: ansible.netcommon collection. ios_config, nxos_config, eos_config modules. Connection: network_cli cho SSH, netconf cho NETCONF. Idempotent operations: apply config only if different. Check mode (dry run) TRƯỚC khi apply. Roles cho reusable configurations.
2. Netmiko — Python SSH for Network: ConnectHandler cho SSH connection. send_command cho show commands. send_config_set cho config changes. TextFSM templates parse output thành structured data. Concurrent execution với threading. Error handling: expect patterns, timeout management. Tốt cho quick scripts và custom automation.
3. NAPALM — Vendor-Agnostic: get_facts(), get_interfaces(), get_bgp_neighbors() — same API cho Cisco, Juniper, Arista. Configuration replace và merge. Rollback support. Compliance validation: compare running config vs desired state. Tích hợp tốt với Ansible (napalm_install_config module).
4. Configuration Validation: Pre-check: verify current state trước khi change. Post-check: verify desired state sau khi change. Automated test suites: ping tests, routing table verification, interface status. Batfish/Catfish cho offline config analysis. pyATS/Genie (Cisco) cho network testing framework.
5. Change Management: Peer review cho network changes (gitops cho network). Maintenance windows cho production changes. Rollback plan cho EVERY change. Change tickets với expected impact. Post-change monitoring period. Gradual rollout: test trên 1 device → group → all.
6. NETCONF/RESTCONF — Modern Approach: NETCONF: XML over SSH, YANG models. RESTCONF: HTTP/REST + JSON/XML, YANG models. Structured data thay vì screen scraping. Transaction support (commit, rollback). Better than CLI parsing cho reliable automation. Adoption tăng nhưng không phải mọi vendor support đầy đủ.

📖 Chuyện Kể Đêm Khuya

🔄 Chuyện Automated Config Push Sai VLAN

Ansible playbook update VLAN assignment cho 50 access switches. Playbook tested trên lab — works. Push to production. 5 phút sau: phones ring nonstop. 200 users mất network access. Reason: playbook variable file có VLAN ID sai — VLAN 100 (production) thay vì VLAN 200 (test). Tất cả access ports chuyển sang VLAN 100 — overloaded VLAN, DHCP exhausted.

Rollback: Ansible playbook restore từ git backup. 10 phút total downtime. Post-mortem: thêm validation step (check VLAN exists, check DHCP pool), peer review cho variable files, gradual rollout (5 switches → verify → next 5).

💡 Bài học: Automation multiplies mistakes. VALIDATE inputs trước khi push. Gradual rollout thay vì all-at-once. Peer review playbooks VÀ variable files. Pre-check/post-check automated tests. Rollback plan tested và ready.

⏰ Chuyện Ansible Playbook Timeout

Playbook upgrade firmware cho 100 switches, 10 concurrent. Switch reboot sau upgrade, Ansible wait cho SSH connection. 20 switches reboot simultaneously → power spike → UPS alarm → 5 switches không boot lại (firmware corrupt). Ansible timeout, report failures.

Root cause: concurrent firmware upgrades cause power spike. Fix: serial: 5 (chỉ 5 switches cùng lúc), wait_for reboot hoàn tất trước khi tiếp, pre-download firmware trước maintenance window, backup running config trước upgrade.

💡 Bài học: Firmware upgrades ≠ config changes. Concurrent upgrades cause resource contention. Serial execution cho risky operations. Backup config TRƯỚC mọi firmware change. Verify firmware integrity (checksum) trước install. Have console access ready.

📋 Chuyện Network as Code Cứu Mạng

Senior network engineer nghỉ việc. Toàn bộ network knowledge trong đầu anh ấy. 300 devices, undocumented configs, manual changes over 5 năm. New engineer join: 'Network diagram ở đâu?' — 'Không có.' 'Config backup?' — 'Trong USB drive ở drawer, dated 2 năm trước.'

Đại Sư implement Network as Code: NAPALM backup tất cả configs to git daily. Ansible playbooks cho standard changes. Netbox cho IPAM và documentation. CI/CD pipeline cho config deployment. 6 tháng sau: toàn bộ network documented, reproducible, và version-controlled.

💡 Bài học: Network knowledge không được ở trong đầu một người. Config as code trong git. Automated backup daily. Documentation as code (Netbox). Network automation không chỉ là efficiency — nó là knowledge preservation và disaster recovery.

⚠️ Những Sai Lầm Kinh Điển

1. Automation without testing: Push untested changes to production. ALWAYS test trên lab/staging environment trước.
2. No rollback plan: Automated change breaks something. How to roll back? Automated rollback cũng cần tested.
3. Hardcode credentials trong playbooks: ansible-vault cho secrets. Environment variables. Vault integration (HashiCorp Vault). NEVER plaintext credentials trong git.
4. No change tracking: Who changed what, when? Git commits cho network changes. Audit trail bắt buộc cho compliance.
5. Automate before understanding: Automate manual process mà không hiểu nó → automate mistakes. Understand, document, THEN automate.
