📚 Thân Thế & Lai Lịch

Linux Thần Thánh sinh ra trong một gia đình Windows. Ngày đầu tiên cài Ubuntu, chàng ta mất 3 giờ để cài driver Wi-Fi — bằng cách compile kernel module từ source. Bố chàng ta hỏi: "Sao không dùng Windows cho dễ?" Chàng ta đáp: "Vì Linux là TỰ DO, bố!"

10 năm sau, chàng ta đã thử hết mọi distro: Ubuntu, Fedora, Arch, Gentoo, Slackware, và thậm chí cả LFS (Linux From Scratch — 3 tuần compile). Chàng ta có thể live trong terminal mà không cần GUI suốt 6 tháng. Desktop environment? "Terminal với tmux là đủ rồi."

Bây giờ chàng ta quản lý 5,000 Linux servers, viết Ansible playbooks trong giấc ngủ, và vẫn tranh luận về systemd vs init mỗi tối thứ Sáu trên Reddit.

📜 Bí Kíp

1. Distro Selection — Chọn Đúng Từ Đầu: RHEL/CentOS Stream/Rocky/Alma cho enterprise (10 năm support). Ubuntu LTS cho cloud-native (5 năm support). Debian cho stability. Arch cho bleeding edge (chỉ dev/personal). Alpine cho containers (5MB base image). ĐỪNG dùng bleeding-edge distro cho production.
2. Kernel Tuning — Hiểu Nhưng Đừng Lạm Dụng: vm.swappiness: giảm cho database servers (10), default 60. net.core.somaxconn: tăng cho high-connection servers (65535). fs.file-max: tăng cho servers handle nhiều files. NHƯNG: đừng copy-paste sysctl settings từ StackOverflow — hiểu trước khi tune. Default kernel settings tốt cho 90% workloads.
3. systemd — Yêu Hay Ghét, Đều Phải Dùng: Unit files cho service management. journalctl cho log analysis. systemd-analyze cho boot performance. Timer units thay thế cron. Socket activation cho on-demand services. Resource control (CPUQuota, MemoryMax) cho service isolation. Đừng tranh luận systemd vs init nữa — systemd thắng rồi.
4. Package Management — Giữ Sạch Hệ Thống: RHEL: dnf, RPM. Ubuntu: apt, dpkg. Pin package versions cho production. KHÔNG mix repositories. Unattended-upgrades cho security patches (cẩn thận test trước). Đừng curl | bash install scripts trên production — package manager hoặc build from source.
5. Filesystem — Chọn Đúng Tool: ext4: default, reliable, proven. XFS: large files, parallel I/O, good for databases. Btrfs: snapshots, compression, checksums. ZFS: ultimate data integrity (nhưng memory-hungry). tmpfs cho /tmp. noatime mount option cho performance. Regular fsck cho health checks.
6. Security Hardening: Disable root SSH login. SSH key-only authentication. fail2ban cho brute force protection. SELinux/AppArmor enabled (ĐỪNG disable vì 'nó gây lỗi' — fix policy thay vì disable). Firewall (iptables/nftables/firewalld) enabled. Automatic security updates. Audit logging enabled.
7. Automation — Đừng Làm Gì Hai Lần: Shell scripts cho simple automation. Ansible cho configuration management. Cloud-init cho instance initialization. Infrastructure as Code cho reproducibility. Nếu bạn SSH vào server để thay đổi config manually, bạn đang làm sai.

📖 Chuyện Kể Đêm Khuya

💀 Chuyện rm -rf / Kinh Hoàng

Junior admin chạy script cleanup: rm -rf $DIR/. Variable $DIR chưa được set → expand thành rm -rf /. Script chạy với root privileges. Trong 30 giây, hệ thống bắt đầu chết: commands not found, services crash, kernel panic.

May mắn: đây là staging server. Không may: staging server cũng chứa test data quan trọng, 2 tuần work mất trắng. Từ đó, team thêm set -euo pipefail vào mọi script, và thêm check: if [ -z "$DIR" ]; then echo 'DIR not set'; exit 1; fi

💡 Bài học: LUÔN dùng set -euo pipefail trong bash scripts. Validate variables trước khi dùng. rm -rf với variable? Thêm safety check. Backup trước khi cleanup. Và nhớ: --preserve-root là default trong modern GNU rm, nhưng đừng rely vào nó.

🔥 Chuyện Kernel Panic Lúc 2 Giờ Sáng

Production server kernel panic sau kernel update. Auto-reboot → kernel panic lại. Loop. 500K users không access được service. Team SSH vào: không được — server boot vào kernel panic loop. Phải console access qua IPMI, boot vào old kernel từ GRUB menu.

Root cause: kernel update incompatible với custom driver (GPU passthrough driver). Lesson: ĐỪNG auto-update kernel trên production. Test kernel update trên staging với đúng hardware/driver configuration trước. Pin kernel version, manual update schedule.

💡 Bài học: Kernel updates cần testing cẩn thận, đặc biệt với custom drivers. Pin kernel version trên production. GRUB menu giữ 2-3 kernel cũ cho rollback. Test trên staging với identical hardware config. IPMI/iLO/iDRAC access là must-have cho bare metal.

⚔️ Chuyện systemd vs init Debate

Team meeting biến thành flame war khi senior engineer đề xuất chuyển từ init scripts sang systemd. 'systemd is bloated!' 'systemd violates Unix philosophy!' Debate kéo dài 3 giờ, không ai viết code. Manager cuối cùng quyết định: 'Dùng systemd vì mọi major distro đều mặc định nó. Meeting adjourned.'

3 tháng sau, init advocate thừa nhận: 'Systemd timer units hay hơn cron, và journalctl actually useful.' Nhưng chàng ta vẫn grumble mỗi khi PID 1 bị đề cập.

💡 Bài học: Kỹ thuật debate là healthy nhưng phải có deadline. Adopt what the ecosystem supports. Systemd là standard — learn nó thay vì fight nó. Focus năng lượng vào solving real problems, không phải ideology wars.

⚠️ Những Sai Lầm Kinh Điển

1. Disable SELinux thay vì fix policy: SELinux 'causes issues' → disable → security hole. Debug với audit2why, tạo custom policy. Permissive mode để test trước khi enforce.
2. Run everything as root: Convenience ≠ security. Least privilege: dedicated service accounts, sudo cho specific commands, không phải sudo su -.
3. Không monitor disk space: / full → system crash. /var/log full → services can't write logs → silent failures. Monitor và alert ở 80%.
4. Copy config từ internet không hiểu: StackOverflow sysctl.conf cho MySQL server applied lên web server. Different workloads cần different tuning. Understand trước khi apply.
5. Không test backup restore: Backup chạy hàng đêm, 2 năm không ai test restore. Khi cần: backup files corrupted. TEST RESTORE REGULARLY.
6. Ignore system logs: /var/log/syslog, journalctl đầy warnings mà không ai đọc. Centralized logging + alerting cho error patterns. Logs là early warning system.
