📚 Thân Thế & Lai Lịch

Đại Sư từng tạo infrastructure bằng console clicks. Click, click, click — 200 resources. Documentation? "Console history." Reproducibility? "Click lại." Disaster recovery? "Click... nhanh hơn."

Khi Terraform xuất hiện, chàng ta viết infrastructure bằng code. terraform plan → review → terraform apply → infrastructure created. Version controlled, reviewable, reproducible. "This is the way," chàng ta tuyên bố.

Nhưng rồi chàng ta gặp State File — cái file JSON chứa toàn bộ trạng thái infrastructure. Mất state file = Terraform không biết gì về existing resources. State file corruption = nightmare. "State file is the most critical file in your infrastructure," chàng ta khắc chữ lên bàn làm việc.

📜 Bí Kíp

1. State Management — Quan Trọng Nhất: Remote state backend (S3 + DynamoDB, Azure Blob, GCS). State locking (DynamoDB cho AWS) prevent concurrent modifications. State encryption at rest. KHÔNG commit state file vào git (chứa secrets). terraform state mv, terraform import cho state manipulation. Backup state trước mọi operation.
2. Modules — Reusable Infrastructure: Modules cho reusable components (VPC module, EKS module). Public Terraform Registry cho community modules. Pin module versions: source = "terraform-aws-modules/vpc/aws" version = "5.0.0". Outputs cho inter-module communication. Module composition cho complex architectures.
3. Workspaces vs Directory Structure: Workspaces: same code, different state (dev, staging, prod). Directory structure: different directories per environment. Common pattern: directories + shared modules. Terragrunt cho DRY multi-environment management. Choose ONE approach, be consistent.
4. CI/CD Integration: terraform plan trong PR review — reviewable change preview. terraform apply sau PR merge. Automated testing: terratest, tftest. Policy enforcement: Sentinel (HashiCorp), OPA. Drift detection: scheduled terraform plan. NEVER terraform apply từ local machine cho production.
5. Best Practices: terraform fmt cho consistent formatting. terraform validate cho syntax checking. tflint cho linting. checkov/tfsec cho security scanning. Resource naming conventions. Tagging strategy enforced. Lifecycle prevent_destroy cho critical resources.
6. OpenTofu — Open-Source Fork: Fork của Terraform sau HashiCorp license change (BSL). Compatible syntax. Community-driven. Drop-in replacement cho hầu hết cases. Consider cho new projects nếu license concerns.

📖 Chuyện Kể Đêm Khuya

💀 Chuyện terraform destroy Production

Junior engineer chạy terraform destroy. Terraform prompt: 'Do you really want to destroy all resources?' Engineer type 'yes' mà không đọc resource list. 47 resources destroyed: VPC, subnets, security groups, RDS database, ECS cluster. Production DOWN.

Recovery: database from backup (RPO 4 giờ = mất 4 giờ data). Rebuild infrastructure từ Terraform code (mất 2 giờ). Total impact: 6 giờ downtime, 4 giờ data loss. Total cost: estimated $200K.

Prevention implemented: lifecycle prevent_destroy cho critical resources. Separate state files cho production. RBAC restrict terraform destroy access. CI/CD pipeline only — no local terraform cho production.

💡 Bài học: lifecycle prevent_destroy cho databases, storage, networking. Separate state per environment. RBAC cho production access. CI/CD only — no local terraform apply/destroy. terraform plan REVIEW mandatory trước apply. Destruction protection ở cloud level (RDS deletion protection, S3 bucket policy).

🗃️ Chuyện State File Corruption

Terraform apply interrupted giữa chừng (network timeout). State file partially written. Next terraform plan: panic, crash, inconsistent state. Team cannot modify infrastructure — every operation fails.

Recovery: download state backup từ S3 versioning (bật versioning là MUST). Restore previous version. terraform refresh để sync state với actual infrastructure. Verify mọi resource. 4 giờ recovery.

💡 Bài học: State file backup là CRITICAL. S3 versioning enabled cho state bucket. DynamoDB state locking prevent concurrent writes. Remote backend thay vì local state. State file chứa secrets — encrypt at rest. Never manually edit state file — use terraform state commands.

🔄 Chuyện Drift Detection Phát Hiện Manual Change

Engineer manually modify security group qua console (thêm rule cho debugging). 3 tháng sau, Terraform plan show: 'security group rule will be removed.' Engineer đã quên manual change. Terraform apply xóa rule — service mất network access.

Fix: scheduled drift detection (terraform plan weekly). Alert on drift. Policy: NO manual changes — all changes through Terraform. Nếu phải manual change → terraform import ngay.

💡 Bài học: Infrastructure drift là silent killer. Scheduled drift detection. Policy: all changes through IaC. terraform import cho resources created manually. CloudTrail/Activity Log detect manual changes. Drift = technical debt = ticking time bomb.

⚠️ Những Sai Lầm Kinh Điển

1. Local state file: Mất laptop = mất state = mất infrastructure tracking. Remote backend ALWAYS. S3 + DynamoDB cho AWS.
2. Không lock state: 2 engineers terraform apply cùng lúc → state corruption. DynamoDB locking cho AWS, built-in cho other backends.
3. Mega state file: 1 state file cho toàn bộ infrastructure. Split state: networking, compute, database separately. Smaller blast radius.
4. Hardcode values: variables.tf, terraform.tfvars cho environment-specific values. No hardcoded AMI IDs, IP addresses, account IDs.
5. Không review terraform plan: terraform apply --auto-approve = YOLO. ALWAYS review plan output. Especially: resources being destroyed.
6. terraform apply trên main branch: Merge to main → auto-apply. Feature branch → terraform plan cho review. Main merge → terraform apply. Sequential, not concurrent.
