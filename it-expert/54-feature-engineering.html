📜 Bí Kíp Luyện Kim

1

Feature Selection — Bớt đi để thêm vào

Quá nhiều features = noise + overfitting + slow training. Ba cách tiếp cận: (1) Filter: correlation, mutual information, chi-square — nhanh nhưng thô. (2) Wrapper: recursive feature elimination (RFE) — chính xác nhưng chậm. (3) Embedded: L1 regularization, tree feature importance — cân bằng nhất. Trick: dùng permutation_importance thay vì feature_importances_ — nó không bị bias bởi cardinality.

2

Encoding Categorical Features

One-Hot: tốt khi cardinality < 10. Target Encoding: tốt cho high cardinality nhưng cẩn thận data leakage — luôn dùng fold-based encoding (encoding trên k-1 folds, apply lên fold còn lại). Ordinal: cho features có thứ tự tự nhiên (low/medium/high). Frequency Encoding: thay category bằng tần suất — đơn giản nhưng mất thông tin nếu hai categories có cùng frequency.

3

Feature Crossing — Tạo combo features

Đôi khi magic nằm ở sự kết hợp: price_per_sqm = price / area, age_income_ratio = age / income. Tree-based models tự tìm interactions, nhưng giúp chúng bằng explicit crossing sẽ nhanh hơn. Dùng PolynomialFeatures cho degree 2, nhưng cẩn thận explosion: 100 features → 5,050 features. Chỉ cross features có domain logic.

4

Temporal Features — Thời gian là vàng

Từ timestamp, tạo: hour_of_day, day_of_week, is_weekend, month, quarter. Cyclical encoding cho periodic features: sin(2π * hour/24), cos(2π * hour/24). Lag features: value_1d_ago, value_7d_ago. Rolling stats: rolling_mean_7d, rolling_std_30d. Diff features: value - value_1d_ago.

5

Feature Store — Hệ thống hóa features

Feast: open-source, tốt cho batch + online serving. Define features as code, version control, share across teams. Tecton: managed, tốt cho real-time features (streaming). Feature store giải quyết: (1) training-serving skew, (2) feature reuse across projects, (3) point-in-time correctness. Năm 2026, bất kỳ team ML nào > 5 người đều nên có feature store.

6

Handling Missing Values — Nghệ thuật điền chỗ trống

Đừng chỉ dùng mean/median imputation. Tốt hơn: KNN Imputation, IterativeImputer (MICE), hoặc tạo feature is_missing rồi impute đơn giản. Với tree-based models, có thể để missing (XGBoost/LightGBM handle native). Quan trọng: hiểu TẠI SAO data missing — MCAR, MAR, hay MNAR? Mỗi loại cần strategy khác nhau.

7

Text Features — Khai thác text data

Ngoài TF-IDF, hãy tạo: text_length, word_count, special_char_count, uppercase_ratio. Với tên sản phẩm: extract brand, category từ text. Năm 2026: dùng sentence embeddings (từ pretrained models) làm features — sentence-transformers cho vector 384-768 chiều, rồi PCA xuống 50-100 chiều để tránh curse of dimensionality.

📖 Chuyện Kể Đêm Khuya

🏆 Chuyện 1: Một feature tăng gấp đôi accuracy

Team anh Đặc Trưng đang build model dự đoán churn cho app thương mại điện tử. 50 features, XGBoost, F1 = 0.45. Cả team loay hoay 2 tuần tune model, thử đủ loại — không khá hơn.

Anh Đặc Trưng ngồi nhìn data 10 phút, rồi tạo MỘT feature: days_since_last_purchase. F1 nhảy lên 0.82. Cả team há hốc. Anh bình thản: "Khách hàng không mua gì 90 ngày thì... churn rồi còn gì."

💡 Bài học: Domain knowledge + creative feature engineering thường powerful hơn tuning model. Trước khi tune hyperparameters, hãy dành thời gian hiểu business logic và tạo features có ý nghĩa. Một feature đúng đáng giá bằng 100 features ngẫu nhiên.
💀 Chuyện 2: Data leakage qua feature — kẻ giết người thầm lặng

Anh xây model dự đoán khách hàng sẽ mua premium plan. Có feature total_premium_features_used. Model đạt AUC 0.99. Tuyệt vời? KHÔNG.

Feature đó chỉ có giá trị > 0 khi khách hàng ĐÃ MUA premium. Model đang "nhìn tương lai." Remove feature, AUC xuống 0.68 — con số thật. Anh mất 1 tuần debug vì feature name trông vô hại.

💡 Bài học: Data leakage qua features là bug nguy hiểm nhất trong ML. Luôn hỏi: "Feature này có available tại thời điểm prediction không?" Nếu model quá tốt (AUC > 0.95 trên bài toán thực), hãy nghi ngờ data leakage trước khi ăn mừng.
🏪 Chuyện 3: Feature Store cứu team khỏi chaos

Trước khi có feature store, 5 data scientists trong team tạo features riêng, mỗi người một cách. Cùng feature "user_age" nhưng người tính theo năm, người tính theo ngày, người tính từ registration date, người tính từ birth date. Training và production dùng khác nhau → model production sai lung tung.

Anh Đặc Trưng setup Feast, define mọi features as code, version control trên Git. 3 tháng sau: zero training-serving skew, features reuse 70% across projects, onboarding member mới từ 2 tuần xuống 2 ngày.

💡 Bài học: Feature store không phải luxury — đó là necessity khi team > 3 người. Nó giải quyết: consistency, reusability, point-in-time correctness, và training-serving skew. Đầu tư vào feature store sớm, tiết kiệm vô số giờ debug sau này.

⚠️ Những Sai Lầm Kinh Điển

❌ Target encoding không fold-based: Encode trên toàn bộ data → data leakage. Luôn dùng k-fold encoding hoặc leave-one-out encoding.
❌ One-hot encoding cho high cardinality: City có 10,000 giá trị → 10,000 columns. Dùng target encoding, frequency encoding, hoặc embedding thay thế.
❌ Fit scaler trên toàn bộ data: StandardScaler fit trên train+test → test data leak vào scaling. Luôn fit trên training set only, transform trên test.
❌ Tạo feature không available at prediction time: Feature dùng data từ tương lai → model production sẽ sai. Kiểm tra temporal validity của mọi feature.
❌ Quá nhiều features không chọn lọc: 1,000 features mà chỉ 50 useful → noise drowns signal. Feature selection là bắt buộc, không phải tùy chọn.
❌ Bỏ qua missing value patterns: Missing data có thể là feature! Tạo is_feature_X_missing column — pattern of missingness thường informative.
