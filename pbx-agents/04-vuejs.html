Thân Thế & Lai Lịch

Câu chuyện của kẻ luyện võ Vue từ thuở sơ khai

Ngày xưa ngày xưa, ở một vùng đất xa xôi gọi là JavaScript Kingdom, có một chàng thiếu niên tên là Vue Đại Hiệp. Chàng lớn lên trong thời kỳ jQuery thống trị thiên hạ — khi mà việc thao tác DOM bằng tay là chuyện thường ngày ở huyện.

Một ngày nọ, chàng tình cờ gặp được Evan You — vị đại sứ sáng tạo ra Vue.js. Evan truyền cho chàng bí kíp Reactivity System, dạy chàng cách dùng ref() và reactive() để biến đổi dữ liệu mà UI tự động cập nhật như có phép thuật.

Từ đó, Vue Đại Hiệp lang thang khắp giang hồ, từ Options API chuyển sang Composition API, từ Vuex chuyển sang Pinia, luyện thành computed(), watch(), watchEffect(), và nắm vững tất cả lifecycle hooks từ onMounted đến onUnmounted. Chàng còn kết bạn với Vue Router để đi khắp các trang, và học cách phối hợp với TypeScript để code thêm vững chắc.

Master setup(), ref(), reactive(), composables. Gom logic như gom đồ ăn vào hộp — gọn gàng, sạch sẽ, tái sử dụng được.

Hiểu rõ Proxy, dependency tracking, effect scheduling. Biết được khi nào Vue "theo dõi" bạn và khi nào nó "bỏ qua" bạn.

computed() cho giá trị phái sinh, watch() cho side effects. Biết khi nào dùng cái nào như biết khi nào ăn cơm, khi nào ăn phở.

Bí Kíp Vue.js

Những điều ta học được sau bao nhiêu lần "deploy xong mới phát hiện bug"
1
ref() vs reactive() — Chọn sai là khổ cả đời
ref() cho primitive values (string, number, boolean) và khi cần .value để truy cập. reactive() cho object/array nhưng KHÔNG BAO GIỜ destructure trực tiếp, vì sẽ mất reactivity. Mẹo: cứ dùng ref() cho mọi thứ, an toàn hơn nhiều. reactive() chỉ dùng khi bạn thực sự hiểu mình đang làm gì.
// SAI - mất reactivity khi destructure
const state = reactive({ count: 0 })
const { count } = state // count không reactive nữa!

// ĐÚNG - dùng ref() hoặc toRefs()
const count = ref(0) // luôn reactive
const { count } = toRefs(state) // vẫn reactive
2
Composables — Vũ khí bí mật của Composition API
Đừng tạo component khổng lồ với 500 dòng code trong setup(). Hãy tách logic ra thành các composable function (useXxx). Đây là lý do Composition API tồn tại — để bạn có thể tái sử dụng logic giữa các component mà không cần mixins (cái thứ độc hại đó).
// composables/useCallStatus.ts
export function useCallStatus() {
  const isOnCall = ref(false)
  const duration = ref(0)
  const startCall = () => { isOnCall.value = true }
  return { isOnCall, duration, startCall }
}
3
watchEffect() vs watch() — Hai anh em khác tính
watchEffect() tự động theo dõi mọi dependency bên trong và chạy ngay lập tức. watch() chỉ theo dõi những gì bạn chỉ định và cho bạn giá trị cũ + mới. Mẹo: watchEffect() cho "tôi muốn làm gì đó khi bất kỳ dependency nào thay đổi". watch() cho "tôi muốn biết cụ thể cái gì thay đổi và giá trị trước là bao nhiêu".
4
Template Refs + nextTick() — Đợi DOM sẵn sàng
Khi bạn thay đổi reactive data, DOM không cập nhật ngay lập tức. Vue "gom" các thay đổi lại và cập nhật một lần. Nếu bạn cần truy cập DOM ngay sau khi thay đổi data, hãy dùng nextTick(). Đây là sai lầm kinh điển của người mới: thay đổi data rồi đọc DOM ngay — và tự hỏi tại sao nó vẫn là giá trị cũ.
const message = ref('Xin chào')
message.value = 'Tạm biệt'

// SAI: DOM chưa cập nhật
console.log(el.textContent) // vẫn là "Xin chào"

// ĐÚNG: đợi nextTick
await nextTick()
console.log(el.textContent) // "Tạm biệt"
5
provide/inject — Chống Prop Drilling
Khi bạn truyền props qua 5-6 tầng component, code sẽ trở nên như cái cây Giáng sinh — đẹp nhưng rối. provide/inject cho phép component ông truyền thẳng data cho component cháu mà không cần qua trung gian. Nhưng nhớ: đừng lạm dụng, vì debug sẽ khó hơn khi bạn không biết data đến từ đâu.
6
defineModel() — Vượt cửa Vue 3.4+
Trước Vue 3.4, two-way binding cho component tự làm là cực hình: defineProps + defineEmits + computed get/set. Giờ chỉ cần defineModel() là xong. Một dòng thay ba mươi dòng. Đây là minh chứng rằng Vue luôn tiến hóa để dev bớt khổ.
// Trước Vue 3.4 - dài dòng dòng
const props = defineProps(['modelValue'])
const emit = defineEmits(['update:modelValue'])
const value = computed({
  get: () => props.modelValue,
  set: (v) => emit('update:modelValue', v)
})

// Vue 3.4+ - gọn như thơ
const value = defineModel<string>()

Cái Bẫy Reactivity — Hay Cách Ông Mất 3 Ngày Debug

Ngày xưa, ông có một cái object lớn như cái nhà, ông bỏ nó vào reactive() đẹp đẽ lắm. Rồi một ngày, ông cần truyền một phần của nó cho component con. Ông vui vẻ destructure: const { name, age } = userState.

Ông thay đổi name, nhưng UI cứ như tường — không nhúc nhích. Ông tưởng Vue bị lỗi. Ông đọc lại docs. Ông hỏi Stack Overflow. Ông thức đến 3 giờ sáng.

Hóa ra, khi destructure reactive object, các giá trị primitive bị "tách rời" khỏi Proxy và mất hết reactivity. Ông phải dùng toRefs() hoặc storeToRefs() với Pinia store. Từ đó, ông không bao giờ destructure reactive object nữa. Ông bảo con cháu: "Đừng bao giờ xắt nhỏ reactivity ra!"

💡 Bài học: Destructure reactive object = mất reactivity. Dùng toRefs() hoặc cứ truy cập qua object gốc.
⚔

Cuộc Chiến v-if vs v-show — Ai Mới Là Võ Lâm Minh Chủ?

Ở một dự án ngày xưa, team ông có hai phe: phe v-if và phe v-show. Phe v-if nói: "Tôi tiết kiệm DOM node, tôi là vương!" Phe v-show đáp: "Tôi toggle nhanh hơn, tôi mới là vương!"

Họ tranh luận 2 tiếng trong cuộc họp. PM ngồi đó không hiểu gì, chỉ biết là deadline sắp đến.

Cuối cùng ông phải đứng dậy phân xử: v-if dùng khi condition ít thay đổi (vì nó destroy/recreate DOM). v-show dùng khi toggle thường xuyên (vì nó chỉ thay đổi CSS display). Trong PBX project, cái call panel toggle liên tục thì dùng v-show. Cái admin settings hiếm khi mở thì dùng v-if.

Từ đó hai phe hòa bình. Nhưng ông vẫn thấy họ liếc nhau về v-if với v-for trên cùng element... đó lại là chuyện khác.

💡 Bài học: v-if cho render có điều kiện (lazy), v-show cho toggle thường xuyên. Và ĐỪNG BAO GIỜ dùng v-if với v-for trên cùng element!
🎃

Ác Mộng Prop Drilling — Khi Props Đi Qua 7 Tầng Địa Ngục

Ông còn nhớ cái dự án đầu tiên dùng Vue. Ông có một cái theme color cần truyền từ App.vue xuống đến một cái button nhỏ xíu nằm sâu tận 7 tầng component.

Ông truyền props: App → Layout → Sidebar → Menu → MenuItem → SubMenu → Button. Mỗi component phải khai báo defineProps rồi truyền tiếp xuống. Code dài như sông Hàn.

Một ngày, PM nói: "Đổi tên prop từ 'themeColor' thành 'primaryColor' nhé." Ông phải sửa 7 file. Ông muốn khóc.

Rồi ông học được provide/inject. Ông provide ở App.vue, inject ở Button — xong. Sau này có Pinia, mọi thứ còn dễ hơn nữa. Ông đặt theme vào store, component nào cần thì lấy. Đời ông không bao giờ drilling nữa.

💡 Bài học: Quá 2-3 tầng props thì nên dùng provide/inject hoặc Pinia store. Prop drilling là đường dẫn ngắn nhất đến burnout.
👻

Ma onUnmounted — Khi Memory Leak Âm Thầm Ăn Mừng

Có một lần, ông làm tính năng realtime cho PBX — lắng nghe SIP events qua WebSocket. Ông đăng ký event listener trong onMounted(), rồi... quên mất onUnmounted().

Mỗi lần user navigate đi rồi quay lại, một listener mới được tạo ra mà cái cũ không bị hủy. Sau 30 phút, tab browser ăn 2GB RAM. User gọi điện bảo: "Sao máy tính tui nóng như lò nướng vậy?"

Từ đó ông có quy tắc bất thành văn: "Có onMounted thì phải có onUnmounted. addEventListener thì removeEventListener. Subscribe thì unsubscribe. Như âm với dương, có vay có trả."

💡 Bài học: LUÔN dọn dẹp trong onUnmounted(). setInterval, addEventListener, WebSocket subscription — tất cả phải được hủy khi component bị destroy.

Những Lỗi Kinh Điển

Những "hố" mà ai cũng phải rơi xuống ít nhất một lần
💥
Quên .value với ref()
Trong script setup, ref() cần .value để đọc/ghi. Trong template thì không cần. Số lượng lần dev viết count++ thay vì count.value++ rồi tự hỏi "tại sao nó không tăng?" nhiều không đếm xuể.
✅ Mẹo: bật ESLint + Vue plugin, nó sẽ nhắc bạn ngay.
😱
Thay đổi prop trực tiếp
Props là ONE-WAY data flow. Thay đổi prop trực tiếp sẽ bị Vue cảnh báo và gây ra bug khó debug. Đừng làm cha nó làm, đừng sửa hàng xóm.
✅ Dùng emit event lên parent hoặc dùng defineModel() cho two-way binding.
🔥
v-for không có :key
Không dùng :key hoặc dùng index làm key. Vue sẽ không biết element nào thay đổi, dẫn đến render sai và bug "ma" xuất hiện khi sort/filter list.
✅ Luôn dùng unique ID làm key. KHÔNG BAO GIỜ dùng index nếu list có thể thay đổi.
👀
Gọi API trong computed()
computed() là synchronous và được cache. Gọi async API trong computed là hành động phạm pháp. Nó sẽ chạy nhiều lần không kiểm soát và không bao giờ return đúng.
✅ Gọi API trong onMounted() hoặc watch(), lưu kết quả vào ref().
🚨
Deep watch mà không cần thiết
watch(obj, cb, { deep: true }) trên object lớn sẽ làm Vue phải duyệt toàn bộ object mỗi khi có bất kỳ thay đổi nào. Performance chết như con hổ bị cảm.
✅ Chỉ watch property cụ thể: watch(() => obj.value.name, cb). Chính xác và nhanh hơn.
🤯
Component khổng lồ 1000+ dòng
Nhồi hết mọi thứ vào một component duy nhất. Template 300 dòng, script 500 dòng, style 200 dòng. Đời mà maintain code này bắt đầu từ khi bạn không nhớ mình viết gì.
✅ Tách thành composables + child components. Mỗi component nên dưới 200 dòng.

Vue.js Trong Dự Án PBX

Cách Vue 3.5 trở thành xương sống của giao diện PBX

Toàn bộ giao diện người dùng của hệ thống PBX được xây dựng trên Vue 3.5 với Composition API và TypeScript. Đây là một hệ thống quản lý tổng đài điện thoại doanh nghiệp, nơi mà mỗi giây đều quan trọng — từ việc hiển thị trạng thái cuộc gọi realtime, quản lý hàng đợi, đến giao diện agent xử lý cuộc gọi.

📞
Giao Diện Cuộc Gọi Realtime
Vue reactivity system là nền tảng để hiển thị trạng thái cuộc gọi (ringing, connected, on-hold) theo thời gian thực. ref() và computed() giúp UI cập nhật tức thì khi SIP events thay đổi.
🌳
Pinia Quản Lý State
Trạng thái agent (available, busy, away), danh sách cuộc gọi, cấu hình hệ thống — tất cả được quản lý tập trung trong Pinia stores. storeToRefs() đảm bảo reactivity khi destructure store data.
🛣
Vue Router & Guards
Điều hướng giữa các trang: Dashboard, Call History, Agent Management, Settings. Navigation guards kiểm tra quyền truy cập và trạng thái đăng nhập trước khi cho phép vào trang.
📈
DevExtreme Components
Bảng dữ liệu, biểu đồ thống kê, form nhập liệu — DevExtreme cung cấp UI components sẵn có. Vue 3 wrapper giúp tích hợp mượt mà với Composition API và reactive data binding.
🔌
Composables Cho Logic Nghiệp Vụ
useCallManager(), useAgentStatus(), useWebSocket() — logic nghiệp vụ được đóng gói trong composables, tái sử dụng across components. Clean code, dễ test, dễ maintain.
🔷
TypeScript An Toàn
Mọi component, mọi composable, mọi store đều có type definitions. defineProps<T>() và defineEmits<T>() đảm bảo rằng data flow giữa các component luôn đúng kiểu, bắt lỗi ngay lúc compile thay vì lúc production.
