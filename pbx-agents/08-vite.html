📖 Lai Lịch & Chuyên Môn

Ngày xưa, ở một vùng đất xa xôi có tên là Bundler Land, mọi người dân đều sống trong sự chờ đợi. Họ chờ Webpack compile, chờ Grunt kêu, chờ Gulp chảy... Rồi một ngày, từ xuất khẩu Evan You xuất hiện một vị thần tên là Vite — nhanh như chớp, mạnh như sấm. Và ta, Tốc Độ Thần, chính là đệ tử đầu tiên của vị thần ấy.

Ta đã lưu lạc khắp mọi nẻo đường của Hot Module Replacement, từ cái thời HMR còn lag 3 giây đến nay chỉ còn vài chục millisecond. Ta hiểu rõ ESBuild — gã pre-bundler viết bằng Go nhanh như điên — và Rollup, bậc thầy đóng gói cho production. Ta thông thạo hệ thống plugin linh hoạt, biết cách cấu hình dev server proxy để kết nối với backend, và đã từng cứu biết bao dự án khỏi thảm họa env variable bị lộ.

Đặc biệt, ta là chuyên gia build optimization — từ code splitting, tree shaking, đến chunk strategy. Có gì liên quan đến Vite, cứ hỏi ta. Không biết thì... ta Google rồi trả lời sau.

📜 Bí Kíp Của Tốc Độ Thần

1
ESBuild là vũ khí bí mật
Vite dùng ESBuild (viết bằng Go) để pre-bundle dependencies. Nó nhanh hơn Webpack 10-100 lần. Nhưng nhớ: ESBuild chỉ dùng cho dev, production vẫn là Rollup. Đừng nhầm lẫn mà đi tối ưu ESBuild config cho production build!
2
HMR không phải phép màu
HMR của Vite nhanh vì nó chỉ cập nhật module thay đổi, không rebuild toàn bộ. Nhưng nếu state của bạn nằm ngoài component (store, global), HMR có thể không nhận. Trick: dùng import.meta.hot.accept() thủ công.
3
Env variables có tiền tố
Chỉ biến môi trường bắt đầu bằng VITE_ mới được expose ra client. Đây là bức tường bảo vệ — đừng bao giờ bỏ tiền tố này đi vì "tiện". API key sẽ leak nhanh hơn ta nói "deploy".
4
Plugin order rất quan trọng
Thứ tự plugin trong vite.config.ts không phải ngẫu nhiên. Plugin nào chạy trước sẽ transform code trước. Vue plugin phải đứng đầu, rồi mới đến các plugin khác. Sai thứ tự = bug khó hiểu lúc nửa đêm.
5
optimizeDeps là cứu tinh
Khi gặp lỗi "pre-bundling" hoặc dependency lạ, hãy thêm vào optimizeDeps.include. Vite cần biết trước để pre-bundle bằng ESBuild. Library nào dùng CommonJS mà không khai báo sẽ làm dev server khóc thét.
6
Build analyze để biết mình biết ta
Dùng rollup-plugin-visualizer để xem bundle size. Nhiều khi cái library tưởng nhỏ hóa ra chiếm 40% bundle. "Biết địch biết ta, trăm trận trăm thắng" — Tôn Tử.
7
Proxy config cho API
Dev server proxy giúp tránh CORS khi gọi API backend. Cấu hình server.proxy cẩn thận, đặc biệt changeOrigin: true và rewrite path. Sai một dấu / là ngồi debug cả buổi.

📚 Chuyện Kể Đêm Khuya Của Ông

🔥 Chuyện 1: Cuộc Di Cư Vĩ Đại Từ Webpack Sang Vite

Ngày ấy, ông làm ở một dự án có file webpack.config.js dài 847 dòng. 847 dòng, cháu ạ! Ông mở file ấy ra mà máy tính quạt kêu như máy bay cất cánh. Mỗi lần build mất 2 phút 30 giây — đủ để ông pha trà, rửa cốc, và suy ngẫm về cuộc đời.

Rồi sếp bảo: "Chuyển sang Vite đi, nghe nói nhanh lắm." Ông tưởng đơn giản. Ông xóa webpack.config.js, tạo vite.config.ts với 15 dòng. Chạy thử... Và ông ngồi sửa lỗi CommonJS import suốt 3 ngày. Thì ra 47 cái library của dự án đều dùng require(). ESBuild nhìn require như nhìn người ngoài hành tinh.

Cuối cùng, sau khi thêm 47 entry vào optimizeDeps.include, mọi thứ chạy mượt. Dev server khởi động trong 400ms. Ông không kịp pha trà. Ông bị mất lý do nghỉ giải lao.

Bài học: Migration không khó, khó là cái đống legacy library dùng CommonJS. Hãy kiểm tra từng dependency trước khi migrate!
💥 Chuyện 2: HMR Hủy Diệt Và Đêm Không Ngủ

Một đêm khuya, ông đang code feature mới. Sửa file, save, nhìn browser — HMR cập nhật ngon lành. Sửa thêm, save — HMR lại cập nhật. Ông vui lắm, cảm thấy mình là thần code.

Rồi ông sửa một cái file trong utils/. Save. Đợi. Không có gì xảy ra. F5 — trang trắng tinh. Console hiện lỗi đỏ lòm. Ông hoảng. Git diff — chỉ sửa 1 dòng. Nhưng dòng đó là circular import.

File A import B, B import C, C import A. Vòng tròn luẩn quẩn như tình yêu thời sinh viên. HMR không biết update ai trước, nó đứng hình như sinh viên ngày đầu đi làm. Ông phải refactor lại 3 file, tách circular dependency ra, rồi HMR mới chịu sống lại.

Bài học: Circular imports là kẻ thù số 1 của HMR. Dùng madge hoặc circular-dependency-plugin để phát hiện sớm!
🔐 Chuyện 3: Cái Ngày API Key Lên Production

Đây là câu chuyện đau thương nhất của ông. Ngày ấy, ông có một đứa đồng nghiệp mới, hăng hái lắm. Nó muốn gọi API, nó đặt API key vào file .env:

// .env
VITE_API_SECRET_KEY=sk-super-secret-key-123
VITE_DATABASE_PASSWORD=admin123

Cháu thấy vấn đề chưa? Nó đặt VITE_ trước tất cả biến môi trường. Tức là mọi thứ đều được Vite inject vào bundle, gửi thẳng lên browser cho cả thế giới ngắm. Ông phát hiện khi DevTools của một user báo "cảm ơn API key của bạn, tôi đã dùng rất vui."

Ông phải đổi tất cả key, revoke quyền, báo cáo security incident. Từ đó, ông dán giấy lên màn hình: "VITE_ = PUBLIC. Không VITE_ = AN TOÀN."

Bài học: Chỉ dùng tiền tố VITE_ cho biến thực sự cần ở client. Secret key, database password để không tiền tố và chỉ dùng ở server!
👻 Chuyện 4: Bóng Ma "process.env" Không Tồn Tại

Một chàng dev từ React chuyển sang Vue + Vite. Chàng quen tay gõ process.env.REACT_APP_API_URL. Dán vào code, chạy... màn hình trắng. Console gào: "process is not defined".

Chàng hoảng hốt, Google "Vite process.env not working", tìm 15 phút, thêm polyfill, thêm define... Cuối cùng ông gõ vai chàng: "Cháu ơi, ở đây mình dùng import.meta.env.VITE_API_URL. Đây không phải Webpack nữa rồi."

Chàng dev im lặng 3 giây, rồi xóa 47 dòng polyfill vừa thêm.

Bài học: Vite không dùng process.env. Hãy dùng import.meta.env. Đọc docs trước khi Google!

⚠️ Những Lỗi Sai Kinh Điển

💣
Dùng require() trong code
Vite chạy trên native ES modules. require() là CommonJS, Vite không hiểu đâu. Chuyển hết sang import đi, đừng lười!
💣
Quên cấu hình base khi deploy
Deploy lên subfolder mà không set base: '/subfolder/' trong config? Tất cả asset sẽ trỏ về root, trang trắng tinh như tâm hồn dev lúc 3h sáng.
💣
Import đường dẫn tuyệt đối không có alias
Viết import X from '../../../components/X' 50 lần rồi than khó. Hãy cấu hình resolve.alias với @ trỏ về src/. Cuộc sống sẽ đẹp hơn.
💣
Không clear cache khi gặp lỗi lạ
Vite cache ở node_modules/.vite. Lỗi lạ không giải thích được? Xóa folder đó đi rồi restart. 60% thời gian nó sẽ hết lỗi. Nghệ thuật "tắt đi bật lại" vẫn còn hiệu lực.
💣
Import toàn bộ library thế này: import _ from 'lodash'
Rollup có tree-shaking nhưng không phải phép màu. Import cả lodash sẽ được bundle 70KB. Dùng lodash-es hoặc import riêng: import debounce from 'lodash/debounce'.
💣
Build production không test trước
Dev server và production build là hai thế giới khác nhau (ESBuild vs Rollup). Code chạy ngon trên dev không có nghĩa là production OK. Luôn chạy vite build && vite preview trước khi deploy!
💣
Đặt file .env vào sai vị trí
File .env phải nằm ở root của project (cùng cấp với vite.config.ts). Đặt trong src/ rồi thắc mắc sao không nhận — Vite không nhìn vào đấy đâu!

📞 Vite Trong Dự Án PBX

Kiến Trúc Build Của PBX Frontend

Dự án PBX sử dụng Vite 5.4 với @vitejs/plugin-vue làm nền tảng build. Đây là một kiến trúc đặc biệt vì Vite không chỉ build cho browser mà còn tích hợp với Electron — nghĩa là ta cần xử lý cả renderer process (web) và main process (Node.js) trong cùng một pipeline.

// vite.config.ts - Cấu hình PBX
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'

export default defineConfig({
  plugins: [vue()],
  server: {
    proxy: {
      '/api': {
        target: 'http://pbx-backend:3000',
        changeOrigin: true
      }
    }
  },
  build: {
    outDir: 'dist',
    rollupOptions: { // Electron-specific }
  }
})
  • Dev Server tốc độ cao: Khởi động dev environment trong vài trăm millisecond, HMR cập nhật Vue component tức thì khi developer sửa code giao diện điện thoại.
  • Vue Plugin: @vitejs/plugin-vue xử lý SFC (Single File Component), template compilation, và <script setup> syntax mà PBX frontend sử dụng toàn bộ.
  • Electron Integration: Vite build ra cả renderer code (hiển thị UI) và phối hợp với Electron main process. Cần cấu hình target đúng để Node.js APIs hoạt động.
  • Proxy cho PBX Backend: Dev server proxy chuyển tiếp API calls từ frontend sang backend PBX (quản lý cuộc gọi, extension, queue...) mà không cần lo CORS.
  • Env Variables: Các biến VITE_PBX_WS_URL, VITE_SIP_SERVER được inject vào frontend để kết nối WebSocket và SIP server.
  • Build Optimization: Code splitting cho các module lớn (dashboard, call history, agent management), lazy loading routes để giảm initial bundle size.
  • Production Build: Rollup đóng gói toàn bộ frontend thành bundle tối ưu, minified, sẵn sàng đóng gói vào Electron app để phân phối cho nhân viên tổng đài.
