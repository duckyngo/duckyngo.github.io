🧓 Câu chuyện của Trưởng Lão

Già này bắt đầu hành trình từ cái thời mà SQL Server còn chạy trên máy chủ to bằng cái tủ lạnh, cháu ạ. Hồi đó, viết một cái stored procedure xong mà nó chạy đúng là vui như Tết — vì debug thì khổ hơn cày ruộng mùa đông.

Trải qua bao nhiêu năm chinh chiến, già đã thuần thục T-SQL như người ta thuần ngựa — từ những câu SELECT đơn giản đến những stored procedure dài hàng nghìn dòng, từ việc tối ưu index cho table vài trăm triệu dòng đến việc xử lý deadlock lúc 3 giờ sáng khi cả team đang ngủ say.

Già thành thạo query optimization — đọc execution plan như đọc truyện kiếm hiệp, nhìn vào là biết ngay chỗ nào bị "table scan" thay vì "index seek". Database migration cho già giống như dọn nhà — phải cẩn thận từng món đồ, đánh rơi cái gì là mất dữ liệu luôn. Transaction và connection pooling là hai bảo bối mà già luôn mang theo bên mình. Còn EF Core? Đó là cây cầu nối giữa thế giới .NET và vương quốc MSSQL mà già canh giữ.

Nếu database là một giang hồ, thì già là trưởng lão của phái SQL Server — đã tu luyện đủ các tuyệt chiêu: Window Functions (Lăng Ba Vi Bộ), CTEs (Càn Khôn Đại Na Di), Dynamic SQL (Hàm Ma Công), và MERGE statement (Hợp Nhất Thần Chưởng). Có cháu hỏi già: "Ông ơi, sao ông không dùng ORM cho khỏe?" Già cười khà khà rồi bảo: "Cháu ơi, ORM như xe đạp có bánh phụ — tiện đấy nhưng đi đua thì phải bỏ ra!"

Bí Kíp Của Trưởng Lão

Bí kíp #1

Index Không Phải Là Thuốc Tiên

Đừng thấy query chậm là cứ thêm index bừa bãi. Mỗi index thêm vào là mỗi lần INSERT/UPDATE phải gánh thêm chi phí. Hãy dùng SET STATISTICS IO ON để xem thật sự cần index ở đâu, cháu ạ.

Bí kíp #2

Execution Plan Là Bản Đồ Kho Báu

Mỗi khi query chạy chậm, đừng ngồi đoán mò. Bật execution plan lên là thấy ngay chỗ nào SQL Server đang "khổ sở" — Table Scan, Key Lookup, hay Sort trên cả triệu dòng. Đọc plan giỏi là đã thắng một nửa.

Bí kíp #3

Stored Procedure Cần Được Yêu Thương

Viết stored procedure mà không có comment, không có error handling, không có SET NOCOUNT ON thì giống như xây nhà mà không có móng. Cháu nhớ luôn dùng TRY...CATCH và log lỗi vào bảng riêng nhé.

Bí kíp #4

Connection Pooling — Đừng Mở Cửa Rồi Quên Đóng

Mỗi kết nối đến SQL Server giống như mở cửa vào nhà kho — phải nhớ đóng lại! Dùng connection pooling đúng cách, set Max Pool Size hợp lý, và LUÔN dispose connection sau khi dùng. Không thì server hết connection là sập hệ thống ngay.

Bí kíp #5

Transaction Ngắn Gọn Là Vàng

Transaction giữ lock — lock lâu thì blocking — blocking nhiều thì deadlock. Giữ transaction ngắn nhất có thể. Đừng bao giờ mở transaction rồi đi gọi API bên ngoài. Già đã thấy nhiều cháu làm vậy rồi khóc lúc 3 giờ sáng.

Bí kíp #6

NOLOCK Không Phải Thần Dược

Cháu nào hay gắn WITH (NOLOCK) vào mọi query cho "nhanh" thì nghe già nói: NOLOCK có thể đọc dirty data, phantom reads, thậm chí bỏ qua cả dòng dữ liệu. Chỉ dùng khi thật sự chấp nhận dữ liệu không chính xác 100%.

Bí kíp #7

EF Core — Bạn Tốt Nhưng Phải Biết Giới Hạn

EF Core sinh ra SQL rất "thật thà" — nhiều khi nó SELECT hết cả bảng trong khi cháu chỉ cần 3 cột. Luôn dùng .Select() projection, .AsNoTracking() cho query chỉ đọc, và kiểm tra SQL được sinh ra bằng logging.

Chuyện #1: Cái Index Bị Lãng Quên Và Sự Nổi Giận Của Khách Hàng

Ngày xưa, có một cháu developer trẻ tuổi, xây xong hệ thống PBX call log đẹp như mơ. Trên môi trường dev, chạy mượt như lụa — vì data mới có 100 dòng. Lên production, bảng CallLog có 50 triệu bản ghi. Một cái query tìm cuộc gọi theo số điện thoại — không có index — chạy mất 47 giây. Khách hàng gọi điện la làng: "Tôi bấm tìm cuộc gọi mà uống hết cốc cà phê vẫn chưa xong!"

Già chạy vào, gõ một dòng duy nhất:

CREATE NONCLUSTERED INDEX IX_CallLog_PhoneNumber
ON dbo.CallLog (PhoneNumber)
INCLUDE (CallDate, Duration, Status);

Query từ 47 giây xuống còn 12 mili-giây. Cháu developer mắt tròn mắt dẹt, hỏi: "Ông ơi, ông làm phép thuật gì vậy?" Già cười: "Không phải phép thuật, cháu ơi. Đây gọi là ĐỌC SÁCH TRƯỚC KHI VIẾT CODE."

💡 Bài học: Luôn tạo index cho các cột trong WHERE, JOIN và ORDER BY. Và test với data thật, đừng test với 100 dòng rồi tự tin lên production!
💀

Chuyện #2: DELETE Không WHERE — Thảm Họa Tồi Tệ Nhất Lịch Sử

Một đêm khuya, có anh DBA trực đêm, buồn ngủ díp mắt. Anh ta cần xóa một vài bản ghi test trong bảng UserMappings. Anh ta gõ:

DELETE FROM dbo.UserMappings
-- WHERE UserId = 999 ← Anh ta QUÊN bỏ comment dòng này

Enter. 478,293 dòng bị xóa sạch. Anh ta tỉnh ngủ ngay lập tức — tỉnh hơn uống 10 ly cà phê. Mồ hôi lạnh chảy dài, tay run bần bật. Anh ta nhìn màn hình, nhìn lại câu query, rồi nhìn lên trần nhà cầu nguyện.

May mắn thay, già đã thiết lập backup tự động mỗi 15 phút. Restore lại từ backup, chỉ mất data 12 phút. Nhưng anh DBA đó từ đấy mỗi lần viết DELETE đều run tay — y như già mỗi lần thấy câu DELETE không có WHERE là tim già đập loạn xạ.

💡 Bài học: LUÔN viết WHERE trước, chạy SELECT trước để kiểm tra, rồi mới đổi thành DELETE. Và backup, backup, BACKUP! Già nói 3 lần vì quan trọng gấp 3!
👻

Chuyện #3: Ma Deadlock Hiện Hồn Lúc 3 Giờ Sáng

Hệ thống PBX chạy ngon lành suốt ban ngày. Nhưng cứ đến 3 giờ sáng — đúng lúc batch job đồng bộ dữ liệu từ hệ thống cũ chạy — thì alert nổ liên tục: "Deadlock detected! Transaction was chosen as the deadlock victim."

Già mở system_health extended event lên xem deadlock graph, thấy hai stored procedure đang ôm nhau khóc — Procedure A lock bảng Users rồi cần bảng Extensions, trong khi Procedure B lock bảng Extensions rồi cần bảng Users. Hai đứa cứ chờ nhau mãi, không đứa nào chịu nhường.

Già sửa bằng cách bắt cả hai procedure access các bảng theo cùng thứ tự (Users trước, Extensions sau), thêm ROWLOCK hint và giảm scope của transaction. Từ đó, 3 giờ sáng yên bình trở lại. Già lại được ngủ ngon.

💡 Bài học: Deadlock xảy ra khi các transaction lock tài nguyên theo thứ tự khác nhau. Quy tắc vàng: luôn access bảng theo cùng một thứ tự trong mọi transaction!
🔥

Chuyện #4: Cursor — Kẻ Thù Giấu Mặt

Có cháu mới ra trường, viết stored procedure đồng bộ call log bằng CURSOR — duyệt từng dòng một, xử lý từng dòng một, như đếm gạo bằng tay. Dữ liệu ít thì không sao, nhưng khi có 2 triệu cuộc gọi mới cần sync, cái procedure chạy mất 4 tiếng 23 phút.

Già ngồi viết lại bằng set-based operation — một cái MERGE statement gọn gàng, sạch sẽ. Kết quả: 47 giây. Cháu developer hỏi: "Ông ơi, sao nhanh vậy?" Già bảo: "Cháu ơi, SQL Server sinh ra để xử lý SET, không phải để chạy vòng lặp. Dùng cursor giống như bắt con trâu đi xe đạp — nó làm được nhưng khổ lắm!"

💡 Bài học: Tránh CURSOR như tránh tà. Hãy nghĩ theo set-based approach. SQL Server được tối ưu để xử lý cả tập dữ liệu cùng lúc, không phải từng dòng một.
⚠️

Những Sai Lầm Kinh Điển

  • SELECT * mọi nơi mọi lúc: Cháu ơi, SELECT * giống như đi siêu thị mà vác cả cái kệ về nhà trong khi chỉ cần mua chai nước mắm. Hãy chỉ SELECT những cột cần thiết!
  • Không dùng parameterized query: Cứ nối chuỗi SQL thẳng vào query là mời SQL Injection vào nhà. Già đã thấy cháu viết "WHERE Name = '" + input + "'" rồi hacker xóa sạch database. Dùng @Parameter đi, cháu!
  • Index quá nhiều trên bảng INSERT nhiều: Bảng call log mỗi giây có hàng chục bản ghi mới — mà gắn 15 index lên đó thì INSERT chậm như rùa. Phải cân bằng giữa đọc nhanh và ghi nhanh, cháu ạ.
  • Không có chiến lược backup: "Ôi, máy chủ hỏng ổ cứng, dữ liệu mất hết!" — Già nghe câu này nhiều đến mức tai đã chai. Full backup hàng ngày, differential backup mỗi 4 giờ, transaction log backup mỗi 15 phút — đó là tối thiểu!
  • Dùng FLOAT cho tiền bạc: Cháu lưu tiền cước cuộc gọi bằng FLOAT, rồi cộng 0.1 + 0.2 ra 0.30000000000000004. Khách hàng nhìn hóa đơn tưởng bị hack. Dùng DECIMAL(18,2) đi, cháu!
  • Quên đóng connection: Mở SqlConnection mà không có using block, không dispose — connection pool cạn kiệt, hệ thống từ chối kết nối mới. Già đã thấy production sập chỉ vì lý do ngớ ngẩn này nhiều lần.
  • Migration không có rollback plan: Chạy ALTER TABLE trên production mà không biết rollback thế nào nếu lỗi. Già luôn viết script UP và DOWN cho mỗi migration. Không có đường lùi thì đừng tiến, cháu ơi!

-- Ví dụ: Query lấy lịch sử cuộc gọi với thông tin user mapping

SELECT
  cl.CallId,
  cl.CallerNumber,
  cl.CalleeNumber,
  cl.StartTime,
  cl.Duration,
  um.DisplayName AS CallerName,
  dept.DepartmentName
FROM dbo.CallLogs cl
INNER JOIN dbo.UserMappings um
  ON cl.CallerExtension = um.ExtensionNumber
LEFT JOIN dbo.Departments dept
  ON um.DepartmentId = dept.Id
WHERE cl.StartTime >= DATEADD(DAY, -7, GETDATE())
ORDER BY cl.StartTime DESC;
