👶 Chương 1: Cậu Bé Viết Code Đầu Tiên

Năm 2012, có một thằng bé mới ra trường, ôm cái laptop cũ, ngồi góc văn phòng gõ code PHP. Thằng bé ấy nghĩ rằng: "Chỉ cần code giỏi là đủ. Code đẹp thì đời đẹp."

Sai. Sai hoàn toàn. Sai đến mức 12 năm sau vẫn còn cười khi nhớ lại.

Ngày đầu tiên đi làm, senior review code, nói đúng một câu: "Code chạy được không có nghĩa là code tốt. Code tốt là code mà người khác đọc được, sửa được, và không chửi mình." Câu nói đó theo ta suốt 12 năm.

Từ PHP sang Java, từ Java sang C#, từ jQuery sang Angular rồi Vue.js — mỗi lần chuyển stack là một lần nhận ra: ngôn ngữ chỉ là công cụ, tư duy mới là vũ khí.

💪 Chương 2: Năm Thứ 3 — Cú Sốc Đầu Tiên

Sau 3 năm code, ta nghĩ mình giỏi lắm rồi. Biết design pattern, biết SOLID, viết unit test đàng hoàng. Được promote lên Senior Developer.

Rồi được giao lead một team 5 người cho dự án đầu tiên. Và đó là lúc ta nhận ra: code giỏi và lead team giỏi là hai kỹ năng hoàn toàn khác nhau.

Team có một bạn junior viết code rất chậm nhưng rất cẩn thận. Có một bạn senior viết nhanh như gió nhưng không bao giờ viết test. Có một bạn mid-level rất giỏi nhưng không bao giờ nói chuyện với ai. Ta phải học cách làm việc với con người — không phải máy tính.

Dự án đó trễ 2 tháng. Không phải vì code khó, mà vì ta không biết cách phân chia công việc, không biết cách communicate requirement, và quan trọng nhất: không biết cách nói "không" với stakeholder khi họ muốn thêm feature vào sprint cuối.

"Kỹ năng code đưa bạn vào phòng. Kỹ năng lead giữ bạn ở lại. Kỹ năng communicate quyết định bạn ngồi ở đâu trong phòng."

🔥 Chương 4: Real-Time Communication — Tình Yêu Và Nước Mắt

Lần đầu tiên làm hệ thống real-time là năm thứ 7. Một hệ thống chat cho enterprise. Nghe đơn giản? WebSocket, message queue, done? Không. Một ngàn lần không.

Real-time system là nơi mà mọi thứ bạn biết về web development truyền thống đều phải xem xét lại. Connection management, message ordering, reconnection strategy, presence detection, typing indicators, read receipts — mỗi cái nghe nhỏ nhưng phức tạp khủng khiếp.

Và rồi đến VoIP. SIP protocol. Asterisk. Codec negotiation. NAT traversal. Echo cancellation. Jitter buffer. Mỗi cái là một vũ trụ riêng.

Đó là lý do tại sao khi nhận dự án PBX cho Coway Korea, ta biết chính xác mình đang bước vào cái gì. Và ta cũng biết: đây sẽ là dự án khó nhất, nhưng cũng sẽ là dự án đáng tự hào nhất.

PHASE 01
⚡ Khởi Động Kỹ Thuật (Technical Kickoff)

Architecture Decision Records (ADR) — Tại sao chúng cứu mạng bạn

Mỗi quyết định kiến trúc quan trọng đều phải được ghi lại. Không phải trong đầu, không phải trong Slack, mà trong một file ADR chính thức. Tại sao? Vì 3 tháng sau, khi có bug production và cả team hỏi "Tại sao mình chọn cách này?" — bạn cần một tài liệu để trả lời thay vì dựa vào trí nhớ mơ hồ.

ADR gồm: Context (tại sao cần quyết định), Decision (chọn gì), Consequences (hệ quả tốt và xấu), Alternatives Considered (đã xem xét những gì khác).

"Không có ADR, mọi cuộc họp technical đều bắt đầu bằng câu: 'Ai nhớ tại sao mình làm thế này?' — và không ai nhớ."

Tech Stack Selection — Tại sao .NET 8 + Vue.js 3 + Electron cho PBX?

  • .NET 8: Performance tốt nhất trong các framework enterprise. SignalR native support. Khách Hàn Quốc quen thuộc với .NET ecosystem. Long-term support đến 2028.
  • Vue.js 3: Composition API + TypeScript = code sạch, dễ test. Ecosystem nhẹ hơn React cho desktop app. DevExtreme có Vue component tốt.
  • Electron: Coway yêu cầu desktop app chạy trên Windows. Electron cho phép share code với web. Integration với hệ thống điện thoại qua SIP.js.
  • SignalR: Real-time communication hub. Tự động fallback từ WebSocket sang SSE sang Long Polling. Scale horizontal với Redis backplane.
  • Asterisk + Docker: PBX engine proven, cộng đồng lớn. Docker cho phép deploy consistent giữa dev/staging/prod.

Proof of Concept — Validate cái gì trước?

Khi bắt đầu dự án PBX, có 100 thứ cần làm. Nhưng PL phải biết chọn 3-5 POC quan trọng nhất:

  • POC 1: SIP.js kết nối với Asterisk qua WebSocket — đây là core của toàn bộ dự án. Nếu cái này không work, toàn bộ kiến trúc phải thay đổi.
  • POC 2: SignalR với 500+ concurrent connections trên .NET 8 — enterprise PBX cần handle nhiều agent cùng lúc.
  • POC 3: Electron app size và memory consumption — Korean enterprise cấp laptop chuẩn 8GB RAM, app không được ngốn quá 500MB.
  • POC 4: IndexedDB sync với backend khi offline/online — agent có thể mất mạng khi đang gọi.
  • POC 5: DevExtreme DataGrid với 10,000+ rows và real-time update — bảng call log phải mượt.

Development Environment — Tiết kiệm 100+ giờ

Một PL giỏi dành 2-3 ngày đầu setup dev environment cho cả team. Docker Compose file chạy một phát lên hết. Hot reload cho cả backend lẫn frontend. Mock SIP server cho dev không cần Asterisk thật. Seed data script tự tạo data test. Nếu developer mới join mà mất hơn 30 phút để chạy được project, đó là lỗi của PL.

Coding Standards & Conventions

  • .editorconfig + eslint + prettier — format tự động, không cãi nhau tab vs space
  • Naming convention: PascalCase cho C#, camelCase cho TypeScript, kebab-case cho Vue component files
  • Commit message convention: feat(module): description — conventional commits
  • PR template bắt buộc: What/Why/How/Testing
  • Code review checklist tự động qua GitHub Actions

Git Branching Strategy — Không gây nội chiến

Với team offshore cho Hàn Quốc, ta dùng chiến lược đơn giản nhưng hiệu quả:

  • main — luôn deployable, chỉ merge qua PR
  • develop — integration branch, auto-deploy staging
  • feature/* — mỗi task một branch, lifetime tối đa 3 ngày
  • hotfix/* — cherry-pick vào cả main và develop
  • Quy tắc vàng: không bao giờ push trực tiếp vào main hoặc develop

Tại sao lifetime tối đa 3 ngày? Vì branch sống lâu hơn 3 ngày sẽ merge conflict với 90% xác suất. Và merge conflict lúc deadline là cơn ác mộng.

Database Design — Quyết định không thể undo

Có những quyết định database mà khi đã chọn, gần như không thể thay đổi trong dự án enterprise: schema naming convention, primary key strategy (GUID vs auto-increment — cho PBX ta chọn GUID vì multi-server sync), soft delete vs hard delete (enterprise Hàn bắt buộc soft delete cho audit trail), timezone handling (UTC everywhere, convert ở UI layer).

API Contract Design — Trước khi viết code

Frontend và backend team cần agree API contract trước. Swagger/OpenAPI spec viết trước, code sau. Tại sao? Vì frontend team có thể mock API và build UI song song với backend. Với dự án có deadline gấp như PBX (May 15!), parallel development là bắt buộc.

Korean Enterprise Requirements — Security từ ngày đầu

  • PIPA (개인정보보호법) compliance bắt đầu từ database design — encrypt PII fields
  • Audit trail cho mọi thao tác CRUD trên data nhạy cảm
  • Session timeout theo tiêu chuẩn KISA (한국인터넷진흥원)
  • Password policy theo hướng dẫn NIS (국가정보원)
  • Data retention policy — logs giữ tối thiểu 5 năm
PHASE 02
🔨 Tháng Đầu Tiên — Xây Nền Móng

Cuộc tranh luận "Scaffold vs Build"

Tháng đầu tiên luôn có cuộc tranh luận: "Nên dùng template/scaffold có sẵn hay build from scratch?" Câu trả lời của PL dày dạn: scaffold cho boilerplate, build cho business logic. Authentication, logging, error handling — dùng library. Call routing, agent management, real-time state — build custom vì đây là core business.

Code Review Culture — Feedback không phá hoại

Có hai loại code review: loại giúp người ta học, và loại khiến người ta muốn nghỉ việc. PL phải thiết lập culture từ đầu:

  • Comment vào code, không phải vào con người: "Có thể dùng early return ở đây" thay vì "Sao viết rối thế?"
  • Mỗi comment nên kèm suggestion hoặc giải thích WHY
  • Phân biệt nit (nhỏ, tùy), suggestion (nên làm), blocker (phải sửa)
  • PR không quá 400 dòng — PR 5000 dòng không ai review kỹ được
  • Auto-approve cho style/format changes
Developer viết 5000 dòng trong một PR

Tuần thứ 2, một developer gửi PR với 5000 dòng thay đổi. Bạn ấy rất tự hào vì "làm xong nguyên module trong 1 tuần". Nhưng khi review, ta nhận ra: module đó có 3 bugs nghiêm trọng, 2 security vulnerabilities, và hoàn toàn không có test.

Phải mất 3 ngày để review, 2 ngày để sửa, và 1 ngày để viết test. Tổng cộng: 6 ngày để "fix" một PR mà đáng lẽ nên được chia thành 8-10 PRs nhỏ, mỗi cái review trong 30 phút.

Bài học: PR lớn = review tệ = bug lọt = production crash. Luôn chia nhỏ PR.

Build vs Library — Câu chuyện DevExtreme

Khi chọn DevExtreme cho DataGrid, team có người phản đối: "Tự build table component đi, không phụ thuộc third-party." PL phải tính toán: tự build DataGrid với sort, filter, group, virtual scroll, export Excel cần ít nhất 3 developer x 2 tháng = 6 man-months. DevExtreme license: $2,000. ROI rõ ràng. Nhưng cũng phải document limitation của DevExtreme để team không lãng phí thời gian cố customize những thứ library không support.

Automated Testing — Bài test hôm nay, ơn cứu mạng ngày mai

  • Unit test cho business logic (call routing rules, agent status transitions, CDR calculations)
  • Integration test cho API endpoints — especially authentication flow
  • E2E test cho critical paths: login → nhận cuộc gọi → chuyển cuộc gọi → kết thúc → xem CDR
  • CI pipeline chạy test trước khi merge — không test pass, không merge. Không ngoại lệ.
  • Code coverage target: 70% cho business logic, 50% cho toàn bộ codebase

SignalR Architecture — Tại sao WebSocket quan trọng cho PBX

PBX system cần real-time ở mọi nơi: trạng thái agent (available/busy/away), cuộc gọi đến, chuyển cuộc gọi, conference, queue status, live dashboard. HTTP polling sẽ tạo hàng ngàn requests/giây. SignalR với WebSocket transport là lựa chọn duy nhất hợp lý.

Kiến trúc ta chọn: Hub-per-feature (CallHub, AgentHub, DashboardHub) thay vì One-Big-Hub. Tại sao? Vì mỗi hub có lifecycle khác nhau, scale khác nhau, và authorization khác nhau.

Docker Setup cho Asterisk — Bài học xương máu

  • Asterisk trong Docker cần --net=host cho SIP UDP — NAT trong Docker là ác mộng
  • Logs phải mount volume ra ngoài — container restart mất hết logs = mất hết evidence
  • Config reload không cần restart container — asterisk -rx "module reload"
  • Health check endpoint tự viết — asterisk -rx "core show channels" để biết system còn sống
  • Resource limit cho container — Asterisk không có giới hạn channel mặc định, nó sẽ ăn hết RAM nếu bạn để

Sprint Velocity Reality Check

Sprint 1: team estimate 80 story points, complete 35. Sprint 2: estimate 60, complete 45. Sprint 3: estimate 50, complete 48. Velocity thật sự thường chỉ bằng 50-60% estimate ban đầu. PL phải chấp nhận thực tế này và plan accordingly, không nên ép team "làm nhanh hơn" — nên tìm cách làm đúng hơn.

PHASE 03
💥 Giai Đoạn Giữa — Khi Mọi Thứ Bắt Đầu Phức Tạp

Cuộc tranh luận vĩ đại: Refactor — Bây giờ hay không bao giờ?

Đến giữa dự án, technical debt bắt đầu tích lũy. Team muốn refactor. PM muốn ship feature. PL phải đứng giữa và đưa ra quyết định khó nhất: refactor cái gì, bỏ qua cái gì, và schedule refactor khi nào.

Quy tắc của ta: chia technical debt thành 3 nhóm:

  • Urgent: ảnh hưởng trực tiếp đến feature sắp build — refactor NGAY
  • Important: sẽ gây vấn đề trong 1-2 tháng — schedule vào sprint tiếp
  • Nice-to-have: code xấu nhưng hoạt động — ghi vào backlog, để sau release

Integration Hell — Khi 5 hệ thống cần nói chuyện

PBX project có 5 hệ thống tích hợp: .NET Backend, Vue.js Frontend, Asterisk PBX, Electron Desktop, và External CRM API. Khi mỗi hệ thống hoạt động riêng lẻ thì đẹp. Khi kết hợp lại: kaboom.

Vấn đề thường gặp: timezone mismatch giữa .NET (UTC) và Asterisk (server local time). Message format không consistent giữa SignalR và SIP events. Error handling khác nhau giữa mỗi layer — Electron swallow errors mà backend không biết.

Giải pháp: Integration Test Environment riêng, chạy toàn bộ stack, test mỗi ngày. Contract testing giữa các service. Centralized logging với correlation ID cho mọi request từ UI đến PBX engine.

Bug mất 3 developer 2 tuần để tìm — và đó là một typo

Giữa dự án, khách Hàn report: "Cuộc gọi đôi khi bị ngắt sau đúng 30 giây." Team debug 2 tuần. Check SignalR connection. Check SIP timeout. Check Asterisk dialplan. Check network. Check Electron. Tất cả đều "bình thường".

Cuối cùng, một junior developer tìm thấy: trong Asterisk dialplan, có dòng Set(TIMEOUT(absolute)=30) thay vì Set(TIMEOUT(absolute)=3000). Ai đó viết 30 thay vì 3000. Đơn vị là giây, không phải millisecond. Một. Cái. Typo.

3 developer x 2 tuần = 6 man-weeks. Chi phí: khoảng $15,000. Nguyên nhân: thiếu configuration validation và thiếu unit test cho dialplan config.

Bài học: Mọi configuration value đều phải có validation. Mọi magic number đều phải có constant. Mọi constant đều phải có unit test.

Performance Optimization Under Pressure

Khi khách Hàn bắt đầu UAT, họ test với 200 concurrent agents. Dashboard chậm 8 giây. Call log query timeout. Agent status update delay 3 giây. Cả team phải optimize trong khi vẫn phải ship feature mới.

  • Thêm Redis cache cho agent status — giảm từ 200ms xuống 5ms
  • Implement database query pagination — không load 100,000 CDR records cùng lúc
  • SignalR group-based broadcasting thay vì broadcast tất cả — giảm 80% network traffic
  • Lazy loading cho Vue.js components — giảm initial bundle size 60%
  • Index optimization cho MSSQL — 5 missing indexes khiến dashboard query chậm 50x

Code Quality vs Delivery Speed — Cuộc chiến vĩnh cửu

PM nói: "Khách cần feature này thứ Sáu." PL biết: "Làm đúng cần đến thứ Ba tuần sau." Ai đúng? Cả hai. Nghệ thuật của PL là tìm middle ground: ship MVP thứ Sáu với test coverage 60%, schedule technical improvement cho sprint sau, document known limitations.

Manage 10 PRs/ngày — Giữ Sạch Sẽ

  • Review PR mỗi sáng 9h và chiều 2h — không để PR chờ quá 4 giờ
  • Auto-assign reviewer dựa trên CODEOWNERS file
  • PR bot check: lint, test, build, coverage — pass hết mới cho review
  • Priority: hotfix > feature blocking others > feature > refactor > documentation
  • Delegate review: senior members review junior's code, PL review architecture changes

Khoảnh Khắc Nhận Ra Kiến Trúc Không Scale

Giữa dự án, ta nhận ra: monolithic SignalR hub không scale ngang được khi cần 1000+ connections. Phải re-architect sang Redis backplane. Quyết định này ảnh hưởng 20+ files, cần 1 tuần refactor giữa sprint. PL phải negotiate với PM: "Nếu không sửa bây giờ, production sẽ crash khi go-live. Tốn 1 tuần bây giờ hay 3 tuần hotfix sau?" PM luôn chọn 1 tuần.

Real-Time System Debugging — Ác Mộng

Debug SignalR + SIP + WebSocket cùng lúc giống như debug 3 ứng dụng chạy song song, nói 3 ngôn ngữ khác nhau, qua 3 protocol khác nhau. Công cụ cần có:

  • sngrep — capture SIP traffic real-time
  • Browser DevTools WebSocket inspector — xem SignalR messages
  • Asterisk CLI core set verbose 5 — xem dialplan execution
  • Centralized logging (ELK stack hoặc Seq) với correlation ID
  • Custom dashboard hiển thị call state machine transitions

Knowledge Sharing Sessions Thực Sự Hiệu Quả

Load Testing — Server Chết Ở 100 Concurrent Calls

Story: Tuần trước go-live, chạy load test. 50 concurrent calls: OK. 80: OK nhưng dashboard chậm. 100: Asterisk bắt đầu drop calls. 120: server OOM, restart.

🔒 PIPA Compliance (개인정보보호법)

Luật Bảo vệ Thông tin Cá nhân Hàn Quốc (PIPA) là một trong những luật privacy nghiêm ngặt nhất thế giới — ngang GDPR, có phần còn strict hơn.

  • Thu thập dữ liệu: Phải có consent rõ ràng, cụ thể cho từng loại data. Không được dùng "pre-checked checkbox".
  • Lưu trữ: PII phải encrypt AES-256 at rest. Mã số chủ dân (주민등록번호) — tuyệt đối KHÔNG lưu trừ khi bắt buộc bởi luật khác.
  • Truyền tải: TLS 1.2+. Không gửi PII qua email không encrypt. API truyền PII phải log access.
  • Xóa: Khi user request xóa, phải xóa trong 5 ngày làm việc. Audit trail giữ lại nhưng anonymize.
  • Cross-border: Data transfer ra ngoài Hàn Quốc phải có user consent VÀ đảm bảo nước nhận có protection tương đương.
  • Breach notification: Report cho PIPC (개인정보보호위원회) trong 72 giờ. Thông báo affected users "ngay lập tức".
  • Vi phạm: Phạt lên đến 5% doanh thu hàng năm. Không đùa được.

Cho dự án PBX: call recording, caller ID, call history — tất cả đều là PII. Phải encrypt, access-control, audit, và retention policy đầy đủ.

📑 Korean Enterprise Security Audit Checklist

Khách enterprise Hàn Quốc thường thuê third-party security firm audit trước go-live. Checklist tiêu biểu:

  • Authentication: MFA option, session timeout 30 phút, account lockout sau 5 lần sai password, password complexity (chữ hoa + thường + số + ký tự đặc biệt, >= 8 ký tự, đổi mỗi 90 ngày)
  • Authorization: Role-based access control (RBAC). Principle of least privilege. Admin functions có separate authentication.
  • Data Protection: Encrypt at rest + in transit. Key rotation mỗi 365 ngày. Secure key storage (Azure Key Vault / AWS KMS).

  • Network: WAF (Web Application Firewall). DDoS protection. Internal network segmentation. VPN cho admin access.
  • Code: No hard-coded credentials. Dependency vulnerability scan. Static code analysis (SonarQube). Regular penetration testing.
  • Infrastructure: OS patches within 30 days of release. Antivirus on all servers. Intrusion detection system.
🇬🇳 Hangul/Unicode Handling

Hangul (한글) xử lý trong software có nhiều cạm bẫy hơn bạn nghĩ:

  • Sorting: Hangul sort theo thứ tự ㄱㄴㄷ... (consonant order). Default Unicode sort có thể sai.

  • Display: Hangul characters rộng hơn Latin. UI layout phải test với Korean text — text overflow xảy ra thường xuyên.
  • File names: Windows + Korean file names + network drives = encoding hell. Always use Unicode APIs.

🔥 Story 1: Kiến Trúc "Hoàn Hảo" Sập Ở 10,000 WebSocket Connections

Năm 2022, ta thiết kế kiến trúc cho hệ thống notification real-time. Trên giấy, mọi thứ hoàn hảo: SignalR hub, Redis backplane, load balancer. Design review pass. Code review pass. Unit test pass. Integration test với 100 connections: pass.

Deploy staging, test với 500 connections: pass. Deploy production. Tuần đầu: 2,000 users, ổn. Tuần hai: 5,000 users, bắt đầu chậm. Tuần ba: 8,000 users, dashboard lag 10 giây. Tuần tư: 10,000 users, server crash.

Root cause: mỗi SignalR connection giữ một in-memory subscription list. Với 10,000 connections x 15 subscriptions/connection = 150,000 objects trong memory. GC (Garbage Collection) chạy liên tục, CPU 100%. Thêm vào đó, Redis backplane serialize/deserialize mỗi message cho mọi connection — CPU và bandwidth nổ tung.

Fix: Re-architect sang group-based messaging. Thay vì mỗi user subscribe 15 topics riêng, group users theo department/role và broadcast theo group. Kết quả: 150,000 subscriptions giảm còn 200 groups. Memory usage giảm 95%. CPU idle ở 20% với 15,000 concurrent connections.

Bài học: Load test phải test ở scale thực tế, không phải 1/100 scale. Và kiến trúc "đúng" ở 100 connections có thể "sai" ở 10,000.
💣 Story 2: Chọn Sai Database — Migrate Giữa Dự Án

Dự án cho khách Hàn, team ban đầu chọn PostgreSQL vì "free, open-source, powerful". Tuyệt vời về mặt kỹ thuật. Nhưng... khách Hàn yêu cầu DBA team của họ quản lý database. DBA team chỉ biết MSSQL. Không negotiate được.

Bài học: Chọn tech stack PHẢI involve khách hàng. Đặc biệt database — đây là thứ operations team của khách sẽ manage hàng ngày. Tech "tốt nhất" không phải lúc nào cũng là "phù hợp nhất".
💨 Story 3: Missing Index Timeout Dashboard Trong Cuộc Họp Ban Giám Đốc

Khách Hàn có cuộc họp board meeting hàng tháng. Giám đốc mở dashboard real-time, show KPI cho board. Tháng đầu: OK. Tháng hai: dữ liệu nhiều hơn, query hơi chậm nhưng chấp nhận được (3 giây). Tháng ba: data tăng 3x, dashboard timeout sau 30 giây. Giữa phòng họp có 20 giám đốc đang nhìn màn hình trắng.

CEO Hàn Quốc gọi trực tiếp CTO SotaTek. Lúc 10 giờ sáng giờ Hàn = 8 giờ sáng giờ Việt Nam. Team PL được đánh thức bởi emergency call.

Root cause: table CallDetailRecords có 2 triệu rows. Dashboard query filter theo StartTime và AgentId. Cả hai column đều không có index. Full table scan × 5 panels = timeout.

Bài học: LUÔN review query execution plan cho mọi dashboard/report query. Thêm index cho mọi column được dùng trong WHERE/ORDER BY/GROUP BY. Và test với production-volume data, không phải dev data 100 rows.
🔐 Story 4: Security Vulnerability 2 Ngày Trước Audit

Dự án enterprise Hàn Quốc, 2 ngày trước security audit. Team chạy security scan (OWASP ZAP) theo routine. Kết quả: Critical: API endpoint /api/users/{id} trả về password hash trong response.

Tại sao? Vì developer dùng AutoMapper map Entity → DTO mà không exclude PasswordHash field. AutoMapper mặc định map tất cả properties cùng tên. Và không ai catch trong code review vì PR diff chỉ show AutoMapper config, không show actual response data.

Fix ngay lập tức: thêm .ForMember(d => d.PasswordHash, opt => opt.Ignore()). Nhưng phải audit toàn bộ 50+ DTOs khác. Tìm thêm 3 endpoints leak sensitive data (email, phone number không masked).

48 giờ marathon fix + test + re-scan. Security audit: PASSED. Nhưng tim PL suýt ngừng đập.

Bài học: Security scan phải chạy TỪ ĐẦU dự án, mỗi sprint, trong CI/CD pipeline. Không phải 2 ngày trước audit. AutoMapper PHẢI dùng explicit mapping, không dùng default convention-based mapping cho API responses.
📨 Story 5: 50,000 Duplicate Notifications Trong 5 Phút

Hệ thống notification real-time. SignalR broadcast message khi có cuộc gọi mới. Một ngày đẹp trời, server restart do Windows Update (đúng, trong production). SignalR reconnection logic trigger: mỗi client reconnect → server gửi "missed notifications" → nhưng bug: server gửi ALL notifications, không chỉ missed ones.

500 users reconnect cùng lúc × 100 notifications mỗi user = 50,000 notifications trong 5 phút. Mỗi notification trigger browser popup + sound. Máy tính của 500 nhân viên Coway đồng loạt phát âm thanh và hiện popup liên tục. IT helpdesk Coway nhận 200 tickets trong 30 phút.

Fix: Implement idempotent notification delivery. Mỗi notification có unique ID. Client track lastReceivedId. Server chỉ gửi notifications có ID > lastReceivedId. Thêm rate limiting: max 10 notifications/giây/user.

Bài học: Real-time systems PHẢI handle reconnection gracefully. Idempotency không phải optional. Rate limiting là bắt buộc. Và tắt Windows Auto-Update trên production servers.
💻 Story 6: Electron App Ăn 4GB RAM Trên Laptop Chuẩn Hàn

Enterprise Hàn Quốc cấp laptop chuẩn cho nhân viên: Intel i5 thế hệ 10, 8GB RAM, 256GB SSD. Electron app chạy trên dev machine (32GB RAM): mượt mà. Chạy trên laptop chuẩn: 4GB RAM sau 2 giờ sử dụng. Máy còn 4GB cho Windows + Outlook + Chrome. Kết quả: máy đơ, nhân viên restart, call drop.

Root cause analysis:

  • Memory leak trong Vue.js: event listener không cleanup khi component unmount → 500 orphan listeners sau 2 giờ
  • Electron renderer process cache DOM nodes cho call history → 10,000 DOM nodes sau 1 ngày
  • IPC messages giữa main/renderer process không garbage collected đúng cách
  • DevExtreme DataGrid cache data rows trong memory thay vì virtual scroll

Fix mất 3 tuần:

  • Implement proper cleanup trong Vue.js onUnmounted lifecycle hook
  • Virtual scrolling cho tất cả lists/tables
  • Periodic cache cleanup mỗi 30 phút
  • Electron webFrame.clearCache() scheduled
  • Memory usage monitoring + alert khi > 1GB

Kết quả: memory usage ổn định 500-800MB sau 8 giờ sử dụng.

Bài học: LUÔN test trên hardware chuẩn của khách. Electron = Chromium = memory hungry. Memory profiling phải là phần bắt buộc của testing, không phải "nice to have".

  • API docs auto-generated từ Swagger — luôn up-to-date vì generated từ code.
  • Runbook — "Khi X xảy ra, làm Y." Step-by-step, copy-pasteable commands.

Không viết: Word documents 50 trang mà không ai mở. Confluence pages nested 5 levels deep. "Architecture overview" mà overview quá tổng quát, chẳng giúp gì.

⚖️ Technical Decision-Making Framework

Khi đứng trước quyết định kỹ thuật quan trọng, ta dùng framework RAPID:

  • R — Requirements: Yêu cầu thực sự là gì? (Không phải yêu cầu bề mặt, mà yêu cầu gốc)
  • A — Alternatives: Liệt kê ít nhất 3 options. Nếu chỉ có 1 option, bạn chưa nghĩ đủ kỹ.
  • P — Pros/Cons: Trade-off matrix. Performance, cost, complexity, team familiarity, long-term maintenance.
  • I — Impact: Quyết định này reversible hay irreversible? Reversible → decide fast. Irreversible → decide carefully.
  • D — Document: Ghi lại trong ADR. Future-you sẽ cảm ơn present-you.
🔄 "Rewrite" vs "Refactor" — Khi Nào Nói Gì

Refactor khi: Code works nhưng messy. Thêm feature mới vào module hiện tại tốn 3x effort. Test coverage đủ để refactor an toàn. Team hiểu business logic trong code.

Rewrite khi: Technology đã end-of-life. Architecture fundamentally wrong (ví dụ: monolith cần chuyển sang microservices VÀ có justification rõ ràng). Không ai trong team hiểu code (bus factor = 0). Cost of maintaining > cost of rewriting.

KHÔNG BAO GIỜ rewrite khi: "Code xấu quá" (refactor thôi). "Tôi muốn dùng framework mới" (không phải lý do business). "Nhanh hơn build lại từ đầu" (90% cases, câu này SAI — vì bạn quên count edge cases đã được handle trong code cũ).

"Mọi cuộc rewrite lớn đều bắt đầu bằng câu 'Lần này mình sẽ làm đúng' — và kết thúc bằng câu 'Thì ra cái cũ nó handle case này...' x 500 lần."
👜 The Bus Factor & Knowledge Distribution

Bus factor: Nếu 1 người trong team bị xe bus đụng (hoặc nghỉ việc, ốm, đi du lịch), project có chết không?

Nếu bus factor = 1 → nguy hiểm cực kỳ. Với dự án PBX:

  • Asterisk/SIP expert: bus factor thường = 1. Giải pháp: pair programming bắt buộc cho VoIP tasks, documentation chi tiết.
  • Database design: bus factor thường = 1-2. Giải pháp: ER diagram + data dictionary luôn up-to-date.
  • Frontend architecture: bus factor tốt hơn = 2-3 nếu team share code review.
  • DevOps/CI/CD: bus factor nguy hiểm = 1. Giải pháp: runbook + recorded walkthrough.

Target: mọi critical area phải có bus factor >= 2. Đạt được bằng: code review cross-team, pair programming rotation, "shadow" on-call.

QUY TAC #01
"Không có test = Không có code"

Năm 2018, deploy feature mới không test. Production crash. 2 giờ downtime. CEO email toàn công ty. Từ đó, CI/CD pipeline có gate: coverage < 70% = build fail. Không ngoại lệ. Kể cả hotfix lúc 3 giờ sáng.

QUY TAC #02
"Log nhiều hơn bạn nghĩ bạn cần. Nhưng đừng log quá nhiều."

Paradox? Không. Log mọi business event (call started, call ended, agent status changed). Không log mọi technical detail (every SQL query, every HTTP header). Structured logging (JSON format). Correlation ID cho mỗi request chain. Và LUÔN log error với đầy đủ context — stack trace, input parameters, user ID.

QUY TAC #03
"Database migration phải forward-only và backward-compatible"

Rename column? Không. Thêm column mới, migrate data, deprecate column cũ, xóa sau 2 releases. Tại sao? Vì khi deploy fails và cần rollback, database migration KHÔNG rollback theo. Column bị rename = cũ code chạy với database mới = crash.

QUY TAC #04
"Configuration > Magic Number. Always."

Mọi số magic trong code (timeout 30 giây, max retry 3, page size 50) phải là configuration. Vì: khách Hàn muốn timeout 60 giây. Team QA cần page size 10 để test pagination. Production cần retry 5 vì network unstable. Nếu hard-code, phải re-deploy. Nếu config, chỉ cần restart.

QUY TAC #05
"Mỗi API endpoint phải có rate limiting"

Ngày xưa ta nghĩ: "API nội bộ, ai mà abuse?" Rồi một developer viết script test gọi API 10,000 lần/giây. Server crash. Lesson: rate limiting cho mọi endpoint. External: 100 req/min. Internal: 1,000 req/min. Admin: 50 req/min (vì admin endpoints thường expensive).

QUY TAC #06
"Đừng bao giờ deploy thứ Sáu"

Thứ Sáu deploy, bug phát hiện thứ Bảy, team off thứ Bảy, hotfix Chủ Nhật, team burn out thứ Hai. Dự án PBX: deploy thứ Ba hoặc thứ Tư. Có đủ weekday buffer để fix. Ngoại lệ duy nhất: security hotfix — thì ngày nào cũng deploy.

QUY TAC #07
"Monitoring không phải afterthought. Nó là Day 1 feature."

Dự án trước, setup monitoring sau go-live. Production crash, không ai biết cho đến khi khách gọi. Bây giờ: health check endpoint + Prometheus metrics + Grafana dashboard + PagerDuty alerts — tất cả setup TRƯỚC sprint 1 kết thúc.

QUY TAC #08
"Mọi external dependency đều PHẢI có circuit breaker"

Gọi external API mà API đó down? Nếu không có circuit breaker, request queue lên, thread pool exhausted, server bạn cũng down theo. Domino effect. Circuit breaker pattern: sau 5 failures, stop calling external API trong 30 giây, thử lại 1 request, nếu OK thì resume. Polly library cho .NET.

QUY TAC #09
"PR nhỏ, merge thường xuyên, conflict ít, deploy tự tin"

PR 50 dòng review 10 phút, confidence 95%. PR 500 dòng review 1 giờ, confidence 70%. PR 5000 dòng review "approve without reading", confidence 20%. Quy tắc: max 400 dòng changed, max 3 files core logic. Nếu hơn, phải split.

QUY TAC #10
"API versioning từ ngày đầu"

URL: /api/v1/agents. Header: X-API-Version: 1. Tại sao? Vì Electron app = desktop app = không thể force update tất cả users cùng lúc. Sẽ có lúc v1 clients và v2 clients coexist. Nếu không version API, bạn sẽ break old clients khi update backend.

QUY TAC #11
"Security là responsibility của MỌI developer, không chỉ security team"

Mọi developer phải biết OWASP Top 10. Code review phải check security. SQL injection, XSS, CSRF — không phải "advanced topics". Đây là BASIC. Trong dự án PBX cho Hàn Quốc: mỗi sprint có 1 security-focused task. Mỗi tháng có 1 security scan. Không negotiate.

QUY TAC #12
"Document WHY, not WHAT. Code tells WHAT."

Comment // increment counter by 1 ← vô dụng, ai cũng thấy counter++. Comment // Increment retry counter. Korean telecom gateway sometimes returns timeout on first try but succeeds on retry. Max 3 retries per KISA guideline. ← vàng ròng.

QUY TAC #13
"Premature optimization is the root of all evil. Nhưng NO optimization is the root of all downtime."

Đừng optimize khi chưa cần. Nhưng PHẢI đo performance từ ngày đầu. Set baseline. Monitor trends. Khi response time tăng 20% so với baseline → investigate. Đừng đợi khách phàn nàn.

QUY TAC #14
"Backup chưa test = Không có backup"

Dự án năm 2020: backup database hàng ngày, tự tin. Đến lúc cần restore: backup file corrupted 3 tháng liền không ai biết. Từ đó: backup test monthly. Restore vào test environment, verify data integrity, ghi log. Automated.

QUY TAC #15
"Đừng giải quyết vấn đề bạn chưa có. Nhưng PHẢI biết vấn đề đó tồn tại."

YAGNI (You Ain't Gonna Need It) là đúng. Nhưng YAGNI không có nghĩa là đóng mắt trước tương lai. Ví dụ: chưa cần microservices? Đúng, đừng build microservices. Nhưng PHẢI design module boundaries sạch để KHI CẦN chuyển sang microservices, không phải rewrite toàn bộ. Đó là nghệ thuật của architecture.

SAI LAM #01
Over-Engineering: Kiến Trúc Du Hành Vũ Trụ Cho App Quản Lý Todo

PL mới thường muốn chứng tỏ bằng kiến trúc phức tạp. Microservices cho 5 users. Event sourcing cho CRUD app. Kubernetes cho 1 server. GraphQL cho 3 endpoints. Kết quả: team mất 2 tháng setup infrastructure, 1 tháng build feature. Khách cancel dự án vì "sao lâu thế?"

Thuốc giải: Bắt đầu đơn giản nhất có thể mà vẫn đáp ứng yêu cầu HIỆN TẠI. Scale khi CẦN, không phải khi "có thể cần".

SAI LAM #02
Under-Testing: "Test Sau Cũng Được"

Mỗi lần PL nói "viết test sau, deadline gấp", một production bug được sinh ra. Test debt tích lũy exponentially. Tháng 1 skip test, có 5 untested features. Tháng 2 có 15. Tháng 3: 30+ untested features, regression everywhere, fix bug này sinh bug khác. Team chạy trên hamster wheel.

Thuốc giải: Test là phần KHÔNG THỂ negotiate trong definition of done. Feature chưa có test = feature chưa xong.

SAI LAM #03
Wrong Abstraction: DRY Quá Mức

Thấy 2 hàm có code giống nhau 80%, refactor thành 1 hàm với 10 parameters và 5 if-else branches. Code "DRY" nhưng unreadable, unmodifiable, và khi business logic của 2 use cases diverge (luôn luôn diverge), phải thêm 5 if-else nữa. Kết quả: một monster function 500 dòng mà không ai dám sửa.

Thuốc giải: "Duplication is far cheaper than the wrong abstraction" — Sandi Metz. Chấp nhận duplication cho đến khi pattern THẬT SỰ rõ ràng (rule of three: duplicate 3 lần mới abstract).

SAI LAM #04
Premature Optimization: Optimize Cái Chưa Ai Dùng

Developer muốn optimize database query từ 10ms xuống 1ms. Nhưng query đó chỉ chạy 1 lần/giờ. Trong khi đó, query khác chạy 10,000 lần/phút và tốn 500ms. Optimize sai chỗ = phí effort.

Thuốc giải: Measure first, optimize second. Profiler, APM, database query analysis — dùng data, không dùng gut feeling.

SAI LAM #05
Không Nói "Không" Với Feature Creep

Khách muốn thêm feature → PL đồng ý vì "chỉ thêm chút thôi" → team overload → quality giảm → bug tăng → khách không happy → khách muốn thêm feature "fix" quality issues → vòng lặp vô tận.

Thuốc giải: Mỗi feature request phải qua impact analysis: effort estimate, risk assessment, scope impact. Trình bày trade-offs cho stakeholder. Để HỌ quyết định priority, nhưng VỚI đầy đủ thông tin.

SAI LAM #06
Ignoring Non-Functional Requirements

Team focus 100% vào feature, quên: performance, security, accessibility, scalability, maintainability. Đến UAT khách nói: "Sao chậm thế?" "Sao không có audit log?" "Sao không responsive?" — tất cả đều "obvious" nhưng không có trong spec vì PL không hỏi.

Thuốc giải: NFR checklist ở đầu mỗi dự án: Performance targets? Security requirements? Accessibility standards? Scalability expectations? Luôn hỏi, luôn document, luôn test.

SAI LAM #07
Single Point of Failure — Cả Architecture Lẫn Team

Một server, không HA. Một developer biết Asterisk, không có backup. Một deployment process chỉ 1 người biết. Bất kỳ cái nào fail → dự án dừng.

Thuốc giải: Redundancy ở mọi level. Ít nhất 2 servers cho production. Ít nhất 2 người biết mỗi critical component. Deployment process automated và documented.

SAI LAM #08
Copy-Paste Driven Development

Stack Overflow answer → copy → paste → nó chạy → commit. Không hiểu code. Không đọc documentation. 3 tháng sau: security vulnerability trong copied code. Hoặc: license violation (GPL code trong proprietary project).

Thuốc giải: Copy-paste OK cho learning. Nhưng mỗi dòng code trong production phải được HIỂU. Code review phải catch "copied-without-understanding" patterns.

SAI LAM #09
Ignoring Tech Debt Interest

Tech debt giống nợ tài chính: có lãi suất. Workaround nhỏ hôm nay = 1 giờ fix. Tháng sau, 3 features build on top of workaround, fix = 1 tuần. 6 tháng sau, fix = rewrite module. Lãi suất compound lên đến mức "trả không nổi".

Thuốc giải: Allocate 15-20% sprint capacity cho tech debt. Mỗi sprint, pick top-priority tech debt items. Không để accumulate quá 2 sprints.

SAI LAM #10
PL Tự Code Quá Nhiều — Quên Đi Lead

PL giỏi code → muốn tự code critical features → không đủ thời gian review, plan, mentor, communicate → team drift, architecture inconsistent, stakeholder confused. PL cuối cùng burn out vì cố làm 2 jobs.

Thuốc giải: PL dành max 30% thời gian code (chủ yếu POC, architecture setup, critical bug fix). 70% cho review, planning, mentoring, communication. Delegate implementation, own architecture and quality.

SAI LAM #11
Không Build Relationship Với Stakeholder Kỹ Thuật Phía Khách

PL chỉ nói chuyện với PM bên khách. Không biết DBA bên khách có constraint gì. Không biết IT Security team yêu cầu gì. Không biết DevOps bên khách deploy thế nào. Đến lúc go-live: surprise từ mọi phía.

Thuốc giải: PL phải có direct channel với counterparts bên khách: DBA, IT Security, DevOps, QA Lead. Ít nhất 1 technical sync meeting mỗi 2 tuần.

SAI LAM #12
"Nó Chạy Trên Máy Tôi" — Không Standardize Environment

Developer A dùng Node 18, Developer B dùng Node 20. Database localhost có data khác staging. Docker version mismatch. Kết quả: bug chỉ xuất hiện trên 1 máy, hoặc chỉ trên staging, hoặc chỉ production. Debug nightmare.

Thuốc giải: Docker Compose cho dev environment. .nvmrc cho Node version. global.json cho .NET version. Seed data script idempotent. "Works on my machine" → "Works on THE machine (Docker)".

📞 Real-Time Call State Management

Call state machine là trái tim của PBX system. Design SAI ở đây = mọi thứ sai.

call-state-machine.txt
# Call State Machine
IDLE → RINGING → CONNECTED → ENDED
         ↓            ↓
     MISSED     ON_HOLD ↔ CONNECTED
                     ↓
              TRANSFERRING → TRANSFERRED
                     ↓
              CONFERENCE → ENDED
  • State phải sync giữa 3 nơi: Asterisk (SIP), .NET server (business logic), Vue.js client (UI). Dùng SignalR push events, KHÔNG polling.
  • State transition validation trên server: client không được jump state (IDLE → CONNECTED bỏ qua RINGING). Server reject invalid transitions.
  • Optimistic UI: UI update ngay khi user click (e.g., click Answer → UI show CONNECTED). Nếu server reject → rollback UI state. User experience mượt hơn.
  • Event sourcing cho call history: mỗi state transition = 1 event. CDR (Call Detail Record) build từ event stream. Debug dễ, audit trail đầy đủ.
  • Timeout handling: Ring timeout (30 giây → auto-route sang agent khác), Hold timeout (5 phút → alert supervisor), Conference max duration (enterprise Hàn Quốc thường limit 60 phút).
🐳 Asterisk/Docker Integration Best Practices
  • Docker Compose setup: Asterisk container + .NET API container + MSSQL container + Redis container. One command: docker compose up
  • Asterisk config management: Config files mount as volumes, NOT baked into image. Change config → restart container, NOT rebuild image.
  • ARI (Asterisk REST Interface) over AMI: ARI là modern, REST-based, WebSocket events. AMI là legacy, TCP-based, text protocol. Cho dự án mới, dùng ARI.
  • Media handling: RTP media nên bypass Asterisk khi có thể (direct media) để giảm load. Chỉ route qua Asterisk khi cần recording hoặc conference mixing.
  • Call recording storage: Record ở Asterisk, upload async lên object storage (S3/Azure Blob), reference trong database. KHÔNG lưu recording trong database.
  • Scalability: Horizontal scale bằng multiple Asterisk instances behind SIP load balancer (Kamailio). Mỗi instance handle 200-300 concurrent calls.
  • Failover: Active-passive Asterisk pair. SIP trunk register cả 2. Kamailio health-check và route sang active instance.
💾 IndexedDB Sync Strategy — Offline-First

Electron app PHẢI hoạt động khi mất network (ít nhất basic functions):

  • Dexie.js wrapper cho IndexedDB — API clean, TypeScript support, migration support.
  • Sync strategy: Server là source of truth. Client sync delta changes khi online. Conflict resolution: server wins (cho PBX, consistency quan trọng hơn availability).
  • Offline capabilities: View call history, view contacts, view agent list. Queue outbound calls for when network restored.
  • Sync mechanism: lastSyncTimestamp per entity type. Server endpoint GET /api/sync/{entity}?since={timestamp} returns delta. Batch sync mỗi 30 giây khi online.
  • Storage management: IndexedDB quota varies. Implement LRU cache — giữ 30 ngày data gần nhất, archive cũ hơn.
  • Encryption: IndexedDB lưu local, ai có access vào laptop có thể đọc. Encrypt PII trước khi lưu vào IndexedDB. Dùng Web Crypto API.

🔒 Security Architecture — PIPA Compliance
  • Authentication: JWT (short-lived, 15 min) + Refresh Token (long-lived, 7 days, stored HttpOnly cookie). Refresh token rotation — mỗi lần refresh, old token invalidated.

📦 Deployment Pipeline — Electron Desktop App

Deploy Electron app cho enterprise Hàn Quốc có nhiều challenge hơn web deploy:

  • Version strategy: SemVer. Backend API version phải compatible với N-1 Electron version (không phải mọi user update cùng lúc).
📁 Code Organization & Module Boundaries

Module boundaries là kiến trúc quan trọng nhất. Sai ở đây, sai ở mọi nơi:

  • Call Module: Inbound/outbound call management, call state machine, call transfer, conference, call recording. Dependency: Agent Module, SIP Service.
  • Agent Module: Agent status management (Available/Busy/Away/Offline), agent skills/queues, break management, agent performance metrics. Dependency: Auth Module.
  • Queue Module: Call queue management, routing rules, priority handling, queue monitoring, overflow rules. Dependency: Call Module, Agent Module.
  • Dashboard Module: Real-time statistics, historical reports, KPI visualization. Dependency: Call Module, Agent Module, Queue Module (READ ONLY).
  • Admin Module: System configuration, user management, SIP trunk management, IVR configuration. Dependency: Auth Module.
  • Auth Module: Authentication, authorization, session management, audit logging. No domain dependency.

Boundary Rules:

  • Modules communicate through defined interfaces (services), NEVER direct database access across modules.
  • Each module owns its database tables. Dashboard module uses read-only views/materialized views of other modules' data.
  • SignalR events are the primary cross-module communication mechanism for real-time updates.
  • Circular dependencies = architecture smell. If Module A depends on Module B AND Module B depends on Module A, extract shared logic into Module C.
