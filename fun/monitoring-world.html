Thế Giới Monitoring

Từ MRTG ping đơn giản đến distributed tracing xuyên microservices - Hành trình 30 năm của ngành giám sát hệ thống. Ba trụ cột Metrics, Logs, Traces và tất cả những gì bạn cần biết để giữ hệ thống luôn sáng đèn xanh.

⌄

Monitoring vs Observability

Monitoring trả lời câu hỏi "Hệ thống có đang hoạt động không?" - bạn biết trước cần theo dõi cái gì. Observability trả lời câu hỏi "Tại sao hệ thống lại hành xử kỳ lạ thế?" - bạn có thể khám phá những vấn đề mà bạn chưa từng nghĩ tới. Monitoring là tập con của Observability. Cả hai đều cần thiết, nhưng observability mới là đích đến.

📈

Metrics

Con số biết kể chuyện. CPU 95%, latency p99 = 2s, request rate 10k/s. Metrics cho bạn bức tranh tổng quan, phát hiện xu hướng bất thường, và trigger alert khi vượt ngưỡng. Nhẹ nhàng, dễ lưu trữ lâu dài.

📜

Logs

Nhật ký chi tiết của mọi sự kiện. "Error: connection refused to db-primary at 14:32:07". Logs cho bạn ngữ cảnh đầy đủ - cái gì, khi nào, ở đâu. Nhưng cũng tốn dung lượng nhất trong ba trụ cột.

🔗

Traces

Theo dấu hành trình của một request xuyên qua 20 microservices. "Request này mất 3 giây vì service-payment gọi API bên thứ ba bị timeout." Traces là mảnh ghép cuối cùng để hiểu hệ thống phân tán.

Insight quan trọng

Ba trụ cột này không hoạt động riêng lẻ. Sức mạnh thực sự nằm ở khả năng correlate giữa chúng: từ một alert (metric), bạn nhảy sang log tương ứng, rồi từ log trace-ID dẫn bạn đến distributed trace để thấy toàn bộ bức tranh. Đây là lý do Grafana đang xây dựng cả ba stack: Mimir (metrics), Loki (logs), Tempo (traces).

Dòng Thời Gian Monitoring

30 năm tiến hóa - từ ping ICMP đến AI-powered observability

1995-2000

MRTG & RRDtool - Khởi nguyên

Multi Router Traffic Grapher ra đời năm 1995, vẽ biểu đồ băng thông mạng đầu tiên trên web. Tobias Oetiker sau đó tạo RRDtool - cơ sở dữ liệu round-robin trở thành nền tảng cho gần như mọi tool monitoring suốt một thập kỷ. Đơn giản nhưng cách mạng: lần đầu tiên sysadmin "nhìn thấy" traffic thay vì chỉ đọc số.

1999-2005

Nagios - Ông vua cảnh báo

Ethan Galstad tạo ra NetSaint (sau đổi tên thành Nagios). Kiến trúc check-based: chạy plugin kiểm tra rồi trả về OK/WARNING/CRITICAL. Hàng nghìn plugin có sẵn. Dashboard xanh/đỏ trở thành biểu tượng NOC (Network Operations Center) khắp thế giới. Nhưng cấu hình bằng file text thuần là cơn ác mộng.

2001-2010

Zabbix & Cacti - Enterprise monitoring

Zabbix (2001) mang đến auto-discovery, template system, và web UI quản lý hoàn chỉnh. Cacti kế thừa RRDtool với giao diện đồ họa đẹp hơn. Đây là thời đại của SNMP polling, agent-based monitoring, và những dashboard đầy biểu đồ traffic trên tường NOC. Zabbix vẫn mạnh mẽ đến tận ngày nay với hàng triệu deployment.

2012-2016

Prometheus & TSDB Revolution

SoundCloud tạo ra Prometheus (2012), lấy cảm hứng từ Borgmon của Google. Pull-based model, PromQL query language mạnh mẽ, multi-dimensional labels. Grafana (2014) xuất hiện như data visualization layer. Cả hai cùng nhau thay đổi hoàn toàn cách ngành nghĩ về monitoring. CNCF project thứ hai sau Kubernetes.

2019-nay

OpenTelemetry & Unified Observability

OpenTracing và OpenCensus hợp nhất thành OpenTelemetry (OTel). Một chuẩn duy nhất cho metrics, logs, và traces. Vendor-neutral instrumentation. eBPF cho phép observe mà không cần sửa code. AI/ML bắt đầu phát hiện anomaly tự động. Thời đại của observability-as-code.

Pull-based / CNCF Graduated

Service discovery tích hợp sẵn cho Kubernetes. Nhưng single-node, không có native clustering - đó là lý do cần Thanos hoặc Cortex.

Dashboard Best Practices

1

USE Method cho Infrastructure

Brendan Gregg đề xuất: cho mỗi resource, theo dõi Utilization (% đang dùng), Saturation (hàng đợi, quá tải), và Errors (lỗi). CPU, Memory, Disk, Network - mỗi thứ đều cần ba chỉ số này. Đây là cách tổ chức dashboard infrastructure tốt nhất.

2

RED Method cho Services

Tom Wilkie (Grafana Labs) đề xuất cho microservices: Rate (requests/second), Errors (% lỗi), Duration (latency). Ba chỉ số này cho mỗi service endpoint. Kết hợp USE + RED = bức tranh toàn cảnh từ infra đến application.

3

Four Golden Signals (Google SRE)

Google SRE Book định nghĩa: Latency, Traffic, Errors, Saturation. Nếu bạn chỉ có thể theo dõi bốn thứ, hãy theo dõi bốn thứ này. Dashboard chính nên bắt đầu từ đây rồi mới drill-down vào chi tiết.

4

Layered Dashboards

Đừng nhồi tất cả vào một dashboard. Tạo hierarchy: Overview (toàn hệ thống, 5-8 panels) -> Service Level (chi tiết từng service) -> Debug Level (deep-dive khi troubleshoot). Dùng link variables để navigate giữa các tầng.

5

Dashboard as Code

Đừng tạo dashboard bằng tay trên UI rồi quên. Dùng Grafana provisioning (YAML + JSON), hoặc Grafonnet (Jsonnet library) để định nghĩa dashboard dưới dạng code. Version control bằng Git. Review bằng PR. Rollback khi cần. Đây là DevOps đúng nghĩa.

6

Đừng Quên Annotations

Annotations đánh dấu thời điểm deploy, restart, config change lên dashboard. Khi latency tăng đột ngột, bạn ngay lập tức thấy "Deploy v2.3.1 lúc 14:30" - và biết ngay nguyên nhân. Tích hợp CI/CD pipeline gửi annotation tự động vào Grafana.

Grafana Loki

Log Aggregation / "Prometheus for Logs"

Triết lý khác biệt hoàn toàn: Loki không index nội dung log. Nó chỉ index labels (giống Prometheus). Khi query, Loki duyệt qua chunks log theo labels rồi mới grep nội dung. Kết quả? Chi phí storage và operation thấp hơn 10x so với ELK. Đánh đổi: query full-text chậm hơn. Nhưng với LogQL, bạn rất ít khi cần grep toàn bộ.

Structured Logging - Viết log đúng cách

Log không cấu trúc: 2024-01-15 14:32:07 ERROR Failed to connect to database server 10.0.1.50:5432 after 3 retries

Structured log (JSON): {"ts":"2024-01-15T14:32:07Z","level":"error","msg":"db connection failed","host":"10.0.1.50","port":5432,"retries":3}

Structured logging là bước tiến hóa quan trọng nhất trong logging. Thay vì viết một chuỗi text tự do mà máy tính phải regex để hiểu, bạn viết key-value pairs rõ ràng. Kết quả: query nhanh hơn hàng trăm lần ({level="error", host="10.0.1.50"} thay vì regex match), aggregation chính xác, và không bao giờ bị parse sai vì log format thay đổi.

Syslog vẫn còn đó

Đừng quên syslog (RFC 5424) - giao thức log cổ điển nhất, vẫn là chuẩn cho network devices, firewalls, switches. Hầu hết mọi thiết bị mạng đều nói syslog. Rsyslog và syslog-ng là hai implementation phổ biến nhất trên Linux. Trong homelab, rsyslog → Promtail → Loki là pipeline rất phổ biến.

On-call Best Practices

1

Alert trên Symptom, không phải Cause

Đừng alert "CPU > 90%". Hãy alert "Latency p99 > 2s" hoặc "Error rate > 5%". CPU cao có thể là bình thường (batch job). Latency cao thì chắc chắn ảnh hưởng user. Alert dựa trên SLO (Service Level Objective) là chuẩn mực hiện đại.

2

Mỗi Alert phải Actionable

Nếu nhận alert mà bạn chỉ "nhìn rồi dismiss" - xóa alert đó đi. Mỗi alert phải kèm runbook: "Khi nhận alert này, làm bước 1, 2, 3." Nếu không có action rõ ràng, nó chỉ là noise và dần dà team sẽ bỏ qua tất cả mọi alert.

3

Đo lường Alert Quality

Theo dõi: số alert mỗi tuần, % alert dẫn đến action thực sự (signal vs noise), thời gian acknowledge trung bình, và số lần bị đánh thức ngoài giờ. Mục tiêu: dưới 2 alert thực sự mỗi ca on-call 12 tiếng. Nhiều hơn = hệ thống hoặc alert rules có vấn đề.

Alert Fatigue là sát thủ thầm lặng

Một nghiên cứu cho thấy khi engineer nhận hơn 5 alerts/ngày không actionable, tỷ lệ bỏ qua alert thực sự quan trọng tăng gấp 3 lần. Đây gọi là alert fatigue - cùng cơ chế tâm lý với "cry wolf". Tốt hơn có 5 alerts chất lượng cao hơn 500 alerts mà 495 cái là noise. Hãy ruthless trong việc xóa alert không cần thiết.

SNMP & Network Monitoring

Thế giới network devices nói ngôn ngữ riêng - và ngôn ngữ đó là SNMP

SNMP - Simple Network Management Protocol

Ra đời từ năm 1988, SNMP là cách giao tiếp giữa Network Management System (NMS) và network devices. Mỗi thiết bị expose một cây MIB (Management Information Base) với hàng nghìn OID (Object Identifier). Ví dụ: .1.3.6.1.2.1.2.2.1.10 là ifInOctets - số byte nhận trên interface. NMS poll các OID này định kỳ hoặc thiết bị gửi SNMP trap khi có sự kiện.

SNMPv1

Phiên bản đầu tiên. Community string truyền plaintext - bảo mật gần như bằng không. Vẫn thấy ở thiết bị cũ. Đừng dùng nếu có lựa chọn khác.

SNMPv2c

Cải tiến performance và data types, nhưng vẫn community string plaintext. Phổ biến nhất hiện tại vì đơn giản. "Đủ dùng" trong mạng nội bộ đáng tin cậy.

SNMPv3

Authentication (SHA/MD5) + Encryption (AES/DES). Đây là phiên bản nên dùng. Nhưng cấu hình phức tạp hơn nhiều: usm users, auth protocols, priv protocols. Nhiều sysadmin vẫn chọn v2c vì "lười".

SNMP Exporter

Bridge giữa SNMP và Prometheus. Đọc MIB, poll SNMP targets, expose metrics ở format Prometheus. File generator chuyển MIB → config YAML. Cần thiết cho ai muốn monitor network devices bằng Prometheus.

gNMI / YANG

Tương lai của network monitoring. gRPC-based, streaming telemetry thay vì polling. Structured data model (YANG). Các vendor lớn (Cisco, Arista, Juniper) đang push mạnh. Nhưng chưa phổ biến ở thiết bị tầm trung và homelab.

NetFlow / sFlow

Không phải monitoring truyền thống mà là flow analysis. Thiết bị mạng sample traffic và gửi flow records: source IP, dest IP, port, bytes, packets. Cực kỳ hữu ích cho traffic analysis, security monitoring, và bandwidth billing.

OpenTelemetry & Distributed Tracing

Chuẩn mở thống nhất cho observability - tương lai đã đến

OpenTelemetry là gì?

Hãy tưởng tượng bạn có 50 microservices, mỗi service dùng một thư viện monitoring khác nhau, gửi data đến backend khác nhau. Chaos. OpenTelemetry (OTel) giải quyết bằng một chuẩn duy nhất: một SDK để instrument code, một Collector để thu thập và forward data, và một protocol (OTLP) để truyền tải. Bạn instrument một lần, gửi đến bất kỳ backend nào: Jaeger, Zipkin, Grafana Tempo, Datadog, New Relic...

Distributed Tracing giải thích đơn giản

User click "Đặt hàng". Request đi qua API Gateway → Auth Service → Order Service → Payment Service → Inventory Service → Notification Service. Mỗi service tạo một span. Tất cả spans cùng một trace ID. Khi lỗi xảy ra, bạn nhìn trace và thấy ngay: "Payment Service mất 8 giây vì gọi API ngân hàng timeout". Không có tracing, bạn phải đọc log từng service rồi tự ghép lại - mệt muốn chết.

🔭

OTel Collector

Vendor-neutral Data Pipeline

OTel Collector là trái tim của kiến trúc OpenTelemetry. Ba thành phần chính: Receivers (nhận data - OTLP, Prometheus, Jaeger, Zipkin, syslog...), Processors (biến đổi, filter, batch, sampling), và Exporters (gửi đi - bất kỳ backend nào). Bạn có thể nhận metrics từ Prometheus format, logs từ syslog, traces từ Jaeger, rồi export tất cả sang Grafana stack. Một tool để trị tất cả.

Khái niệm cốt lõi trong Distributed Tracing

Trace

Một trace đại diện cho toàn bộ hành trình của một request. Nó bao gồm nhiều spans từ nhiều services, được liên kết bởi một trace ID duy nhất.

Span

Một đơn vị công việc trong trace. Mỗi span có tên, thời gian bắt đầu/kết thúc, attributes, và parent span ID. Span lồng nhau tạo thành cây (tree).

Context Propagation

Cách trace-ID và span-ID được truyền từ service này sang service khác qua HTTP headers (W3C Trace Context, B3) hoặc gRPC metadata. Không có propagation = không có trace.

Sampling

Ở quy mô lớn, trace 100% requests là không khả thi (quá nhiều data). Head sampling (quyết định ngay đầu), tail sampling (quyết định sau khi thấy kết quả), và adaptive sampling là các chiến lược phổ biến.

Những Hiểu Nhầm Phổ Biến

Những quan niệm sai lầm có thể khiến bạn đi sai hướng ngay từ đầu

Sai lầm

"Có Grafana dashboard đẹp là đã monitoring tốt"

Dashboard đẹp mà không ai nhìn thì vô nghĩa. Dashboard mà không có alert rules cũng vô nghĩa. Rất nhiều team dành cả tuần làm dashboard tuyệt đẹp rồi... không bao giờ mở lại.

Thực tế

Dashboard chỉ là một phần của monitoring. Quan trọng hơn là: alert rules chất lượng, runbooks cho mỗi alert, on-call process rõ ràng, và incident review sau mỗi sự cố. Dashboard là công cụ debug, không phải mục tiêu.

Sai lầm

"Prometheus thay thế được mọi monitoring tool"

Prometheus rất mạnh cho metrics nhưng nó KHÔNG phải là giải pháp toàn diện. Nó không có log management, tracing hạn chế, SNMP polling cần exporter riêng, và long-term storage phải dùng tool khác.

Thực tế

Monitoring stack hiện đại cần nhiều tools phối hợp: Prometheus cho metrics, Loki/ELK cho logs, Jaeger/Tempo cho traces, Alertmanager cho alerting, Grafana cho visualization. Đó là lý do gọi là "stack" chứ không phải "tool".

Sai lầm

"Thu thập càng nhiều metrics càng tốt"

Khi bạn có 500.000 time-series mà chỉ thực sự nhìn 200 cái, bạn đang lãng phí storage, CPU, và tiền. High cardinality metrics (ví dụ: metric theo user_id) có thể giết chết Prometheus.

Thực tế

Thu thập có chủ đích. Bắt đầu với Four Golden Signals cho mỗi service, USE method cho infrastructure. Chỉ thêm metrics khi có câu hỏi cụ thể cần trả lời. Review và xóa metrics không dùng thường xuyên. Fewer, better metrics > many useless metrics.

Sai lầm

"Monitoring và Observability là hai thứ hoàn toàn khác"

Nhiều người (và vendor marketing) cố tình tách biệt chúng thành hai concept không liên quan, ám chỉ monitoring là "cũ" và observability là "mới".

Thực tế

Monitoring là tập con của observability. Bạn không thể có observability mà không monitoring. Monitoring trả lời "có vấn đề không?", observability giúp trả lời thêm "vấn đề là gì, ở đâu, tại sao?". Cả hai bổ sung cho nhau, không thay thế nhau.

Sai lầm

"ELK stack miễn phí hoàn toàn"

Elasticsearch, Logstash, Kibana đều open-source (SSPL license từ 2021). Nhưng chi phí thực sự nằm ở infrastructure: Elasticsearch cần RAM rất nhiều, Logstash cần CPU, và data volume tăng nhanh chóng mặt.

Thực tế

Với 100GB logs/ngày, bạn cần ít nhất 3 Elasticsearch nodes x 16GB RAM + hot/warm/cold tiering + curator để xóa data cũ. Chi phí infra thực tế có thể lên hàng nghìn đô/tháng. Đó là lý do Loki ra đời - giải quyết vấn đề chi phí của log aggregation.

Sai lầm

"Uptime 99.99% là đủ"

Nhiều team đặt target "four nines" mà không hiểu nó thực sự có nghĩa gì. 99.99% = chỉ 52.6 phút downtime mỗi năm. Để đạt được con số này, bạn cần toàn bộ stack (infra, app, network) đều đạt mức đó.

Thực tế

Hãy dùng SLO (Service Level Objectives) thay vì uptime đơn thuần. "99.9% requests phải có latency < 500ms" hữu ích hơn "99.99% uptime". SLO cho phép error budget: bạn biết chính xác còn bao nhiêu "ngân sách lỗi" để deploy tính năng mới hay thí nghiệm.

Sai lầm

"Average (mean) latency là metric đủ tốt"

Average latency 200ms nghe có vẻ ổn. Nhưng nếu 95% requests mất 50ms và 5% requests mất 3 giây, average vẫn là ~200ms. Bạn đã che giấu trải nghiệm tệ của 5% users.

Thực tế

Dùng percentiles: p50, p90, p95, p99. p99 = 2s nghĩa là 1% requests chậm nhất mất hơn 2 giây. Prometheus histogram + histogram_quantile() là cách chuẩn để tính percentiles. Luôn hiển thị p50 và p99 cạnh nhau trên dashboard.

Sai lầm

"Monitoring setup xong rồi là xong"

Team dành 2 tuần setup monitoring stack, vui mừng, rồi không ai đụng lại trong 6 tháng. Alert rules cũ, dashboard không phản ánh kiến trúc mới, metrics mới không được thêm.

Thực tế

Monitoring là sản phẩm sống, cần liên tục bảo trì. Mỗi feature mới cần metrics mới. Mỗi incident cần review: "Tại sao monitoring không phát hiện sớm hơn?". Tốt nhất là coi monitoring như code: version control, code review, CI/CD cho dashboards và alert rules.

Cheat Sheet

PromQL và LogQL - những query bạn sẽ dùng hàng ngày

PromQL - Prometheus Query Language

CPU Usage per Instance

Tính % CPU đang sử dụng cho mỗi instance. Dùng rate() cho counter metrics.

100 - (avg by(instance) ( rate(node_cpu_seconds_total {mode="idle"}[5m] ) * 100 ))

Memory Usage %

Phần trăm RAM đã sử dụng. MemAvailable chính xác hơn MemFree.

(1 - ( node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes )) * 100

HTTP Request Rate by Status

Requests per second grouped theo HTTP status code.

sum by(status_code) ( rate(http_requests_total[5m]) )

Error Rate %

Tỷ lệ requests trả về 5xx. Nền tảng cho SLO monitoring.

sum(rate( http_requests_total {status_code=~"5.."}[5m] )) / sum(rate( http_requests_total[5m] )) * 100

Latency Percentile (p99)

99th percentile latency từ histogram. Quan trọng hơn average.

histogram_quantile(0.99, sum by(le) ( rate(http_request_duration_seconds_bucket[5m]) ) )

Disk Space Prediction

Dự đoán bao giờ disk đầy dựa trên trend 7 ngày.

predict_linear( node_filesystem_avail_bytes {mountpoint="/"}[7d], 3600 * 24 * 30 ) < 0

Top 5 Pods by Memory

5 pod dùng nhiều memory nhất trong Kubernetes.

topk(5, container_memory_usage_bytes {namespace="production"} )

Alert Rule: High Error Rate

Cấu trúc alert rule trong Prometheus.

- alert: HighErrorRate expr: | sum(rate(http_requests_total {code=~"5.."}[5m])) / sum(rate( http_requests_total[5m])) > 0.05 for: 5m labels: severity: critical

LogQL - Loki Query Language

Lọc log theo label

Query cơ bản: chọn log stream theo labels.

{namespace="production", container="api-server"} |= "error"

Regex Filter

Lọc log content bằng regex pattern.

{job="nginx"} |~ "status=(4|5)\\d{2}" !~ "healthcheck"

JSON Parse + Filter

Parse JSON log rồi filter theo field.

{app="payment-service"} | json | level="error" | duration > 5s

Log Metrics: Rate

Đếm số error logs per second - biến logs thành metrics.

sum(rate( {namespace="production"} |= "error" [5m] )) by (container)

Pattern Match + Unwrap

Extract giá trị số từ log để tính aggregation.

{job="app"} | json | unwrap duration | avg_over_time( {job="app"} | json | unwrap duration [5m] )

Top Error Messages

Tìm các error message xuất hiện nhiều nhất.

topk(10, sum by(msg) ( count_over_time( {app="api"} | json | level="error" [1h] ) ) )

Quick Reference: Khi nào dùng gì?

Tình huống Dùng gì Tại sao
Latency tăng đột ngột Metrics (Prometheus) → Traces (Tempo) Metrics phát hiện, traces chỉ rõ service nào chậm
Lỗi 500 tăng vọt Metrics → Logs (Loki) Metrics trigger alert, logs cho thấy error message cụ thể
Switch mạng bị flapping SNMP + Syslog SNMP trap thông báo link down/up, syslog cho chi tiết
Disk sắp đầy Metrics + predict_linear() PromQL dự đoán khi nào disk hết dựa trên trend
Xác định bottleneck trong microservices Distributed Traces (OTel) Traces hiển thị waterfall view - thấy ngay service nào chậm
Post-incident analysis Logs + Metrics + Traces (correlated) Cả ba trụ cột kết hợp cho bức tranh toàn cảnh
Capacity planning Long-term metrics (VictoriaMetrics) Cần data nhiều tháng để thấy growth trend
