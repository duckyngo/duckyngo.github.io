01 — Định nghĩa

G-AGX là gì?

Core telephony platform phía đối tác — không phải app của mình.

Hiểu đơn giản, G-AGX là hệ thống CTI/PBX phía đối tác đã xây xong, chịu trách nhiệm xử lý toàn bộ nghiệp vụ điện thoại. Trong meeting, phía G-AGX nói rất rõ: hệ thống của họ đã được phát triển hoàn tất. Bên mình không làm trực tiếp vào G-AGX, mà sẽ kết nối từ PC App + backend sang G-AGX thông qua interface.

🔐 Authentication server↔server 📞 Make call / Click-to-call ⚡ Real-time call events 📊 Ringing / Answered / Hangup ❌ Busy / Missed call

💡 Nói theo kiểu business-tech

G-AGX không phải là app của mình. Nó là core telephony platform / call control platform phía đối tác.

Bên mình chỉ build application layer và integration layer để gọi API, nhận event, rồi xử lý UI/logic ở phía Coway.

03 — Luồng giao tiếp

Outbound & Inbound

Hai chiều giao tiếp — hai luồng hoàn toàn khác nhau.

Outbound Từ mình → G-AGX

Ví dụ: user bấm nút gọi điện trên PC App.

Step 1
User thao tác trên PC App — bấm "Gọi" hoặc click-to-call
Step 2
App gọi backend của mình — Interface Gateway
Step 3
Backend thực hiện auth/check — verify session, map user → extension
Step 4
Backend gọi API của G-AGX — truyền terminal, caller, callee number
Step 5
G-AGX điều khiển PBX — thực hiện cuộc gọi thật
API Parameters // G-AGX make call API cần đúng 3 tham số: { "terminal_number": "1001", // device/phone vật lý "caller_extension": "8501", // số nội bộ người gọi "callee_number": "0901234567" // số đích } // G-AGX không tự đoán → mình phải truyền đúng
Inbound Từ G-AGX → Mình

Chiều ngược lại — event callback / receive notification. Khi cuộc gọi thay đổi trạng thái, G-AGX đẩy event real-time về backend.

Event
Cuộc gọi thay đổi trạng thái: ringing → answered → hangup
Push
G-AGX đẩy event real-time về endpoint của backend mình
Process
Backend parse payload, cập nhật trạng thái, đẩy qua WebSocket cho App
Display
PC App hiển thị real-time — user thấy trạng thái cuộc gọi thay đổi

⚠️ Yêu cầu từ phía G-AGX

Backend của mình phải có public endpoint / callback URL để nhận event. Phía G-AGX nói rõ: muốn test phần này, bên mình phải cung cấp endpoint URL/domain, vì hiện tại họ mới đang để sample/test endpoint thôi.

04 — Authentication

Xác thực giữa
hai server

Minimum server-to-server authentication — không phải user login.

Ban đầu trong meeting có chút nhầm lẫn, nhưng sau đó họ chốt: Authentication là cần thiết. Mục đích là minimum server-to-server authentication — có API 인증 체크 (auth check).

🔒

Server-to-Server Auth Check

Trước khi gọi bất kỳ business API nào (make call, transfer...), backend của mình phải đi qua bước auth/check với G-AGX. Đây là lớp xác thực tối thiểu, tránh API bị gọi vô tội vạ từ nguồn không hợp lệ.

🔑

Admin ID/PW — Credential tạm thời

Trong màn hình admin có ID/PW liên quan đến auth check. Phía họ nói giá trị hiện tại là tạm thời — lúc open chính thức sẽ phải chốt lại credential thật.

🛡️

Dev Server vs Production Server

Sample trong document cũ đang trỏ production server → gọi vào sẽ timeout. Muốn test phải dùng development server info mới đúng.

§
05 — Server Design

Common Gateway vs
Interface Gateway

Phía G-AGX nhìn server của mình theo 2 vai trò khác nhau.

Common Gateway

Kênh Request / Response chính

Server entry chung cho mọi request/API từ phía Coway sang G-AGX.

  • Nhận request từ backend của mình
  • Điểm gọi chính cho auth, make call…
  • Nơi G-AGX gửi lại API response
  • Kiểm tra IP/domain đã đăng ký (whitelist)
  • Server phía Coway, không phải G-AGX
Interface Gateway(s)

Kênh nhận Event Realtime

Nhóm endpoint nhận event broadcast từ G-AGX khi cuộc gọi có thay đổi.

  • Đăng ký được 1~5 server endpoints
  • Event broadcast đến TẤT CẢ server active
  • Không phải load balancing round-robin
  • Là fan-out broadcast
  • Mỗi endpoint cần trạng thái kết nối "ON"
Tóm tắt // Hai vai trò, hai mục đích: Common Gateway = "kênh command — mình gửi, họ trả lời" Interface Gateway = "kênh event — họ đẩy, mình nhận" // Đây là thiết kế tách biệt request flow và event flow
§
06 — Broadcast Fan-out

Tại sao 1 Common +
5 Interface Gateway?

Đây là chỗ rất dễ hiểu sai. Không phải load balancer — là broadcast.

Event Broadcast Model — Fan-out, không phải Round-robin

Source G-AGX
──→ ──→ ──→ ──→ ──→
Interface GW #1 — Active
Interface GW #2 — Active
Interface GW #3 — Active
Interface GW #4 — Inactive
Interface GW #5 — Inactive
⚠ Cùng 1 event được gửi đến TẤT CẢ endpoint active — không chọn 1 cái bất kỳ

Logic ban đầu: request từ Coway sang G-AGX chỉ cần 1 cổng chung. Nhưng event trả về có thể nhiều, realtime, liên tục — nên họ tách riêng và hỗ trợ nhiều endpoint vì trước đây phía Coway có vẻ đã hình dung mô hình backend phân tán.

Góc nhìn kỹ thuật hiện đại: Nếu hạ tầng dùng ELB / reverse proxy / autoscaling / k8s pods thì không nhất thiết phải giữ đúng tư duy "5 endpoint vật lý riêng". Hoàn toàn có thể thiết kế 1 logical endpoint, còn scale phía sau bằng hạ tầng cloud.

Broadcast vào 5 nơi cùng lúc không giống cách scaling hiện đại. Nhưng đó là thiết kế của G-AGX — mình cần hiểu để adapt, không cần giữ nguyên.

§
08 — Terminal / Extension Mapping

Dependency lớn nhất
nằm ở đây

Không chốt mapping sớm → dev sẽ bị nghẽn.

Khi gọi điện, API cần terminal number, caller extension, callee number. Nhưng app không tự biết terminal/caller nếu không có mapping. Phía G-AGX nói rõ: phần mapping đó là nội bộ Coway, có thể liên quan đến HR/org information.

🔑 User Login
→
👤 Employee Info
→
📞 Extension
→
📱 Terminal/Device

🚨 Critical Dependency — Phải chốt sớm

Muốn chạy thật, bên mình phải làm rõ: user login bằng gì → từ login map ra extension thế nào → extension map sang terminal/device ra sao → lấy dữ liệu từ HR API hay nguồn nào?

Đây là dependency lớn — không chốt sớm là dev sẽ bị nghẽn hoàn toàn.

§
09 — Câu chốt hạ

Khi giải thích với
khách hoặc nội bộ

G-AGX là hệ thống telephony core đã được phía đối tác xây dựng hoàn tất.

Bên chúng ta không phát triển phần tổng đài lõi, mà phát triển PC App và backend integration để kết nối với G-AGX.

Kiến trúc tích hợp gồm 2 chiều: một là command/API từ backend của chúng ta sang G-AGX, hai là event realtime từ G-AGX callback về endpoint của chúng ta.

Backend của chúng ta đóng vai trò interface gateway — vừa thực hiện auth và API orchestration, vừa nhận và xử lý call events.
