Thế Giới Network Interface

Từ card mạng vật lý đến virtual NIC, từ Ethernet 10BASE-T đến 100GbE, từ bonding đến VLAN tagging. Tất cả kiến thức bạn cần để làm chủ network interface trong homelab và production — nơi mỗi packet đều đi qua.

⌄

Network Interface là gì thực sự?

Network Interface (NIC — Network Interface Card/Controller) là điểm giao tiếp giữa máy tính và mạng. Nó hoạt động ở Layer 1 (Physical) và Layer 2 (Data Link) của mô hình OSI. Mỗi NIC có một địa chỉ MAC duy nhất (48-bit), đóng vai trò như “số chứng minh thư” của thiết bị trên mạng LAN.

Điểm mấu chốt

Trong Linux, mọi thứ đều là interface. Dù là card mạng vật lý, cầu nối ảo, VLAN sub-interface hay VPN tunnel — kernel đều biểu diễn chúng dưới dạng network interface. Hiểu cách kernel quản lý interface là chìa khóa để làm chủ networking trên Linux. Lệnh ip link show cho bạn thấy tất cả.

OSI Layer 1 & 2 — Nền tảng

Layer 1 (Physical): Tín hiệu điện/quang, tốc độ truyền, loại cáp (Cat5e, Cat6, fiber OM3/OM4), cổng kết nối (RJ45, SFP+, QSFP28). NIC chuyển đổi giữa tín hiệu vật lý và frame số.
Layer 2 (Data Link): Địa chỉ MAC, Ethernet frame (preamble + header + payload + FCS), ARP resolution, VLAN tagging (802.1Q). Đây là nơi NIC “sống” — nó đọc và ghi Ethernet frames.

Homelab Insight

Trong homelab, bạn sẽ gặp cả physical lẫn virtual NIC. Một server Proxmox điển hình có thể có 2 NIC vật lý (enp3s0, enp4s0) nhưng chạy hàng chục veth pairs cho containers, nhiều bridge cho VMs, và VLAN sub-interfaces. Nắm vững cả hai thế giới là bắt buộc.

Dòng Thời Gian Network Interface

Từ sợi cáp đồng trục đầu tiên đến 800GbE và WiFi 7 — hơn 50 năm tiến hóa không ngừng của network interface

1973–1980

Ethernet ra đời

Robert Metcalfe phát minh Ethernet tại Xerox PARC (1973). Cáp đồng trục (coax), tô pô bus, tốc độ 2.94 Mbps. Năm 1980, DEC-Intel-Xerox (DIX) xuất bản chuẩn Ethernet II. Đây là nền tảng của mọi mạng LAN hiện đại.

1983–1990

IEEE 802.3 và 10BASE-T

IEEE chuẩn hóa Ethernet thành 802.3 (1983). 10BASE-T (1990) mang Ethernet lên cáp xoắn đôi RJ45, thay thế cáp đồng trục. Tốc độ 10 Mbps. Star topology với hub trung tâm. NIC trở thành card mở rộng ISA/PCI phổ biến nhất.

1995–1999

Fast Ethernet & Wi-Fi

100BASE-TX (Fast Ethernet, 1995) — tăng 10x tốc độ với cùng cáp Cat5. IEEE 802.11 (1997) — Wi-Fi ra đời với 2 Mbps. 802.11b (1999) đạt 11 Mbps, wireless NIC bắt đầu phổ biến. Switch thay thế hub. Full-duplex trở thành chuẩn.

1999–2006

Gigabit Ethernet & VLAN

1000BASE-T (1999) — 1 Gbps trên cáp Cat5e/Cat6. IEEE 802.1Q (1998) — chuẩn VLAN tagging. Linux bonding module xuất hiện. NIC bắt đầu hỗ trợ offloading (checksum, TSO). Server NIC có nhiều port (dual/quad). Đây là kỷ nguyên “vàng” của enterprise networking.

2007–2016

10GbE, SR-IOV & Virtualization

10GBASE-T (2006) và SFP+ modules. SR-IOV cho phép chia NIC vật lý thành nhiều virtual functions. Open vSwitch (OVS, 2009) cho SDN. Wi-Fi 802.11ac (2013) đạt 1.3 Gbps. Container networking (Docker, 2013) tạo bùng nổ veth pairs và bridge. DPDK (2010) bypass kernel cho throughput cực cao.

2017–nay

25/40/100/400GbE & SmartNIC

25GbE trở thành chuẩn server mới (thay 10G). 100GbE trong datacenter. 400GbE và 800GbE cho spine/core. SmartNIC (DPU) với ARM CPU tích hợp (NVIDIA BlueField, AMD Pensando). Wi-Fi 6E (2020) và Wi-Fi 7 (2024). XDP/eBPF cho programmable networking. Đây là kỷ nguyên của “NIC thông minh”.

Virtual Interfaces (Linux Kernel)

🔄

Loopback (lo)

Virtual / Local / Always Up

Interface đặc biệt luôn tồn tại, địa chỉ 127.0.0.1/8 (và ::1/128 cho IPv6). Traffic gửi đến lo không bao giờ rời khỏi máy. Dùng cho inter-process communication, testing, và các service bind local. Mọi network namespace đều có lo riêng.

Luôn có sẵn 127.0.0.1 Local only

Virtual / Pair / Container Networking

Luôn tạo thành cặp đôi — packet vào đầu này sẽ ra đầu kia. Nền tảng của container networking: một đầu trong container (eth0), đầu kia gắn vào bridge trên host. Docker và Kubernetes dùng veth pairs cho mọi container/pod. Hiểu veth là hiểu 80% container networking.

TAP / TUN

Virtual / Userspace / VPN

TUN (Layer 3): Nhận/gửi IP packets. Dùng cho VPN routing (WireGuard, OpenVPN tun mode). TAP (Layer 2): Nhận/gửi Ethernet frames. Dùng cho VM bridging (QEMU/KVM), OpenVPN tap mode. Cả hai cho phép userspace programs xử lý network traffic — cầu nối giữa kernel networking và ứng dụng.

VPN VM networking TUN = L3 TAP = L2
🌉

Bridge Interface

Virtual / Switch / L2 Forwarding

Software switch trong kernel — kết nối nhiều interface lại như một switch vật lý. Forwarding dựa trên MAC address table. Nền tảng của VM/container networking: vmbr0 trong Proxmox, docker0 trong Docker, br0 cho KVM. Hỗ trợ STP, VLAN filtering, và nhiều tính năng switch.

VM/Container cơ bản VLAN-aware L2 switch STP support
🤝

Bond Interface

Virtual / Aggregation / Redundancy

Gộp nhiều NIC vật lý thành một interface logic. Mục đích: tăng bandwidth (link aggregation) hoặc tăng độ tin cậy (failover). 7 mode khác nhau (balance-rr, active-backup, balance-xor, broadcast, 802.3ad, balance-tlb, balance-alb). 802.3ad (LACP) là chuẩn enterprise. Chi tiết ở phần Bonding bên dưới.

Redundancy Bandwidth x2+ 7 modes Cần switch hỗ trợ (LACP)

VLAN Sub-Interface

Virtual / Tagged / 802.1Q

Interface con của physical/bridge interface, thêm VLAN tag (802.1Q) vào mỗi frame. Tên dạng: eth0.10, enp3s0.100, vmbr0v20. Cho phép một NIC vật lý tham gia nhiều VLAN khác nhau. Trong Proxmox VLAN-aware bridge, config đơn giản hơn nhiều. Chi tiết ở phần VLAN.

macvlan / ipvlan

Virtual / Sub-Interface / Container

macvlan: Tạo interface con với MAC address riêng. Container/VM nhìn như thiết bị thật trên mạng (có IP riêng, MAC riêng). 4 mode: bridge, vepa, private, passthru. ipvlan: Tương tự nhưng dùng chung MAC, khác IP. Tốt hơn khi switch giới hạn số MAC. Nhẹ hơn bridge cho container networking.

Bonding / Teaming

Gộp nhiều NIC thành một — 7 mode với đặc điểm hoàn toàn khác nhau. Chọn sai mode = không đạt được gì

Mode 0: balance-rr

Round-Robin

Gửi packet luân phiên qua từng slave. Tăng bandwidth thực sự cho 1 connection. Nhưng dễ gây out-of-order packets (vì đi nhiều đường khác nhau). Cần switch hỗ trợ EtherChannel/port-channel. Ít dùng trong production vì TCP không thích out-of-order.

Tăng BW thực Out-of-order Cần switch config

Mode 1: active-backup

Failover

Chỉ 1 slave active, các slave còn lại standby. Khi active chết, backup lên thay. Không tăng bandwidth nhưng độ tin cậy cao nhất. Không cần switch hỗ trợ gì đặc biệt. Lựa chọn số 1 cho homelab và production đơn giản.

Không cần switch đb Độ tin cậy cao Không tăng BW

Mode 2: balance-xor

XOR Hash

Dùng hash (src MAC XOR dst MAC) để chọn slave. Mỗi “flow” luôn đi qua cùng slave → không out-of-order. Tăng aggregate bandwidth (nhiều flow khác nhau đi nhiều đường). Cần switch hỗ trợ. Tương tự LACP nhưng không có protocol negotiation.

Không out-of-order Hash-based Cần switch config

Mode 3: broadcast

Broadcast

Gửi mỗi packet trên TẤT CẢ slaves. Fault tolerance tối đa nhưng không tăng bandwidth (thực ra giảm vì dupplicate). Use case cực kỳ hạn chế — chỉ khi cần đảm bảo packet đến được đích bằng mọi giá.

Max fault tolerance Lãng phí BW Rất ít dùng

Mode 4: 802.3ad (LACP)

Link Aggregation Control Protocol

Chuẩn vàng của bonding. Cả hai bên (host và switch) đều chạy LACP protocol để tự động negotiation. Tăng bandwidth + fault tolerance. Hash-based distribution (L2, L3, L3+4). Cần switch hỗ trợ LACP (hầu hết managed switch). Nếu có managed switch, dùng mode này.

Chuẩn enterprise Auto negotiation BW + Failover Cần managed switch

Mode 5: balance-tlb

Adaptive Transmit Load Balancing

Cân bằng tải outgoing traffic dựa trên tải của từng slave. Incoming vẫn đi qua 1 slave (active). Không cần switch hỗ trợ — chỉ cần driver NIC hỗ trợ thay đổi MAC. Tăng outgoing bandwidth mà không cần config switch. Lý tưởng cho homelab với unmanaged switch.

Không cần switch đb Tăng outgoing BW Incoming không balance

Mode 6: balance-alb

Adaptive Load Balancing

Bao gồm balance-tlb + cân bằng cả incoming (bằng ARP negotiation). Không cần switch hỗ trợ gì. Tăng cả incoming lẫn outgoing bandwidth. Lý tưởng cho homelab không có managed switch. Nhược điểm: chỉ hoạt động với IPv4 (ARP-based). Best choice cho homelab unmanaged.

Không cần switch đb Tăng cả 2 chiều IPv4 only (ALB) Best cho homelab

LACP Deep Dive

802.3ad LACP hoạt động bằng LACPDU (LACP Data Unit) — các gói tin trao đổi mỗi 1 giây (fast) hoặc 30 giây (slow). Cả switch và host đều thống nhất: port nào thuộc group nào, trạng thái active/standby. Hash policy quyết định cách phân bổ traffic: xmit_hash_policy=layer3+4 (hash src/dst IP + port) cho phân bổ tốt nhất. Lưu ý: 1 TCP connection vẫn chỉ đi qua 1 link — LACP tăng aggregate bandwidth, không tăng single-flow bandwidth.

Homelab Insight: Chọn bonding mode

Có managed switch (MikroTik, UniFi Pro, Cisco)? → Mode 4 (LACP). Chỉ có unmanaged switch? → Mode 6 (balance-alb) hoặc Mode 1 (active-backup). Chỉ cần failover? → Mode 1. Proxmox cluster? → LACP cho storage network, active-backup cho management. Đừng phức tạp hóa nếu không cần thiết.

VLAN — Phân Đoạn Mạng

IEEE 802.1Q — cách chia một mạng vật lý thành nhiều mạng logic, cô lập traffic, tăng bảo mật

802.1Q VLAN Tagging

VLAN tag là 4 bytes được chèn vào Ethernet frame header: TPID (0x8100, 2 bytes) + TCI (2 bytes chứa Priority, DEI, và VLAN ID 12-bit = 4094 VLANs tối đa). Frame có tag gọi là tagged frame, không có tag là untagged frame.

Cấu trúc Ethernet Frame với 802.1Q Tag
  +----------+----------+------+------+------+---------+-----+
  | Dst MAC  | Src MAC  | TPID | PCP  | VLAN | EthType | ... |
  | 6 bytes  | 6 bytes  |0x8100| 3bit |  ID  | 2 bytes |Payload
  |          |          |      | +DEI | 12bit|         |     |
  +----------+----------+------+------+------+---------+-----+
                         |--- 802.1Q Tag (4 bytes) ---|

  Trunk port:  Gửi/nhận tagged frames (nhiều VLAN trên 1 link)
  Access port: Strip tag khi ra, thêm tag khi vào (1 VLAN)
  Native VLAN: VLAN mặc định cho untagged traffic trên trunk

VLAN trong Linux

Tạo VLAN sub-interface: ip link add link eth0 name eth0.100 type vlan id 100. Kernel sẽ tự động thêm/gỡ 802.1Q tag. Mỗi VLAN sub-interface hoạt động như một interface độc lập — có IP riêng, routing riêng. Cần NIC parent ở chế độ trunk (nhận tagged frames từ switch).

Proxmox VLAN-Aware Bridge

Proxmox hỗ trợ VLAN-aware bridge — thay vì tạo nhiều bridge/sub-interface, chỉ cần 1 bridge với bridge-vlan-aware yes. VM/CT chỉ cần chỉ định VLAN tag trong config. Đơn giản hơn, sạch hơn. Config trong /etc/network/interfaces:
auto vmbr0
iface vmbr0 inet static
bridge-ports enp3s0
bridge-stp off
bridge-vlan-aware yes
bridge-vids 2-4094

VLAN Use Cases trong Homelab

VLAN 10: Management

Proxmox host, router admin, switch admin, iDRAC/IPMI. Chỉ các thiết bị quản trị mới vào được. Firewall chặn tất cả từ VLAN khác. Đây là VLAN quan trọng nhất — mất nó là mất quyền kiểm soát.

VMs và containers chạy services (web, database, media). Cho phép giao tiếp nội bộ nhưng firewall kiểm soát truy cập từ ngoài. Tách biệt khỏi user network để hạn chế attack surface.

Bridging — Software Switch

Từ Linux bridge đơn giản đến Open vSwitch — nền tảng của virtualization và container networking

Linux Bridge (brctl/bridge)

Kernel module, đơn giản, ổn định. Học MAC addresses tự động. Hỗ trợ STP, VLAN filtering. Mặc định trong Proxmox (vmbr0), Docker (docker0). Đủ cho 99% homelab.

Đơn giản Ổn định Mặc định

Open vSwitch (OVS)

Software switch mạnh mẽ, hỗ trợ OpenFlow, tunneling (VXLAN, GRE, Geneve), QoS, mirroring. Dùng trong OpenStack, Kubernetes (OVN). Phức tạp hơn Linux bridge nhiều.

SDN/OpenFlow Tunneling Phức tạp

macvlan (Bridge mode)

Tạo sub-interface với MAC riêng trực tiếp trên physical NIC. Không cần bridge, performance tốt hơn. Nhưng parent không giao tiếp được với child. Dùng cho Docker macvlan networks.

Hiệu suất cao MAC riêng Parent isolation

Linux Bridge vs OVS — Khi nào chọn gì?

Linux Bridge: Homelab, Proxmox, đơn giản, ít config, ổn định. OVS: Khi cần SDN, overlay networking (VXLAN), OpenFlow rules, hoặc dùng với OpenStack/OVN. Rule of thumb: Nếu không biết cần OVS để làm gì, bạn không cần nó. Linux bridge + VLAN-aware là đủ cho 99% trường hợp homelab.

Network Namespaces

Cô lập hoàn toàn network stack — nền tảng của container networking, mỗi namespace là một “máy tính mạng” riêng

Namespace = Isolation

Mỗi network namespace có riêng: interfaces, routing table, iptables rules, ARP table, /proc/net. Mặc định, tất cả process chạy trong namespace “default”. Docker/Kubernetes tạo namespace riêng cho mỗi container/pod. Và veth pair là “ống dẫn” nối giữa các namespace.

Container Networking Internals — Điều gì xảy ra bên trong Docker?
  Host Namespace (default)          Container Namespace
  +-----------------------+          +---------------------+
  | eth0 (physical NIC)   |          | eth0 (veth-inside)  |
  | IP: 192.168.1.100     |          | IP: 172.17.0.2      |
  |                       |          |                     |
  | docker0 (bridge)      |          | lo (127.0.0.1)      |
  | IP: 172.17.0.1        |          |                     |
  |   |                   |          | routing table riêng  |
  |   +-- veth-outside ---+----- veth pair -----+  iptables riêng    |
  |                       |          |                     |
  | iptables NAT          |          +---------------------+
  | (MASQUERADE)          |
  +-----------------------+

  Container gửi packet ra internet:
  eth0 (container) → veth pair → docker0 bridge → iptables NAT → eth0 (host) → Internet

Hands-on: Tạo namespace thủ công

Bạn có thể tự tạo namespace và kết nối chúng — đây chính xác là những gì Docker làm bên trong:
ip netns add ns1 — tạo namespace
ip link add veth0 type veth peer name veth1 — tạo veth pair
ip link set veth1 netns ns1 — chuyển 1 đầu vào namespace
ip netns exec ns1 ip addr add 10.0.0.2/24 dev veth1 — cấu hình IP
ip netns exec ns1 ip link set veth1 up — bật interface
Hiểu flow này là hiểu 80% container networking.

Sai lầm phổ biến

“Chỉ cần 1 IP trên mỗi interface”

Sai! Một interface có thể có nhiều IP (secondary addresses). Rất hữu ích cho: multi-homing, migration, service binding.

Thực tế

Multiple IPs trên 1 NIC

ip addr add 10.0.0.1/24 dev eth0
Cả 3 IP đều hoạt động đồng thời. Không cần “eth0:1” kiểu cũ.

DHCP vs Static — Khi nào dùng gì?

DHCP: Client devices, laptops, IoT. Tiện lợi, tự động. Static: Servers, hypervisors, routers, switches, DNS/DHCP server. Bắt buộc cho infrastructure. DHCP Reservation: Giải pháp trung gian — device luôn nhận cùng IP từ DHCP server dựa trên MAC address. Tốt cho homelab devices cần IP cố định nhưng không muốn config static.

Persistent Configuration

Lệnh ip chỉ là runtime — mất khi reboot. Để persistent:
Debian/Ubuntu: /etc/network/interfaces hoặc Netplan (/etc/netplan/*.yaml)

Luôn test bằng lệnh ip trước, rồi mới ghi vào config file.

Performance Tuning

Tối ưu hóa NIC — từ ring buffer, offloading, RSS/RPS đến Jumbo frames. Biến 10G thành thật sự 10G

📦

Ring Buffers

NIC Hardware / Packet Queue

NIC lưu packets vào ring buffer (hàng đợi vòng) trước khi kernel xử lý. Nếu kernel chậm hơn NIC → buffer đầy → packet drop. Tăng ring buffer size giảm drop nhưng tăng latency. Kiểm tra: ethtool -g eth0. Set: ethtool -G eth0 rx 4096 tx 4096.

Giảm packet drop Tăng latency nếu quá lớn
⚙

Offloading (TSO/GSO/GRO)

NIC/Kernel / CPU Reduction

TSO (TCP Segmentation Offload): NIC tự chia packet lớn thành MTU-sized segments, giảm tải CPU. GSO (Generic Segmentation Offload): Tương tự TSO nhưng trong kernel. GRO (Generic Receive Offload): Gộp nhiều small packets thành 1 lớn trước khi đưa lên stack. Checksum offload: NIC tính checksum thay CPU. Kiểm tra: ethtool -k eth0.

Giảm CPU usage Tăng throughput Tắt nếu debug packet
🔀

RSS / RPS / RFS

Multi-Queue / CPU Distribution

RSS (Receive Side Scaling): NIC phân phối packets vào nhiều RX queues → nhiều CPU cores xử lý song song. Cần NIC hỗ trợ (multi-queue). RPS (Receive Packet Steering): Software RSS — khi NIC không hỗ trợ multi-queue. RFS (Receive Flow Steering): Gửi packet đến CPU core đang chạy app xử lý → giảm cache miss.

Multi-core scaling RSS = hardware RPS = software
📏

MTU / Jumbo Frames

Frame Size / Efficiency

MTU mặc định: 1500 bytes. Jumbo frames: 9000 bytes. Giảm overhead (header ratio nhỏ hơn), giảm số packets, giảm interrupts. Lý tưởng cho storage network (iSCSI, NFS), Ceph. Bắt buộc TẤT CẢ thiết bị trên path đều hỗ trợ (NIC, switch, router). Set: ip link set eth0 mtu 9000.

Tăng 10-20% throughput Giảm CPU Tất cả phải hỗ trợ Không dùng cho internet
⚡

Interrupt Coalescing

NIC / Latency vs Throughput

Mỗi packet đến, NIC tạo 1 interrupt → nhiều packets = nhiều interrupts = CPU bận. Coalescing: NIC chờ gộp nhiều packets rồi mới tạo 1 interrupt. Giảm CPU nhưng tăng latency. Cho throughput workload (storage, backup): tăng coalescing. Cho latency workload (VoIP, trading): giảm coalescing. Set: ethtool -C eth0 rx-usecs 100.

Giảm interrupts Tăng latency Trade-off
🚀

XDP / DPDK / eBPF

Kernel Bypass / Programmable

XDP (eXpress Data Path): Xử lý packet ngay tại driver level, trước khi vào kernel stack. Cực nhanh (millions pps). DPDK: Bypass kernel hoàn toàn, userspace networking. Dùng cho router, firewall, load balancer hiệu suất cao. eBPF: Programmable hook trong kernel. Dùng bởi Cilium (Kubernetes CNI).

Millions pps XDP = in-kernel DPDK = userspace Phức tạp

Homelab Insight: Performance Tuning Checklist

1. Kiểm tra link speed: ethtool eth0 | grep Speed
2. Kiểm tra drops: ethtool -S eth0 | grep -i drop
3. Tăng ring buffer nếu có drops: ethtool -G eth0 rx 4096
4. Bật Jumbo frames cho storage VLAN: ip link set eth0 mtu 9000
5. Kiểm tra offloading: ethtool -k eth0 (TSO, GRO nên on)
6. Kiểm tra multi-queue: ethtool -l eth0
Làm từng bước, đo lường sau mỗi thay đổi với iperf3.

Troubleshooting

Các công cụ và quy trình debug network interface — từ Layer 1 lên Layer 4

1

Layer 1: Link vật lý

ethtool eth0: Kiểm tra speed, duplex, link detected. Nếu “Link detected: no” → kiểm tra cáp, SFP module, switch port. ip link show: state UP/DOWN. dmesg | grep eth: Lỗi driver, firmware. NIC LED: xanh = link, nhấp nháy = traffic, tắt = không link.

2

Layer 2: MAC & Switch

ip neigh show: ARP table — có thấy MAC của gateway không? Nếu INCOMPLETE/FAILED → L2 có vấn đề. bridge fdb show: MAC table của bridge. arping: Test ARP level. Kiểm tra VLAN tag đúng không, trunk port đúng không.

3

Layer 3: IP & Routing

ip addr show: IP đúng không, subnet mask đúng không? ip route show: Default gateway có không? Route đến đích đúng interface không? ping gateway: Reach được gateway không? traceroute: Packet đi đường nào, kẹt ở đâu?

4

Layer 4: Port & Connection

ss -tlnp: Service có đang listen không? Trên port nào? Bind 0.0.0.0 hay 127.0.0.1? ss -tnp: Kết nối đang active. nft list ruleset / iptables -L: Firewall có đang chặn không? tcpdump -i eth0 port 80: Xem thực tế packet có đến không.

5

NIC Statistics

ethtool -S eth0: Thống kê chi tiết từ NIC — rx/tx errors, drops, CRC errors, collisions. ip -s link show eth0: Thống kê từ kernel. cat /proc/interrupts | grep eth: Interrupt distribution across CPUs. Số liệu này cho biết NIC có đang “khỏe” không.

6

tcpdump — Vũ khí cuối cùng

Khi tất cả các công cụ khác không cho đủ thông tin, dùng tcpdump. tcpdump -i eth0 -nn: Xem tất cả traffic. tcpdump -i eth0 host 10.0.0.1: Lọc theo IP. tcpdump -w capture.pcap: Lưu ra file, mở bằng Wireshark để phân tích sâu. Nếu packet đến NIC nhưng app không thấy → firewall/routing. Nếu packet không đến NIC → vấn đề L1/L2/switch.

Common Issues trong Homelab

1. “NIC không nhận”: Kiểm tra driver (lspci -v | grep -i net), cài firmware (apt install firmware-linux-nonfree).
2. “Link 1G thay vì 10G”: Cáp Cat5e chỉ hỗ trợ 1G. Cần Cat6a cho 10GBASE-T. Hoặc SFP+ module không tương thích.
3. “VLAN không thông”: Switch port chưa config trunk, hoặc chưa cho phép VLAN ID đó.
4. “Bond không tăng speed”: LACP chưa config trên switch, hoặc chỉ test 1 flow (1 flow = 1 link).
5. “Container không ra internet”: iptables MASQUERADE chưa bật, hoặc ip_forward = 0.

Cheat Sheet

Copy-paste commands cho mọi tình huống — từ cơ bản đến nâng cao

Xem tất cả interfaces

Liệt kê và kiểm tra trạng thái

# Tất cả interfaces ip link show ip -br link show # Chi tiết với IP ip -br addr show # Chỉ UP interfaces ip link show up # Interface cụ thể ip addr show dev eth0 # Thống kê traffic ip -s link show eth0

Cấu hình IP

Thêm/xóa IP, bật/tắt interface

# Bật interface ip link set eth0 up # Thêm IP ip addr add 192.168.1.100/24 dev eth0 # Xóa IP ip addr del 192.168.1.100/24 dev eth0 # Thêm default gateway ip route add default via 192.168.1.1 # Thay đổi MTU ip link set eth0 mtu 9000 # Thay đổi MAC ip link set eth0 address aa:bb:cc:dd:ee:ff

Tạo VLAN interface

802.1Q sub-interface

# Tạo VLAN 100 trên eth0 ip link add link eth0 \ name eth0.100 type vlan id 100 ip addr add 10.100.0.1/24 dev eth0.100 ip link set eth0.100 up # Xem VLAN info cat /proc/net/vlan/eth0.100 # Xóa VLAN interface ip link del eth0.100

Tạo Bridge

Software switch cho VM/CT

# Tạo bridge ip link add br0 type bridge ip link set br0 up # Thêm interface vào bridge ip link set eth0 master br0 ip link set veth0 master br0 # VLAN-aware bridge ip link add br0 type bridge \ vlan_filtering 1 # Gán IP cho bridge ip addr add 192.168.1.1/24 dev br0 # Xem bridge members bridge link show

Tạo Bond (LACP)

Link aggregation với 802.3ad

# Tạo bond ip link add bond0 type bond \ mode 802.3ad # Thêm slaves ip link set eth0 down ip link set eth0 master bond0 ip link set eth1 down ip link set eth1 master bond0 # Bật bond ip link set bond0 up ip addr add 192.168.1.100/24 dev bond0 # Xem trạng thái bond cat /proc/net/bonding/bond0

Tạo veth pair + namespace

Container networking thủ công

# Tạo namespace ip netns add myns # Tạo veth pair ip link add veth0 type veth \ peer name veth1 # Chuyển 1 đầu vào namespace ip link set veth1 netns myns # Config trong namespace ip netns exec myns \ ip addr add 10.0.0.2/24 dev veth1 ip netns exec myns \ ip link set veth1 up ip netns exec myns \ ip link set lo up # Config đầu host ip addr add 10.0.0.1/24 dev veth0 ip link set veth0 up

Kiểm tra sức khỏe NIC

# Thông tin cơ bản ethtool eth0 # Thống kê chi tiết ethtool -S eth0 # Ring buffer ethtool -g eth0 # Offload settings ethtool -k eth0 # Driver info ethtool -i eth0 # Nhấp nháy LED (tìm port) ethtool -p eth0 5 # Test NIC ethtool -t eth0

Debug mạng ở mọi level
