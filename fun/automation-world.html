Infrastructure as Code là gì thực sự?

Infrastructure as Code (IaC) - Nôm na là bạn mô tả toàn bộ hạ tầng bằng code, thay vì click chuột hay gõ lệnh thủ công. Server, network, firewall, DNS, user - tất cả đều được định nghĩa trong file text, lưu trữ trong Git, và có thể tái tạo bất cứ lúc nào.

Insight quan trọng

IaC không chỉ là "viết script để cài server". Nó là một triết lý vận hành - mọi; thay đổi đều được review, version, test, và rollback. Hãy nghĩ nó như Git cho hạ tầng - tương tự cách developer quản lý source code, SysAdmin/DevOps quản lý infrastructure bằng code. Không còn "ờ, hình như hôm trước tôi s𞻚 gì đó trên server này..."

Ba trụ cột của IaC

📝 Declarative

Bạn mô tả trạng thái mong muốn (desired state), công cụ tự tìm cách đạt được. Ví dụ: "Tôi muốn 3 VM chạy Ubuntu 22.04" - Terraform sẽ tự tạo đúng 3 con.

🔄 Idempotent

Chạy 1 lần hay 100 lần, kết quả như nhau. Không sợ "chạy lại sẽ hỏng". Đây là điểm khác biệt lớn nhất so với shell script thường.

🤝 Version Control

Mọi; thay đổi đều đi qua Git. Ai s𞻚? S𞻚 gì? Lúc nào? Rollback về phiên bản trước chỉ cần git revert.

Dòng Thời Gian Automation

30 năm phát triển - từ shell script viết tay đến GitOps tự động hoàn toàn

1990s

Kỷ nguyên Shell Scripts

Mọi; SysAdmin đều có bộ sưu tập; shell script riêng. Bash, Perl, expect - mỗi người một kiểu. Không ai hiểu script của người khác. "Chạy trên máy tôi được mà" là câu nói kinh điển.

1993

CFEngine - Đi Trước Thời Đại

Mark Burgess tạo ra CFEngine tại Đại học Oslo. Đây là công cụ Configuration Management đầu tiên của nhân loại. Khái niệm "desired state" và "convergence" ra đời từ đây. Quá đi trước thời đại nên ít người hiểu.

2005

Puppet - Configuration Management cho mọi;i người

Luke Kanies tạo Puppet với ngôn ngữ DSL riêng. Agent-based, master-slave architecture. Lần đầu tiên CM trở nên dễ tiếp cận;. Google, Twitter, Spotify đều dùng.

2009

Chef - Ruby cho Infra

Adam Jacob mang Ruby vào thế giới infra. "Recipes" và "Cookbooks" - đặt tên theo ẩm thực. Mạnh mẽ nhưng learning curve dốc. Facebook dùng Chef quản lý hàng triệu server.

2012

Ansible - Cách Mạng Agentless

Michael DeHaan tạo Ansible. Không cần agent, chỉ cần SSH. YAML đọc hiểu như tiếng Anh. Red Hat mua năm 2015 với giá $150M. Trở thành công cụ CM phổ biến nhất thế giới. Đây là "iPhone moment" của Configuration Management.

2014

Terraform - Infrastructure Provisioning

HashiCorp ra mắt Terraform với ngôn ngữ HCL. Lần đầu tiên, bạn có thể mô tả toàn bộ infra đa cloud trong một file. Plan → Apply → Destroy. State file quản lý trạng thái. Thế giới IaC thay đổi mãi mãi.

2018-nay

GitOps - Git là nguồn sự thật; duy nhất

Weaveworks định nghĩa GitOps. ArgoCD, FluxCD ra đời. Mọi; thay đổi infra đều qua Pull Request. Cluster tự đồng đồng bộ với Git. Không ai SSH vào server nữa. Đây là đỉnh cao của IaC.

📖 Playbook

File YAML mô tả các task cần thực hiện. Mỗi play nhắm vào một nhóm host. Mỗi task gọi một module.

# playbook.yml
- name: Cấu hình web server
  hosts: containers
  become: yes
  tasks:
    - name: Cài Nginx
      apt:
        name: nginx
        state: present
        update_cache: yes

    - name: Copy config
      template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify: Restart Nginx

  handlers:
    - name: Restart Nginx
      service:
        name: nginx
        state: restarted

📁 Roles

Cấu trúc thư mục chuẩn để tái sử dụng code. Một role = một chức năng (webserver, database, monitoring...). Chia sẻ qua Ansible Galaxy.

roles/
  webserver/
    tasks/
      main.yml        # Tasks chính
    handlers/
      main.yml        # Handlers
    templates/
      nginx.conf.j2   # Jinja2 templates
    files/
      ssl-cert.pem    # File tĩnh
    vars/
      main.yml        # Biến
    defaults/
      main.yml        # Giá trị mặc định

🔐 Ansible Vault

Mã hóa dữ liệu nhạy cảm (passwords, API keys, certificates). Encrypt cả file hoặc từng biến. Không bao giờ commit mật; khẩu dạng plain text!

# Mã hóa file
ansible-vault encrypt secrets.yml

# Sửa file đã mã hóa
ansible-vault edit secrets.yml

# Chạy playbook với vault
ansible-playbook site.yml \
  --ask-vault-pass

# Dùng vault password file
ansible-playbook site.yml \
  --vault-password-file ~/.vault_pass

Infrastructure Provisioning

Tạo và quản lý tài nguyên hạ tầng - VM, network, storage, DNS - bằng code

Configuration Management vs Provisioning

CM (Ansible, Puppet) trả lời: "Server đã có, cấu hình nó thế nào?"
Provisioning (Terraform, Pulumi) trả lời: "Tạo server ở đâu, bao nhiêu, loại gì?"
Thực tế, chúng bổ sung nhau: Terraform tạo VM → Ansible cấu hình VM.

ArgoCD vs FluxCD - Chọn cái nào?

ArgoCD nếu bạn cần UI đẹp, multi-cluster, và team lớn cần visibility. FluxCD nếu bạn thích CLI, muốn nhẹ nhàng, và đã dùng Helm/Kustomize nhiều. Cho homelab? Cả hai đều tuyệt vời, nhưng ArgoCD có UI sẽ vui hơn khi khám phá.

Ansible cho Proxmox Homelab

Thực chiến: tự động hóa cụm Proxmox "Mimosa-Offices" bằng Ansible

Tại sao Ansible + Proxmox?

Proxmox có API REST hoàn chỉnh. Ansible có module community.general.proxmox và community.general.proxmox_kvm. Kết hợp lại: bạn có thể tạo VM, container, cấu hình network, quản lý storage - tất cả bằng YAML. Không cần mở WebUI nữa.

Ví dụ 1: Tạo LXC Container trên Proxmox
# create-lxc.yml
- name: Tạo LXC containers cho homelab
  hosts: localhost
  gather_facts: false
  vars:
    proxmox_host: "pve1.mimosa.lab"
    proxmox_user: "root@pam"
    api_token_id: "ansible"
    api_token_secret: "{{ vault_proxmox_token }}"

  tasks:
    - name: Tạo Docker LXC
      community.general.proxmox:
        vmid: 200
        hostname: "docker01"
        node: "pve1"
        api_host: "{{ proxmox_host }}"
        api_user: "{{ proxmox_user }}"
        api_token_id: "{{ api_token_id }}"
        api_token_secret: "{{ api_token_secret }}"
        ostemplate: "local:vztmpl/debian-12-standard_12.2-1_amd64.tar.zst"
        storage: "local-zfs"
        disk: "50"
        cores: 4
        memory: 4096
        swap: 512
        netif: '{"net0":"name=eth0,bridge=vmbr0,ip=192.168.1.50/24,gw=192.168.1.1"}'
        features:
          - nesting=1
          - keyctl=1
        state: present
      register: lxc_result

    - name: Khởi động container
      community.general.proxmox:
        vmid: 200
        api_host: "{{ proxmox_host }}"
        api_user: "{{ proxmox_user }}"
        api_token_id: "{{ api_token_id }}"
        api_token_secret: "{{ api_token_secret }}"
        state: started
Ví dụ 2: Cấu hình tất cả Proxmox nodes
# configure-pve-nodes.yml
- name: Cấu hình cụm Proxmox Mimosa-Offices
  hosts: proxmox
  become: yes
  tasks:
    - name: Xóa enterprise repo (không có subscription)
      file:
        path: /etc/apt/sources.list.d/pve-enterprise.list
        state: absent

    - name: Thêm no-subscription repo
      apt_repository:
        repo: "deb http://download.proxmox.com/debian/pve bookworm pve-no-subscription"
        filename: pve-no-subscription
        state: present

    - name: Cập; nhật; hệ thống
      apt:
        update_cache: yes
        upgrade: dist
      register: upgrade_result

    - name: Cài các gói cần thiết
      apt:
        name:
          - vim
          - htop
          - iotop
          - tmux
          - curl
          - wget
          - net-tools
          - lm-sensors
        state: present

    - name: Cấu hình NTP
      template:
        src: templates/timesyncd.conf.j2
        dest: /etc/systemd/timesyncd.conf
      notify: Restart timesyncd

    - name: Tạo API token cho Terraform/Ansible
      shell: |
        pveum user token add root@pam ansible \
          --privsep 0 --expire 0 2>/dev/null || true
      changed_when: false

  handlers:
    - name: Restart timesyncd
      service:
        name: systemd-timesyncd
        state: restarted
Ví dụ 3: Tự động backup và monitoring
# backup-and-monitor.yml
- name: Cấu hình backup và monitoring cho homelab
  hosts: proxmox
  become: yes
  tasks:
    - name: Tạo schedule backup hàng đêm
      cron:
        name: "Backup all VMs"
        hour: "2"
        minute: "0"
        job: "vzdump --all --mode snapshot --storage backup-nfs --compress zstd --mailnotification always --mailto admin@mimosa.lab"

    - name: Cài Prometheus node exporter
      apt:
        name: prometheus-node-exporter
        state: present

    - name: Bận; node exporter service
      service:
        name: prometheus-node-exporter
        state: started
        enabled: yes

    - name: Tạo script kiểm tra cluster health
      copy:
        dest: /usr/local/bin/check-cluster.sh
        mode: "0755"
        content: |
          #!/bin/bash
          # Kiểm tra quorum
          pvecm status | grep -q "Quorate:.*Yes"
          if [ $? -ne 0 ]; then
            echo "CRITICAL: Cluster lost quorum!"
            exit 2
          fi
          echo "OK: Cluster healthy"
          exit 0

    - name: Schedule health check mỗi 5 phút
      cron:
        name: "Cluster health check"
        minute: "*/5"
        job: "/usr/local/bin/check-cluster.sh >> /var/log/cluster-health.log 2>&1"

Pro tip cho Proxmox + Ansible

Khi dùng Ansible với Proxmox, hãy tạo API token thay vì dùng password. Token không hết hạn, có thể giới hạn quyền, và không cần lưu mật khẩu root. Lưu token trong Ansible Vault, không bao giờ commit dạng plain text. Cấu trúc thư mục gợi ý: homelab-ansible/ với roles riêng cho proxmox, docker, monitoring, backup.

Những Hiểu Nhầm Phổ Biến

Ít nhất 80% người mới học IaC sẽ mắc phải những sai lầm này

Sai

"IaC chỉ dành cho cloud lớn, homelab không cần"

Nhiều người nghĩ IaC chỉ cần khi có hàng trăm server trên AWS/Azure.

Đúng

Homelab càng cần IaC! Vì bạn là người duy nhất quản lý, nếu quên cấu hình gì đó sau 6 tháng, không ai nhớ giúp. IaC là "tài liệu sống" của hạ tầng. Với 5 node Proxmox, Ansible tiết kiệm hàng chục giờ mỗi tháng.

Sai

"Ansible và Terraform làm cùng một việc"

Rất nhiều người hỏi: "Nên dùng Ansible hay Terraform?" như thể chỉ cần chọn một.

Đúng

Chúng làm hai việc khác nhau. Terraform tạo hạ tầng (VM, network, storage). Ansible cấu hình hạ tầng (cài phần mềm, s𞻚 config file). Workflow lý tưởng: Terraform tạo VM → cloud-init bootstrap → Ansible cấu hình chi tiết.

Sai

"Viết shell script cũng là IaC"

Tôi đã có 200 dòng bash script cài đặt server, vậy là IaC rồi chứ?

Đúng

Shell script không idempotent - chạy lần 2 có thể hỏng. Không có error handling tử tế. Không có state tracking. Khó test. IaC tools đã giải quyết hết những vấn đề này. Shell script có vị trí riêng (glue code), nhưng không nên là công cụ chính để quản lý infra.

Sai

"IaC mất nhiều thời gian hơn làm tay"

Click vài cái là xong, viết YAML mất cả tiếng đồng hồ!

Đúng

Lần đầu viết IaC: chậm hơn. Lần thứ 2, 3, 10: nhanh hơn gấp nhiều lần. Và quan trọng hơn: không bao giờ sai. Rebuild toàn bộ homelab sau disaster? 30 phút với IaC vs 2 ngày làm tay. Đầu tư ban đầu, hưởng lợi mãi mãi.

Sai

"Terraform state file không quan trọng"

Nhiều người commit state file vào Git hoặc thậm chí xóa nó đi.

Đúng

State file là "bộ nhớ" của Terraform. Mất nó = Terraform không biết gì đã tồn tại, sẽ tạo lại tất cả hoặc xung đột. Không bao giờ commit state vào Git (chứa secrets!). Dùng remote backend: S3, Consul, hoặc đơn giản là một NFS share trên homelab.

Sai

"GitOps thì phải có Kubernetes"

Tôi không dùng K8s nên GitOps không liên quan đến tôi.

Đúng

GitOps là triết lý, không phải công cụ. Bạn hoàn toàn có thể áp dụng GitOps với Ansible + Git + CI/CD: push thay đổi vào repo → CI chạy ansible-playbook tự động → infra được cập; nhật;. ArgoCD/FluxCD chỉ là implementation cho K8s, không phải định nghĩa của GitOps.

Sai

"Ansible chậm, không dùng cho production"

SSH vào từng server một thì chậm chết.

Đúng

Ansible có forks (song song nhiều host), pipelining, async tasks, và Mitogen (plugin tăng tốc 2-7x). Với 5-50 host của homelab, Ansible chạy trong vài giây. NASA, Red Hat, hàng nghìn enterprise dùng Ansible trong production hàng ngày.

Sai

"Cứ dùng công cụ mới nhất là tốt nhất"

Pulumi mới hơn Terraform, chắc tốt hơn. Crossplane còn mới hơn nữa!

Đúng

Công cụ tốt nhất là công cụ bạn thành thạo và cộng đồng hỗ trợ mạnh. Terraform + Ansible đã được chứng minh hàng triệu lần. Trước khi chạy theo công cụ mới, hãy hỏi: "Nó giải quyết vấn đề gì mà công cụ hiện tại không làm được?"

Cheat Sheet

Những lệnh cần nhớ nhất - Ansible và Terraform

Ansible Commands

Kiểm tra kết nối

Ping tất cả host trong inventory

ansible all -m ping -i inventory.yml

Chạy Playbook

Thực thi playbook với vault password

ansible-playbook site.yml \ -i inventory.yml \ --ask-vault-pass

Dry Run (Check Mode)

Xem sẽ thay đổi gì mà không thực thi

ansible-playbook site.yml --check --diff

Chỉ chạy trên 1 host

Giới hạn playbook cho host cụ thể

ansible-playbook site.yml \ --limit pve1.mimosa.lab

Chạy Ad-hoc Command

Thực thi lệnh nhanh không cần playbook

ansible proxmox -m shell \ -a "uptime" -i inventory.yml

Xem biến của host

Liệt kê tất cả facts của một host

ansible pve1.mimosa.lab -m setup \ -i inventory.yml

Mã hóa file Vault

Encrypt file chứa secrets

ansible-vault encrypt secrets.yml ansible-vault decrypt secrets.yml ansible-vault edit secrets.yml ansible-vault view secrets.yml

Cài Role từ Galaxy

Tải và cài đặt role từ Ansible Galaxy

ansible-galaxy install geerlingguy.docker ansible-galaxy collection install \ community.general

Liệt kê Tasks

Xem danh sách tasks mà không chạy

ansible-playbook site.yml --list-tasks ansible-playbook site.yml --list-hosts

Terraform Commands

Khởi tạo Project

Tải providers và khởi tạo backend

terraform init terraform init -upgrade

Xem thay đổi (Plan)

Xem sẽ tạo/s𞻚/xóa gì

terraform plan terraform plan -out=tfplan terraform plan -target=proxmox_vm.web

Áp dụng thay đổi

Thực thi plan đã tạo

terraform apply terraform apply tfplan terraform apply -auto-approve

Xóa tài nguyên

Xóa tất cả hoặc chỉ định

terraform destroy terraform destroy \ -target=proxmox_vm.test

Quản lý State

Xem, di chuyển, xóa resource trong state

terraform state list terraform state show proxmox_vm.web terraform state rm proxmox_vm.old terraform state mv old_name new_name

Import tài nguyên có sẵn

Đưa resource đã tồn tại vào state

terraform import \ proxmox_vm_qemu.existing \ pve1/qemu/100

Validate và Format

Kiểm tra cú pháp và format code

terraform validate terraform fmt terraform fmt -recursive

Xem Output

Xem giá trị output đã định nghĩa

terraform output terraform output -json terraform output vm_ip_address

Workspace (Multi-env)

Quản lý nhiều môi trường với workspace

terraform workspace list terraform workspace new staging terraform workspace select production

Workflow gợi ý cho Homelab

Bước 1: Dùng Terraform tạo VM/LXC trên Proxmox (infrastructure provisioning).
Bước 2: Dùng cloud-init cấu hình cơ bản khi boot (SSH key, hostname, network).
Bước 3: Dùng Ansible cấu hình chi tiết (packages, services, config files).
Bước 4: Lưu tất cả vào Git repo. Dùng GitHub Actions hoặc Woodpecker CI để tự động chạy khi có thay đổi.
Đây là "GitOps cho homelab" - đơn giản, hiệu quả, và bạn sẽ không bao giờ mất cấu hình nữa.
