Vương Trạng Thái — Bậc Thầy State Management

"State là trái tim của ứng dụng. Quản lý sai thì ứng dụng nhồi máu cơ tim."

Redux / RTK Zustand Pinia MobX XState Jotai / Recoil

📜 Thân Thế & Lai Lịch

Vương Trạng Thái — người ta nói ông sinh ra đã có hai state: khóc và bú. Từ hai state đơn giản đó, ông dần phát triển thêm hàng triệu states phức tạp hơn trong suốt cuộc đời coding.

Ông từng là tín đồ Redux cuồng nhiệt, viết action creators, reducers, selectors, middleware, và tự hào vì "predictable state container". Cho đến một ngày, ông nhận ra mình đã viết 47 files chỉ để thêm một counter vào app. Moment of clarity đó đã thay đổi cuộc đời ông.

Ông bắt đầu hành trình tìm kiếm "state management vừa đủ" — không quá phức tạp như Redux, không quá đơn giản như useState rải rác. Qua nhiều năm, ông ngộ ra rằng: 80% state nên ở local, 15% ở component tree (Context/composables), và chỉ 5% thực sự cần global store. Câu nói đúc kết: "Tốt nhất là khi bạn KHÔNG CẦN state management library."

🗝️ Bí Kíp

Bí Kíp #1: Phân loại State — Không phải state nào cũng equal

Server State: Data từ API (users, products). Dùng React Query / SWR / TanStack Query — chúng handle caching, revalidation, optimistic updates. KHÔNG cần Redux cho server state. Client State: UI state (modal open/close, theme, sidebar). Đa số local component state đủ. URL State: Filters, pagination, search params. Dùng URL search params — shareable, bookmarkable. Form State: Input values, validation. Dùng form libraries (React Hook Form, Formik). Phân loại đúng = giảm 70% complexity.

Bí Kíp #2: Redux Toolkit — Redux đúng cách

Redux thuần = boilerplate hell. Redux Toolkit (RTK) = Redux done right. createSlice auto-generate action creators + reducers. createAsyncThunk cho async operations. RTK Query cho API caching (thay thế React Query trong Redux ecosystem). Immer built-in — viết "mutative" code nhưng immutable under the hood: state.items.push(item) thay vì spread hell. Nếu BẮT BUỘC dùng Redux (existing codebase), dùng RTK. Nếu dự án mới, cân nhắc alternatives nhẹ hơn.

Bí Kíp #3: Zustand — Minimalism at its Finest

Zustand: 1KB, không boilerplate, không Provider wrapper, không context. Tạo store: create((set) => ({ count: 0, increment: () => set(s => ({ count: s.count + 1 })) })). Done. Hook-based API, TypeScript-friendly, middleware system cho persist/devtools. Zustand auto-detect sự thay đổi và chỉ re-render components dùng slice state thay đổi. Perfect cho 90% use cases. Redux mạnh hơn cho complex state logic (middleware chain, time-travel debugging), nhưng Zustand đủ cho đa số apps.

Bí Kíp #4: Pinia — Vue's Official State Management

Pinia thay thế Vuex, designed cho Vue 3 Composition API. Stores = reactive objects + actions. Không cần mutations (Vuex mutations là overhead không cần thiết). TypeScript support out of the box. Devtools integration tuyệt vời. Plugin system cho persist, reset, undo/redo. So với Vuex: ít boilerplate hơn 60%, TS inference tốt hơn, modular by default (mỗi store là một file, import khi cần).

Bí Kíp #5: State Machines với XState — Khi state phức tạp cần structure

Khi state có nhiều transitions phức tạp (checkout flow, multi-step wizard, authentication), dùng state machines. XState cho phép define tất cả possible states, transitions, guards (conditions), và side effects. Impossible states become impossible — bạn không thể ở state "loading" VÀ "error" cùng lúc. Visualizer tool cho phép designer và PM hiểu flow. Overkill cho simple state, nhưng life-saver cho complex workflows.

Bí Kíp #6: Atomic State — Jotai, Recoil

Atomic state: mỗi piece of state là một "atom" độc lập. Components subscribe atom cụ thể — chỉ re-render khi atom đó thay đổi. Derived state qua selectors/computed atoms. Bottom-up approach (ngược với Redux top-down). Jotai: minimalist, 2KB, Zustand-like simplicity nhưng atomic model. Tốt khi nhiều components cần share small pieces of state mà không muốn centralized store.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Redux Boilerplate Hell — 47 Files cho 1 Counter

Năm 2019, team tôi setup Redux cho dự án mới theo "best practices": folder structure theo feature, separate files cho actions, action types, reducers, selectors, sagas. Thêm feature counter cần: counterActions.js, counterTypes.js, counterReducer.js, counterSelectors.js, counterSaga.js, rồi register trong rootReducer.js, rootSaga.js.

Junior dev mới join, hỏi: "Tôi muốn thêm field 'name' vào user profile, phải sửa bao nhiêu files?" Trả lời: 7 files. Junior: "Ở dự án trước tôi chỉ cần setState..." Senior: "ĐÂY LÀ ENTERPRISE PATTERN!"

6 tháng sau, codebase có 400+ files Redux-related. Thời gian implement feature: 30% viết logic, 70% viết boilerplate. Team chuyển sang Zustand — 400 files giảm còn 20 files. Feature delivery speed tăng gấp 3. Giám đốc hỏi: "Sao giờ mới đổi?"

💡 Bài học: Boilerplate không phải "enterprise quality" — nó là friction. Redux Toolkit giảm 80% boilerplate. Nhưng nếu app không cần Redux (đa số apps), dùng solution nhẹ hơn. Complexity phải justify bởi actual requirements, không phải "best practices" trên blog.

📖 Chuyện 2: Prop Drilling — 7 Tầng Địa Ngục

Component tree: App → Layout → Sidebar → Menu → MenuItem → SubMenuItem → Icon. Icon component cần biết current theme (dark/light). Giải pháp "nhanh": pass theme prop từ App qua 7 levels.

Tuần sau, Icon cần thêm user.language. Pass thêm prop qua 7 levels. Tuần sau nữa, notifications.count. Mỗi intermediate component nhận props mà nó không dùng — chỉ forward xuống. Sửa prop name ở top → sửa 7 files. Quên sửa 1 file → runtime error.

React Context giải quyết được nhưng tạo re-render cho mọi consumer khi bất kỳ value nào trong context thay đổi. Giải pháp đúng: tách context theo concern (ThemeContext, UserContext, NotificationContext) + dùng Zustand cho cross-cutting state.

💡 Bài học: Prop drilling 2-3 levels = OK. Hơn 3 levels = cần giải pháp khác. Options: React Context (cho infrequently changing state), Zustand/Jotai (cho frequently changing state), Component Composition (invert control, pass rendered components thay vì data). Đừng tạo "God Context" chứa mọi thứ.

📖 Chuyện 3: Optimistic Update Gone Wrong

Social media app, user click "Like" → UI update ngay (optimistic) → API call background. UX mượt mà, user happy. Cho đến khi API fail silently.

Scenario: User like 10 posts liên tiếp. Network glitch → 3 likes fail. UI vẫn hiển thị liked. User refresh → 3 posts quay lại "not liked". User: "BUG! Tôi đã like rồi mà!"

Tệ hơn: user unlike → like → unlike trong 1 giây. Race condition: unlike request đến server trước like request (network reordering). Server state: liked. Client state: not liked. Desync vĩnh viễn cho đến khi refresh.

Fix: implement proper rollback trên failure, debounce rapid actions, dùng mutation queue với ordering guarantee (React Query's useMutation with onError rollback). Và LUÔN handle error case trong optimistic updates.

💡 Bài học: Optimistic updates = great UX nhưng cần error handling cẩn thận. Luôn implement: 1) Rollback on failure (revert UI state), 2) Retry logic, 3) Conflict resolution cho rapid interactions. React Query / TanStack Query có built-in support cho optimistic updates với proper rollback.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Đưa MỌI THỨ vào global store

Form input value, modal visibility, tooltip position — không cần trong Redux/Zustand. Global store cho shared, cross-component state. Local state cho component-specific state. Quy tắc: nếu chỉ 1 component dùng → local state. Nếu unmount component xóa state là OK → local state.

❌ Sai Lầm #2: Mutate state trực tiếp

state.user.name = "new name" — React/Vue (ngoài reactive system) KHÔNG detect được mutation trực tiếp. Phải tạo new reference: spread operator hoặc Immer. RTK/Zustand cho phép "mutative" syntax nhờ Immer proxy — nhưng hiểu rằng under the hood vẫn immutable.

❌ Sai Lầm #3: Không normalize data

API trả về nested objects: post → comments → users → posts (circular). Store chứa duplicated data — update user name phải tìm ở 10 chỗ khác nhau. Normalize: flat structure, IDs reference, lookup by ID. RTK's createEntityAdapter hoặc normalizr library.

❌ Sai Lầm #4: Dùng Context cho frequently changing state

React Context re-render MỌI consumer khi value thay đổi — kể cả consumers chỉ dùng một phần nhỏ. Mouse position trong Context? Mỗi pixel move = re-render toàn bộ tree. Dùng external store (Zustand) với selector cho fine-grained subscriptions.

❌ Sai Lầm #5: Sync state giữa nhiều stores

UserStore update → phải sync sang CartStore → phải sync sang NotificationStore. State duplication + sync logic = bugs guaranteed. Single source of truth: mỗi piece of data chỉ live ở MỘT nơi. Derived/computed từ source thay vì duplicate.
