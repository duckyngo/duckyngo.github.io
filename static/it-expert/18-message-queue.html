📜 Thân Thế & Lai Lịch

Phạm Hàng Đợi lớn lên trong thời đại 'gọi trực tiếp, đợi phản hồi'. Service A gọi Service B, đợi. B gọi C, đợi. Chain 5 services, mỗi cái 200ms = 1 giây latency minimum. Một service down = cả chain down. Ngày ông phát hiện message queues, ông như tìm được chân lý: decouple services, async processing, buffer burst traffic, guaranteed delivery. Nhưng với great power comes great responsibility — message ordering, exactly-once delivery, dead letter queues, partition rebalancing, consumer group management. Ông từng debug 3 ngày trời vì messages 'biến mất' — thực ra nằm trong dead letter queue mà không ai monitor. Từ đó, ông viết rule bằng máu: 'MỌI dead letter queue phải có alerting.' Motto: 'Async không phải fire-and-forget — nó là fire-and-make-sure-it-arrives.'

🗝️ Bí Kíp

Bí Kíp #1: RabbitMQ vs Kafka — Hai triết lý khác nhau

RabbitMQ: traditional message broker, smart broker + dumb consumers. Messages consumed → deleted from queue. Routing flexible (direct, topic, fanout, headers). Tốt cho: task queues, RPC, flexible routing. Kafka: distributed event log, dumb broker + smart consumers. Messages retained (configurable retention). Consumers track offset — replay possible. Tốt cho: event streaming, log aggregation, high-throughput pipelines, event sourcing. Rule: RabbitMQ cho traditional messaging (work queues, pub/sub). Kafka cho event streaming (event log, replay, analytics pipeline).

Bí Kíp #2: At-least-once vs Exactly-once vs At-most-once

At-most-once: fire and forget, có thể mất message. Dùng cho: logs, metrics (mất 1 data point OK). At-least-once: guaranteed delivery, có thể duplicate. Consumer PHẢI idempotent. Dùng cho: đa số use cases (orders, notifications, tasks). Exactly-once: guaranteed delivery + no duplicates. Kafka Transactions support exactly-once within Kafka ecosystem. Expensive, complex. Dùng cho: financial transactions, inventory updates. Practice: design for at-least-once + idempotent consumers — simplest reliable approach.

Bí Kíp #3: Dead Letter Queue (DLQ) — Nơi messages cuối cùng

Message processing fail (poison message, bug, dependency down) → retry N times → still fail → move to Dead Letter Queue. DLQ PHẢI monitored: alert khi DLQ có messages. DLQ PHẢI có process xử lý: manual review, automated retry after fix, discard with logging. ĐỪNG để DLQ accumulate silently — đó là data loss ẩn. Pattern: main queue → retry queue (with delay) → DLQ (after max retries). Include original error, timestamp, retry count trong DLQ message.

Bí Kíp #4: Message Ordering — Khó hơn bạn nghĩ

Kafka: ordering guaranteed WITHIN partition. Messages cùng partition key → same partition → ordered. Khác partition key → no ordering guarantee. RabbitMQ: ordering within single queue + single consumer. Multiple consumers → no ordering (consumers process at different speeds). Nếu cần strict ordering: single partition/consumer (limits throughput) hoặc design system to be order-independent (idempotent, commutative operations).

Bí Kíp #5: Consumer Groups & Scaling

Kafka consumer groups: multiple consumers chia partitions. 10 partitions + 5 consumers = 2 partitions/consumer. Add consumer → rebalance → partitions redistribute. Cẩn thận: rebalancing = temporary pause → message processing delay. Tối ưu: number of consumers ≤ number of partitions. RabbitMQ: competing consumers — multiple consumers trên same queue, messages distributed round-robin. Simpler scaling model.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Kafka Partition Rebalance Storm

100 partitions, 10 consumers. Deploy new consumer version: rolling update → consumer shutdown → rebalance → new consumer start → another consumer shutdown → rebalance again → repeat for all 10 consumers. 10 rebalances trong 5 phút. Mỗi rebalance: 30-60 giây pause. During rebalance: messages not consumed, producer side queue builds up, dependent services timeout waiting for responses. Cascade: order processing delayed 30 phút during peak hours. 5,000 orders stuck. Fix: implement cooperative rebalancing (incremental, less disruptive). Increase session.timeout.ms và heartbeat.interval.ms. Blue-green deployment thay vì rolling update cho consumers. Static partition assignment cho critical consumers.

💡 Bài học: Kafka consumer rebalancing là critical operational concern. Plan deployments: rolling update during low-traffic periods, cooperative rebalancing protocol, adequate timeout configs. Monitor consumer lag — if lag increases during deployment, something is wrong. Consider: static partition assignment cho latency-sensitive consumers.

📖 Chuyện 2: Dead Letter Queue Overflow — 2 Triệu Messages Không Ai Biết

Order notification service: process order events → send email. Bug trong email template → parsing error → message fail → retry 3x → DLQ. Nhưng: NO ALERTING on DLQ. Bug xảy ra Friday afternoon. Monday morning: DLQ có 2 TRIỆU messages. 2 triệu customers không nhận được order confirmation email. Customer support nhận 500 calls: 'Tôi đặt hàng rồi nhưng không nhận được xác nhận, đơn hàng có thành công không?' Trust damaged. Manual work: replay 2M messages after fix. Replay mất 6 giờ. After incident: DLQ alerting mandatory cho mọi queue. DLQ dashboard. Automated DLQ replay tool. DLQ count included trong daily ops report.

💡 Bài học: Dead Letter Queue PHẢI có alerting — threshold alert (> 100 messages), rate alert (> 50 messages/minute). Monitoring DLQ = monitoring reliability. DLQ không phải 'ignore bucket' — nó là 'investigate immediately bucket'. Include DLQ monitoring trong operational runbook.

📖 Chuyện 3: Message Ordering Bug — Đơn Hàng Bị Cancel Trước Khi Tạo

E-commerce: order events pushed to Kafka. Event sequence: OrderCreated → OrderPaid → OrderShipped. Consumer processes events. Nhưng: OrderCreated partition key = user_id, OrderPaid partition key = order_id. Khác partition key → khác partition → khác processing order. Result: consumer nhận OrderPaid TRƯỚC OrderCreated → 'Order not found' error → OrderPaid moved to DLQ. Tệ hơn: OrderCancelled arrive trước OrderCreated → cancel nonexistent order → ignore. Rồi OrderCreated arrive → order exists nhưng should be cancelled → nobody cancels it. 30% orders trong trạng thái inconsistent. Fix: dùng order_id làm partition key cho TẤT CẢ order-related events → same partition → ordered processing.

💡 Bài học: Message ordering critical cho stateful processing. Rule: ALL events related to same entity PHẢI dùng same partition key → same partition → ordered. Design partition key carefully — entity ID (order_id, user_id) not attribute (payment_type, status). Test ordering assumptions explicitly trong integration tests.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Fire and forget cho critical operations

Publish message thành công ≠ message processed thành công. Implement acknowledgment, delivery confirmation, consumer health monitoring. Critical messages cần guaranteed delivery + processing confirmation.

❌ Sai Lầm #2: Không set message TTL

Messages without expiry → queue grows forever → disk full → broker crash. Set TTL appropriate cho use case: notifications 24h, tasks 7 days, events 30 days. Expired messages → DLQ hoặc drop with logging.

❌ Sai Lầm #3: Consumer không idempotent

Network retry, consumer restart, rebalance → duplicate delivery. Consumer process message twice → duplicate order, double charge. Implement idempotency: deduplication key, check-before-process, upsert instead of insert.

❌ Sai Lầm #4: Single consumer cho critical queue

Consumer crash → messages accumulate → processing delay. Minimum 2 consumers cho redundancy. Monitor consumer lag — alert khi lag exceeds threshold.

❌ Sai Lầm #5: Quá nhiều queues/topics

100 queues cho 100 message types → operational overhead (monitoring, alerting, capacity planning each). Group related messages: order.created, order.paid → 'orders' topic with message type header. Fewer topics = less operational burden.
