📚 Thân Thế & Lai Lịch

Hiệu Năng Đại Sư từng là developer viết code "chạy được là được." Cho đến khi ứng dụng chàng ta viết serve 100 users thì mượt, 1000 users thì lag, 10000 users thì sập. CTO hỏi: "Tại sao?" Chàng ta trả lời: "Hệ thống cần upgrade?" CTO: "Không, code cần optimize."

Từ đó, chàng ta bắt đầu hành trình giác ngộ: perf, strace, flamegraph, htop, vmstat, iostat — những tools mà trước đây chàng ta chưa bao giờ nghe. Chàng ta nhận ra rằng performance tuning là detective work: tìm bottleneck, measure, optimize, repeat.

Sau 15 năm, chàng ta có thể nhìn vào htop output 3 giây và nói: "Database server, IOwait cao, cần SSD. Team dùng SELECT * phải không?" — đúng 95% thời gian.

📜 Bí Kíp

1. CPU — Scheduling & Optimization: mpstat cho per-CPU stats. top/htop cho realtime monitoring. Context switches cao = quá nhiều processes competing. CPU steal > 5% trên VM = noisy neighbor. NUMA awareness cho multi-socket servers. CPU pinning (taskset/cgroups) cho latency-sensitive workloads. irqbalance cho interrupt distribution.
2. Memory — Hiểu Để Tối Ưu: free -h là bước đầu tiên. Buffer/cache là TỐT — kernel tận dụng RAM. vm.swappiness=10 cho database servers (giảm swapping). Transparent Huge Pages (THP) — disable cho databases (fragmentation issues). OOM killer: tune oom_score_adj cho critical processes. Memory leaks: valgrind, address sanitizer.
3. Disk I/O — Thường Là Bottleneck: iostat -x 1 cho per-disk stats. await > 10ms = slow disk. %util > 80% = saturated. Scheduler: noop/none cho SSD, mq-deadline cho HDD. readahead tuning cho sequential workloads. Direct I/O cho databases. noatime,nodiratime mount options. NVMe > SSD > HDD — hardware matters.
4. Network — Tuning Cho Throughput: netstat/ss cho connection stats. TIME_WAIT sockets quá nhiều = cần tcp_tw_reuse. TCP window scaling cho WAN. MTU optimization (jumbo frames trong datacenter). Socket buffer sizes (net.core.rmem_max, wmem_max). BBR congestion control cho WAN. Interrupt coalescing cho high-throughput.
5. Application Profiling: CPU profiling: perf record + flamegraph. Memory profiling: valgrind, heaptrack. Database: EXPLAIN ANALYZE mọi slow query. Connection pooling cho database connections. Async I/O cho I/O-bound applications. Caching: Redis/Memcached cho repeated computations. Profile TRƯỚC khi optimize — đừng guess.
6. Benchmarking — Đo Đúng Cách: fio cho disk benchmarks. iperf3 cho network benchmarks. wrk/ab cho HTTP benchmarks. sysbench cho database benchmarks. Benchmark trước và sau optimization. Kiểm soát variables: cùng data, cùng conditions. Benchmark production-like workload, không phải synthetic. 3 runs minimum, report p50/p95/p99.

📖 Chuyện Kể Đêm Khuya

😰 Chuyện Swappiness Giết Chết Database

PostgreSQL server chạy trên VM 32GB RAM, database buffer pool 24GB. Sau 2 tháng, performance degrade dần. Queries chậm gấp 10x. Team tăng RAM lên 64GB — vẫn chậm. Tăng lên 128GB — hơi khá hơn nhưng vẫn chậm so với ban đầu.

Đại Sư SSH vào, chạy free -h: swap usage 40GB. vm.swappiness=60 (default). Kernel swap out database buffer pool ra disk để cache file system data. Database read từ swap = disk I/O = chậm kinh khủng. Fix: vm.swappiness=10, restart PostgreSQL, performance trở lại bình thường. Mất 5 phút diagnostic, tiết kiệm 128GB RAM upgrade không cần thiết.

💡 Bài học: Database servers: vm.swappiness=10 hoặc thấp hơn. Monitor swap usage — nếu database server đang swap, performance sẽ terrible. RAM upgrade không giúp nếu kernel swap out useful data. Understand Linux memory management trước khi throw hardware at problem.

🔧 Chuyện ulimit Quá Thấp

Application server bắt đầu refuse connections lúc peak hour. Error: 'Too many open files.' Server có 128GB RAM, 32 cores, load average 0.5 — resources dồi dào. Nhưng ulimit -n: 1024. Mỗi connection = 1 file descriptor. Application cần 5000+ connections peak = hit limit.

Fix nhanh: ulimit -n 65535 trong service file. Permanent: /etc/security/limits.conf. Nhưng root cause: application không close connections properly — connection leak. Fix leak + tăng ulimit = problem solved. Nếu chỉ tăng ulimit mà không fix leak, limit sẽ hit lại ở number cao hơn.

💡 Bài học: Default ulimits quá thấp cho servers. Check và tune: nofile (65535), nproc (65535). Nhưng tăng limits không phải always the answer — fix resource leaks trước. lsof | wc -l cho biết đang dùng bao nhiêu file descriptors.

📊 Chuyện Flamegraph Phát Hiện 1 Dòng Code

API response time p99: 2 giây. Team optimize database queries, add caching, upgrade hardware — p99 giảm từ 2s xuống 1.5s. Vẫn chưa đạt target 500ms. 3 sprint tốn cho optimization mà kết quả minimal.

Đại Sư chạy perf record 60 giây, generate flamegraph. Phát hiện: 70% CPU time trong 1 function — JSON serialization. Developer dùng reflection-based JSON library (slow) thay vì code-generated serializer. Thay JSON library: p99 giảm từ 1.5s xuống 200ms. 1 dòng code thay đổi > 3 sprint optimization.

💡 Bài học: PROFILE trước khi optimize. Flamegraph cho thấy WHERE time is spent. 80/20 rule: 80% performance gain từ 20% (thường ít hơn) code changes. Đừng optimize blind — measure, identify bottleneck, fix, measure again.

⚠️ Những Sai Lầm Kinh Điển

1. Optimize mà không measure: Guessing bottleneck = waste time. Profile first. Measure baseline, set target, optimize, measure result.
2. Premature optimization: Code chưa có users đã optimize for millions. Ship first, optimize khi có real traffic data. 'Premature optimization is the root of all evil.' — Knuth.
3. Throw hardware at problem: Server chậm → thêm RAM, thêm CPU. Nhưng problem là slow query scan full table. Hardware không fix bad algorithms.
4. Copy sysctl.conf từ blog: Blog nói 'optimal Linux tuning' cho MySQL. Bạn chạy PostgreSQL trên NVMe. Different workload, different tuning needed.
5. Benchmark trên dev machine: i9 laptop benchmark: 100K requests/sec! Production c5.large: 5K requests/sec. Benchmark trên production-like environment.
