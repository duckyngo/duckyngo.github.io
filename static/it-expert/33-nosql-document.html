📚 Thân Thế & Lai Lịch

Tài Liệu Kỳ Nhân, tên thật là Trần Văn Tự Do, sinh ra trong một gia đình mà ai cũng nói chuyện lung tung không theo thứ tự. Bố kể chuyện luôn nhảy từ topic này sang topic khác. Mẹ viết danh sách mua sắm mà mỗi item lại có sub-items lồng nhau 5 cấp.

Hồi làm dev, lão chán cái kiểu phải ALTER TABLE mỗi khi thêm một field mới. Lần đầu gặp MongoDB năm 2010, lão khóc vì sung sướng: "Cuối cùng cũng có database hiểu được gia đình tôi!"

Nhưng sau 15 năm chiến đấu, lão đã thấm thía rằng "schema-less" thực ra nghĩa là "schema-on-read" — tức là schema vẫn tồn tại, chỉ là bạn không biết nó ở đâu cho đến khi mọi thứ nổ tung.

📜 Bí Kíp

Bí Kíp #1: Embedding vs Referencing — Nghệ Thuật Chọn Lựa

Embed khi: data thuộc về nhau (địa chỉ trong user), ít thay đổi, và bạn luôn đọc cùng nhau. Reference khi: data dùng chung giữa nhiều documents, thay đổi thường xuyên, hoặc document sẽ vượt quá 16MB limit. Quy tắc đơn giản: nếu bạn sẽ JOIN ở application code, có lẽ nên embed.

Bí Kíp #2: Schema Validation — "Schema-less" Không Có Nghĩa "Luật-less"

MongoDB 3.6+ có JSON Schema validation. Dùng nó! db.createCollection("users", { validator: { $jsonSchema: {...} } }). Không có validation = mỗi dev tự chế schema riêng = chaos. Firestore có Security Rules — đừng bỏ qua.

Bí Kíp #3: Index Trong MongoDB — Không Khác RDBMS Mấy

Compound index: thứ tự field quan trọng (ESR rule — Equality, Sort, Range). Covered query khi tất cả fields trong query đều nằm trong index. Sparse index cho optional fields. TTL index cho data tự xóa. .explain("executionStats") là bạn thân nhất.

Bí Kíp #4: Aggregation Pipeline — MapReduce Đã Chết

Quên MapReduce đi. Aggregation Pipeline mới là vua: $match → $group → $project → $sort. Pipeline stage có thể tận dụng index (đặt $match đầu tiên!). $lookup cho "JOIN" nhưng đừng lạm dụng — nếu bạn $lookup nhiều quá, có lẽ bạn cần RDBMS.

Bí Kíp #5: Replica Set — Minimum 3 Node

Primary + Secondary + Arbiter là minimum. Đừng bao giờ chạy standalone MongoDB trong production. Write concern "majority" cho dữ liệu quan trọng. Read preference "secondaryPreferred" cho analytics query — đừng đánh primary bằng report.

Bí Kíp #6: Sharding — Chọn Shard Key Sai Là Xong Đời

Shard key không thể thay đổi (trước MongoDB 5.0). Shard key kém = hot shard. Tránh monotonic shard key (timestamp, auto-increment). Hashed shard key phân tán đều nhưng mất range query. Compound shard key thường là lựa chọn tốt nhất.

Bí Kíp #7: CouchDB & Firestore — Hai Thái Cực

CouchDB: master-master replication, offline-first, perfect cho mobile sync. Firestore: real-time listeners, tích hợp Firebase Auth, billing theo document reads/writes. Chọn CouchDB khi cần sync offline. Chọn Firestore khi cần real-time + nhanh MVP. Nhưng nhớ đọc pricing trước khi production.

📖 Chuyện Kể Đêm Khuya

🌪️ Chuyện 1: MongoDB Schema Chaos — Khi Mọi Dev Tự Chế Schema Riêng

Startup có 8 dev, dùng MongoDB "vì schema-less, linh hoạt". Không ai define schema validation. Sau 1 năm, collection users có: một số document có field phone, một số có phoneNumber, một số có phone_number, một số có tel. Field age: có doc lưu string "25", có doc lưu number 25, có doc lưu { years: 25, months: 3 }.

Khi cần làm analytics report, anh data engineer khóc ròng. Mỗi query phải $or 4 field names, $type check 3 types. Lão phải viết migration script chạy 3 ngày liên tục để normalize toàn bộ collection.

💡 Bài học: "Schema-less" không có nghĩa "không cần nghĩ về schema". Luôn define schema convention, dùng schema validation, và enforce bằng code review.
💸 Chuyện 2: Firestore Billing Surprise — Hóa Đơn Đến, Tim Ngừng Đập

Anh dev xây app to-do list với Firestore. Mọi thứ đẹp đẽ, real-time update mượt mà. Nhưng anh viết một cái dashboard listing ALL tasks của ALL users với real-time listener. Mỗi khi bất kỳ user nào thêm task, tất cả documents trong collection được đọc lại.

1000 users × 100 tasks × mỗi lần update = 100,000 document reads. Mỗi phút có ~10 updates = 1 triệu reads/phút = 1.4 tỷ reads/ngày. Hóa đơn cuối tháng: $18,000 USD. Cho một app to-do list. Anh CEO startup gọi lão lúc 2 giờ sáng, giọng run run.

💡 Bài học: Firestore tính tiền theo document reads. Real-time listeners trên collection lớn = phá sản. Luôn paginate, limit queries, và dùng composite index thay vì đọc toàn bộ collection.
🔗 Chuyện 3: Embedding Quá Sâu — Document 16MB

Anh dev quyết định embed mọi thứ vào document user: orders, reviews, messages, notifications, activity log. "Đọc một lần lấy hết, nhanh!" — anh tự hào nói. Cho đến khi user "power user" có 50,000 orders. Document user đạt 16MB limit — MongoDB từ chối ghi thêm.

Tệ hơn: mỗi lần đọc profile user = load 16MB data. Response time cho API /user/profile: 8 giây. Lão phải redesign, tách orders thành collection riêng, reference bằng userId. Mất 2 tuần migration + downtime.

💡 Bài học: Embed có giới hạn. Data tăng trưởng không giới hạn (orders, logs, messages) nên reference, không embed. Quy tắc: nếu array trong document có thể vượt 100 items, tách ra collection riêng.

⚠️ Những Sai Lầm Kinh Điển

❌ Dùng MongoDB làm relational database

$lookup tứ phía, reference khắp nơi, biến MongoDB thành SQL server chạy chậm hơn 10 lần. Nếu data của bạn heavily relational, hãy dùng... relational database.

❌ Không tạo index

"MongoDB nhanh mà cần gì index" — nhanh khi collection có 1000 docs. Khi lên 10 triệu docs, full collection scan mỗi query. Giống RDBMS: không index = không performance.

❌ Dùng MongoDB mà bật mặc định "acknowledged" write concern

Write concern w:1 (default) = chỉ primary xác nhận. Node primary crash trước khi replicate = mất data. Dữ liệu quan trọng luôn dùng w:"majority".

❌ Lưu file trong MongoDB GridFS khi không cần

GridFS cho file > 16MB. Nhưng nếu bạn chỉ cần lưu avatar 200KB, hãy dùng object storage (S3). GridFS chunk files = overhead không cần thiết cho small files.

❌ Quên cấu hình Firestore Security Rules

Default rules cho phép mọi người đọc/ghi. Bạn vừa tạo một database công cộng. Bot crawl sẽ tìm thấy và phá hoại — hoặc tệ hơn, đọc hết dữ liệu nhạy cảm.

❌ Không giới hạn document size khi embed

Embed array tăng trưởng vô hạn. Document lớn dần, query chậm dần, cho đến khi chạm 16MB wall. Luôn set giới hạn và có strategy khi data outgrow document.
