📚 Thân Thế & Lai Lịch

Đại Sư bắt đầu với Nginx làm web server, rồi phát hiện nó có thể làm reverse proxy và load balancer. "Wait, 1 tool làm 3 việc?" Từ đó chàng ta nghiện load balancing.

Chàng ta đã thử mọi algorithm: round-robin, least-connections, weighted, IP hash, consistent hashing. Chàng ta biết khi nào dùng L4 (TCP), khi nào dùng L7 (HTTP). Chàng ta có thể configure HAProxy trong giấc ngủ — và thực sự đã làm vậy một lần khi bị gọi lúc 3 giờ sáng.

Giờ đây với service mesh (Envoy, Istio), load balancing đã phức tạp hơn bao giờ hết. Nhưng core concepts vẫn giữ nguyên: distribute traffic evenly, detect failures, và đừng gửi users tới dead servers.

📜 Bí Kíp

1. L4 vs L7 — Chọn Đúng Layer: L4 (TCP/UDP): fast, simple, không biết content. NAT-based hoặc DSR (Direct Server Return). Dùng cho: database connections, raw TCP, non-HTTP protocols. L7 (HTTP/HTTPS): content-aware, path routing, header manipulation. Dùng cho: web applications, API routing, SSL termination. L7 phổ biến hơn cho modern web apps.
2. Health Checks — Phát Hiện Server Chết: Active health checks: LB gửi requests tới backend, check response. Passive health checks: monitor actual traffic cho errors. HTTP health check: GET /health → 200 OK. Check BEYOND 'process running': verify database connectivity, downstream services. Interval, threshold, timeout tuning quan trọng — quá aggressive = false positives.
3. HAProxy — Enterprise Load Balancer: High performance, battle-tested. Runtime API cho dynamic configuration. Built-in stats page. ACLs cho complex routing. Rate limiting, connection queuing. HAProxy + keepalived cho HA pair. Lua scripting cho custom logic. HAProxy 2.x: HTTP/2 support, dynamic servers.
4. Nginx — Web Server + Reverse Proxy: upstream blocks cho backend pools. proxy_pass cho reverse proxy. ssl_certificate cho TLS termination. proxy_cache cho caching. rate limiting với limit_req_zone. Plus (commercial) cho dynamic configuration, health checks. OpenResty cho Lua scripting.
5. Envoy — Cloud-Native Proxy: xDS APIs cho dynamic configuration. Circuit breaking, retries, timeouts. Distributed tracing integration. Hot restarts cho zero-downtime config changes. gRPC support. Foundation for service meshes (Istio). Complex nhưng powerful cho microservices.
6. Session Persistence: Sticky sessions (cookie, IP hash) keep user on same backend. Trade-off: session persistence vs load distribution. Stateless applications = no sticky sessions needed = best for scaling. If must use: cookie-based > IP hash (NAT breaks IP hash). Better: externalize session to Redis/Memcached.

📖 Chuyện Kể Đêm Khuya

🍪 Chuyện Sticky Session Phá Autoscaling

E-commerce site dùng sticky sessions (cookie-based). Black Friday: traffic tăng 10x, autoscaler thêm 8 servers. Nhưng existing sessions vẫn gắn với 2 servers cũ. 2 servers cũ CPU 100%, 8 servers mới idle 5%. Users on old servers: slow response. Users on new servers: fast. 80% users chậm.

Root cause: sticky sessions prevent load redistribution. Fix: externalize sessions to Redis cluster, remove sticky sessions. All servers share session state. Autoscaling works correctly. Performance equal across all servers.

💡 Bài học: Sticky sessions là anti-pattern cho auto-scaling. Externalize state (Redis, database) cho stateless application servers. Load balancer distribute evenly. Nếu PHẢI dùng sticky sessions, set reasonable TTL và monitor per-server load.

❤️ Chuyện Health Check False Positive

HAProxy health check: TCP connect to port 8080. Backend app crash nhưng Nginx trên server vẫn listen port 8080 → health check PASS. Traffic gửi tới crashed app → 502 errors cho users. HAProxy nghĩ server healthy vì TCP connect thành công.

Fix: chuyển từ TCP health check sang HTTP health check: GET /health → kiểm tra application status, database connection, dependencies. Health endpoint return 503 nếu app unhealthy dù Nginx healthy.

💡 Bài học: TCP health check chỉ verify port is open, không verify application is working. HTTP health check endpoint phải verify actual application health: database connectivity, downstream services, business logic readiness. Deep health check vs shallow health check.

🔄 Chuyện Traefik Auto-Discovery Magic

Team migrate từ Nginx manual config sang Traefik với Docker provider. Traefik auto-discover containers qua Docker labels. Deploy new service → Traefik automatically route traffic. Magic! Cho đến khi developer deploy test container với label 'traefik.enable=true' và duplicate hostname → Traefik route production traffic tới test container. Users see test data.

Fix: separate Traefik instances cho production và non-production. Namespace constraints cho Docker labels. Deployment review process cho Traefik labels.

💡 Bài học: Auto-discovery powerful nhưng dangerous nếu không constrained. Separate production và non-production service discovery. Review labels/annotations trước deploy. Traefik routers phải có explicit constraints (namespace, label filters).

⚠️ Những Sai Lầm Kinh Điển

1. No health checks: Traffic gửi tới dead servers. Always configure health checks — HTTP deep health check for web apps.
2. SSL termination ở wrong place: SSL termination ở LB: backend traffic unencrypted (OK trong private network). End-to-end encryption: re-encrypt to backend. Choose based on security requirements.
3. Không monitor backend latency: LB happy nhưng backends slow. Monitor p50/p95/p99 latency per backend. Slow backend affects all users routed to it.
4. Single load balancer: LB is single point of failure. Active-passive pair (keepalived/VRRP) hoặc cloud-managed LB cho HA.
5. Over-complicated routing: 50 ACL rules, 20 backends, path-based + header-based + cookie-based routing. Simplify. Document routing logic. Test edge cases.
