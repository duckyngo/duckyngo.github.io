🗝️ Bí Kíp

Bí Kíp #1: Jetpack Compose — Tương lai UI Android

Declarative UI: describe UI based on state, Compose handle rendering. Composable functions: @Composable fun Greeting(name: String). State hoisting: state owned by parent, events bubbled up. Recomposition: chỉ re-render composables có state thay đổi. Side effects: LaunchedEffect, DisposableEffect, SideEffect. Navigation: NavHost + NavController. Interop: Compose trong XML (ComposeView), XML trong Compose (AndroidView). 2025: Compose là default — XML layout cho legacy code.

Bí Kíp #2: Kotlin Coroutines — Async Done Right

Coroutines: lightweight threads cho async operations. suspend fun: function có thể pause/resume. viewModelScope: coroutine auto-cancel khi ViewModel cleared. lifecycleScope: auto-cancel khi lifecycle destroyed. Dispatchers: Main (UI), IO (network/disk), Default (CPU). Flow: reactive streams, cold by default, collect trong lifecycle-aware scope. StateFlow/SharedFlow cho UI state. Channel cho one-time events.

Bí Kíp #3: Dependency Injection — Hilt/Koin

Hilt (recommended by Google): Dagger-based, compile-time DI, Android-specific scopes (@AndroidEntryPoint, @HiltViewModel). Less boilerplate than raw Dagger. Koin: runtime DI, Kotlin-first, simpler setup, no annotation processing. Tradeoff: Hilt = compile-time safety + more setup. Koin = simpler + runtime errors. Recommendation: Hilt cho production apps (compile-time error detection). Koin cho prototypes, small apps. Manual DI cho very simple apps.

Bí Kíp #4: Room Database — SQLite Made Easy

Room: ORM wrapper over SQLite. Compile-time query validation (typo trong SQL → compile error, không phải runtime crash). Entity = data class + @Entity annotation. DAO = interface với @Query, @Insert, @Update, @Delete. Database migration: Migration(1, 2) với ALTER TABLE. Flow/LiveData integration: database changes auto-emit updates. Encrypted Room: SQLCipher. Testing: in-memory database cho unit tests.

Bí Kíp #5: Android Architecture Components

ViewModel: survive configuration changes (rotation). LiveData/StateFlow: lifecycle-aware observable data. WorkManager: guaranteed background work (survive app kill, device reboot). DataStore: modern SharedPreferences replacement (Proto DataStore cho typed data, Preferences DataStore cho key-value). Paging 3: large data sets với RecyclerView/LazyColumn. Navigation Component: type-safe navigation, deep links.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Fragment Backstack Nightmare

App với 5 screens, mỗi screen là Fragment. User navigate: A → B → C → B → D → C. Backstack: A, B, C, B, D, C. User press Back: C → D → B → C → B → A. User confused: 'Tại sao tôi thấy screen B hai lần?' Fix: popUpTo + inclusive trong Navigation Component. Nhưng business logic phức tạp hơn: B cần different state khi navigate từ A vs từ D. Kết quả: Fragment arguments chaos, SavedStateHandle mismatches. Team chuyển sang single-Activity architecture với Navigation Component (hoặc Compose Navigation): centralized navigation graph, type-safe arguments, predictable back stack. Fragment transactions manual = đau đầu guaranteed.

💡 Bài học: Single-Activity architecture là best practice. Navigation Component (XML hoặc Compose) manage back stack, arguments, deep links. Đừng manual FragmentTransaction trừ khi absolutely necessary. Backstack behavior: test EVERY navigation path — forward, backward, deep link, process death restoration.

📖 Chuyện 2: Android Fragmentation Testing — 200 Devices

App trông đẹp trên Pixel 7. Deploy. Bug reports: Samsung Galaxy S21 — notch handling sai. Xiaomi Redmi Note 10 — WebView version cũ, JavaScript feature không support. Oppo A15 — camera API khác (custom Camera2 implementation). Tablet Samsung — layout vỡ ở landscape. Worse: Android 10 behavior khác Android 13 cho background location permission. Android 12 thêm splash screen API — app có 2 splash screens (custom + system). Android 14 thay đổi foreground service restrictions. Firebase Test Lab: run tests trên 50+ devices automatically. BrowserStack: manual testing trên specific devices. CI integration: mỗi PR run tests trên 5 representative devices.

💡 Bài học: Android fragmentation là REALITY — accept và plan for it. Minimum: test trên 5 representative devices (high-end Samsung, mid-range Xiaomi, low-end device, tablet, latest Android + oldest supported). Firebase Test Lab cho automated testing. OEM-specific issues: check Samsung Internet, Xiaomi MIUI restrictions, Huawei HMS. Set minSdkVersion carefully — mỗi level drop = significant user base drop.

📖 Chuyện 3: Coroutine Leak — The Silent Killer

App dùng GlobalScope.launch cho API call. User navigate away → coroutine vẫn chạy → update UI element đã destroyed → crash: 'View not attached to window manager.' Worse: coroutine call API in loop → user leave app → API calls continue → battery drain + data usage. Fix: dùng viewModelScope (auto-cancel khi ViewModel cleared) hoặc lifecycleScope (auto-cancel khi lifecycle destroyed). KHÔNG dùng GlobalScope trừ application-level background work. Structured concurrency: parent cancel → children cancel.

💡 Bài học: KHÔNG dùng GlobalScope. Dùng viewModelScope cho ViewModel operations, lifecycleScope cho UI operations, custom CoroutineScope với proper cancellation cho background work. Structured concurrency ensures no leaked coroutines. Test: verify coroutines cancel khi expected (ViewModel cleared, Fragment destroyed).

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Hardcode dimensions trong dp

android:layout_width='360dp' — giả sử screen width = 360dp. Samsung Galaxy Fold: 280dp (folded). Tablet: 800dp. Dùng ConstraintLayout percentages, weight, hoặc Compose responsive modifiers.

❌ Sai Lầm #2: Ignore process death

Android kill app process khi memory low. User quay lại → app recreate từ đầu. Nếu state không saved (ViewModel saved state, onSaveInstanceState) → user mất progress. Test: Developer Options > Don't keep activities.

❌ Sai Lầm #3: Main thread network call

NetworkOnMainThreadException trên Android >= 3.0. Nhưng disk I/O trên main thread vẫn cho phép (chỉ slow, không crash). StrictMode detect: enable trong debug builds. MỌII I/O: coroutine với Dispatchers.IO.

❌ Sai Lầm #4: Không handle configuration changes

Screen rotation → Activity recreated → state lost. ViewModel survive rotation, SavedStateHandle survive process death. Không handle = user rotate phone → form data lost → rage quit.

❌ Sai Lầm #5: Dùng deprecated APIs mà không check

Android evolve nhanh — deprecated API có thể bị remove. AsyncTask deprecated, dùng Coroutines. SharedPreferences ok nhưng DataStore better. Check API level, dùng AndroidX compat libraries.
