📜 Thân Thế & Lai Lịch

Nguyễn Lịch Trình bắt đầu với cron jobs — crontab trên Linux server, đơn giản và hiệu quả. Cho đến khi server reboot giữa đêm, cron job không chạy, daily report không gửi, sếp không có data cho meeting 8h sáng. 'Cron job của anh đâu?' — câu hỏi ám ảnh suốt sự nghiệp. Ông upgrade lên Celery (Python), rồi Bull (Node.js), rồi Temporal cho complex workflows. Mỗi tool dạy ông bài học: Celery dạy distributed task processing, Bull dạy Redis-based simplicity, Temporal dạy durable execution. Nhưng bài học lớn nhất là: 'Task scheduling đúng giờ thì invisible. Sai giờ thì VISIBLE cho toàn bộ company.' Motto: 'Idempotent, retriable, observable — ba tính chất mà mọi scheduled task phải có.'

🗝️ Bí Kíp

Bí Kíp #1: Idempotency — Chạy 1 lần hay 10 lần, kết quả phải giống nhau

Scheduled task có thể chạy lại (retry after failure, duplicate trigger, clock drift). Nếu task 'send daily report email' chạy 2 lần → user nhận 2 emails = bad. Implement: check nếu report đã gửi cho ngày đó → skip. Idempotency key: task_name + date + parameters = unique identifier. Check trước khi execute, mark completed sau khi success. Database unique constraint hoặc distributed lock.

Bí Kíp #2: Celery — Python's Task Queue Standard

Celery: distributed task queue cho Python. Broker: Redis hoặc RabbitMQ. Result backend: Redis, PostgreSQL, hoặc disabled (fire-and-forget). Beat scheduler cho periodic tasks. Canvas: chain, group, chord cho complex task workflows. Cẩn thận: Celery memory leaks (long-running workers), task serialization gotchas (don't pass ORM objects, pass IDs), timezone handling (USE_TZ=True, CELERY_TIMEZONE config). Monitor: Flower dashboard cho real-time worker monitoring.

Bí Kíp #3: BullMQ — Node.js Queue Powerhouse

BullMQ: Redis-based queue cho Node.js. Features: delayed jobs, rate limiting, prioritized jobs, repeatable jobs (cron-like), sandboxed processors (separate process per job type). Dashboard: bull-board hoặc arena. Advantages over raw Redis: reliable (BRPOPLPUSH atomic move), retry with backoff, job lifecycle events. Cẩn thận: Redis persistence config critical — RDB + AOF cho durability. Redis cluster mode cho scalability.

Bí Kíp #4: Temporal — Durable Execution Revolution

Temporal: durable execution platform. Workflow code = regular code (Go, Java, TypeScript, Python) nhưng state persisted automatically. Activity fails → Temporal retry. Server crash → workflow resume from last checkpoint. Long-running workflows (days, months) OK. Saga patterns, timer-based logic, human-in-the-loop — all built-in. Use cases: order fulfillment, subscription billing, CI/CD pipelines. Tradeoff: operational complexity (Temporal server cluster), learning curve (workflow/activity separation).

Bí Kíp #5: Airflow — Data Pipeline Orchestration

Airflow: DAG-based workflow orchestration. Define tasks + dependencies → Airflow schedule + execute + monitor. Built for data pipelines: ETL, ML training, report generation. Operators: BashOperator, PythonOperator, custom operators, cloud-specific operators (AWS, GCP). UI: DAG visualization, task logs, retry, backfill. Cẩn thận: Airflow is ORCHESTRATOR not EXECUTOR — heavy computation should run on external systems (Spark, Kubernetes), Airflow just triggers and monitors.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Cron Job Chạy Hai Lần — The Double Email Incident

Cron job gửi daily digest email, schedule: 0 8 * * * (8h sáng). Server timezone set UTC. Developer nghĩ mình set giờ Việt Nam. Kết quả: email gửi lúc 3h chiều thay vì 8h sáng. Fix timezone → nhưng quên rằng Daylight Saving Time (DST) shift 1 giờ ở một số nước. Users ở Mỹ nhận email lúc 7h hoặc 9h tùy mùa. Để fix cho mọi timezone → phải per-user scheduling → cron job đơn giản thành distributed scheduling system. Tệ hơn: server có 2 instances (load balanced), cả 2 đều run cron → user nhận 2 emails. Fix: distributed lock (Redis SETNX) — chỉ 1 instance execute task. Hoặc dedicated scheduler instance (single point of failure nhưng đơn giản hơn).

💡 Bài học: Cron jobs trên multiple instances = duplicate execution. Solutions: distributed lock (Redis/ZooKeeper), dedicated scheduler node, hoặc task queue (Celery Beat, BullMQ repeatable). Timezone: LUÔN store/process in UTC, convert to user timezone chỉ khi display. Test cron expression trên crontab.guru.

📖 Chuyện 2: Task Stuck in Queue Forever

Celery worker process task: call external API → API timeout 30 seconds → worker stuck. Celery soft_time_limit: not set (default: no limit). Worker holds task indefinitely → prefetch count reached → no new tasks processed. Queue grows → 50,000 tasks pending → system effectively dead. Alert fires 2 giờ sau (vì alert based on queue size > 10,000, not growth rate). 2 giờ notifications không gửi, reports không generate, webhooks không deliver. Fix: soft_time_limit=300 (5 phút), time_limit=600 (10 phút hard kill). External API calls: own timeout (requests.get(url, timeout=10)). Alert on queue growth RATE, không chỉ absolute size.

💡 Bài học: LUÔN set task timeout — soft limit (graceful cleanup) + hard limit (force kill). External calls within tasks MUST have own timeout. Monitor: queue size, processing rate, task duration p99, failed task rate. Alert on anomalies, not just thresholds.

📖 Chuyện 3: Airflow DAG từ Địa Ngục — 200 Tasks, 50 Dependencies

Data team build Airflow DAG cho daily data pipeline. Start simple: 5 tasks. 1 năm sau: 200 tasks, 50 cross-dependencies, 3 conditional branches, retry logic everywhere. DAG visualization: spaghetti. Nobody understands full flow. One task failure → cascade → 50 downstream tasks fail → entire daily pipeline broken. Debug: read through 200 task logs to find root cause (usually: source data format changed). Fix one task → re-run pipeline → 4 giờ to complete → data available lúc 2h chiều thay vì 8h sáng. Stakeholders unhappy. Refactor: break mega-DAG into smaller DAGs with clear interfaces. Implement data quality checks (Great Expectations) between stages. Alert on first failure, not after cascade.

💡 Bài học: Large DAGs = maintenance nightmare. Keep DAGs small and focused (< 30 tasks). Split large pipelines into multiple DAGs with data contracts between them. Implement data quality checks as gates between stages. Monitor: DAG duration trend, task success rate, SLA compliance.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Cron job without distributed lock

Multiple instances → multiple executions. Use Redis SETNX, ZooKeeper, database advisory lock. Or better: use proper scheduler (Celery Beat, BullMQ repeatable) with single scheduler instance.

❌ Sai Lầm #2: No task timeout

Task calls external service → service hangs → task hangs forever → worker pool exhausted → queue dead. ALWAYS set task timeout: soft limit + hard limit. External calls: own timeout shorter than task timeout.

❌ Sai Lầm #3: Retry without backoff

Task fails → retry immediately → fails again → retry → fails → retry... infinite loop hammering failing service. Implement exponential backoff: 1s, 2s, 4s, 8s, 16s... Max retries: 3-5. After max retries: DLQ.

❌ Sai Lầm #4: Schedule critical tasks at exact hour

Everyone schedules tasks at :00 — midnight, every hour. All tasks compete for resources. Stagger schedules: 3 * * * * (at :03), 17 * * * * (at :17). Spread load across the hour.

❌ Sai Lầm #5: No observability for scheduled tasks

Task silently fails → nobody knows until stakeholder complains. Dashboard: last run time, status, duration. Alert: task didn't run (SLA miss), task failed, task running longer than usual.
