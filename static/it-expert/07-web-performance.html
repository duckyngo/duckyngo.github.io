🗝️ Bí Kíp

Bí Kíp #1: Core Web Vitals — Ba chỉ số quyết định

LCP (Largest Contentful Paint): thời gian render element lớn nhất — target < 2.5s. Optimize: preload hero image, dùng fetchpriority='high', optimize server response time. FID/INP (Interaction to Next Paint): thời gian phản hồi user interaction — target < 200ms. Optimize: break long tasks, dùng Web Workers cho heavy computation. CLS (Cumulative Layout Shift): layout shift không mong muốn — target < 0.1. Optimize: set width/height cho images, reserve space cho ads/lazy content, dùng content-visibility.

Bí Kíp #2: Bundle Size — Kẻ thù số 1

Dùng bundlephobia.com check size trước khi npm install. moment.js = 300KB → dùng date-fns hoặc dayjs (2KB). lodash full = 70KB → import specific: import debounce from 'lodash/debounce'. Webpack Bundle Analyzer / Vite's rollup-plugin-visualizer để thấy bundle composition. Tree-shaking chỉ hoạt động với ESM — check thư viện có module field trong package.json.

Bí Kíp #3: Lazy Loading & Code Splitting

Route-based splitting: mỗi page là một chunk, load khi navigate. Component splitting: heavy components (chart, editor, map) lazy load khi cần. React.lazy() + Suspense, Vue defineAsyncComponent(). Image lazy loading: native loading='lazy' cho images below fold. Intersection Observer cho custom lazy load logic. Prefetch: <link rel='prefetch'> cho resources likely needed next.

Bí Kíp #4: Image Optimization — Low-hanging Fruit

Images thường chiếm 50%+ page weight. Format: WebP (30% nhỏ hơn JPEG), AVIF (50% nhỏ hơn). <picture> element cho format fallback. Responsive images: srcset + sizes — serve 400px image cho mobile, 1200px cho desktop. CDN image transformation (Cloudinary, imgix, Vercel/og). Blur placeholder (LQIP) cho perceived performance. NEVER serve 4000x3000 image cho 200x200 thumbnail.

Bí Kíp #5: Critical Rendering Path

Browser: HTML → DOM, CSS → CSSOM, DOM + CSSOM → Render Tree → Layout → Paint. CSS blocking rendering — inline critical CSS, defer non-critical. JavaScript blocking parsing — defer hoặc async attributes. Font loading: font-display: swap — show fallback font ngay, swap khi custom font load. Preconnect: <link rel='preconnect' href='https://api.example.com'> — establish connection early.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Bundle 15MB — Khi node_modules Lên Production

Startup launch ngày D-Day, PM check trang load: 47 giây trên 4G. DevTools Network tab: main bundle 15MB. 15 MEGABYTE JavaScript. Phân tích: full lodash (70KB), moment.js với ALL locales (300KB), three.js cho một animation nhỏ (600KB), và — kinh hoàng nhất — ai đó import toàn bộ AWS SDK (3MB) client-side chỉ để generate S3 presigned URL. Sau 1 tuần audit: thay lodash bằng lodash-es + tree-shaking, thay moment bằng dayjs, move three.js vào dynamic import, move S3 logic sang API route. Bundle: 15MB → 280KB. Load time: 47s → 1.8s. CEO: 'Vậy trước giờ chúng ta ship cái gì cho user?'

💡 Bài học: Audit bundle size TRƯỚC khi nó thành vấn đề. Integrate bundle size check vào CI — fail PR nếu bundle tăng > 10KB không giải thích. Dùng import cost VS Code extension để thấy cost ngay khi import. Quy tắc: nếu library > 50KB, phải có justification.

📖 Chuyện 2: Image Optimization Fail — 8MB Hero Image

Designer gửi hero image: 8MB, 6000x4000px PNG. Developer dùng nguyên, không resize, không compress. Mobile user trên 3G: 2 phút load hero image. LCP: 45 giây. Page bị Google đánh xuống rank vì Core Web Vitals fail. Sau khi optimize: resize 1920px max width, convert WebP (200KB), AVIF cho browser hỗ trợ (120KB), blur placeholder 20px inline SVG. LCP giảm từ 45s xuống 1.2s. Lesson: một image thay đổi = page rank thay đổi.

💡 Bài học: Images là low-hanging fruit performance. Checklist: 1) Đúng kích thước (không serve 4K cho mobile), 2) Đúng format (WebP/AVIF), 3) Compression (quality 80% đủ cho web), 4) Lazy load below-fold, 5) CDN delivery. Modern frameworks (Next.js Image, Nuxt Image) handle tự động — dùng chúng.

📖 Chuyện 3: Lighthouse Audit Emergency

Client gửi email: 'Google nói site chúng tôi slow, đang giảm rank.' Lighthouse audit: Performance 23, Accessibility 41, Best Practices 56. Trang load 12 giây, render-blocking CSS 8 files, JavaScript 3MB, no caching headers. 2 tuần sprint optimization: inline critical CSS, defer rest. Code split per route. Enable gzip/brotli. Set cache headers (static assets: 1 year, HTML: no-cache). Remove unused CSS (PurgeCSS). Add image lazy loading. Preconnect external domains. Kết quả: Performance 96, load time 1.4s. Google rank tăng từ page 3 lên page 1 trong 6 tuần. Organic traffic tăng 300%.

💡 Bài học: Web performance = SEO = revenue. Google dùng Core Web Vitals làm ranking signal. Mỗi 100ms delay = 1% conversion loss (Amazon data). Performance audit nên là part of sprint, không phải emergency response.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Optimize trước khi measure

Đừng đoán bottleneck — MEASURE. Lighthouse, WebPageTest, Chrome DevTools Performance tab. Profile trước, optimize chỗ impact lớn nhất trước. Optimize function chạy 1 lần/ngày ít value hơn optimize function chạy mỗi render.

❌ Sai Lầm #2: Preload EVERYTHING

<link rel='preload'> mọi thứ = preload không gì cả. Browser có priority system — khi bạn preload 20 resources, browser không biết cái nào quan trọng. Chỉ preload: hero image, critical font, above-fold resources.

❌ Sai Lầm #3: Không set cache headers

Static assets (JS, CSS, images) nên cache 1 year + content hash trong filename. HTML nên no-cache hoặc short TTL. Không cache = mỗi visit download lại toàn bộ. Quá cache = user thấy version cũ sau deploy.

❌ Sai Lầm #4: Third-party scripts không kiểm soát

Analytics, ads, chat widgets — mỗi cái thêm 50-500KB. Tổng: 2MB third-party JS. Load chúng async, defer, hoặc qua Partytown (off-main-thread). Audit định kỳ — remove scripts không dùng.

❌ Sai Lầm #5: Render-blocking CSS trên mobile

CSS file 300KB block rendering cho cả desktop lẫn mobile. Media query split: <link rel='stylesheet' media='(min-width: 768px)' href='desktop.css'> — mobile không download desktop CSS.
