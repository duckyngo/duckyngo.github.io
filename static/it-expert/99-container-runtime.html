Chuyên Gia containerd, CRI-O & Podman
"Docker là gateway drug. Sau Docker, bạn sẽ tìm hiểu containerd, CRI-O, Podman, và tự hỏi: 'Tại sao có nhiều cách chạy container thế?'"
📦 containerd 🔴 CRI-O 🦭 Podman 📋 OCI Spec 🔓 Rootless 🔐 Container Security 🆚 Container vs VM 🏗️ Buildah/Skopeo

📚 Thân Thế & Lai Lịch

Vận Hành Container Sư từng nghĩ Docker là duy nhất. "Container = Docker, Docker = Container." Rồi chàng ta phát hiện: Docker chỉ là MỘT implementation. Bên dưới Docker là containerd, bên dưới containerd là runc, và tất cả follow OCI (Open Container Initiative) specification.

Khi Kubernetes deprecate Docker runtime (kubernetes 1.24), chàng ta panic: "Docker chết rồi?!" Rồi chàng ta hiểu: Kubernetes dùng CRI (Container Runtime Interface), và Docker không implement CRI trực tiếp — containerd và CRI-O thì có. Docker images vẫn hoạt động — chỉ là runtime layer thay đổi.

Rồi Podman xuất hiện: "Docker-compatible nhưng daemonless, rootless, không cần root." Chàng ta thử và phải lòng. "podman run" thay thế "docker run" — alias docker=podman và cuộc sống tiếp tục.

📜 Bí Kíp

1. OCI Specification — Tiêu Chuẩn Chung: OCI Runtime Spec: how to run containers (runc, crun, youki). OCI Image Spec: how to build/distribute images. OCI Distribution Spec: how to push/pull images. Mọi container tool follow OCI = images portable giữa Docker, Podman, containerd, CRI-O. Build once, run anywhere (for real this time).
2. containerd — Industry Standard Runtime: Graduated CNCF project. Used by Docker Engine (as daemon). Used by Kubernetes (via CRI plugin). Handles: image pull, container lifecycle, networking, storage. nerdctl: Docker-compatible CLI cho containerd. Lightweight hơn Docker Engine (không có Docker CLI, Docker Compose overhead).
3. Podman — Daemonless & Rootless: No daemon = no single point of failure. Rootless containers: run without root privileges. Docker-compatible CLI: alias docker=podman works. Pods concept (group of containers — like Kubernetes pods). Podman Compose cho docker-compose compatibility. Systemd integration: podman generate systemd cho service files. Best cho: development, CI/CD, security-conscious environments.
4. CRI-O — Kubernetes-Focused: Minimal runtime cho Kubernetes only. Implements CRI directly. Lighter than containerd (doesn't do everything containerd does). Used by OpenShift (Red Hat). Stable, well-tested. Choose if: running OpenShift hoặc want minimal Kubernetes runtime.
5. Rootless Containers — Security Best Practice: Container processes run as non-root user on HOST. Even container escape = unprivileged user = limited damage. Podman natively rootless. Docker rootless mode available nhưng more complex. Limitation: cannot bind to ports < 1024 without capabilities. Solution: rootlesskit port forwarding hoặc use high ports.
6. Container vs VM Security: VMs: hardware-level isolation, separate kernel. Containers: shared kernel, namespace/cgroup isolation. Container escape = access to host kernel. gVisor (Google): userspace kernel cho containers. Kata Containers: lightweight VMs running containers. Firecracker (AWS): microVMs. For multi-tenant: VM isolation > container isolation.

📖 Chuyện Kể Đêm Khuya

🔄 Chuyện Docker to Podman Migration

Enterprise mandate: remove Docker from all servers due to Docker Desktop licensing changes. 500 developers dùng Docker daily. Migration plan: install Podman, alias docker=podman, done? Not quite.

Issues discovered: Docker Compose không compatible 100% (podman-compose vs podman compose). Docker-in-Docker cho CI/CD không work natively. Docker socket mount cho development tools breaks. Rootless mode require subuid/subgid configuration. Migration mất 3 tháng thay vì planned 2 tuần.

💡 Bài học: Docker to Podman migration phức tạp hơn 'just alias'. Test ALL workflows: development, CI/CD, Docker Compose, Docker-in-Docker. Podman compose (v2) vs podman-compose (Python) — different tools. rootless containers cần additional host configuration. Plan migration carefully với pilot group.

🔓 Chuyện Container Escape Vulnerability

Security team discover CVE trong runc — container escape vulnerability. Attacker with container access có thể overwrite runc binary trên host → gain host root access. 2000 containers running trên shared Kubernetes cluster. All containers potentially affected.

Incident response: patch runc immediately (emergency maintenance window). Audit container images cho malicious content. Review pod security policies (restrict privileged containers). Enable runtime security monitoring (Falco). Lesson: container isolation is NOT equivalent to VM isolation.

💡 Bài học: Container runtime vulnerabilities = host compromise potential. Patch container runtimes ngay khi CVE published. Restrict privileged containers (PodSecurityPolicy/Standards). Runtime security monitoring (Falco, Sysdig). For multi-tenant: consider VM-level isolation (Kata Containers, Firecracker).

🏗️ Chuyện Buildah Cho CI/CD Không Root

CI/CD pipeline cần build Docker images. Truyền thống: Docker-in-Docker (DinD) — mount Docker socket (security risk) hoặc run Docker daemon trong container (complex). Security team reject cả hai options.

Đại Sư introduce Buildah: build OCI images without Docker daemon. Rootless build. No socket mount needed. Skopeo cho image operations (copy, inspect). CI/CD pipeline: buildah bud → buildah push. Faster builds (no daemon overhead), more secure, same output images.

💡 Bài học: Buildah + Skopeo = Docker build/push without Docker daemon. Perfect cho CI/CD: no privileged containers needed. Rootless builds possible. Same Dockerfile syntax. Output = standard OCI image, pushable to any registry. Kaniko (Google) là alternative cho Kubernetes-native builds.

⚠️ Những Sai Lầm Kinh Điển

1. Docker socket mount cho everything: /var/run/docker.sock mount = root access to host. Only mount when absolutely necessary. Use alternatives: Podman socket, Buildah.
2. Ignore container runtime CVEs: runc, containerd CVEs = host security risk. Monitor và patch immediately. Subscribe to security advisories.
3. Privileged containers in production: --privileged = almost no isolation. Use specific capabilities (--cap-add) instead. Audit privileged containers.
4. Not using rootless when possible: Rootless containers significantly reduce blast radius of container escape. Default to rootless, use root only when needed.
5. Mixing container runtimes: Docker trên dev, containerd trên staging, CRI-O trên production. Subtle behavior differences. Standardize runtime across environments.
