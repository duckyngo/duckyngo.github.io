Lão mang "gen thủ kho" vào data engineering. Năm 2014, lão được giao xây data warehouse cho công ty 500 nhân viên. Bắt đầu bằng MySQL + Excel reports (đau đớn). Chuyển sang Redshift (ổn hơn). Rồi Snowflake (giác ngộ). Giờ lão quản lý warehouse chứa 50TB data, phục vụ 200 analysts hàng ngày.

Câu nói yêu thích: "Warehouse tốt là warehouse mà analyst tự phục vụ được mà không cần gọi data engineer lúc 11 giờ đêm."

📜 Bí Kíp

Bí Kíp #1: Star Schema — Ngôi Sao Dẫn Đường

Fact table ở giữa (events, transactions — nhiều rows). Dimension tables xung quanh (users, products, dates — ít rows, rich attributes). fact_orders JOIN dim_customers JOIN dim_products JOIN dim_dates. Star schema dễ hiểu, query nhanh (ít JOINs), tối ưu cho columnar storage. Snowflake schema = normalized dimensions — chỉ dùng khi thực sự cần.

Bí Kíp #2: Slowly Changing Dimensions (SCD)

Type 1: overwrite — mất history. Type 2: new row + effective dates — giữ full history, complex. Type 3: thêm cột current/previous — giữ 1 level history. 90% cases dùng Type 2. dbt snapshot module handle SCD Type 2 tự động. Luôn có effective_from, effective_to, is_current columns.

Bí Kíp #3: Snowflake — Tách Compute và Storage

Storage (S3) tách biệt Compute (virtual warehouses). Scale compute lên/xuống independent. Nhiều warehouses đọc cùng data. Auto-suspend warehouse khi idle. Time travel (query data ở thời điểm quá khứ). Zero-copy cloning cho dev/test. Nhưng: bill theo compute time + storage, có thể surprise.

Bí Kíp #4: BigQuery — Serverless, Pay-per-Query

Không manage infrastructure. Tính tiền theo bytes scanned (on-demand) hoặc slots (flat-rate). Partition by date = giảm scan = giảm bill. Cluster by frequently filtered columns. Materialized views tự auto-refresh. Nhưng: không kiểm soát compute allocation, cold start delay cho scheduled queries.

Bí Kíp #5: ClickHouse — Speed Demon Cho Analytics

Columnar storage, vectorized execution, hardware-optimized. Nhanh hơn Redshift/BigQuery cho nhiều analytics workloads. Open source, self-hosted hoặc ClickHouse Cloud. MergeTree engine với ORDER BY = super fast range queries. Nhưng: không tốt cho high-concurrency short queries, updates/deletes hạn chế.

Bí Kíp #6: Materialized Views — Pre-compute Kết Quả

Query phức tạp chạy mỗi ngày 100 lần? Tạo materialized view: pre-compute và lưu kết quả. Snowflake auto-refresh MVs. BigQuery MVs tự optimize queries sử dụng chúng. ClickHouse MVs populate on insert. Trade-off: storage vs query time. Dùng cho dashboards và repeated queries.

📖 Chuyện Kể Đêm Khuya

💸 Chuyện 1: Snowflake Bill Shock

Company dùng Snowflake, size X-Small warehouse. Mọi thứ ổn, bill $2K/tháng. Rồi team data science bắt đầu chạy ML training queries trên warehouse — scale lên 4XL. Quên auto-suspend. Warehouse chạy 24/7 dù chỉ dùng 2 giờ/ngày.

Hóa đơn cuối tháng: $47,000. CFO gần ngất. Lão rush vào: set auto-suspend 5 minutes, tạo resource monitors với alerts, tách warehouse cho data science (auto-suspend aggressively). Tháng sau: $3,500. CFO thở phào nhưng bắt lão phải review bill hàng tuần.

💡 Bài học: Cloud data warehouse = pay as you use. Auto-suspend PHẢI bật. Resource monitors + spending alerts PHẢI có. Tách warehouses theo workload. Review bills weekly, không phải monthly surprise.
🔍 Chuyện 2: Query Scan Toàn Bộ Warehouse

Analyst viết query: SELECT * FROM fact_orders WHERE customer_name = 'Nguyễn Văn A'. Fact table 5TB, không partition, không cluster. BigQuery scan 5TB → bill $25 cho 1 query. Analyst chạy query 20 lần (sửa typo, thêm filter) → $500 trong 1 buổi chiều.

Lão optimize: partition fact_orders by order_date, cluster by customer_id. Cùng query chỉ scan 50MB thay vì 5TB. Cost: $0.00025 thay vì $25. 100,000x rẻ hơn. Và lão thêm rule: mọi fact table PHẢI có partition column trong WHERE clause.

💡 Bài học: Trong BigQuery, partition + clustering = giảm cost 1000x. LUÔN partition fact tables by date. Cluster by frequently filtered columns. Require partition filter trong WHERE clause.
📊 Chuyện 3: Dimension Table Không Có History

Dim_customers chỉ lưu current state (SCD Type 1). Customer thay đổi tier từ "Silver" sang "Gold". Mọi historical orders giờ đều JOIN với tier "Gold". Report quý trước: doanh thu Gold tier tăng 200%. CEO vui mừng. Thực tế: doanh thu không đổi, chỉ là customers được upgrade tier và history bị overwrite.

Lão phải rebuild dim_customers thành SCD Type 2, restore history từ event logs (may mà còn). 2 tuần rebuild + rerun reports. CEO nhận report đúng: doanh thu tăng 15%, không phải 200%. CEO bớt vui, nhưng CFO rất vui vì báo cáo tài chính chính xác.

💡 Bài học: Dimension tables hầu hết nên SCD Type 2. History matters cho accurate analytics. Overwriting dimensions = rewriting history = sai reports. Đặc biệt quan trọng: pricing, customer segments, product categories.

⚠️ Những Sai Lầm Kinh Điển

❌ SELECT * FROM fact_table

Fact table 5TB, SELECT * = scan toàn bộ = chậm + đắt. Chọn cột cần dùng, filter by partition column, limit khi explore.

❌ Không partition fact tables

Bảng tỷ rows không partition = mọi query scan tất cả. Partition by date/time là minimum. Cluster/sort key theo common filter columns.

❌ Quá nhiều JOINs trong dashboard query

Dashboard query JOIN 12 tables, chạy mỗi 30 giây. 200 users mở dashboard = 200 × 12 JOINs mỗi 30s. Pre-aggregate vào materialized view hoặc summary table.

❌ Dev/test dùng production warehouse

Analyst test query trên production warehouse, scan 10TB, chậm mọi production queries. Tạo separate warehouse cho dev/test. Snowflake zero-copy clone cho test data.

❌ Không document tables và columns

500 tables, không ai biết dim_xyz_v2_final_REAL là gì. Data catalog + descriptions cho mọi table và column. dbt docs generate tự động từ YAML.
