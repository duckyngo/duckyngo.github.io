📜 Thân Thế & Lai Lịch

Trần Văn Thẻ Đóng sinh ra vào thuở hồng hoang của Internet, khi mà người ta còn dùng <table> để layout cả trang web và nghĩ rằng <marquee> là đỉnh cao nghệ thuật. Thời niên thiếu, ông từng viết trang web bằng Notepad trên Windows 98, mỗi lần F5 refresh là tim đập như trống trận.

Ông tu luyện qua ba đại kiếp nạn: Kiếp đầu là "Cuộc chiến IE6" — khi mà CSS viết cho Chrome chạy đẹp nhưng sang IE6 thì vỡ tan như gương. Kiếp hai là "Thời kỳ jQuery thống trị" — khi mà ai không biết $(document).ready() thì bị coi là dân ngoại đạo. Kiếp ba là "Khai ngộ ES6" — khi arrow function xuất hiện và ông ngộ ra rằng this không phải lúc nào cũng là cái mình nghĩ.

Ngày nay, Thẻ Đóng là bậc trưởng lão được kính trọng trong giới web, nổi tiếng với câu nói: "Nếu trang web của anh không chạy được khi tắt JavaScript, thì anh chưa hiểu HTML."

🗝️ Bí Kíp

Bí Kíp #1: Semantic HTML — Viết cho máy đọc, không chỉ cho người đọc

Đừng bao giờ biến trang web thành "div soup". Mỗi thẻ HTML đều có ý nghĩa riêng: <article> cho bài viết, <nav> cho điều hướng, <aside> cho sidebar, <main> cho nội dung chính. Khi bạn dùng đúng thẻ, screen reader hiểu được cấu trúc trang, Google bot index tốt hơn, và developer sau bạn không muốn resign. Quy tắc vàng: Nếu có thẻ HTML native làm được việc đó, đừng tự tạo bằng <div> + JavaScript.

Bí Kíp #2: CSS Specificity — Cuộc chiến độ ưu tiên

CSS specificity là hệ thống tính điểm: inline style (1000 điểm) > ID (100) > class/attribute/pseudo-class (10) > element/pseudo-element (1). Khi hai rule xung đột, rule có specificity cao hơn thắng. Đừng bao giờ dùng !important trừ khi đang ghi đè CSS của thư viện bên thứ ba — nó giống như dùng búa tạ để đóng đinh treo tranh. Thay vào đó, hãy nắm vững specificity và viết selector đủ cụ thể. Mẹo: Dùng BEM naming convention (.block__element--modifier) để giữ specificity phẳng và dễ quản lý.

Bí Kíp #3: JavaScript Event Loop — Hiểu để không bị "async" đánh bại

JavaScript là single-threaded nhưng non-blocking nhờ event loop. Khi bạn gọi setTimeout(fn, 0), fn không chạy ngay lập tức — nó được đẩy vào task queue và chờ call stack trống. Microtask queue (Promise callbacks) luôn được xử lý trước macrotask queue (setTimeout, setInterval). Đây là lý do tại sao Promise.resolve().then(() => console.log('B')) luôn chạy trước setTimeout(() => console.log('A'), 0). Nắm vững event loop là chìa khóa để debug async code mà không phát điên.

Bí Kíp #4: Closures — Hàm nhớ lâu hơn người yêu cũ

Closure xảy ra khi một hàm "nhớ" được scope nơi nó được tạo ra, ngay cả khi hàm đó được thực thi ở nơi khác. Đây là nền tảng của module pattern, data privacy, và factory functions. Cẩn thận với closures trong vòng lặp — cái bẫy kinh điển khi dùng var trong for loop với setTimeout. Giải pháp: dùng let (block-scoped) thay vì var (function-scoped), hoặc dùng IIFE để tạo scope mới cho mỗi iteration.

Bí Kíp #5: Prototypal Inheritance — Thừa kế kiểu JavaScript

Không giống class-based OOP trong Java/C#, JavaScript dùng prototype chain. Mọi object đều có [[Prototype]] link tới object khác. Khi truy cập property không tồn tại trên object, engine tìm ngược lên prototype chain. class syntax trong ES6 chỉ là syntactic sugar — bên dưới vẫn là prototype. Hiểu điều này giúp bạn debug được những bug kỳ lạ khi method "biến mất" hoặc instanceof cho kết quả bất ngờ.

Bí Kíp #6: CSS Layout — Flexbox cho hàng, Grid cho lưới

Quy tắc đơn giản: Flexbox cho layout 1 chiều (hàng HOẶC cột), Grid cho layout 2 chiều (hàng VÀ cột). Đừng cố ép Flexbox làm grid layout phức tạp — bạn sẽ kết thúc với flex-wrap + calc() hack xấu xí. Grid Template Areas (grid-template-areas) là công cụ mạnh nhất mà ít ai dùng — nó cho phép bạn "vẽ" layout bằng tên, dễ đọc và dễ thay đổi. Kết hợp container queries (thay vì chỉ media queries) để component tự responsive theo container chứ không phải viewport.

Bí Kíp #7: Modern JavaScript Patterns — Viết ít, làm nhiều

Nắm vững destructuring, optional chaining (?.), nullish coalescing (??), và structuredClone() cho deep copy. Dùng AbortController để cancel fetch requests — đừng để request zombie ám ảnh app. Proxy và Reflect là công cụ meta-programming mạnh mẽ cho reactivity system (Vue 3 dùng nó). Và nhớ: Array.from({length: n}, (_, i) => i) tạo mảng số thứ tự nhanh hơn bạn nghĩ.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Cơn Ác Mộng z-index

Năm 2019, team tôi nhận dự án redesign một trang thương mại điện tử. Mọi thứ đẹp lung linh cho đến khi tester report: "Modal xuất hiện ĐẰNG SAU dropdown menu." OK đơn giản, tăng z-index lên. Nhưng dropdown có z-index 999. Modal tăng lên 9999. Rồi tooltip cần lên trên modal: 99999. Rồi ai đó thêm notification overlay: 999999.

Chỉ sau 2 tuần, chúng tôi có một "cuộc chạy đua z-index" với giá trị cao nhất là z-index: 2147483647 — đúng, giá trị integer max! Và khi hai element cùng z-index max, chúng tôi ngồi đó nhìn nhau, không biết làm gì.

Vấn đề thực sự? Chúng tôi không hiểu stacking context. Mỗi khi element có position: relative/absolute + z-index, hoặc transform, opacity < 1, filter, nó tạo ra stacking context MỚI. Z-index chỉ có ý nghĩa TRONG cùng stacking context. Con z-index 999999 trong stacking context cha z-index 1 vẫn thua con z-index 2 trong stacking context cha z-index 2.

💡 Bài học: Đừng chạy đua z-index. Hãy hiểu stacking context. Tạo một file z-index tokens với các layer có tên rõ ràng (base: 1, dropdown: 10, modal: 20, toast: 30). Ít level hơn = ít đau đầu hơn.

📖 Chuyện 2: Callback Hell — Hành Trình Xuống Địa Ngục

Thời chưa có Promise, tôi từng viết một feature upload ảnh: đọc file → resize → upload → lưu URL vào DB → gửi notification. Mỗi bước là một callback lồng nhau. Code trông như "pyramid of doom" — thụt lề sâu đến mức phải scroll ngang.

Nhưng kinh hoàng thực sự là error handling. Mỗi level callback cần try-catch riêng. Quên bắt error ở bất kỳ level nào → error biến mất vào hư không, silent failure. Debug như tìm kim trong đống rơm — mà đống rơm đang cháy.

Khi junior hỏi "callback hell là gì", tôi chỉ mở file đó ra. Cậu ta im lặng 10 giây rồi nói: "Anh ơi, cái này là code hay là nghệ thuật trừu tượng?" 250 dòng code, 15 level callback lồng nhau, 3 nơi error bị nuốt im lặng. Sau đó tôi refactor sang Promise rồi async/await — file giảm từ 250 dòng xuống 45 dòng.

💡 Bài học: Async/await không chỉ là syntactic sugar — nó cứu mạng developer. Luôn dùng try-catch với async/await, và đừng quên handle rejection ở top level. Promise.allSettled() khi cần chạy nhiều async tasks song song mà không muốn một failure làm sập tất cả.

📖 Chuyện 3: Keyword "this" — Kẻ Phản Bội Thầm Lặng

Senior developer mới join team, 10 năm Java, tự tin viết JavaScript class. Code trông sạch, logic rõ ràng. Nhưng khi pass method làm callback cho event listener — BÙM — this là undefined.

"Nhưng tôi đang gọi this.handleClick mà!" — anh ta la lên. Đúng, nhưng khi JavaScript tách method ra khỏi object và pass nó như callback, this binding bị mất. Trong Java, this luôn là instance. Trong JavaScript, this phụ thuộc vào CÁCH hàm được gọi, không phải NƠI hàm được định nghĩa.

Anh ta mất 2 ngày debug, thử đủ cách: .bind(this), self = this, arrow function trong constructor. Cuối cùng ngộ ra rằng arrow function không có this riêng — nó "inherit" this từ enclosing scope. Từ đó, mọi class method dùng làm callback đều viết dạng arrow function: handleClick = () => { ... }.

💡 Bài học: Trong JavaScript, this được xác định lúc runtime bởi cách gọi hàm (call site), không phải lúc viết code. 4 quy tắc theo thứ tự ưu tiên: new binding → explicit binding (call/apply/bind) → implicit binding (method call) → default binding (global/undefined). Arrow function là ngoại lệ — luôn dùng this của scope cha.

📖 Chuyện 4: Khi CSS Animation Giết Chết Performance

Marketing yêu cầu homepage "sống động hơn" — mỗi card phải có hiệu ứng parallax, mỗi button phải pulse, background phải có particles bay. Tôi — đầy nhiệt huyết — dùng setInterval thay đổi left và top cho 200 particles.

Kết quả: 3 FPS trên điện thoại. Fan quay vù vù. Pin tụt từ 80% xuống 20% trong 5 phút. Trang web đẹp tuyệt vời — nếu bạn nhìn screenshot. Còn trên thiết bị thật, nó giống slideshow hơn là animation.

Vấn đề: left, top, width, height trigger layout recalculation (reflow). Mỗi frame, browser phải tính lại vị trí của TẤT CẢ element. Giải pháp: dùng transform: translate() và opacity — hai property duy nhất được GPU accelerate, chạy trên compositor thread, không trigger reflow.

💡 Bài học: Chỉ animate transform và opacity. Dùng will-change để hint browser chuẩn bị GPU layer. requestAnimationFrame thay vì setInterval cho animation. Và nhớ: animation đẹp nhất là animation mà user không nhận ra — smooth 60fps, subtle, purposeful.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Dùng == thay vì ===

0 == "" là true. null == undefined là true. [] == false là true. JavaScript type coercion là cánh cửa dẫn đến dimension khác. Luôn dùng strict equality === trừ khi bạn có lý do RẤT cụ thể.

❌ Sai Lầm #2: Quên rằng CSS class order không quyết định priority

class="red blue" hay class="blue red" không khác nhau. Thứ tự trong STYLESHEET mới quyết định. Nếu .blue được define SAU .red trong CSS file, .blue thắng — bất kể thứ tự trong HTML.

❌ Sai Lầm #3: innerHTML cho user input — mở cửa cho XSS

Bạn element.innerHTML = userInput? Chúc mừng, bạn vừa cho hacker inject <script> tag vào trang web. Dùng textContent cho text thuần, hoặc sanitize HTML trước khi render.

❌ Sai Lầm #4: Viết CSS selector quá cụ thể

body div.container div.content article.post div.text p span.highlight — specificity cao chót vót, không ai override được, và khi HTML structure thay đổi một chút thì CSS vỡ toàn bộ. Giữ selector phẳng, tối đa 2-3 level.

❌ Sai Lầm #5: Không hiểu hoisting

var declarations được hoist lên đầu function nhưng GIÁ TRỊ thì không. function declarations được hoist hoàn toàn. let/const có temporal dead zone — truy cập trước khi declare sẽ throw ReferenceError thay vì cho undefined. Khác nhau tinh tế nhưng gây bug khó tìm.

❌ Sai Lầm #6: Dùng for...in cho array

for...in iterate qua enumerable properties — bao gồm cả prototype chain! Dùng for...of cho arrays, hoặc tốt hơn: .map(), .filter(), .reduce() cho functional approach. for...in chỉ nên dùng cho object (và kèm hasOwnProperty check).

❌ Sai Lầm #7: Nghĩ rằng margin: auto center mọi thứ

margin: 0 auto chỉ center theo chiều ngang, và chỉ khi element có width cố định. Muốn center cả hai chiều? display: grid; place-items: center; — 2 dòng, xong. Đừng struggle với margin-top 50% minus half-height hack nữa.
