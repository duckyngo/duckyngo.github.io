📚 Thân Thế & Lai Lịch

Tân SQL Tiên Phong, tên thật là Ngô Phân Tán, là người lớn lên giữa hai thế hệ. Cha lão là DBA Oracle cổ điển, tin rằng "single-node RDBMS là đủ cho mọi thứ". Mẹ lão... à, mẹ lão không biết gì về database, nhưng bà hay nói: "Con phải biết kết hợp truyền thống và hiện đại."

Lão bắt đầu sự nghiệp bằng sharding MySQL — chia 1 database thành 64 shards. Application code phải routing query đến đúng shard. Cross-shard JOIN? Không có. Cross-shard transaction? Bằng 2PC tự viết, buggy kinh hoàng. Lão mất 3 năm tuổi trẻ cho cái sharding framework đó.

Rồi CockroachDB và TiDB xuất hiện: "Distributed SQL, horizontal scaling, ACID transactions, PostgreSQL/MySQL compatible." Lão khóc: "Nếu có thứ này 3 năm trước, tóc tôi đã không bạc."

📜 Bí Kíp

Bí Kíp #1: NewSQL Là Gì — Và Không Phải Là Gì

NewSQL = distributed database + SQL interface + ACID transactions + horizontal scaling. Lấy cảm hứng từ Google Spanner. KHÔNG phải "SQL database chạy trên nhiều server" (đó là sharding). NewSQL tự động shard, tự động rebalance, tự động handle distributed transactions. Application code viết như single-node SQL.

Bí Kíp #2: CockroachDB — Survival Là Tên Giữa

PostgreSQL-compatible wire protocol. Mọi data replicated ≥ 3 bản. Zone configs cho geographic placement. Serializable isolation mặc định (strongest). Tự động range splitting khi data lớn. Nhược: latency cao hơn single-node PostgreSQL (network roundtrips cho consensus). Dùng khi: multi-region deployment, zero-downtime requirements.

Bí Kíp #3: TiDB — MySQL Nhưng Scale Ngang

MySQL-compatible. Architecture: TiDB (SQL layer, stateless) + TiKV (distributed KV store, Raft consensus) + PD (placement driver). Scale TiDB nodes cho more connections. Scale TiKV nodes cho more storage. MySQL tools (mysqldump, MySQL Workbench) hoạt động. Migration từ MySQL: thay connection string, done (lý thuyết là vậy...).

Bí Kíp #4: Vitess — MySQL Sharding, Kubernetes Native

Không phải NewSQL thuần túy — Vitess là sharding middleware cho MySQL. YouTube dùng Vitess. Tự động shard splitting, connection pooling, query routing. Chạy trên Kubernetes native. Khi bạn đã có MySQL và cần scale mà không muốn migrate sang database mới — Vitess là lựa chọn pragmatic.

Bí Kíp #5: Distributed Transactions — Latency Trade-off

Distributed transaction = consensus protocol (Raft/Paxos) = network roundtrips = latency. Single-node PostgreSQL commit: ~0.5ms. CockroachDB commit (3-node, same region): ~5-10ms. Cross-region: ~50-200ms. Accept latency trade-off hoặc design around it: batch writes, read-only replicas cho reads.

Bí Kíp #6: Khi Nào Cần NewSQL, Khi Nào Không

CẦN: multi-region deployment, data > 1TB và growing, need horizontal write scaling, zero-downtime DDL. KHÔNG CẦN: data < 100GB (single PostgreSQL node đủ), read-heavy workload (dùng read replicas), đội nhỏ (operational complexity cao). NewSQL giải quyết vấn đề scaling, không phải vấn đề performance trên small data.

📖 Chuyện Kể Đêm Khuya

🔧 Chuyện 1: Từ 64 MySQL Shards Đến CockroachDB

Fintech company, 64 MySQL shards, sharding key = user_id. Application code: 1 routing layer, 1 cross-shard query layer, 1 distributed transaction coordinator — tất cả tự viết. 50,000 lines of sharding infrastructure code. 3 engineers full-time maintain.

Bugs: cross-shard transfers occasionally lose money (2PC bugs). Rebalancing shards khi 1 shard quá lớn = downtime. Adding new shard = migration weekend. Team kiệt sức.

Lão đề xuất migrate sang CockroachDB. 6 tháng migration, song song chạy dual-write. Kết quả: xóa 50,000 lines sharding code, 3 engineers chuyển sang feature development, zero-downtime schema changes, automatic rebalancing. CTO ôm lão.

💡 Bài học: Application-level sharding tạo ra enormous complexity. NewSQL databases encapsulate complexity này. Nếu bạn đang tự build sharding — hãy evaluate NewSQL trước khi đầu tư thêm effort.
🌏 Chuyện 2: Multi-Region Deployment — Vật Lý Không Nói Dối

Startup muốn "global deployment": users ở Việt Nam, Mỹ, và Châu Âu. Dùng CockroachDB multi-region. Setup: 3 regions, 9 nodes. Enable locality-aware leases. Write latency: 200-300ms cho cross-region transactions. Users Việt Nam thấy chậm khi transaction cần quorum từ 3 regions.

Lão optimize: REGIONAL BY ROW — mỗi row được pin vào region của user. Vietnamese users' data stays in Vietnam, US data stays in US. Cross-region chỉ khi user Việt Nam truy cập data Mỹ. Write latency cho local operations giảm từ 200ms xuống 15ms.

💡 Bài học: Distributed ≠ fast everywhere. Ánh sáng mất 130ms đi từ Việt Nam sang Mỹ (round trip). Design data locality: pin data gần users. REGIONAL BY ROW trong CockroachDB, placement policies trong YugabyteDB.
😅 Chuyện 3: "Thay Connection String Là Xong" — Sự Thật Phũ Phàng

Team migrate từ MySQL 8 sang TiDB. "MySQL compatible mà, đổi connection string thôi!" Tuần đầu: 90% queries chạy OK. Tuần hai phát hiện: AUTO_INCREMENT behavior khác (gaps), một số MySQL-specific syntax không support, stored procedures phức tạp không compatible.

Tệ nhất: query performance profile khác. Query chạy 2ms trên MySQL mất 50ms trên TiDB (distributed overhead). Nhưng query chạy 30 giây trên MySQL (vì data quá lớn cho 1 node) chạy 2 giây trên TiDB. Lão phải tune queries, adjust indexes, và educate team: "TiDB không phải MySQL nhanh hơn — nó là MySQL khác."

💡 Bài học: "Compatible" không có nghĩa "identical". Test mọi query, check edge cases, benchmark performance. Migration plan cần compatibility testing phase dài hơn bạn nghĩ. Luôn có rollback plan.

⚠️ Những Sai Lầm Kinh Điển

❌ Dùng NewSQL cho database nhỏ

Data 10GB, 100 concurrent users? Single PostgreSQL instance xử lý thoải mái. NewSQL thêm operational complexity mà không mang lại lợi ích. Đừng dùng búa tạ đập ruồi.

❌ Expect single-node latency

Distributed consensus = network roundtrips. P99 latency sẽ cao hơn single-node. Design application cho slightly higher latency: batch operations, async where possible.

❌ Ignore data locality

Multi-region setup nhưng không configure data placement. All writes routed to leader region. Other regions chờ consensus. Configure locality: zone configs, regional tables, geo-partitioning.

❌ Assume 100% SQL compatibility

CockroachDB "PostgreSQL compatible" nhưng không support tất cả PostgreSQL features (extensions, PLpgSQL fully). TiDB "MySQL compatible" nhưng edge cases khác. Test, test, test.

❌ Under-provision nodes

3 nodes minimum, nhưng 3 nodes = mất 1 node là chỉ còn bare minimum quorum. 5 nodes cho production. Separate storage và compute resource planning.
