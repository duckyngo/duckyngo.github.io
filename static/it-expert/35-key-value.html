📚 Thân Thế & Lai Lịch

Chìa Khóa Thần Tốc, tên thật là Lê Quang Nhanh, từ nhỏ đã sống theo phương châm "nhanh là tất cả". Ăn nhanh, nói nhanh, chạy nhanh, và đặc biệt — mất kiên nhẫn với bất cứ thứ gì chậm. Lão từng ném laptop ra cửa sổ vì website load mất 5 giây.

Năm 2009, khi gặp Redis, lão đã tìm được tri kỷ. Một database trả kết quả trong microseconds? Sub-millisecond latency? Lão ôm cái laptop (laptop mới, cái cũ đã bay ra cửa sổ) và khóc: "Cuối cùng có thứ gì đó nhanh bằng tôi."

Sau 17 năm, lão đã cache mọi thứ có thể cache — và một số thứ không nên cache. Lão là người tin rằng: "Mọi vấn đề performance đều có thể giải quyết bằng thêm một lớp cache. Trừ vấn đề quá nhiều cache."

📜 Bí Kíp

Bí Kíp #1: Cache Patterns — Cache Aside, Write Through, Write Behind

Cache Aside: App check cache → miss → đọc DB → ghi cache. Đơn giản, phổ biến nhất. Write Through: ghi cache + DB cùng lúc. Nhất quán nhưng chậm hơn. Write Behind: ghi cache, async ghi DB sau. Nhanh nhất nhưng risk mất data. Chọn pattern dựa trên trade-off giữa consistency và performance.

Bí Kíp #2: Eviction Policies — Khi RAM Hết, Ai Sẽ Bị Đuổi?

allkeys-lru: đuổi key ít dùng nhất — safe default cho cache. volatile-lru: chỉ đuổi key có TTL. allkeys-lfu (Redis 4+): đuổi key ít frequent nhất — tốt hơn LRU cho workload có hot keys. noeviction: trả error khi hết RAM — dùng khi data không thể mất (session store).

Bí Kíp #3: Redis Data Structures — Không Chỉ Là GET/SET

Strings cho cache đơn giản. Hashes cho objects (user profile). Lists cho queues. Sets cho unique collections. Sorted Sets cho leaderboards, rate limiting. HyperLogLog cho count distinct (chỉ tốn 12KB cho billions of items). Streams cho event log. Biết đúng data structure = giảm 90% complexity.

Bí Kíp #4: Redis Persistence — RDB vs AOF

RDB: snapshot định kỳ, nhanh restore, có thể mất data giữa 2 snapshots. AOF: log mọi write, ít mất data, file lớn, restore chậm. Kết hợp cả hai: RDB cho fast restore + AOF cho durability. Redis 7+ có Multi-Part AOF, hiệu quả hơn rất nhiều.

Bí Kíp #5: Redis Cluster vs Sentinel

Sentinel: HA cho single-master setup, tự động failover. Cluster: sharding data trên nhiều masters, scale write. Sentinel khi data < 25GB. Cluster khi cần scale beyond 1 node. Redis Cluster tối thiểu 6 nodes (3 master + 3 replica). Valkey giữ nguyên kiến trúc này.

Bí Kíp #6: etcd — Không Phải Cache, Là Nguồn Sự Thật

etcd là key-value store dùng Raft consensus, backbone của Kubernetes. Mỗi key change = consensus across cluster. Không dùng cho high-throughput data — tối đa vài nghìn writes/second. Dùng cho: config, service discovery, distributed locking, leader election. Luôn compact + defrag định kỳ.

Bí Kíp #7: Memcached — Đơn Giản Là Sức Mạnh

Khi bạn chỉ cần cache GET/SET đơn giản, Memcached nhẹ hơn Redis. Multi-threaded (Redis single-threaded cho commands). Slab allocation hiệu quả cho objects cùng size. Nhưng: không persist, không pub/sub, không data structures. Dùng khi cần pure cache, không gì khác.

📖 Chuyện Kể Đêm Khuya

🔥 Chuyện 1: Redis OOM Lúc 3 Giờ Sáng

Hệ thống e-commerce cache product data trong Redis. Mọi thứ chạy ngon lành 6 tháng. Một đêm, lúc 3 giờ sáng, PagerDuty kêu: Redis OOM (Out of Memory). Toàn bộ website down vì cache layer chết.

Điều tra: anh dev mới cache session data VÀ user activity log vào cùng Redis instance. Không set TTL. Không set maxmemory-policy. 6 tháng session data tích tụ: 62GB trên instance 64GB RAM. Một đêm traffic cao = boom.

Lão fix: tách Redis instances (cache vs session vs queue), set TTL cho mọi key, cấu hình maxmemory-policy allkeys-lru, alert khi memory > 80%. Và viết lên bảng quy tắc: "KHÔNG TTL = KHÔNG DEPLOY".

💡 Bài học: Luôn set TTL cho cache keys. Luôn cấu hình maxmemory và eviction policy. Tách Redis instances theo use case. Monitor memory usage với alerts.
⚡ Chuyện 2: Cache Stampede — Khi Cả Nghìn Request Đánh Cùng Lúc

Cache popular product vói TTL 5 phút. Sản phẩm hot nhất có 10,000 views/phút. Khi key expire: 10,000 requests đồng thời đều miss cache, đều query database, đều tính toán, đều ghi lại cache. Database bị 10,000 identical queries cùng lúc → CPU 100% → timeout → cascade failure.

Đây gọi là Cache Stampede (hay Thunder Herd). Lão dùng 3 chiến thuật: 1) Mutex lock — chỉ 1 request được rebuild cache, còn lại chờ. 2) Probabilistic early expiration — random refresh trước khi TTL hết. 3) Stale-while-revalidate — trả data cũ trong khi refresh ngầm.

💡 Bài học: Cache stampede là vấn đề thực tế ở high traffic. Implement locking hoặc probabilistic refresh. Đặc biệt nguy hiểm với expensive-to-compute cache entries.
🔐 Chuyện 3: Redis Exposed Ra Internet

Startup deploy Redis trên cloud, quên cấu hình firewall. Redis mặc định không có password, bind 0.0.0.0. Một sáng thức dậy: tất cả keys bị xóa, thay bằng 1 key chứa message: "Your data has been backed up. Send 2 BTC to recover."

Lão check access log: bot scan port 6379 (Redis default), kết nối vào, FLUSHALL, set ransom message. Toàn bộ session, cache, queue data — mất sạch. May mà cache có thể rebuild từ database. Nhưng 50,000 user sessions mất = 50,000 user phải login lại.

💡 Bài học: KHÔNG BAO GIỜ expose Redis ra internet. Luôn set password (requirepass), bind localhost hoặc private network, dùng firewall. Redis không được thiết kế để chống lại untrusted access.
🔄 Chuyện 4: etcd Disk Full Và Kubernetes Cluster Chết

Kubernetes cluster chạy 200 pods. etcd lưu toàn bộ cluster state. Không ai configure compaction. etcd database size tăng dần: 2GB, 4GB, 6GB. Đến khi chạm default quota 8GB — etcd từ chối writes. Kubernetes không thể schedule pods, không thể update deployments. Cluster đóng băng.

Panic. Lão phải manual compact etcd, defrag, tăng quota. 2 tiếng downtime. Từ đó, lão setup auto-compaction mỗi giờ và monitor etcd db size.

💡 Bài học: etcd cần maintenance: auto-compaction, periodic defrag, monitor db size. Default 8GB quota có thể không đủ cho large clusters. etcd health = Kubernetes health.

⚠️ Những Sai Lầm Kinh Điển

❌ Dùng Redis làm primary database

"Redis nhanh quá, dùng luôn làm main DB!" — cho đến khi server restart và data biến mất. Redis là cache/store bổ trợ. Primary database cần ACID guarantees mà Redis không cung cấp (trừ khi bạn rất biết mình đang làm gì).

❌ Lưu large objects trong Redis

Value 10MB? Redis single-threaded — 1 large value block tất cả requests. Giữ values nhỏ (< 100KB). Large objects → compress hoặc lưu ở object storage, chỉ cache reference.

❌ KEYS * trên production

KEYS * scan toàn bộ keyspace, block Redis cho đến khi xong. 10 triệu keys = vài giây block = hàng nghìn request timeout. Dùng SCAN cursor-based thay thế.

❌ Không set maxmemory

Redis dùng hết RAM → OS swap → latency tăng 1000x → Redis chậm hơn database → mục đích cache không còn ý nghĩa. Luôn set maxmemory = 70-80% available RAM.

❌ Cache everything, invalidate nothing

Cache user profile, user profile thay đổi, cache vẫn trả data cũ. User đổi tên "Nguyễn Văn A" thành "Nguyễn Văn B" nhưng cả hệ thống vẫn gọi anh là "A" trong 30 phút. Cache invalidation là bài toán khó — nhưng phải giải.

❌ Dùng Redis Pub/Sub cho reliable messaging

Redis Pub/Sub là fire-and-forget. Subscriber offline = message mất. Không có acknowledgment, không có retry. Cần reliable messaging? Dùng Redis Streams, hoặc tốt hơn — Kafka/RabbitMQ.
