Quản Lý Artifact Đại Sư

Chuyên Gia Nexus, Artifactory & Harbor
"Artifact management giống như quản lý kho: biết cái gì ở đâu, version nào, ai đưa vào, và quan trọng nhất — dọn dẹp kho định kỳ trước khi hết chỗ."
📦 Nexus Expert 🏗️ Artifactory Pro ⚓ Harbor Registry 🔐 Image Signing 🗑️ Cleanup Policies 📋 Versioning 🔍 Vulnerability Scan

📚 Thân Thế & Lai Lịch

Đại Sư nhớ thời "artifact storage" là shared NFS mount. Build artifacts: jar files, Docker images, npm packages — tất cả trên NFS. No versioning, no access control, no cleanup. "ai xóa artifact của tôi?!" — hàng ngày.

Khi Nexus/Artifactory ra đời, chàng ta thấy ánh sáng: repository management, access control, proxy caching, vulnerability scanning. "Mọi artifact cần một nhà," chàng ta nói.

Harbor cho container images đặc biệt powerful: vulnerability scanning built-in, image signing, replication, garbage collection. "If Docker Hub is your only registry, you are one outage away from no deployments," chàng ta cảnh báo.

📜 Bí Kíp

1. Container Registry — Harbor: Open-source, CNCF graduated. Vulnerability scanning (Trivy integrated). Image signing (Cosign/Notary). Replication (multi-registry sync). Robot accounts cho CI/CD. Tag retention policies. Garbage collection cho disk space. Quotas per project.
2. Nexus Repository: Support multiple formats: Maven, npm, Docker, PyPI, NuGet, Helm. Proxy repositories cache upstream packages. Hosted repositories cho internal artifacts. Group repositories combine proxy + hosted. Cleanup policies cho disk management. RBAC cho access control.
3. Artifactory — Enterprise Grade: Universal repository manager. Build info integration. Smart remote repositories (cache + metadata). Virtual repositories. Xray cho security scanning. Build promotion workflows. Multi-site replication. Enterprise features: access federation, HA.
4. Versioning Strategy: Semantic versioning (semver): MAJOR.MINOR.PATCH. Immutable versions: never overwrite published version. Snapshot/development versions cho pre-release. Tag conventions: v1.2.3, latest (mutable), sha-abc123. Container image tags: version + git SHA cho traceability.
5. Cleanup Policies: Retention: keep last N versions. Age-based: delete artifacts older than X days. Usage-based: delete unused artifacts. Docker: tag retention + garbage collection. Automated cleanup policies prevent disk exhaustion. BUT: never auto-delete production-referenced artifacts.

📖 Chuyện Kể Đêm Khuya

💾 Chuyện Harbor Storage Full

Harbor registry running 2 năm without cleanup policy. Storage: 2TB Docker images. Disk 95% full. New image pushes fail. CI/CD pipelines break. Developers cannot deploy. Emergency: delete old images manually — nhưng which images are still in use?

Fix: audit running containers (kubectl get pods -o jsonpath image references). Tag retention policy: keep 10 tags per repository. Garbage collection (Harbor built-in). Result: 2TB → 400GB. Set up automated cleanup schedule.

💡 Bài học: Container registries need cleanup policies from DAY ONE. Tag retention: keep last N tags. Garbage collection schedule (weekly). Monitor disk usage với alerts. Know which images are deployed (don't delete active images). Automation prevents disk full emergencies.

📦 Chuyện Wrong Artifact Deployed

CI/CD pipeline build artifact, tag as 'latest', push to registry. Two teams building same image name simultaneously. Race condition: Team A pushes, Team B overwrites 5 seconds later. Team A deploys: gets Team B's image. Feature mismatch. Users confused. 2 giờ debugging 'why is my code not deployed?'

Fix: immutable tags (version + git SHA). Never overwrite tags. CI/CD use unique tags: v1.2.3-abc1234. Deployment references exact tag, not 'latest'.

💡 Bài học: NEVER use mutable tags (latest) cho production deployments. Immutable tags: version-gitsha. Registry immutable tag setting prevent overwrites. CI/CD pipeline generate unique tags. Traceability: image tag → git commit → build log.

🔍 Chuyện Vulnerability Scan Block Deployment

Harbor integrated Trivy scanning. New policy: block deployment of images with CRITICAL CVEs. Monday morning: ALL deployments blocked. Every image has CRITICAL CVE — base image (ubuntu:20.04) has new critical CVE published over weekend.

Entire engineering team cannot deploy. Exception process takes 2 giờ approval. Fix: update base images. But lesson: scan policies need exception handling, not blanket blocks.

💡 Bài học: Vulnerability scan policies need nuance. Block CRITICAL for new deployments, alert for existing. Exception process for false positives and accepted risks. Base image update strategy (automated rebuild). Don't let perfect be enemy of good — accept-risk with documentation vs block everything.

⚠️ Những Sai Lầm Kinh Điển

1. No cleanup policy: Registry grows indefinitely. Disk full = deployments stopped. Cleanup policies from day one.
2. Using Docker Hub for everything: Rate limits, outages, no vulnerability scanning. Private registry (Harbor, ECR, GCR) cho production images.
3. Mutable tags: latest tag overwrites = unpredictable deployments. Immutable, versioned tags only.
4. No image signing: Anyone can push to registry. Unsigned images in production. Cosign/Notary cho image verification.
5. Single registry: Registry outage = no deployments. Multi-site replication cho HA. Cache registries in CI/CD.
