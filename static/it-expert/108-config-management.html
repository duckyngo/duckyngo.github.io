📚 Thân Thế & Lai Lịch

Đại Sư bắt đầu với bash scripts: ssh server "sed -i 's/old/new/g' /etc/config.conf". Works cho 1 server. 10 servers? Viết loop. 500 servers? Bash script 2000 dòng, error handling "pray and hope". Idempotency? "Run nó lại lần nữa và cầu nguyện."

Ansible thay đổi cuộc đời: YAML playbooks, agentless (SSH), idempotent modules. "Describe desired state, not steps." Chàng ta viết playbooks cho everything: user management, package installation, service configuration. Cuộc sống organized hơn.

Chàng ta cũng học Puppet (Ruby DSL, agent-based, strong enforcement) và Chef (Ruby, code-as-infrastructure). Mỗi tool có philosophy khác nhau, nhưng goal giống nhau: consistent, automated, repeatable server configuration.

📜 Bí Kíp

1. Ansible — Agentless Simplicity: SSH-based, no agent installation. YAML playbooks human-readable. Modules cho everything: package, service, file, template. Idempotent: run multiple times, same result. Roles cho reusable configuration. Galaxy cho community roles. ansible-vault cho encrypted secrets. Ad-hoc commands cho quick tasks.
2. Idempotency — Core Concept: Apply configuration multiple times → same end state. package: present, service: started — not 'install package', 'start service'. Check current state before action. Templates overwrite only if changed. Handlers run only when triggered. Idempotency = safe to re-run = safe to automate.
3. Inventory Management: Static inventory: INI or YAML files. Dynamic inventory: cloud APIs (AWS EC2, Azure VMs). Groups cho role-based targeting. Host variables và group variables. Inventory plugins cho custom sources. Đừng hardcode IPs — use DNS names hoặc dynamic inventory.
4. Roles — Reusable Units: Roles encapsulate: tasks, handlers, templates, files, defaults, vars. Galaxy roles cho common configurations. Role dependencies cho composition. Role testing: molecule cho unit tests. Ansible Collections cho distribution. Pin role versions cho reproducibility.
5. Best Practices: Playbooks trong git — version controlled. ansible-lint cho style checking. Molecule cho testing. Staging → production workflow. Tags cho selective execution. Check mode (--check) cho dry run. Diff mode (--diff) cho change preview. NEVER store secrets in plain text — ansible-vault.

📖 Chuyện Kể Đêm Khuya

🎭 Chuyện Ansible Playbook Chạy Sai Hosts

Playbook update nginx config. Inventory group [webservers] include production VÀ staging. Engineer chạy playbook targeting 'webservers' — both environments updated simultaneously. Staging config (debug mode enabled) pushed to production. Production nginx serving debug info to users.

Fix: separate inventory files per environment. --limit cho targeted execution. Mandatory --check trước --apply. CI/CD pipeline enforce environment targeting.

💡 Bài học: SEPARATE inventories per environment. --limit flag cho targeted hosts. --check mode trước apply. CI/CD pipeline restrict targeting. Group naming include environment: prod_webservers, staging_webservers.

👨‍🍳 Chuyện Chef Cookbook Conflict

Two Chef cookbooks manage same config file: cookbook_a sets max_connections = 100, cookbook_b sets max_connections = 500. Run list order determines winner. Tuesday: cookbook_a runs last → 100. Thursday: cookbook_b runs last → 500. Config flip-flops. Application behavior inconsistent.

Root cause: overlapping responsibility. Fix: single cookbook owns each config file. Attribute precedence hierarchy understood by all. Wrapper cookbooks cho customization. Code review cho attribute conflicts.

💡 Bài học: Ownership clarity: one tool/cookbook owns each configuration file. Avoid multiple tools managing same resource. Attribute/variable precedence documented. Code review catch conflicts. Testing catch drift.

🔐 Chuyện Ansible Vault Cứu Secrets

Team stored database passwords in group_vars/all.yml — plain text, committed to git. Git history preserved passwords forever, even after 'fixing' by removing them. Anyone with repo access = all production credentials.

Migration: ansible-vault encrypt all secrets. Rotate ALL exposed credentials. git filter-branch to remove history (complex, error-prone). Lesson learned: vault from day one, not after incident.

💡 Bài học: ansible-vault cho MỌI secrets từ ngày đầu. Git history preserves deleted secrets forever. Rotate credentials khi exposed. Vault password management: vault password file (gitignored) hoặc integration với HashiCorp Vault. No exceptions for 'temporary' plain text secrets.

⚠️ Những Sai Lầm Kinh Điển

1. Config management chỉ một lần: Setup server once, never run config management again. Configuration drift. Run regularly (cron) cho enforcement.
2. No testing: Push untested playbook/cookbook to production. Molecule/Test Kitchen cho automated testing. Staging trước production.
3. Ansible trong interactive mode: Expect user input during playbook run. Automation should be non-interactive. Default values, fail nếu missing.
4. Massive playbooks: 1 playbook 500 tasks. Split into roles, use includes. Single responsibility per role.
5. Plain text secrets: ansible-vault, Chef encrypted data bags, Puppet hiera-eyaml. NO EXCEPTIONS for plain text credentials.
