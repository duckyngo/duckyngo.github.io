Cảnh Báo Đại Sư

Chuyên Gia PagerDuty, OpsGenie & Alertmanager
"Alert fatigue: khi bạn nhận 500 alerts mỗi đêm, bạn sẽ bắt đầu ignore tất cả. Kể cả cái alert thực sự quan trọng lúc 3 giờ sáng."
🔔 Alertmanager 📟 PagerDuty 👁️ OpsGenie 🔇 Alert Fatigue 📋 Routing Rules 🔄 Escalation 👥 On-Call 📊 SLO-Based Alerting

📚 Thân Thế & Lai Lịch

Cảnh Báo Đại Sư từng on-call với 500 alerts mỗi đêm. Phone vibrate liên tục. 95% false positives. 4% informational. 1% actual problems. Nhưng cái 1% đó nằm trong đống 500 alerts và Đại Sư phải tìm nó như tìm kim trong bể cỏ.

"Alert fatigue is real," chàng ta nói sau 6 tháng on-call. "Tôi bắt đầu ignore alerts. Rồi production sập và tôi ngủ qua alert vì đã conditioned để ignore." Giác ngộ: better alerts > more alerts.

Giờ chàng ta thiết kế alerting systems theo triết lý: "Every alert should be ACTIONABLE. If you don't need to do something about it RIGHT NOW, it's not an alert — it's a notification."

📜 Bí Kíp

1. Alert Design Philosophy: Every alert = someone needs to take action NOW. If no action needed: dashboard, not alert. If action can wait: ticket, not page. Actionable: alert includes what happened, why it matters, how to fix (runbook link). Urgency levels: page (immediate), ticket (business hours), notification (FYI).
2. Alertmanager — Prometheus Alerting: Group related alerts (reduce noise). Inhibit: suppress child alerts when parent fires. Silence: temporary mute during maintenance. Route: different teams, different channels. Templates cho alert messages. Integration: PagerDuty, Slack, email, webhooks.
3. On-Call Best Practices: Rotation schedule: weekly, balanced. Handoff procedures: document active issues. Escalation policies: if not acknowledged in 10 min, escalate. Follow-the-sun cho global teams. Compensation cho on-call duty. Maximum on-call hours/month. On-call should not be punishment.
4. SLO-Based Alerting: Alert when error budget burning too fast. Multi-window multi-burn rate alerts. 1-hour window: 14.4× burn rate = page immediately. 6-hour window: 6× burn rate = page. This catches both sudden spikes and slow degradation. Based on Google SRE book practices.
5. Alert Routing: Team-based routing: database alerts → DBA team. Service-based: payment service → payments team. Severity-based: critical → page, warning → Slack. Business hours routing: page off-hours only cho critical. Deduplication: same alert once, not 100 times.

📖 Chuyện Kể Đêm Khuya

📟 Chuyện 500 Alerts Lúc 3 Giờ Sáng

Network switch failure → 50 services lose connectivity → 50 services throw errors → each service has 10 alert rules → 500 alerts fire simultaneously. On-call engineer phone: 500 PagerDuty notifications. Cannot identify root cause in noise. Spends 30 minutes ack-ing alerts before investigating.

Fix: alert grouping (by cluster, by failure domain). Inhibition rules: network alert inhibits service alerts. Root cause detection: alert on symptoms (network connectivity) not effects (individual service errors). Post-fix: 500 alerts → 3 alerts cho same scenario.

💡 Bài học: Alert grouping và inhibition CRITICAL cho operational sanity. Group by failure domain. Inhibit downstream alerts khi upstream alert fires. Root cause alerts > symptom alerts. Less is more — 3 actionable alerts > 500 noise alerts.

🔇 Chuyện Alert Storm Từ Một Failure

Deployment introduces memory leak. Memory grows slowly over 4 hours. Hit 80% → alert. 85% → alert. 90% → alert. 95% → alert. OOM → restart → healthy → memory grows again → repeat. On-call engineer receives 15 alerts over 4 hours for SAME issue. 'I know! I'm working on it!' — but each alert still pages.

💡 Bài học: Alert deduplication: don't re-fire same alert until resolved. Alertmanager group_wait, group_interval, repeat_interval tuning. Acknowledge = stop re-paging. Snooze/silence cho known issues under investigation. Alert should have clear resolution criteria.

👥 Chuyện On-Call Burnout

Team 3 người, on-call rotation mỗi tuần. 100+ alerts/đêm on-call. 2 người nghỉ việc vì burnout. 1 người còn lại: on-call MỌI ĐÊM. Quality giảm, response time tăng, incidents longer. CTO hỏi: 'Why is reliability getting worse?'

Fix: hire 2 more engineers. Reduce alert noise (500 → 30 per night). Automate common fixes (self-healing). On-call compensation. Mental health support. On-call cap: max 1 week per month.

💡 Bài học: On-call burnout = reliability risk. Minimum team size cho sustainable on-call: 4-5 people. Reduce alert noise FIRST. Automate common responses. Compensate on-call fairly. Monitor on-call health: alert volume trends, response times, pages per person.

⚠️ Những Sai Lầm Kinh Điển

1. Alert without runbook: Alert fires. Engineer: 'What do I do?' Runbook link in EVERY alert. Step-by-step response procedure.
2. Too sensitive thresholds: CPU > 50% alert. Server normally runs 60%. 100 false alerts/day. Set thresholds based on actual baselines + SLOs.
3. No escalation: Alert fires, nobody responds, nobody knows. Escalation: 5 min → primary, 15 min → secondary, 30 min → manager.
4. Alert on every metric: 500 alert rules. Most never fire, some fire constantly. Audit alert rules quarterly. Delete unused, tune noisy.
5. Same severity for everything: Everything is critical. Nothing is critical. Severity levels matter: critical (page), warning (Slack), info (dashboard).
