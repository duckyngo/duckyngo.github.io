📜 Bí Kíp

Bí Kíp #1: Inverted Index — Bí Mật Đằng Sau Tốc Độ

Thay vì scan mọi document tìm keyword (forward search), inverted index tạo mapping: keyword → list of documents chứa nó. Giống mục lục sách. "Elasticsearch" → [doc1, doc5, doc99]. Query chỉ cần lookup index, không scan data. Đây là lý do search engine nhanh hơn SQL LIKE millions of times.

Bí Kíp #2: Analyzers — Nghệ Thuật Xử Lý Văn Bản

Analyzer = Character Filters + Tokenizer + Token Filters. Ví dụ: "Điện thoại Samsung Galaxy S24" → lowercase → "điện thoại samsung galaxy s24" → remove diacritics → "dien thoai samsung galaxy s24" → tokenize → ["dien", "thoai", "samsung", "galaxy", "s24"]. Custom analyzer cho tiếng Việt: ICU analyzer + ASCII folding + synonym filter.

Bí Kíp #3: Relevance Scoring — TF-IDF & BM25

BM25 (default trong Elasticsearch): Term Frequency (từ xuất hiện nhiều = relevant hơn) × Inverse Document Frequency (từ hiếm = quan trọng hơn) × Field Length (document ngắn chứa keyword = relevant hơn document dài). Tune bằng: boosting fields (title^3), function_score, decay functions cho recency.

Bí Kíp #4: Cluster Architecture — Shard & Replica

Primary shards: phân tán data (set lúc tạo index, không đổi được). Replicas: bản sao cho HA và read throughput. Quy tắc: mỗi shard 20-40GB. Index 200GB → 5-10 primary shards. Số replicas tùy availability requirement. 3 data nodes minimum cho production. Dedicated master nodes khi cluster > 5 nodes.

Bí Kíp #5: Meilisearch & Typesense — Search Cho Mortals

Không phải ai cũng cần Elasticsearch cluster 20 nodes. Meilisearch: instant search, typo-tolerant, zero config, perfect cho small-medium apps. Typesense: similar nhưng có geo-search và faceting tốt hơn. Cả hai deploy single binary, dễ hơn Elasticsearch 100 lần. Dưới 10 triệu documents? Xem xét nghiêm túc.

Bí Kíp #6: OpenSearch — Elasticsearch Nhưng Tự Do

Fork từ Elasticsearch 7.10 khi Elastic đổi license. AWS managed service. Community-driven, Apache 2.0 license. Feature tương đương Elasticsearch cho hầu hết use cases. Nếu dùng AWS, OpenSearch managed service là lựa chọn hợp lý. Nếu tự host, cả hai đều OK.

📖 Chuyện Kể Đêm Khuya

🔴 Chuyện 1: Elasticsearch Cluster Red — Đúng Lúc Black Friday

E-commerce, Elasticsearch cluster 5 nodes. Black Friday, traffic search tăng x8. Cluster status chuyển từ Green → Yellow → RED. Search trả error 503. 80% revenue đến từ search — mất search = mất tiền.

Root cause: index "products" có 1 primary shard (default cũ). 1 shard = 1 node xử lý mọi search request. Node đó quá tải, shard bị unassigned, cluster red. Lão phải reindex toàn bộ 5 triệu products sang index mới với 5 shards. Reindex mất 2 tiếng. 2 tiếng Black Friday mất search = mất $500K revenue.

Từ đó, lão luôn plan shard count trước: planned_data_size / 30GB = number_of_primary_shards. Và load test trước mỗi sale event.

💡 Bài học: Số shards quyết định khả năng scale. 1 shard = bottleneck. Plan shard count dựa trên expected data size. Load test trước peak events. Có runbook cho cluster recovery.
💣 Chuyện 2: Mapping Explosion — Index Chết Vì Quá Nhiều Fields

Team log mọi thứ vào Elasticsearch. Mỗi log entry có dynamic mapping — Elasticsearch tự detect field type. Vấn đề: mỗi microservice log với schema khác nhau. Service A: {"user_id": 123}. Service B: {"userId": "abc"}. Service C: {"user": {"id": 123, "name": "An"}}.

Sau 3 tháng: index có 50,000 fields. Mỗi field có mapping metadata. Cluster state size: 2GB. Master node OOM khi load cluster state. Mỗi new document indexing phải check 50,000 field mappings → slow to crawl.

Lão phải: disable dynamic mapping, define strict mapping, normalize log format, set index.mapping.total_fields.limit. Migration mất 1 tuần.

💡 Bài học: KHÔNG BAO GIỜ dùng dynamic mapping cho logs. Define explicit mapping. Set total_fields.limit. Normalize log schema trước khi index. Mapping explosion kills clusters slowly then suddenly.
🇻🇳 Chuyện 3: Tiếng Việt Search — Có Dấu, Không Dấu, Cả Hai

User search "dien thoai" phải tìm được "điện thoại". Search "áo dài" phải tìm được "ao dai". Và ngược lại. Elasticsearch mặc định không hiểu tiếng Việt.

Lão xây custom analyzer: ICU tokenizer cho Unicode normalization + ASCII folding filter (bỏ dấu) + synonym filter + lowercase. Index mỗi field 2 lần: 1 bản có dấu, 1 bản không dấu. Search dùng multi_match trên cả 2 fields. Kết quả: "dien thoai", "điện thoại", "Điện Thoại", "DIEN THOAI" — đều tìm được. User hạnh phúc, lão tự hào.

💡 Bài học: Search tiếng Việt cần custom analyzer: ICU + ASCII folding + synonym. Index cả dạng có dấu và không dấu. Test với real user queries, không phải test data lý tưởng.

⚠️ Những Sai Lầm Kinh Điển

❌ Dùng Elasticsearch làm primary database

Elasticsearch không có transactions, eventual consistency cho reads, document updates = delete + re-index. Dùng cho search/analytics, source of truth vẫn phải là RDBMS hoặc document store.

❌ 1 shard cho index lớn

1 shard = 1 Lucene index = 1 thread xử lý. Index 100GB trên 1 shard = bottleneck. Nhưng quá nhiều shards cũng chết (mỗi shard tốn ~50MB heap). Sweet spot: 20-40GB per shard.

❌ Dynamic mapping cho production

Ai đó log field "price" là string, rồi number → mapping conflict → document bị reject. Define explicit mapping, set dynamic: strict.

❌ Query everything, filter nothing

Query context tính relevance score (tốn CPU). Filter context không tính score (nhanh, cacheable). Dùng filter cho exact matches (status, date range), query cho full-text search.

❌ Deep pagination (from: 10000, size: 10)

Elasticsearch phải sort tất cả matching documents trên tất cả shards để lấy page 1001. Giới hạn mặc định: 10,000. Dùng search_after cho deep pagination, hoặc scroll API cho export.

❌ Không monitor cluster health

Cluster Yellow = có unassigned replicas. Yellow lâu → node chết → Red → data loss. Monitor cluster health, shard allocation, JVM heap, disk usage. Grafana + Prometheus exporter là minimum.
