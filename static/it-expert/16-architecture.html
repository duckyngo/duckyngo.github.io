Monolith Microservices Modular Monolith Serverless Event-Driven CAP Theorem

📜 Thân Thế & Lai Lịch

Nguyễn Kiến Trúc bắt đầu sự nghiệp xây monolith — một ứng dụng, một database, deploy lên một server. Simple, effective, profitable. Rồi microservices hype ập đến, và CTO yêu cầu: 'Chuyển sang microservices vì Netflix dùng.' 18 tháng sau, team 8 người maintain 23 microservices. Mỗi service cần: riêng database, riêng CI/CD, riêng monitoring, riêng oncall rotation. Distributed transactions? Saga pattern. Service discovery? Consul. Logging? Centralized ELK stack. Tracing? Jaeger. Team dành 60% thời gian cho INFRASTRUCTURE thay vì PRODUCT. Ông ngộ ra: 'Microservices là organizational solution, không phải technical solution. 8 người không cần 23 services. Monolith with good boundaries > distributed mess.' Từ đó ông trở thành advocate cho 'right-sized architecture' — architecture phù hợp với team size, product stage, và actual requirements.

🗝️ Bí Kíp

Bí Kíp #1: Monolith First — Đừng premature microservices

Martin Fowler: 'Don't start with microservices.' Monolith cho phép: nhanh iterate, simple deployment, easy debugging, no network overhead, ACID transactions. Khi nào tách? Khi team > 20-30 devs, khi deploy frequency bị bottleneck bởi team coupling, khi scaling cần independent cho specific service. Tách module thành service khi có CLEAR boundary và INDEPENDENT deployment need. Tách vì 'best practice' = premature optimization.

Bí Kíp #2: Modular Monolith — Best of Both Worlds

Structure: monolith deployment nhưng internal organization theo bounded contexts. Mỗi module: riêng domain logic, riêng database schema (hoặc schema separation), riêng public API (interface). Modules communicate qua in-process calls (nhanh) nhưng through defined interfaces (loose coupling). Khi cần tách thành microservice → extract module, chuyển in-process call sang HTTP/gRPC. Much easier than refactoring tangled monolith.

Bí Kíp #3: Microservices — Khi nào thực sự cần

Prerequisites: 1) Team đủ lớn để own separate services, 2) CI/CD mature, 3) Observability infrastructure (logging, tracing, metrics), 4) Container orchestration (Kubernetes). Mỗi service: single responsibility, own data store, independently deployable. Communication: synchronous (HTTP/gRPC) cho query, asynchronous (message queue) cho commands/events. Cẩn thận: distributed monolith = worst of both worlds (coupled services + distributed complexity).

Bí Kíp #4: Serverless — Pay per Use

AWS Lambda, Cloudflare Workers, Vercel Functions. Ưu: zero infrastructure management, auto-scale to zero, pay only when invoked. Nhược: cold start latency (100ms-2s), execution time limits (15 min Lambda), vendor lock-in, debugging difficulty, state management (must use external store). Dùng cho: event-driven workloads, webhooks, scheduled tasks, low-traffic APIs. KHÔNG dùng cho: long-running processes, real-time systems, high-throughput low-latency services.

Bí Kíp #5: Event-Driven Architecture

Events = immutable facts that happened. Event-driven: services react to events, not direct calls. Loose coupling: publisher doesn't know subscribers. Event Sourcing: store events as source of truth, derive state from event replay. CQRS: separate read model (optimized for queries) from write model (optimized for commands). Cẩn thận: eventual consistency (read after write might see stale data), debugging event chains complex, event schema evolution challenging.

Bí Kíp #6: CAP Theorem & Distributed Systems

CAP: Consistency, Availability, Partition tolerance — choose 2 (in reality: choose between C and A when partition happens). CP systems (strong consistency): banking, inventory → correct data always, may be unavailable during partition. AP systems (high availability): social media, caching → always available, may serve stale data. PACELC: extends CAP — when no Partition, choose between Latency and Consistency. Understand tradeoffs → make informed architecture decisions.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Premature Microservices — Distributed Monolith

Startup 5 devs, MVP chưa có product-market fit. CTO (ex-Google): 'We build microservices from day 1!' 6 services cho một TODO app: User Service, Auth Service, Task Service, Notification Service, File Service, API Gateway. Thực tế: mỗi change cần update 3-4 services. Deploy cần coordinate timing. Transaction across services = saga nightmare cho simple operations. One developer maintain 2 services = context switching hell. Local development: docker-compose với 6 services + 3 databases + message broker + service mesh = laptop fan quay vù vù. Sau 8 tháng, chưa launch được MVP. Competitor dùng Rails monolith launch trong 2 tháng, đã có 10,000 users. Team merge lại thành modular monolith: ship features gấp 4 lần, deploy đơn giản, debug dễ dàng.

💡 Bài học: Microservices tax: service mesh, distributed tracing, saga patterns, API gateways, container orchestration, separate CI/CD per service. Nếu team < 10 devs VÀ product chưa proven → monolith (hoặc modular monolith). Microservices cho organizational scaling, KHÔNG cho technical scaling (horizontal scale monolith vẫn OK cho đa số workloads).

📖 Chuyện 2: Distributed Monolith — Worst of Both Worlds

Company 'migrate' sang microservices bằng cách: tách code thành services nhưng tất cả services share cùng 1 database, deploy cùng lúc, và call nhau synchronously cho mọi operation. Kết quả: distributed monolith — complexity của distributed system + coupling của monolith. Change 1 table schema → break 5 services. Deploy 1 service → must deploy all (vì shared DB dependencies). Latency tăng 10x (in-process calls → HTTP calls) nhưng coupling vẫn tight. Team đùa: 'Chúng ta đã thành công chuyển từ 1 monolith thành 12 monoliths nói chuyện qua mạng.' Fix (18 months): implement database-per-service, extract shared data vào events, replace synchronous calls với message queue cho non-critical paths.

💡 Bài học: Microservice PHẢI own data riêng. Shared database = shared schema = coupled services = distributed monolith. Database per service + event-driven communication cho data sync. Nếu chưa sẵn sàng cho database separation → chưa sẵn sàng cho microservices.

📖 Chuyện 3: Serverless Cold Start trong Payment Flow

Payment processing trên AWS Lambda. Average response: 50ms (warm). Nhưng cold start: 2 seconds. User click 'Pay' → 2 second wait → user click again → double payment. Rate: 1 in 100 transactions bị double charge. Total: $50,000 double-charged trong 1 tháng. Root cause: Lambda cold start + no idempotency key + no client-side disable-button-on-click. Triple failure. Fix: implement idempotency key (client generate unique key per transaction), provisioned concurrency cho payment Lambda (eliminate cold start, nhưng cost tăng), client disable button after click. Bigger lesson: critical path (payment, inventory) cần guaranteed low latency → provisioned compute, not serverless cold start.

💡 Bài học: Serverless cold start = unacceptable cho latency-sensitive operations (payments, real-time). Options: provisioned concurrency ($$), keep-warm pings (hack), hoặc dùng always-on compute cho critical paths. Idempotency keys cho MỌII payment/mutation endpoint — regardless of architecture.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Microservices cho team 3 người

3 devs, 15 microservices = mỗi dev own 5 services. Oncall rotation? 1 người cho 5 services. Vacation? Remaining 2 cover 7.5 services each. ĐỪNG. Monolith hoặc modular monolith.

❌ Sai Lầm #2: Synchronous chain: A → B → C → D

Request từ client → Service A → Service B → Service C → Service D. Latency = sum of all. One service down = entire chain fails. Break synchronous chains: async where possible, circuit breaker for resilience, cache for read-heavy paths.

❌ Sai Lầm #3: Không có observability trước khi tách services

Debug monolith: log file + debugger. Debug microservices: distributed tracing (Jaeger/Zipkin), centralized logging (ELK/Loki), metrics (Prometheus). Setup observability TRƯỚC khi tách services, không phải sau khi production incident.

❌ Sai Lầm #4: Event-driven everywhere

Async events cho mọi communication → eventual consistency everywhere → user submit form → 'processing...' → refresh page → data not there yet → confused user. Sync cho user-facing operations, async cho background processing.

❌ Sai Lầm #5: Ignore failure modes

Network unreliable, services crash, databases timeout. Design for failure: circuit breakers, retries with backoff, bulkheads, graceful degradation. 'Happy path only' architecture = production incidents guaranteed.
