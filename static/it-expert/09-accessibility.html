Email đó thay đổi cuộc đời ông. Ông bắt đầu học dùng screen reader — bịt mắt duyệt web bằng VoiceOver suốt 1 tuần. Trải nghiệm đó khiến ông rơi nước mắt: 90% websites không navigable bằng keyboard, form không có labels, images không có alt text, modal mở mà focus vẫn ở behind. Từ đó ông trở thành Hiệp Sĩ Hòa Nhập, tự nguyện audit accessibility cho mọi project, đào tạo dev teams, và viết linter rules bắt buộc. Motto: 'Accessibility không phải feature — nó là quyền cơ bản.'

🗝️ Bí Kíp

Bí Kíp #1: Semantic HTML — Nền tảng Accessibility

Screen readers dùng HTML semantics để navigate. <nav> = navigation landmark, user nhảy tới nhanh. <main> = nội dung chính. <button> = focusable, clickable by keyboard (Enter/Space). <div onclick> = KHÔNG focusable, keyboard users bị mắc kẹt. Quy tắc: dùng native HTML elements trước. Chỉ cần ARIA khi native element không đủ. 'No ARIA is better than bad ARIA.'

Bí Kíp #2: ARIA Roles, States, Properties

role='dialog' cho modal, aria-expanded='true/false' cho dropdown, aria-live='polite' cho dynamic content updates (toast notifications). aria-label khi text label không visible, aria-labelledby khi label là element khác. aria-hidden='true' cho decorative elements. Cẩn thận: sai ARIA worse than no ARIA — role='button' trên div nhưng không handle keyboard = broken for everyone.

Bí Kíp #3: Keyboard Navigation

MỌI interactive element phải focusable và operable bằng keyboard. Tab order logical (đừng dùng tabindex > 0). Trap focus trong modal (Tab/Shift+Tab cycle trong modal, không escape ra ngoài). Escape key đóng modal/dropdown. Arrow keys cho menu/tabs. Skip to main content link cho screen reader users bypass navigation. Test: unplug mouse, navigate bằng keyboard — nếu bạn mắc kẹt ở đâu, user cũng vậy.

Bí Kíp #4: Color Contrast & Visual Design

WCAG AA: text contrast ratio 4.5:1 (normal text), 3:1 (large text 18px+ bold). Đừng dùng color alone để convey information — thêm icon, pattern, text. Red/green colorblind = 8% nam giới. Test: Chrome DevTools > Rendering > Emulate vision deficiencies. Tools: axe DevTools, WAVE, Stark (Figma plugin). Dark mode cũng cần đủ contrast — đừng dùng gray text trên dark gray background.

Bí Kíp #5: Forms & Error Handling Accessible

Label PHẢI liên kết với input (<label for='email'> hoặc wrap input). Error messages: aria-describedby link error text với input, aria-invalid='true' khi validation fail. Dùng role='alert' hoặc aria-live cho error messages xuất hiện dynamic. Autocomplete attributes: autocomplete='email', autocomplete='new-password' — giúp browser và screen reader autofill đúng.

Bí Kíp #6: Testing Accessibility

Automated (40% issues): axe-core, Lighthouse a11y audit, ESLint plugin jsx-a11y. Manual (60% issues): keyboard navigation test, screen reader test (VoiceOver trên Mac, NVDA trên Windows), zoom 200% test, high contrast mode test. User testing: mời người khuyết tật test — automated tools không thể thay thế real user experience.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Form Không Có Labels — Mê Cung Cho Screen Reader

Trang đăng ký có 8 input fields. Designer muốn 'clean design' — không labels, chỉ placeholder text. Trông đẹp trên screenshot. Screen reader user mở form: 'Edit text. Edit text. Edit text. Edit text...' Không biết field nào là gì. Placeholder biến mất khi bắt đầu gõ — user quên field này hỏi gì. Tệ hơn: password field có requirement 'Tối thiểu 8 ký tự, 1 chữ hoa, 1 số' — nhưng chỉ hiển thị khi validation fail, và error message không linked với input. Screen reader user gõ password, submit, fail, KHÔNG BIẾT TẠI SAO fail. Fix: thêm visible labels (hoặc ẩn bằng CSS nhưng vẫn accessible qua sr-only class), aria-describedby cho requirements, aria-invalid + role='alert' cho error messages.

💡 Bài học: Labels là BẮT BUỘC, không phải optional. Placeholder KHÔNG thay thế label (biến mất khi type, contrast thấp, screen reader support inconsistent). Nếu design không muốn visible labels — dùng aria-label hoặc visually hidden labels (.sr-only). Mỗi input cần: label + description + error message accessible.

📖 Chuyện 2: Div Soup Accessibility Nightmare

SPA hiện đại, mọi thứ là <div>. Navigation bar = div với onClick handlers. Buttons = div styled like buttons. Headings = div với font-size lớn. Table data = grid of divs. Screen reader user: không thấy landmarks (no nav, no main), không tab được vào 'buttons' (div không focusable), heading navigation không hoạt động (không có h1-h6), data table không navigable by row/column. Kết quả: user khiếm thị hoàn toàn không dùng được app. Audit phát hiện 147 accessibility violations. Team mất 3 tháng refactor: thay div bằng semantic elements, thêm ARIA cho custom widgets, implement keyboard navigation. Tổng chi phí fix > chi phí build accessible từ đầu gấp 5 lần.

💡 Bài học: Build accessible từ đầu LUÔN rẻ hơn fix sau. Checklist mỗi component: 1) Dùng semantic HTML? 2) Keyboard focusable? 3) Screen reader understandable? 4) Color contrast OK? Tích hợp a11y linting vào CI — catch issues trước khi merge.

📖 Chuyện 3: Color-only Status Indicators

Dashboard hiển thị server status bằng màu: xanh = OK, vàng = warning, đỏ = error. Đẹp, intuitive — cho người phân biệt được màu. Red-green colorblind user (8% nam giới): xanh và đỏ trông GIỐNG NHAU. Tất cả servers trông 'OK' trong khi thực tế 3 servers đang down. Incident kéo dài 2 giờ vì on-call engineer colorblind không nhận ra alert. Fix: thêm icons (checkmark, warning triangle, X mark) + text labels bên cạnh color indicators.

💡 Bài học: KHÔNG BAO GIỜ dùng color alone để convey information. Luôn có secondary indicator: icon, text, pattern, shape. Test với colorblind simulation (Chrome DevTools > Rendering > Emulate vision deficiencies). Applies cho: status indicators, form validation, charts, data visualization.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Dùng div thay button

div + onClick KHÔNG focusable bằng keyboard, KHÔNG triggered bằng Enter/Space, screen reader KHÔNG announce là button. Dùng <button> — free keyboard support, free ARIA role, free focus styling.

❌ Sai Lầm #2: Alt text 'image.jpg' hoặc trống cho content images

Decorative images: alt='' (empty, screen reader skip). Content images: alt text mô tả NỘI DUNG, không phải filename. 'Biểu đồ doanh thu Q3 tăng 25%' thay vì 'chart.png' hoặc 'biểu đồ'.

❌ Sai Lầm #3: tabindex > 0 để 'fix' focus order

tabindex='5' đặt element lên đầu tab order — nhưng phá vỡ natural flow. Chỉ dùng: tabindex='0' (add to tab order), tabindex='-1' (programmatic focus only). Fix DOM order thay vì hack tabindex.

❌ Sai Lầm #4: Autoplay video/audio

Screen reader user: đang navigate bằng audio → video autoplay → KHÔNG NGHE ĐƯỢC screen reader. Luôn muted autoplay hoặc no autoplay. Provide controls dễ tìm để pause/stop.

❌ Sai Lầm #5: Modal không trap focus

Modal mở → Tab key → focus thoát ra element behind modal → user tương tác với hidden content. Implement focus trap: Tab cycle trong modal, Escape đóng modal, focus return to trigger element khi đóng.
