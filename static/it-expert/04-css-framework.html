📜 Thân Thế & Lai Lịch

Phạm Mỹ Thuật lớn lên trong thời đại "pixel-perfect" — khi mà designer gửi PSD file và developer phải match từng pixel một. Ông từng dùng float-based layout, clearfix hack, và table-based design. Khi Bootstrap ra đời, ông tưởng mình đã tìm được chân lý — cho đến khi MỌI website trên Internet đều trông giống nhau.

Rồi Tailwind CSS xuất hiện, và cộng đồng chia thành hai phe: "Utility-first là thiên tài!" vs "Inline style với extra steps!". Ông đứng giữa hai phe, uống trà, quan sát, rồi thầm nghĩ: "Cả hai đều đúng, cả hai đều sai."

Ngày nay, ông chuyên tư vấn cho các startup về design system — biến đống CSS lộn xộn thành hệ thống có tổ chức. Motto: "CSS framework tốt nhất là cái mà team bạn consistent dùng được."

🗝️ Bí Kíp

Bí Kíp #1: Utility-First (Tailwind) vs Component-Based (Bootstrap) — Chọn ai?

Tailwind: Viết style trực tiếp trong HTML bằng utility classes. Ưu điểm: không cần đặt tên class, không context switching, tree-shaking loại bỏ CSS không dùng. Nhược điểm: HTML dài dòng, learning curve cho class names. Bootstrap: Pre-built components (.btn, .card, .modal). Ưu điểm: nhanh, ít custom, documentation tốt. Nhược điểm: mọi site trông giống nhau, khó customize deeply. Quy tắc: Tailwind cho custom design, Bootstrap cho admin panel/internal tools, Material UI cho design system consistency.

Bí Kíp #2: Design Tokens — Ngôn ngữ chung giữa Design và Dev

Design tokens là bộ giá trị gốc: colors, spacing, typography, shadows — defined 1 lần, dùng everywhere. Tailwind config, CSS custom properties, hoặc tools như Style Dictionary. Khi designer nói "primary-500", developer biết chính xác là #6366f1. Khi rebrand, thay đổi 1 file tokens → cả app update. Đừng hardcode #6366f1 rải rác 500 chỗ — dùng var(--color-primary-500) hoặc Tailwind's text-primary-500.

Bí Kíp #3: Dark Mode — Không chỉ đổi background

Dark mode đúng cách: không chỉ invert colors. Text trắng trên nền đen → contrast quá cao → mỏi mắt. Dùng xám nhẹ cho text (#e0e0e0), dark gray cho background (#1a1a2e). Shadows trong dark mode dùng darker shade thay vì rgba black. Images cần giảm brightness. Tailwind dark: prefix hoặc CSS prefers-color-scheme. Lưu preference trong localStorage và respect system setting mặc định.

Bí Kíp #4: Tailwind CSS nâng cao — Beyond Basics

@apply để extract repeated utility patterns vào CSS class — nhưng ĐỪNG lạm dụng (nếu @apply everywhere thì tại sao dùng Tailwind?). Tailwind plugins cho custom utilities. group-hover, peer-checked cho state-based styling. JIT (Just-In-Time) mode arbitrary values: w-[237px], bg-[#1da1f2]. Tailwind Merge library để handle conflicting classes khi components compose. cva (Class Variance Authority) cho type-safe component variants.

Bí Kíp #5: CSS-in-JS — Khi nào dùng, khi nào tránh

Styled-components, Emotion: CSS viết trong JS, scoped tự động, dynamic styles dễ dàng. Nhưng: runtime overhead (JS phải parse CSS), không cache được CSS riêng, SSR complexity. Zero-runtime CSS-in-JS alternatives: Vanilla Extract, Panda CSS, StyleX — compile-time CSS extraction, best of both worlds. Quy tắc: nếu dùng Tailwind, không cần CSS-in-JS. Nếu cần programmatic styles, dùng zero-runtime solutions.

Bí Kíp #6: Headless UI Libraries — Logic without Style

Radix UI, Headless UI, React Aria, Ark UI: cung cấp behavior + accessibility, KHÔNG có styling. Bạn style theo ý muốn bằng Tailwind/CSS. So với Material UI: MUI cho bạn cả behavior + style (opinionated), Radix cho bạn behavior only (flexible). Khi project cần custom design → headless + Tailwind. Khi cần ship nhanh, consistent design → full component library (MUI, Ant Design, Mantine).

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: className Dài 200 Ký Tự

Junior dev push code, PR review: một button component có className: "flex items-center justify-center gap-2 px-6 py-3 bg-gradient-to-r from-indigo-500 via-purple-500 to-pink-500 text-white font-semibold rounded-xl shadow-lg hover:shadow-xl hover:scale-105 active:scale-95 transition-all duration-300 ease-in-out disabled:opacity-50 disabled:cursor-not-allowed focus:ring-2 focus:ring-offset-2 focus:ring-indigo-500 dark:focus:ring-offset-gray-900"

Senior review comment: "Bạn vừa viết cả một paragraph trong className." Junior reply: "Nhưng mà... Tailwind way?" Senior: "Tailwind way là extract thành component, không phải viết tiểu thuyết trong HTML attribute."

Team tạo rule: className > 80 ký tự → extract thành component hoặc dùng @apply. Từ đó, codebase sạch hơn, PR review dễ hơn, và không ai phải scroll ngang để đọc một cái button nữa.

💡 Bài học: Tailwind utility classes tuyệt vời cho prototyping và one-off styles. Nhưng khi pattern lặp lại → extract thành component (React/Vue component, không phải CSS class). @apply chỉ dùng cho base styles, component composition cho complex patterns. Dùng Prettier plugin tailwind để auto-sort class order.

📖 Chuyện 2: Bootstrap Version Upgrade — Cuộc Đại Tu Đau Thương

Dự án dùng Bootstrap 3, quyết định upgrade lên Bootstrap 5. "Chỉ update version thôi mà" — PM nói. 2 tháng sau, team vẫn đang fix regression bugs.

Bootstrap 3 → 4: Grid system đổi từ float sang flexbox. Tất cả .col-xs-* thành .col-*. Glyphicons bị remove. Panel → Card. .pull-left → .float-start. Bootstrap 4 → 5: jQuery bị remove hoàn toàn. .ml-* → .ms-* (margin-start for RTL). .left/.right → .start/.end.

Mỗi thay đổi nhỏ nhưng rải rác 500 files. Find & Replace giải quyết 70%, 30% còn lại cần manual check vì class names trùng với business logic. Sau vụ này, team viết wrapper components cho Bootstrap — để lần upgrade sau chỉ cần sửa wrappers.

💡 Bài học: Đừng dùng CSS framework classes trực tiếp rải rác trong codebase. Tạo abstraction layer: design system components wrap framework classes. Khi upgrade framework, chỉ sửa abstraction layer. Và LUÔN đọc migration guide trước khi update major version.

📖 Chuyện 3: "Pixel Perfect" Meeting từ Địa Ngục

Designer gửi Figma mockup, yêu cầu "pixel perfect". Developer implement bằng Tailwind, responsive trên mọi kích thước. Cuộc họp review: designer mở Figma bên trái, website bên phải, zoom 400%, overlay hai màn hình: "Cái button này lệch 2 pixel."

Dev: "Nhưng nó responsive, trên mobile hiển thị đẹp, accessible, có focus ring..." Designer: "2 pixel. Lệch. Sửa." Dev sửa, thêm mt-[2px]. Designer: "Giờ trên tablet lệch 1 pixel." Dev: *cầm laptop ra ban công hít thở*

Giải pháp: team tạo shared design tokens, designer dùng 4px grid system (spacing increment = 4px), developer map vào Tailwind spacing scale. Không còn arbitrary pixel values, cả hai nói cùng ngôn ngữ.

💡 Bài học: "Pixel perfect" là myth trên responsive web — mỗi device render khác nhau. Thay vào đó, establish design system với consistent spacing, typography scale, và color palette. Designer và developer dùng cùng token system. Kết quả: consistent design, không còn debate 2 pixel.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Import toàn bộ CSS framework

@import 'bootstrap' → 250KB CSS, dùng 10% utilities. Tailwind JIT mode tự động tree-shake. Bootstrap cần custom import hoặc PurgeCSS. Material UI hỗ trợ tree-shaking qua named imports. Luôn audit CSS bundle size.

❌ Sai Lầm #2: Override framework styles bằng !important

Khi MUI button không đúng màu, bạn dùng !important. Rồi hover state cần !important. Rồi disabled state. Dừng lại — dùng theme customization API, CSS custom properties, hoặc styled() wrapper. !important là dấu hiệu bạn đang fight framework thay vì work with it.

❌ Sai Lầm #3: Không dùng CSS custom properties cho theming

:root { --brand-primary: #6366f1; } → thay đổi 1 chỗ, cả app update. Tốt hơn là hardcode hex values 500 lần. Custom properties cascade, có thể override per component/section, và JavaScript đọc/ghi được qua getComputedStyle.

❌ Sai Lầm #4: Dùng Tailwind nhưng không config theme

Tailwind mặc định có palette, spacing, breakpoints — nhưng project bạn có brand colors riêng. Không config tailwind.config.js → dùng arbitrary values everywhere bg-[#1a2b3c] → mất consistency. Config theme trước, dùng semantic names: bg-brand-primary.

❌ Sai Lầm #5: Quên responsive testing trên thiết bị thật

Chrome DevTools responsive mode !== thiết bị thật. Touch targets khác mouse hover. Virtual keyboard push content lên. Notch/safe area trên iPhone. Luôn test trên ít nhất 2-3 thiết bị thật trước launch.
