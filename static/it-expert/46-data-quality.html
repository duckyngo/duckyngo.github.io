📚 Thân Thế & Lai Lịch

Chất Lượng Dữ Liệu Quan, tên thật là Kiểm Tra Nghiêm, sinh ra trong gia đình QC (Quality Control) nhà máy. Mẹ kiểm tra chất lượng ở nhà máy may, bố kiểm tra ở nhà máy điện tử. Bữa cơm gia đình thường xuyên có các cuộc thảo luận như: "Hôm nay tỷ lệ lỗi 0.3%, giảm từ 0.5% tuần trước."

Vào ngành data, lão áp dụng tinh thần QC: mọi data phải qua kiểm tra trước khi "xuất xưởng". Đồng nghiệp ban đầu ghét lão vì lão block pipeline khi data không đạt chuẩn. Nhưng sau khi lão chặn được 1 batch data sai gây sai report gửi cho investor, mọi người bắt đầu tôn trọng.

Giờ lão là người mà team data gọi khi có vấn đề: "Anh Nghiêm ơi, data hôm nay trông... lạ." Và lão luôn trả lời: "Data không bao giờ 'lạ'. Chỉ có bạn chưa hiểu nó thôi."

📜 Bí Kíp

Bí Kíp #1: Data Contracts — Hợp Đồng Giữa Producer Và Consumer

Data contract = thỏa thuận formal: schema (tên, type, nullable), SLA (freshness, completeness), ownership. Producer cam kết data đúng contract. Consumer biết expect gì. Khi contract bị break → alert, không phải "surprise" lúc 3 giờ sáng. Implement bằng JSON Schema, Protobuf, hoặc YAML spec.

Bí Kíp #2: 6 Dimensions Của Data Quality

Completeness: thiếu data không? Accuracy: data đúng không? Consistency: data nhất quán giữa systems? Timeliness: data fresh không? Uniqueness: có duplicates không? Validity: data tuân theo rules/format? Measure cả 6, không chỉ 1-2.

Bí Kíp #3: Great Expectations — Framework Test Data

Viết "expectations" như unit tests cho data: expect_column_values_to_not_be_null("email"), expect_column_values_to_be_between("age", 0, 150), expect_column_pair_values_to_be_in_set("country", "currency", {...}). Chạy trong pipeline, fail = stop pipeline. Data Docs tự generate documentation.

Bí Kíp #4: Data Catalog — Bản Đồ Kho Báu

DataHub, Amundsen, OpenMetadata, Atlan. Catalog lưu: table descriptions, column descriptions, ownership, lineage, quality scores, usage statistics. Analyst tự tìm data mà không cần hỏi. Lineage: biết data đến từ đâu, ảnh hưởng downstream nào. Investement sớm, return lâu dài.

Bí Kíp #5: Data Profiling — Hiểu Data Trước Khi Dùng

Profile data mới: distribution (mean, median, mode, outliers), null rate, cardinality, patterns. Phát hiện anomaly sớm: "Ủa, cột age có giá trị -5 và 999?" hoặc "Tại sao 40% emails là NULL?" Profile định kỳ detect drift: "Tháng này null rate tăng từ 2% lên 30% — có gì đó sai ở source."

Bí Kíp #6: Anomaly Detection — Biết Trước Khi Stakeholder Biết

Row count giảm 50% so với yesterday? Average order value tăng 10x? NULL rate tăng đột ngột? Automated anomaly detection catch issues trước khi CEO mở dashboard và thấy số lạ. Tools: Monte Carlo, Soda, hoặc custom checks dùng statistical thresholds.

📖 Chuyện Kể Đêm Khuya

🤯 Chuyện 1: NULL vs Empty String — Cơn Ác Mộng Triền Miên

System A lưu "no phone" là NULL. System B lưu là empty string "". System C lưu là "N/A". System D lưu là "0000000000". Warehouse merge 4 systems: cột phone_number có 4 representations khác nhau cho cùng 1 meaning.

Report "% customers có phone": System A = 60%, System B = 100% (empty string ≠ NULL), tổng hợp = 78%. Marketing gọi campaign cho "customers có phone" → gọi vào "", "N/A", "0000000000". Kết quả: 30% failed calls, waste $50K. Lão phải tạo standardization layer: tất cả biến thể → NULL or valid phone format. Regex validation. 2 tuần cleanup.

💡 Bài học: NULL, empty string, "N/A", "0", "none" — đều khác nhau trong database nhưng cùng ý nghĩa business. Standardize representation trong pipeline. Validate data types và patterns. Data contracts giữa systems phải specify how "missing" is represented.
📋 Chuyện 2: Undocumented Columns — "Ai Biết Cột Này Là Gì?"

Warehouse có 500 tables, 8000 columns. Không có documentation. Analyst mới vào, mở table fact_transactions: cột flag_1, flag_2, status_cd, type_x. Hỏi team: không ai biết. DBA cũ đã nghỉ việc 2 năm trước. Codebase không có comment.

Analyst dùng flag_1 (nghĩ là "active flag") trong report. Hóa ra flag_1 = "test transaction flag". Report bao gồm test data → số liệu sai 20%. CEO ra quyết định dựa trên report sai. Hậu quả: sai strategy 1 quý.

Lão bắt buộc: mọi column phải có description trong dbt YAML, DataHub catalog enrichment, weekly data catalog review sessions.

💡 Bài học: Undocumented columns are landmines. Invest in documentation: column descriptions, business definitions, data lineage. Nếu bạn không biết column nghĩa gì — ĐỪNG dùng nó trong report. Hỏi trước, dùng sau.
📊 Chuyện 3: Data Quality Score — Từ Cảm Tính Đến Định Lượng

Mỗi tuần meeting stakeholders: "Data quality hôm nay thế nào?" Team data: "Ờ... tốt?" Không ai đo, không ai biết cụ thể. Cho đến khi CEO hỏi: "Tôi có thể tin 100% vào số liệu trên dashboard không?" Im lặng.

Lão xây Data Quality Score: mỗi table có score 0-100 dựa trên: freshness (data mới nhất bao lâu), completeness (null rate), uniqueness (duplicate rate), validity (pass business rules). Dashboard hiển thị score mỗi table. Weekly report: "Data quality score: 94/100. 3 issues detected, 2 resolved." CEO hài lòng. Trust tăng.

💡 Bài học: "Trust" trong data phải measurable, không phải cảm tính. Build data quality scorecard. Track trends. Report regularly. Khi score giảm → investigate immediately. Quality score = trust currency.

⚠️ Những Sai Lầm Kinh Điển

❌ Validate chỉ ở cuối pipeline

Data sai từ source → transform sai → load sai → phát hiện ở dashboard = quá muộn. Validate ở mỗi layer: source, after extract, after transform, after load.

❌ Không track data lineage

Bug trong report, không biết data đến từ đâu, qua mấy transformations, ảnh hưởng bao nhiêu downstream tables. Lineage = debugging superpower.

❌ "Data quality là việc của data team"

Sai. Data quality là responsibility của EVERYONE: source systems, application devs, data engineers, analysts. Data contracts enforce shared responsibility.

❌ Quá strict validation = block everything

Reject toàn bộ batch vì 1 row lỗi? 999,999 rows đúng bị stuck. Tiered approach: critical rules = block, warning rules = flag + proceed, info rules = log.

❌ Không có data ownership

Table dim_customers — ai own? Ai fix khi sai? Khi "mọi người" own = không ai own. Assign clear ownership per table/domain.
