📚 Thân Thế & Lai Lịch

Đường Ống Dữ Liệu Sư, tên thật là Đặng Chuyển Hoá, hồi bé thích nhất là chơi trò xây kênh dẫn nước từ bồn rửa mặt ra sân. Bố mẹ la mắng, nhưng ông nội — một kỹ sư thủy lợi — lại khen: "Thằng này có gen thủy lợi!"

Vào nghề data, lão phát hiện pipeline dữ liệu cũng giống hệ thống thủy lợi: nguồn (source), kênh dẫn (pipeline), bể chứa (destination), van điều khiển (transformations), và bộ lọc (validation). Và giống hệ thống thủy lợi — nếu bị rò rỉ, hậu quả rất nghiêm trọng nhưng không ai biết cho đến khi đã quá muộn.

Sau 12 năm xây pipeline, lão có câu nói trứ danh: "80% thời gian làm data là dọn dẹp data. 15% là xây pipeline. 5% là phân tích. Và 200% là debug tại sao pipeline fail lúc 3 giờ sáng."

📜 Bí Kíp

Bí Kíp #1: ETL vs ELT — Thứ Tự Quyết Định Mọi Thứ

ETL: Extract → Transform → Load. Transform trước khi load vào warehouse. Tốt khi target storage đắt hoặc hạn chế compute. ELT: Extract → Load → Transform. Load raw data vào warehouse, transform bằng SQL (dbt). Tốt khi warehouse có compute mạnh (BigQuery, Snowflake). Trend hiện tại: ELT thắng nhờ cloud warehouse rẻ và mạnh.

Bí Kíp #2: Idempotency — Chạy Lại Không Lo

Pipeline PHẢI idempotent: chạy 1 lần hay 100 lần, kết quả giống nhau. Cách làm: dùng upsert (INSERT ON CONFLICT UPDATE), replace partition thay vì append, dedup bằng unique key. Nếu pipeline fail giữa chừng và retry, dữ liệu không bị duplicate hay corrupt.

Bí Kíp #3: Data Validation — Trust But Verify

Validate ở 3 điểm: sau Extract (source data hợp lệ?), sau Transform (logic đúng?), sau Load (row counts match? checksums match?). Great Expectations library cho Python. dbt tests cho SQL. Minimum: null checks, unique checks, referential integrity, value range checks.

Bí Kíp #4: Incremental Processing — Đừng Load Lại Mọi Thứ

Full load 100 triệu rows mỗi ngày khi chỉ có 10,000 rows mới? Dùng incremental: CDC (Change Data Capture) bằng Debezium, hoặc watermark column (updated_at > last_run). Incremental giảm thời gian chạy 100x và giảm load lên source database.

Bí Kíp #5: Error Handling — Dead Letter Queue

Record lỗi không nên crash toàn bộ pipeline. Pattern: ghi record lỗi vào dead letter table/queue, tiếp tục xử lý records còn lại, alert khi error rate > threshold, retry dead letters sau. Pipeline chạy xong = success + partial errors list, không phải all-or-nothing.

Bí Kíp #6: Monitoring & Alerting — Biết Trước Khi Boss Biết

Metrics cần monitor: row count (so với expected), latency (job chạy lâu bất thường?), freshness (data mới nhất bao lâu rồi?), error rate. Alert khi: job fail, row count lệch > 10%, freshness > SLA. Dùng Soda, Monte Carlo, hoặc custom checks.

📖 Chuyện Kể Đêm Khuya

💀 Chuyện 1: ETL Job Corrupt Production Data

Anh data engineer viết ETL job: đọc từ MySQL, transform, ghi vào data warehouse. Nhưng anh mắc sai lầm chết người: dùng production MySQL connection thay vì read replica. Và pipeline có bug — thay vì chỉ SELECT, nó chạy UPDATE (do copy-paste sai function).

Pipeline chạy lúc 2 giờ sáng. Cập nhật 5 triệu records trong bảng orders — set status = 'pending' cho mọi order đã completed. Sáng hôm sau: khách hàng complain hàng loạt "đơn hàng đã giao sao vẫn hiện pending?" Customer service mất 3 ngày để fix thủ công.

💡 Bài học: ETL connections đến source PHẢI read-only. Dùng read replica, không bao giờ production primary. Tạo database user riêng cho ETL với chỉ SELECT permission. Review pipeline code kỹ như review production code.
🕳️ Chuyện 2: Silent Data Loss — Mất Dữ Liệu Âm Thầm

Pipeline sync user data từ CRM sang warehouse. Chạy ổn 6 tháng. Một ngày, team analytics phát hiện: số users trong warehouse ít hơn CRM 15%. Ai đó hỏi: "Từ bao giờ?" Không ai biết. Không có monitoring.

Điều tra: 3 tháng trước, CRM vendor thay đổi API pagination — trả max 100 records/page thay vì 1000. Pipeline code giữ nguyên page_size=1000 → API trả 100 → pipeline nghĩ "chỉ có 100 records, done" → bỏ qua 900 records mỗi page. 3 tháng × 300 pages/run = 270,000 users bị mất.

💡 Bài học: Validate row counts sau mỗi extraction. So sánh source count vs extracted count. Alert khi lệch. Silent data loss là loại bug nguy hiểm nhất vì không ai biết cho đến khi đã quá muộn.
🔄 Chuyện 3: Duplicate Data — Chạy Lại = Gấp Đôi

Pipeline append data vào warehouse mỗi giờ. Job fail lúc 3pm, team retry lúc 4pm. Nhưng job không idempotent — retry ghi thêm 1 lần nữa data của 3pm. Revenue report cuối ngày: doanh thu tăng 15% so với hôm qua. CEO vui mừng khôn xiết. Cho đến khi phát hiện: doanh thu thực ra giảm 5%, 15% kia là duplicate data.

Lão phải: tìm và xóa duplicates (dùng ROW_NUMBER() PARTITION BY), redesign pipeline dùng MERGE/UPSERT thay vì INSERT, thêm dedup logic. Và viết email xin lỗi CEO.

💡 Bài học: Pipeline PHẢI idempotent. Dùng UPSERT/MERGE, hoặc replace full partition, hoặc dedup bằng unique key. Test bằng cách chạy pipeline 2 lần liên tiếp — kết quả phải giống nhau.

⚠️ Những Sai Lầm Kinh Điển

❌ Không có retry logic

Network blip, API timeout, database connection reset — transient errors xảy ra thường xuyên. Không retry = pipeline fail vì lý do tạm thời. Implement exponential backoff retry (3 attempts, 1s/5s/30s delay).

❌ Hard-coded credentials trong code

Database password trong ETL script commit lên git. Secret management: dùng environment variables, Vault, AWS Secrets Manager, hoặc Airflow connections.

❌ Full load khi incremental là đủ

Load lại 100 triệu rows mỗi ngày khi chỉ có 1000 rows thay đổi. Tốn time, tốn compute, tốn money, tạo load lên source database.

❌ Không log pipeline metadata

Pipeline chạy xong nhưng không biết: chạy bao lâu? bao nhiêu rows? từ timestamp nào đến timestamp nào? Khi debug, không có gì để check. Log mọi run metadata.

❌ Transform quá complex trong 1 step

1 pipeline: extract 5 sources, join, filter, aggregate, format, load — tất cả trong 1 giant script. Khi fail, không biết fail ở đâu. Chia thành stages nhỏ, mỗi stage có checkpoint.
