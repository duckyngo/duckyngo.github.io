📜 Thân Thế & Lai Lịch

Trần Táo Khuyết bắt đầu với Objective-C — ngôn ngữ mà khi nhìn lần đầu ai cũng hỏi: 'Đây là code hay là bảng mã Morse?' Bracket hell: [[[[object method1] method2] method3] method4]. Rồi Swift ra đời năm 2014, ông khóc vì hạnh phúc. Ông đã survive qua mọi drama của Apple ecosystem: Auto Layout ra đời (constraint hell bắt đầu), Swift 1 → 2 → 3 migration pain, Storyboard vs programmatic UI wars, và cuối cùng SwiftUI revolution. App Store review process đã reject app ông 17 lần — mỗi lần một lý do khác nhau, có lần chỉ vì 'UI not aligned with Human Interface Guidelines'. Motto: 'Trên iOS, user kỳ vọng perfection. 60fps animation, instant response, zero crash. Bất kỳ cái gì kém hơn đều nhận 1-star review.'

🗝️ Bí Kíp

Bí Kíp #1: SwiftUI vs UIKit — Khi nào dùng gì

SwiftUI: declarative UI, preview canvas, less code, cross-Apple-platform (iOS, macOS, watchOS, tvOS). Bắt đầu projects mới bằng SwiftUI. UIKit: imperative, mature, full control, complex custom UI. Cần cho: complex collection views, custom transitions, UIKit-only APIs. Hybrid: SwiftUI hosting UIKit (UIViewRepresentable) và UIKit hosting SwiftUI (UIHostingController). 2025: SwiftUI đủ mature cho 90% use cases. UIKit cho edge cases và legacy codebase.

Bí Kíp #2: Swift Concurrency — async/await + Actors

Swift 5.5+: structured concurrency. async/await thay completion handlers (goodbye callback hell). Task groups cho parallel work. Actor isolation: data race prevention tại compile time — actor properties chỉ accessible từ actor context. @MainActor cho UI updates — compiler đảm bảo UI code chạy trên main thread. Sendable protocol đánh dấu types safe for concurrent access. Tương lai iOS development = Swift Concurrency everywhere.

Bí Kíp #3: App Architecture — MVVM, TCA, Clean Architecture

MVVM + SwiftUI: View (SwiftUI view) → ViewModel (@Observable class) → Model (data structs). Simple, works cho đa số apps. TCA (The Composable Architecture): functional, composable, testable — mỗi feature = State + Action + Reducer + Environment. Opinionated nhưng consistent. Clean Architecture: Use Cases → Repositories → Data Sources. Cho complex domain logic. Recommendation: MVVM cho apps đơn giản, TCA cho apps cần predictable state, Clean Architecture cho enterprise.

Bí Kíp #4: Core Data vs SwiftData vs Realm

Core Data: Apple's ORM, powerful nhưng verbose, NSManagedObjectContext threading rules complex. SwiftData (iOS 17+): modern Core Data wrapper, macro-based (@Model), SwiftUI integration native. Dùng SwiftData cho new projects targeting iOS 17+. Realm: third-party, zero-copy architecture (fast reads), live objects, sync capability (MongoDB Atlas). SQLite direct: khi cần full SQL control hoặc cross-platform database.

Bí Kíp #5: Performance Optimization

Instruments: Time Profiler (CPU bottlenecks), Allocations (memory leaks), Leaks (retain cycles), Core Animation (rendering performance). SwiftUI: avoid expensive body computations, use @State wisely (granular state = less re-rendering), List lazy by default nhưng ForEach trong ScrollView KHÔNG lazy — dùng LazyVStack. Image: downsize on load, cache with NSCache, dùng prepareThumbnail(). Network: URLSession caching, pagination, prefetching.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: App Store Rejection Saga — 17 Lần Bị Từ Chối

Submit app lần 1: 'Guideline 4.0 - Design: UI elements do not conform to Human Interface Guidelines.' Sửa UI, submit lại. Lần 2: 'Guideline 2.1 - Performance: App crashed on iPad Pro 12.9 inch.' Fix crash (auto layout constraint conflict ở landscape). Lần 3: 'Guideline 5.1.2 - Data Use: Privacy policy link required.' Thêm privacy policy. Lần 7: 'Guideline 3.1.1 - In-App Purchase: Feature X requires IAP.' Implement IAP cho premium features. Lần 12: 'Guideline 2.3 - Accurate Metadata: Screenshots not match current version.' Update screenshots. Lần 17: 'APPROVED.' Team ăn mừng bằng nước mắt. Tổng thời gian: 3 tháng từ submit đầu đến approved. Giờ app review nhanh hơn (24-48h) nhưng rejection vẫn đau.

💡 Bài học: Đọc App Store Review Guidelines TRƯỚC khi code, không phải trước khi submit. Common rejections: IAP issues, privacy policy missing, metadata mismatch, crash on specific devices, design guideline violations. TestFlight beta testing để catch issues early. App Store Connect response clear và specific — đọc kỹ, fix đúng, đừng argue.

📖 Chuyện 2: Auto Layout Constraint Hell

Designer gửi complex UI: card layout, nested scroll views, dynamic height cells, floating action button, safe area handling. Developer implement bằng Auto Layout constraints programmatically. 200 constraints cho 1 screen. Ambiguous layout warnings trong console. Một ngày, constraint conflict: Unable to simultaneously satisfy constraints với 15 constraints listed. Debug: nào là leading, trailing, top, bottom, width, height, centerX, centerY — constraint nào conflict với constraint nào? Xcode Constraint debugger KHÔNG helpful cho 200 constraints. Team chuyển sang SnapKit (DSL cho Auto Layout): code readable hơn, ít error-prone hơn. Sau đó SwiftUI ra đời — declarative layout, không constraint. Dev: 'Tại sao phải chịu đau 10 năm?'

💡 Bài học: Auto Layout constraints programmatically = verbose và error-prone. Solutions: SnapKit/Cartography (Swift DSL), Interface Builder (visual nhưng merge conflict hell), hoặc SwiftUI (declarative, no constraints). Cho legacy UIKit: group constraints logically, name constraints cho debugging, dùng UIStackView giảm 80% constraints.

📖 Chuyện 3: Memory Leak từ Closure Retain Cycle

App bắt đầu lag sau 30 phút sử dụng. Memory usage tăng liên tục. Instruments > Leaks: retain cycle. Closure trong UIViewController capture self strongly → ViewController không dealloc → tất cả subviews, images, data models giữ trong memory. 50 screens navigate qua trong 30 phút = 50 ViewControllers alive trong memory, mỗi cái hold 5-10MB assets. Memory: 500MB. iOS kill app. Fix: [weak self] trong mọi closure capture ViewController. [unowned self] khi guarantee self alive. Deinit logging: deinit { print('VC deallocated') } — nếu không thấy log = leak.

💡 Bài học: Retain cycle là memory leak phổ biến nhất trên iOS. Rule: closure capture object → dùng [weak self]. Delegate pattern → weak reference. Timer → invalidate trong deinit. NotificationCenter → removeObserver. Add deinit logging cho EVERY ViewController. Run Instruments Leaks check định kỳ.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Force unwrap everywhere

let name = user!.name! — crash waiting to happen. Dùng guard let, if let, nil coalescing (??). Force unwrap chỉ OK khi 100% guaranteed non-nil (IBOutlet after viewDidLoad).

❌ Sai Lầm #2: Blocking main thread

Network call, heavy computation trên main thread → UI freeze → user nghĩ app crash. MỌII I/O và computation nặng phải chạy trên background queue. UI updates PHẢI trên main queue. Swift Concurrency (@MainActor) enforce tại compile time.

❌ Sai Lầm #3: Massive View Controller

ViewController 3000 dòng: networking + UI + business logic + animation + gesture handling. Tách: MVVM (ViewModel cho logic), Coordinator (navigation), Service classes (networking). SwiftUI naturally enforces smaller views.

❌ Sai Lầm #4: Không test trên device thật

Simulator ≠ real device. Camera, GPS, push notifications, performance, memory constraints — tất cả khác. Test trên ít nhất 2 devices: newest + oldest supported model.

❌ Sai Lầm #5: Ignore accessibility

VoiceOver users tăng. Label mọi UI element, test VoiceOver navigation, support Dynamic Type (font size tùy chỉnh), High Contrast mode. Apple review team test accessibility — fail = rejection.
