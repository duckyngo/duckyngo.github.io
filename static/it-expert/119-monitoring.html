📚 Thân Thế & Lai Lịch

Giám Sát Toàn Diện Đại Sư tin rằng: "You can not improve what you do not measure. But measuring wrong things gives you false confidence." Chàng ta đã thấy dashboards toàn xanh trong khi users khóc vì slow performance.

Chàng ta phát minh ra "3 pillars + 1": Metrics (what), Logs (why), Traces (where) + RUM (how users actually experience). Server-side metrics nói hệ thống healthy. RUM nói users experience 5-second page loads vì JavaScript bundle 2MB. "Server fast, frontend fat," chàng ta summarize.

SLI/SLO/SLA là ngôn ngữ chung giữa engineering và business. "99.9% availability" = 43 minutes downtime/tháng. Error budget = how much failure allowed. Burn rate = how fast you are using error budget. "SLOs turn reliability into a business conversation," chàng ta nói.

📜 Bí Kíp

1. SLI/SLO/SLA Framework: SLI (Indicator): measurable metric (e.g., request latency p99, error rate, availability). SLO (Objective): target for SLI (e.g., p99 latency < 200ms, error rate < 0.1%). SLA (Agreement): contractual commitment with consequences (e.g., 99.9% uptime or credits). SLI ⊂ SLO ⊂ SLA. Define SLIs first, set SLOs, then SLAs.
2. Error Budgets: Error budget = 100% - SLO. 99.9% SLO = 0.1% error budget = 43 minutes/month. If error budget exhausted: freeze deployments, focus on reliability. If error budget remaining: deploy fast, experiment. Error budget creates balance between velocity and reliability. Track burn rate: how fast budget consumed.
3. APM — Application Performance Monitoring: Server-side instrumentation: response times, throughput, error rates. Transaction tracing: end-to-end request flow. Code-level visibility: slow methods, exceptions. Database query performance. External call monitoring. Tools: New Relic, Datadog APM, Elastic APM, OpenTelemetry.
4. RUM — Real User Monitoring: Client-side JavaScript monitoring. Actual user experience: page load time, interaction delay, visual stability. Core Web Vitals: LCP, FID/INP, CLS. Geographic distribution of performance. Browser/device breakdown. Real data > synthetic data cho user experience.
5. Synthetic Monitoring: Automated tests from external locations. Uptime monitoring: is site accessible? Transaction monitoring: can users complete workflows? Performance monitoring: baseline response times. Available when no real users (off-hours). Complement to RUM, not replacement. Tools: Grafana Synthetic Monitoring, Datadog, Checkly.
6. Capacity Planning: Current usage trends → future capacity needs. Headroom: 30-40% buffer cho spikes. Growth projections: user growth → resource growth. Load testing validates capacity model. Auto-scaling helps but needs limits. Regular capacity reviews: monthly for fast-growth, quarterly for stable.

📖 Chuyện Kể Đêm Khuya

📊 Chuyện SLO Vi Phạm Mà Không Ai Biết

SLO: 99.9% availability. Actual: 99.7% cho 3 tháng liên tiếp. Nobody noticed vì: (1) no SLO monitoring dashboard, (2) no error budget tracking, (3) no automated alerts on SLO burn. Customers noticed — 3 enterprise customers escalate complaints. CTO finds out from CUSTOMERS, not engineering.

Fix: SLO dashboard visible to all. Error budget burn rate alerting. Weekly SLO review meetings. Automated reports to stakeholders. Engineering priorities adjusted based on error budget remaining.

💡 Bài học: SLOs without monitoring = meaningless numbers on paper. Dashboard + alerts + regular reviews. Error budget tracking automates reliability prioritization. SLO violations should be detected by engineering BEFORE customers complain.

👤 Chuyện RUM Phát Hiện Thực Tế User

Server metrics: p99 latency 50ms. Dashboard: green. But support tickets: 'Site is slow!' RUM deployed → discovery: actual page load time p99 = 8 seconds. Root causes: 3MB JavaScript bundle (2 seconds parse), 15 API calls per page load (waterfall), unoptimized images (1.5 seconds). Server fast, frontend performance terrible.

Fix: code splitting (lazy loading), API call batching, image optimization. Page load p99: 8s → 1.5s. Customer satisfaction improved dramatically.

💡 Bài học: Server-side metrics ≠ user experience. RUM captures REAL user performance including network, browser, frontend. Core Web Vitals cho UX metrics. Frontend optimization often bigger impact than backend. Measure what users experience, not what servers experience.

💰 Chuyện Error Budget Cứu Velocity

Engineering team working hard on new features but SLO 99.9% thường xuyên bị vi phạm (99.5% actual). Management: 'Freeze all feature work, fix reliability!' Engineers frustrated: 6 tháng chỉ fix bugs, no new features, product stagnant.

Đại Sư implement error budget policy: if error budget remaining (>50%): prioritize features. If error budget low (20-50%): balance features/reliability. If error budget exhausted (<20%): reliability sprint. Result: data-driven prioritization. Teams self-manage reliability vs velocity tradeoff.

💡 Bài học: Error budgets give teams AUTONOMY to balance velocity and reliability. Data-driven, not emotion-driven decisions. Error budget policy documented and agreed by engineering + product. Sustainable pace: neither always features nor always reliability. SRE culture enabler.

⚠️ Những Sai Lầm Kinh Điển

1. SLOs without buy-in: Engineering sets SLOs alone. Product/business doesn't know or care. SLOs must be agreed by engineering + product + business. Shared ownership.
2. 100% availability SLO: Unrealistic and counterproductive. 100% = no deployments, no changes. 99.9% or 99.95% reasonable cho most services. Error budget enables velocity.
3. Only synthetic monitoring: Synthetic tests pass, real users fail. Synthetic ≠ real user conditions. Use both synthetic (baseline) and RUM (reality).
4. Ignore frontend performance: Backend optimized, frontend 5-second load. Users don't care about server response time — they care about page load time.
5. Monitoring as afterthought: Build → deploy → 'oh we should add monitoring.' Monitoring in design phase. Observability as first-class requirement.
