📚 Thân Thế & Lai Lịch

Cột Rộng Đại Sư, tên thật là Phạm Đức Khổng Lồ, từ nhỏ đã có sở thích kỳ lạ: sưu tập mọi thứ. Tem thư, nắp chai, lá cây — gì cũng giữ. Và ông sắp xếp chúng theo hệ thống mà chỉ mình ông hiểu: nhóm theo loại, rồi xếp theo ngày, mỗi nhóm lại có nhóm con.

Năm 2008, khi Facebook mở nguồn Cassandra, lão biết đây là "chân ái". Một database có thể lưu trữ petabyte dữ liệu, phân tán trên hàng ngàn node, không có single point of failure? Lão đã tìm thấy niết bàn.

Sau 16 năm, lão đã vận hành clusters từ 3 node đến 3000 node. Lão từng nói: "Cassandra giống như nuôi rồng — mạnh mẽ vô cùng, nhưng không hiểu nó thì nó sẽ đốt cháy bạn."

📜 Bí Kíp

Bí Kíp #1: Partition Key — Chìa Khóa Sinh Tử

Partition key quyết định data nằm trên node nào. Key tốt phân tán đều data. Key tệ = hot partition = một node chết, cluster sống. Ví dụ: dùng country làm partition key khi 80% user ở Mỹ = node Mỹ quá tải. Dùng user_id hoặc compound key (country, user_id).

Bí Kíp #2: Data Modeling — Thiết Kế Từ Query, Không Từ Entity

Ngược với RDBMS: trong Cassandra, bạn thiết kế bảng dựa trên cách bạn sẽ query, không phải entity relationship. Một query = một bảng. Denormalize là bình thường. Duplicate data là bình thường. Đây không phải RDBMS — đừng normalize.

Bí Kíp #3: Consistency Level — Chọn Trade-off

ONE: nhanh nhất, ít an toàn nhất. QUORUM: đọc/ghi > N/2+1 nodes, cân bằng tốt. ALL: an toàn nhất, chậm nhất, một node chết = cả query fail. Quy tắc vàng: Read CL + Write CL > Replication Factor = Strong Consistency. Ví dụ: RF=3, Write QUORUM + Read QUORUM = 2+2 > 3 = OK.

Bí Kíp #4: Compaction Strategy — Chọn Sai Là Disk Cháy

SizeTieredCompaction: tốt cho write-heavy workload. LeveledCompaction: tốt cho read-heavy, ít space amplification. TimeWindowCompaction: cho time-series data. ScyllaDB có Incremental Compaction — hybrid approach. Monitor disk usage, vì compaction cần temporary space gấp đôi.

Bí Kíp #5: DynamoDB — Serverless Nhưng Không Lo-less

On-demand mode cho traffic unpredictable. Provisioned mode + auto-scaling cho traffic ổn định (rẻ hơn). GSI (Global Secondary Index) cho query patterns khác primary key. Nhưng mỗi GSI = một bản copy data = chi phí x2. DAX cache cho hot data — giảm latency từ ms xuống microseconds.

Bí Kíp #6: ScyllaDB — Cassandra Trên Steroid

API tương thích Cassandra nhưng viết bằng C++ thay vì Java. Không có GC pause — đây là killer feature. Shard-per-core architecture tận dụng tối đa hardware. Cùng workload, ScyllaDB thường cần 1/3 đến 1/5 số node so với Cassandra. Nhưng community nhỏ hơn nhiều.

📖 Chuyện Kể Đêm Khuya

🔥 Chuyện 1: Hot Partition Làm Chảy Cluster

Công ty game online dùng Cassandra lưu player activity. Partition key: game_id. Vấn đề: công ty chỉ có... 1 game. Tất cả 500 triệu events/ngày dồn vào 1 partition, trên 1 node. Node đó CPU 100%, latency tăng từ 2ms lên 30 giây. Các node khác ngồi chơi.

Lão đổi partition key thành (game_id, date, bucket) với bucket = hash(player_id) % 100. Data phân tán đều trên 100 partitions × 365 ngày. Latency trở về 2ms. Nhưng phải migrate 50TB data — mất 1 tuần chạy migration liên tục.

💡 Bài học: Partition key phải có high cardinality. Nếu chỉ có vài giá trị, thêm bucket hoặc compound key. Test distribution trước khi production bằng cách check partition size: mỗi partition không nên vượt 100MB.
💸 Chuyện 2: DynamoDB Throttling Giữa Black Friday

E-commerce dùng DynamoDB provisioned mode. Bình thường 1000 RCU/WCU là đủ. Black Friday: traffic tăng x50 trong 5 phút. Auto-scaling cần 5-15 phút để react. Trong 5 phút đó: ProvisionedThroughputExceededException tràn ngập. Giỏ hàng không load được, checkout fail, khách hàng bỏ đi.

Lão chuyển sang on-demand mode trước mỗi sale event, sau đó chuyển lại provisioned. Đắt hơn 20% nhưng không bao giờ throttle. Bài học thương đau: "Tiết kiệm $200/tháng rồi mất $200,000 doanh thu trong 1 ngày."

💡 Bài học: DynamoDB auto-scaling không phải instant. Với spike traffic, dùng on-demand mode hoặc pre-warm bằng cách tăng provisioned capacity trước event. Đừng để auto-scaling bảo vệ bạn trong những lúc quan trọng nhất.
☠️ Chuyện 3: Tombstone Storm — Khi Xóa Data Mà Data Không Chịu Đi

Team dùng Cassandra lưu session data, TTL 24 giờ. Mỗi ngày có 100 triệu sessions tạo ra và tự động expired. Vấn đề: mỗi expired row tạo 1 tombstone. Cassandra phải scan qua tombstones khi đọc. Sau vài tháng: 1 tỷ tombstones. Read latency tăng vọt. Đến khi vượt tombstone_warn_threshold (default 1000), Cassandra bắt đầu warning liên tục.

Tệ hơn: compaction không đuổi kịp tốc độ tạo tombstones mới. Disk usage tăng phi mã dù "data chỉ giữ 24 giờ". Lão phải tune compaction, giảm gc_grace_seconds, và redesign schema dùng TimeWindowCompaction.

💡 Bài học: Trong Cassandra, DELETE không xóa data ngay — nó tạo tombstone. TTL cũng tạo tombstone. High-churn data cần TimeWindowCompaction và gc_grace_seconds phù hợp. Monitor tombstone count per read.

⚠️ Những Sai Lầm Kinh Điển

❌ Dùng Cassandra cho data nhỏ

Cassandra dưới 10GB data trên 3 nodes? Overhead của distributed system lớn hơn lợi ích. PostgreSQL single node sẽ nhanh hơn, đơn giản hơn, dễ vận hành hơn 100 lần.

❌ SELECT * FROM table (full table scan)

Trong Cassandra, điều này yêu cầu coordinator gửi request đến TẤT CẢ nodes. Cluster 100 nodes = 100 requests. Timeout guaranteed. Dùng partition key trong WHERE clause, luôn luôn.

❌ Dùng ALLOW FILTERING

Khi Cassandra từ chối query, nó có lý do. ALLOW FILTERING biến query thành full scan. Dev thêm ALLOW FILTERING "cho chạy được cái đã" rồi quên remove. Production chết khi data lớn.

❌ Partition quá lớn

Nhét hàng triệu rows vào 1 partition. Cassandra cảnh báo khi partition > 100MB. Partition 1GB = query trên partition đó timeout. Thiết kế partition size trước khi data modeling.

❌ Không monitor GC pause (Cassandra/Java)

Cassandra chạy trên JVM. GC pause dài = node bị đánh dấu "down" bởi gossip protocol. Node "sống lại" sau GC = confuse cluster. Tune JVM GC settings (G1GC, heap size). Hoặc chuyển sang ScyllaDB.

❌ DynamoDB: Scan thay vì Query

Scan đọc toàn bộ table, tốn RCU khủng khiếp. Query chỉ đọc items matching partition key. Thiết kế table và GSI sao cho mọi access pattern đều dùng Query, không bao giờ Scan.

❌ Không test với realistic data volume

Dev test với 1000 rows, mọi thứ nhanh. Production có 1 tỷ rows, mọi thứ chết. Wide column databases cần benchmark với production-level data volume và traffic pattern.
