📚 Thân Thế & Lai Lịch

Số Liệu Đại Sư từng monitor bằng cách SSH vào servers chạy top, free -h, df -h. Mỗi sáng, 30 phút SSH qua 20 servers. "Manual monitoring," chàng ta gọi. Thực ra đó là "hope-based observability."

Rồi Prometheus + Grafana thay đổi cuộc đời: scrape metrics tự động, store time-series, query với PromQL, visualize với Grafana dashboards. Đẹp, real-time, alerting. "Tôi nhìn dashboards thay vì SSH," chàng ta hạnh phúc.

Cho đến khi Prometheus OOM vì high cardinality metrics. 10 triệu unique time series. Memory: 30GB. Cost: tears. "Not all metrics are created equal," chàng ta giác ngộ. "Some metrics are more expensive than others."

📜 Bí Kíp

1. Prometheus — Pull-Based Metrics: Scrape targets qua HTTP. Time-series database (TSDB). PromQL cho querying. Service discovery (Kubernetes, consul, file-based). Federation cho scaling. Thanos/Cortex/Mimir cho long-term storage + HA. Recording rules cho pre-computed metrics. Alert rules cho alerting.
2. PromQL — Querying: Instant vectors, range vectors, scalar. rate() cho per-second rate of counter. histogram_quantile() cho percentiles. by/without cho aggregation. offset cho time shift. Common patterns: rate(http_requests_total[5m]), histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])). Practice PromQL — it is essential.
3. Grafana — Visualization: Dashboards cho metrics visualization. Multiple datasources: Prometheus, Loki, Elasticsearch, CloudWatch. Variables cho dynamic dashboards. Alerting (Grafana Alerting hoặc Prometheus Alertmanager). Annotations cho events. Dashboard as code (grafonnet, Terraform). Dashboard folders + permissions cho organization.
4. Cardinality — The Silent Killer: Cardinality = number of unique time series. High cardinality = high memory, slow queries. Causes: user_id labels, request_path labels, UUID labels. Rule: labels should have bounded cardinality. Monitor cardinality: prometheus_tsdb_head_series. Drop high-cardinality labels with metric_relabel_configs.
5. Dashboard Best Practices: USE method (Utilization, Saturation, Errors) cho resources. RED method (Rate, Errors, Duration) cho services. Golden signals: latency, traffic, errors, saturation. Dashboard hierarchy: overview → service → component. Avoid: too many panels, rainbow colors, vanity metrics.
6. Datadog — SaaS Alternative: Agent-based, easy setup. 700+ integrations. APM, infrastructure, logs unified. Pricing per host + custom metrics. Expensive at scale but reduces ops burden. Good for: teams without dedicated monitoring infra team. Consider: costs grow with infrastructure size.

📖 Chuyện Kể Đêm Khuya

📈 Chuyện High Cardinality Giết Prometheus

Developer add label path={{request_path}} to HTTP metrics. 50,000 unique URLs = 50,000 time series PER metric. 10 metrics × 50,000 paths = 500,000 time series. Prometheus memory: 8GB → 30GB → OOM crash. Metrics lost. Alerting down. Nobody knows systems are failing.

Fix: remove path label (use histogram cho latency distribution instead). Implement cardinality limits. Metric relabeling to drop high-cardinality labels before storage. Monitor prometheus_tsdb_head_series cho early warning.

💡 Bài học: Label values MUST have bounded cardinality. No user IDs, no request paths, no UUIDs as labels. If needed: log them, don't metric them. Monitor Prometheus memory và series count. Cardinality explosion = Prometheus death. Set series limits per scrape target.

🎨 Chuyện Dashboard Nói Dối

Grafana dashboard shows CPU 30%, memory 50%, disk 40%. Everything green. But users complaining: 'app is slow!' Dashboard metrics are AVERAGES across 10 servers. Reality: 1 server at 100% CPU (slow), 9 servers at 22% CPU (fine). Average: 30%. Dashboard hides the problem.

Fix: show per-instance metrics, not just averages. Add p99 percentiles. Heat maps cho distribution. Alert on per-instance thresholds, not averages.

💡 Bài học: Averages lie. Show distributions (percentiles, heat maps). Per-instance metrics alongside aggregates. p99 latency more important than average latency. Dashboard designed to surface problems, not hide them. If dashboard always green, it's probably wrong.

🔔 Chuyện Alert Storm 500 Alerts

Monitoring setup: alert cho every metric threshold. CPU > 80%, memory > 80%, disk > 80%, latency > 100ms, error rate > 1%, ... per service, per instance. 50 services × 10 instances × 5 alerts = 2,500 potential alerts. Network issue: 1 upstream dependency fails → 30 services error → 500 alerts fire simultaneously. On-call engineer: 500 PagerDuty notifications in 5 minutes. Phone literally vibrating off desk.

💡 Bài học: Alert design: alert on symptoms (user-facing impact), not causes. Aggregated alerts: 'web-tier error rate high' not 'server-1 error, server-2 error...'. Alert grouping, inhibition, silencing trong Alertmanager. Runbook link in every alert. Alert fatigue = missed real alerts.

⚠️ Những Sai Lầm Kinh Điển

1. Alert on every metric: Too many alerts = alert fatigue = ignored alerts = missed incidents. Alert on symptoms, investigate causes.
2. No recording rules: Complex PromQL query in dashboard = slow rendering. Recording rules pre-compute common queries. Faster dashboards, reusable metrics.
3. Prometheus without persistent storage: Pod restart = data loss. PVC cho TSDB storage. Thanos/Mimir cho long-term và HA.
4. Dashboard without context: Numbers without baseline. What is normal CPU? Add historical comparisons, baselines, SLO targets to dashboards.
5. Monitoring only infrastructure: CPU, memory, disk — but no application metrics. Business metrics (orders/sec, conversion rate) equally important.
