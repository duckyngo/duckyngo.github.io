📜 Thân Thế & Lai Lịch

Đỗ Thí Nghiệm — người đàn ông mà mỗi khi commit code đều run test trước. Ông bắt đầu sự nghiệp ở thời kỳ "test bằng F5 và mắt thường". Click click, thấy OK, deploy. Rồi production bug xuất hiện lúc 3h sáng — cái bug mà nếu có test thì phát hiện trong 30 giây.

Từ đó ông thề sẽ test MỌII THỨ. Thời kỳ cuồng test: 95% code coverage, test từng pixel, mock từng function. Test suite chạy 45 phút. Developer push code xong đi uống cà phê, quay lại CI vẫn chưa xong. Rồi ông ngộ ra: coverage 95% không có nghĩa là app hoạt động đúng — chỉ có nghĩa là code ĐƯỢC CHẠY, không phải code ĐÚNG.

Ngày nay ông theo triết lý: "Test the behavior, not the implementation. Test what matters, not what's easy to test." Ông nổi tiếng với câu: "Nếu refactor code mà test fail, test của bạn test sai thứ."

🗝️ Bí Kíp

Bí Kíp #1: Test Pyramid — Nền móng vững chắc

Đáy (nhiều nhất): Unit tests — test functions, utilities, hooks riêng lẻ. Nhanh, stable, dễ debug. Giữa: Integration tests — test components interact với nhau, form submit, API calls. Realistic hơn. Đỉnh (ít nhất): E2E tests — test flow user thực tế (đăng nhập → tìm sản phẩm → checkout). Chậm nhưng confidence cao. Tỷ lệ lý tưởng: 70% unit, 20% integration, 10% E2E. Đừng invert pyramid (nhiều E2E, ít unit) — CI sẽ chạy cả tiếng.

Bí Kíp #2: Testing Library — Test như user dùng

React Testing Library triết lý: "The more your tests resemble the way your software is used, the more confidence they give you." Đừng test implementation details (internal state, method calls). Test behavior: render component → interact (click, type) → assert outcome (text on screen, element visibility). Query by role (getByRole('button', {name: 'Submit'})) thay vì test ID — vì user không biết test ID, user biết "nút Submit".

Bí Kíp #3: Playwright vs Cypress — Chọn ai cho E2E?

Playwright: Multi-browser (Chromium, Firefox, WebKit), multi-tab support, network interception, parallel execution, auto-wait, trace viewer. Microsoft maintain. Chạy ngoài browser (Node.js). Cypress: Developer experience tuyệt vời, time-travel debugging, real-time reload. Chạy TRONG browser. Nhược: chỉ Chromium-based, single-tab only, không test cross-origin dễ dàng. Quy tắc: Playwright cho CI/CD, comprehensive testing. Cypress cho DX, interactive development. 2025: Playwright đang thắng cuộc chiến.

Bí Kíp #4: Vitest — Jest nhưng nhanh hơn

Vitest dùng Vite's transform pipeline — nhanh hơn Jest đáng kể cho Vite-based projects. Jest-compatible API — migrate gần như zero effort. Native ESM support, TypeScript out of box, HMR cho test files, in-source testing. Browser mode cho DOM testing thật. Nếu project dùng Vite → dùng Vitest, không cần suy nghĩ. Nếu project dùng Webpack → Jest vẫn ổn. Đừng chuyển test runner chỉ vì "mới hơn".

Bí Kíp #5: Mocking Strategies — Mock đúng, test đúng

Mock quá nhiều = test không reflect reality. Mock quá ít = test chậm, flaky, dependent on external services. Quy tắc: Mock network requests (MSW - Mock Service Worker là gold standard), mock timers, mock random. KHÔNG mock component internals, KHÔNG mock everything. MSW intercept ở network level — code fetching vẫn chạy thật, chỉ response là giả. Integration test với MSW = high confidence without external dependency.

Bí Kíp #6: Visual Regression Testing

Logic test pass nhưng CSS broken → production trông thảm hại. Visual regression: capture screenshot, compare với baseline. Tools: Playwright built-in screenshot comparison, Percy, Chromatic (for Storybook). Tích hợp CI: mỗi PR auto-capture screenshots, diff với main branch, reviewer approve visual changes. Đặc biệt hữu ích khi update CSS framework, refactor shared components, hoặc redesign.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Flaky Tests — Bóng Ma Trong CI Pipeline

CI pipeline có 500 E2E tests. Mỗi lần chạy, 3-5 tests random fail. Re-run: pass. Team thêm retry logic: re-run failed tests lên tới 3 lần. "Fixed!" — không, chỉ MASK vấn đề.

Nguyên nhân thực: race conditions. Test click button → assert result NGAY LẬP TỨC — nhưng API call chưa xong. Trên máy developer nhanh: pass (API response < 100ms). Trên CI server chậm: fail (API response > 200ms). Fix: dùng Playwright's auto-wait, waitForResponse, expect(locator).toBeVisible() thay vì sleep(1000).

Sau 2 tuần fix flaky tests, CI reliability từ 70% lên 99%. Deploy confidence tăng vọt. Team không còn ignore CI failures với câu "chắc flaky test thôi" — vì giờ khi CI fail, it means something is actually wrong.

💡 Bài học: Flaky test nguy hiểm hơn no test — vì team mất trust vào test suite và bắt đầu ignore failures. Root causes thường gặp: race conditions, hardcoded waits, shared test state, timezone/locale differences. Fix root cause, KHÔNG retry mask. Mỗi flaky test cần RCA (root cause analysis).

📖 Chuyện 2: Snapshot Testing Gone Wrong

Team adopt snapshot testing cho React components. expect(component).toMatchSnapshot() — một dòng test, coverage tăng vù vù. Manager happy nhìn coverage report: "95%! Tuyệt vời!"

Thực tế: developer thay đổi component → snapshot fail → developer chạy jest --updateSnapshot → pass → commit. KHÔNG AI ĐỌC snapshot diff. Snapshots dài 500 dòng HTML — ai đọc 500 dòng diff trong PR review? Kết quả: snapshot tests thành rubber stamp, không catch bug nào cả.

Tệ hơn: bug slip through vì snapshot file quá lớn, reviewer auto-approve. Một ngày, CSS class name typo khiến toàn bộ styling vỡ — snapshot "updated" nhưng không ai nhận ra btn-priamry thay vì btn-primary.

💡 Bài học: Snapshot testing có chỗ dùng (serialized data, small outputs) nhưng KHÔNG thay thế behavior testing. Nếu snapshot > 20 dòng → quá lớn, không ai review. Dùng inline snapshots cho small, focused assertions. Prefer expect(screen.getByRole('button')).toHaveTextContent('Submit') thay vì snapshot cả component.

📖 Chuyện 3: "Chỉ Test Happy Path" — Khi Murphy Gõ Cửa

Login form tested: nhập email + password → click submit → redirect to dashboard. Test pass. Ship it! Production: user nhập email có space ở cuối → validation pass → API reject → infinite loading. User nhập password có emoji → backend crash. User double-click submit → 2 requests → duplicate session.

Team chỉ test happy path — điều hiếm khi xảy ra trong real life. Users sáng tạo hơn bạn tưởng: paste từ Excel (có hidden characters), autofill browser thêm whitespace, slow network khiến user double-click, ad blocker chặn API endpoint.

Sau incident, team tạo "Adversarial Testing Checklist": empty input, whitespace only, special characters, double submit, network timeout, API error responses, concurrent access. Mỗi feature đều test negative cases.

💡 Bài học: Happy path testing cho bạn false confidence. Test: empty/null inputs, boundary values, error responses, timeout, concurrent actions, invalid data types. Dùng property-based testing (fast-check) để auto-generate edge cases. Rule: nếu bug đã xảy ra, viết regression test TRƯỚC khi fix.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Test implementation details

Test expect(setState).toHaveBeenCalledWith({count: 1}) — refactor từ useState sang useReducer → test fail dù behavior giống hệt. Test behavior (expect(screen.getByText('1')).toBeVisible()), không test internals.

❌ Sai Lầm #2: sleep(3000) trong E2E tests

Hardcoded waits = slow tests + still flaky. Dùng proper waits: await page.waitForSelector(), await expect(element).toBeVisible(). Playwright/Cypress auto-retry assertions — tận dụng nó.

❌ Sai Lầm #3: Không clean state giữa các tests

Test A tạo user → Test B dùng user đó → Test B pass. Run Test B ALONE → fail. Test order dependency = nightmare. Mỗi test phải setup own state, teardown sau khi xong. beforeEach, afterEach là bạn.

❌ Sai Lầm #4: Coverage metric là KPI

80% coverage với meaningful tests > 95% coverage với snapshot tests. Coverage đo lines EXECUTED, không đo lines TESTED CORRECTLY. Một assertion expect(true).toBe(true) sau rendering toàn bộ app → high coverage, zero value.

❌ Sai Lầm #5: Không test accessibility

Logic tests pass, visual tests pass, nhưng keyboard user không navigate được. Tích hợp axe-core vào test suite: expect(await axe(container)).toHaveNoViolations(). Test keyboard navigation trong E2E: Tab, Enter, Escape flows.
