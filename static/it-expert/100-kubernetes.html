📚 Thân Thế & Lai Lịch

Chưởng Môn nhớ ngày đầu tiên đọc Kubernetes documentation: "A pod is the smallest deployable unit that can be created and managed in Kubernetes." Chàng ta đọc lại 5 lần vẫn không hiểu. Rồi chàng ta đọc thêm: "A service is an abstract way to expose an application running on a set of pods as a network service." Càng đọc càng confused.

Chàng ta quyết định cài minikube và thử. kubectl apply -f deployment.yaml. Pod running! kubectl expose deployment. Service created! kubectl get pods. STATUS: Running! Niềm vui ngắn chẳng tày gang khi pod restart liên tục vì OOMKilled — chàng ta quên set memory limits.

15 năm sau, chàng ta đã deploy hàng trăm clusters, viết hàng vạn dòng YAML, và vẫn phải Google "kubernetes service types" mỗi tháng. Kubernetes phức tạp, nhưng nó giải quyết orchestration problems tốt hơn bất kỳ thứ gì khác. "Hate the YAML, love the platform," chàng ta nói.

📜 Bí Kíp

1. Architecture — Control Plane & Data Plane: Control plane: kube-apiserver, etcd, scheduler, controller-manager. Data plane: kubelet, kube-proxy, container runtime. etcd: source of truth — backup etcd = backup cluster state. API server: mọi communication qua REST API. Scheduler: assign pods to nodes dựa trên resources, affinity, taints. Understand architecture để troubleshoot effectively.
2. Pods — Smallest Unit: Pod = 1+ containers sharing network namespace, volumes. Sidecar containers cho logging, proxying. Init containers cho initialization. Pod lifecycle: Pending → Running → Succeeded/Failed. Resource requests (scheduling guarantee) vs limits (maximum). ALWAYS set requests AND limits cho production pods. Liveness probes: restart unhealthy containers. Readiness probes: exclude from service endpoints.
3. Services — Networking Abstraction: ClusterIP: internal only (default). NodePort: expose on node port (30000-32767). LoadBalancer: cloud load balancer. ExternalName: DNS alias. Headless service (clusterIP: None): direct pod addressing cho StatefulSets. Service discovery: DNS (my-service.my-namespace.svc.cluster.local). Ingress cho HTTP/HTTPS routing.
4. Deployments — Declarative Updates: Rolling update strategy: maxSurge, maxUnavailable. kubectl rollout status, rollout history, rollout undo. Pod Disruption Budgets (PDB) cho maintenance safety. Update strategy: RollingUpdate (default, zero-downtime) vs Recreate (downtime, useful cho incompatible versions). Replicas, scaling, auto-scaling integration.
5. StatefulSets — Stateful Workloads: Stable network identity (pod-0, pod-1...). Ordered deployment and scaling. Persistent Volume Claims per pod. Dùng cho: databases, Kafka, Elasticsearch. headless service bắt buộc. volumeClaimTemplates cho automatic PVC creation. CẨNTHẬN: deleting StatefulSet KHÔNG delete PVCs — data preserved.
6. Networking — CNI & Policies: CNI plugins: Calico (popular, network policies), Cilium (eBPF, fastest), Flannel (simple). Pod-to-pod communication: flat network, every pod reachable. NetworkPolicies: firewall rules for pods. Default: all traffic allowed. Apply deny-all then whitelist. DNS: CoreDNS cho service discovery. Ingress controllers: Nginx, Traefik, Istio Gateway.
7. RBAC — Security: Roles/ClusterRoles: define permissions. RoleBindings/ClusterRoleBindings: assign to users/service accounts. Principle of least privilege. Namespace-scoped Roles cho team isolation. Service accounts cho workload identity. Audit logging cho compliance. ĐỪNG dùng cluster-admin cho regular operations.

📖 Chuyện Kể Đêm Khuya

🌪️ Chuyện Pod Eviction Storm

Kubernetes cluster 50 nodes, 2000 pods. Node running hot — memory pressure. Kubelet start evicting pods (BestEffort first, then Burstable). Evicted pods rescheduled trên OTHER nodes → other nodes become memory-pressured → cascade eviction. 30% pods evicted within 10 minutes.

Root cause: pods without memory requests/limits → scheduled anywhere → node overcommitted. Fix: mandatory resource requests/limits (LimitRange, ResourceQuota). Proper capacity planning. Pod Priority Classes cho critical workloads. Monitor node resource utilization.

💡 Bài học: ALWAYS set resource requests AND limits. Requests = scheduling guarantee. Limits = maximum allowed. Without them: nodes overcommit → cascade evictions. LimitRange enforce defaults. ResourceQuota enforce namespace totals. Capacity planning: leave headroom cho spikes.

🕸️ Chuyện Service Mesh Meltdown

Team deploy Istio service mesh cho 'better observability and security.' 200 microservices, mỗi cái có Envoy sidecar. Result: pod count double (200 app + 200 sidecar). Memory usage +40%. Latency +20ms per hop (sidecar overhead). Control plane (istiod) consuming 8GB RAM. Engineers spent more time debugging Istio than actual applications.

CTO: 'We added complexity to reduce complexity?' Đại Sư recommend: start with Linkerd (lighter) hoặc Cilium (sidecar-less). Hoặc: implement mTLS và observability incrementally, không phải all-at-once full mesh.

💡 Bài học: Service mesh powerful nhưng complex. Start small: mTLS first, then observability, then traffic management. Linkerd simpler than Istio. Cilium sidecar-less (eBPF) cho lowest overhead. Evaluate: does your organization NEED service mesh? 10 microservices probably don't.

📋 Chuyện YAML Hell

Deployment 500 dòng YAML, mỗi environment khác nhau (dev, staging, prod). Developer maintain 3 copies. Thay đổi 1 field → update 3 files. Drift between environments. Bug in production chỉ vì staging YAML thiếu 1 environment variable.

Đại Sư introduce: Kustomize cho environment overlays (base + patches). Helm cho templated deployments. CI/CD validate YAML trước apply. Sealed Secrets cho encrypted secrets trong git. ArgoCD cho GitOps — YAML trong git = source of truth.

💡 Bài học: Raw YAML cho nhiều environments = maintenance nightmare. Kustomize: overlays approach, good for simple differences. Helm: templating approach, good for reusable packages. GitOps (ArgoCD/Flux): git = desired state. Choose ONE approach và standardize. YAML linting trong CI/CD catch errors sớm.

⚠️ Những Sai Lầm Kinh Điển

1. Không set resource limits: Pod eat all node memory → OOM → cascade eviction. ALWAYS set requests AND limits. Use LimitRange cho defaults.
2. Latest tag cho container images: kubectl rollout restart pull same 'latest' tag — but image changed. Pin specific versions. Image pull policy: IfNotPresent + versioned tags.
3. Secrets trong YAML không encrypted: base64 is NOT encryption. Sealed Secrets, SOPS, External Secrets Operator cho encrypted secrets. Don't commit raw Secrets to git.
4. Không có PodDisruptionBudgets: Node drain evict tất cả pods cùng lúc. PDB ensure minimum available replicas during maintenance.
5. Single replica deployments: 1 replica = 0 availability during updates/failures. Minimum 2 replicas cho production. 3+ cho critical services.
6. Ignore namespace isolation: Everything trong default namespace. No RBAC, no ResourceQuota, no NetworkPolicy. Namespaces cho team/environment isolation.
7. kubectl apply từ local machine: Developer kubectl apply trực tiếp lên production. No review, no audit, no rollback. GitOps approach: all changes qua git PR → ArgoCD sync.
