📜 Thân Thế & Lai Lịch

Lê Siêu Khung từng là một React developer bình thường, cho đến ngày khách hàng hỏi: "Sao Google không index được trang web?" Đó là khoảnh khắc ông phát hiện rằng SPA thuần client-side render chỉ trả về một file HTML trống với một <div id="root"></div> — Google bot nhìn vào chỉ thấy hư vô.

Từ đó, ông bước vào con đường SSR, trải qua đủ thể loại đau khổ: hydration mismatch, server/client state desync, build time 45 phút cho 10,000 trang. Ông tu luyện qua 5 meta framework, mỗi cái cho ông một vết sẹo và một bài học. Ngày nay, ông có thể nhìn vào bất kỳ trang web nào và nói chính xác nó dùng rendering strategy gì, chỉ bằng cách View Source.

Triết lý của ông: "Không phải trang nào cũng cần SSR, không phải trang nào cũng cần SPA. Hãy để content quyết định rendering strategy."

🗝️ Bí Kíp

Bí Kíp #1: SSR vs SSG vs ISR vs CSR — Chọn đúng vũ khí

SSR (Server-Side Rendering): Mỗi request, server render HTML fresh. Tốt cho nội dung dynamic, personalized. Chi phí: server compute mỗi request. SSG (Static Site Generation): Build time render HTML, serve từ CDN. Tốt cho blog, docs, marketing. Chi phí: build time tăng theo số trang. ISR (Incremental Static Regeneration): SSG + revalidation sau N giây. Best of both worlds nhưng caching complexity cao. CSR (Client-Side Rendering): Trình duyệt render. Tốt cho app behind auth, dashboards. SEO = zero. Quy tắc vàng: dùng SSG cho public content, SSR cho personalized, CSR cho authenticated app.

Bí Kíp #2: Next.js App Router — Kỷ nguyên mới

App Router (Next.js 13+) thay đổi hoàn toàn mental model: Server Components mặc định, "use client" khi cần interactivity. File-based routing với nested layouts, loading states, error boundaries tích hợp. generateStaticParams cho static generation, revalidateTag cho on-demand revalidation. Server Actions cho form handling không cần API route. Cẩn thận: learning curve dốc, nhiều patterns khác hoàn toàn Pages Router. Đừng migrate existing app sang App Router trừ khi có thời gian.

Bí Kíp #3: Nuxt 3 — Vue's Meta Framework Powerhouse

Nuxt 3 với Nitro server engine cho phép deploy everywhere: Node.js, Edge, Serverless, Deno, Bun. Auto-imports cho composables và components — không cần import statement. useFetch và useAsyncData handle SSR data fetching với deduplication tự động. Server routes trong /server/api/ tạo API endpoints. Nuxt Content cho markdown-based content. Hybrid rendering: mỗi route có thể có rendering strategy khác nhau.

Bí Kíp #4: Astro — Content-first, Zero JS by default

Astro ship ZERO JavaScript mặc định — chỉ HTML/CSS. Components interactive được "hydrate" theo strategy: client:load (ngay lập tức), client:idle (khi idle), client:visible (khi scroll vào viewport), client:media (khi media query match). Island Architecture: mỗi interactive component là một "island" độc lập. Dùng bất kỳ UI framework nào (React, Vue, Svelte) trong cùng 1 project. Perfect cho content sites, blogs, docs, marketing.

Bí Kíp #5: Remix — Web Fundamentals First

Remix philosophy: dùng web platform APIs thay vì reinvent. Forms dùng native <form>, data loading qua loader functions, mutations qua action functions. Nested routing với parallel data loading — mỗi route segment fetch data riêng, không waterfall. Progressive enhancement: app hoạt động KHÔNG CẦN JavaScript (form submit, navigation). Error boundaries per route segment. Cần JavaScript cho enhanced UX, không phải base functionality.

Bí Kíp #6: Hydration — Quá trình biến HTML thành app

Hydration là quá trình JavaScript "attach" event listeners và state vào HTML đã được server render. Nó là bottleneck lớn nhất của SSR: browser phải download JS, parse, execute, rồi reconcile với DOM. Partial hydration (Astro islands), progressive hydration (lazy hydrate on interaction), và resumability (Qwik) là 3 approaches giải quyết vấn đề này. Next.js React Server Components reduce hydration bằng cách không ship JS cho server-only components.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Hydration Mismatch — Cơn Ác Mộng Đêm Deploy

Một đêm thứ Sáu (đúng rồi, deploy thứ Sáu), trang e-commerce mới launch với Next.js SSR. Trang product hiển thị đúng trong 0.5 giây đầu, rồi... FLASH — toàn bộ content biến mất rồi hiện lại khác. Giá sản phẩm từ "500,000 VND" nhảy thành "Loading...".

Nguyên nhân: component hiển thị thời gian "3 phút trước" — server render lúc 20:00:00, client hydrate lúc 20:00:03. React phát hiện mismatch → re-render toàn bộ subtree. Nhưng lúc re-render, client state chưa có data → "Loading...". Users thấy giá biến mất, panic, gọi hotline. CS team nhận 200 cuộc gọi trong 1 giờ.

Fix: suppressHydrationWarning cho dynamic content, dùng useEffect để update client-only values sau hydration, và QUAN TRỌNG NHẤT: không bao giờ deploy thứ Sáu tối.

💡 Bài học: Server-rendered HTML phải KHỚP CHÍNH XÁC với client initial render. Bất kỳ sự khác biệt nào (timestamps, random values, browser-only APIs) sẽ gây hydration mismatch. Dùng useEffect cho client-only logic, suppressHydrationWarning cho timestamps, và test SSR output locally trước khi deploy.

📖 Chuyện 2: SEO Trước Thời SSR — Thời Kỳ Đen Tối

Năm 2018, startup chi 50 triệu/tháng cho Google Ads. CEO hỏi: "Sao organic traffic = 0?" Kiểm tra: SPA thuần React, View Source chỉ thấy <div id="root"></div>. Google bot render JavaScript bằng headless Chrome, nhưng crawl budget có hạn — và nó ưu tiên crawl trang có HTML sẵn.

Giải pháp "tạm thời" của team: dùng Prerender.io — service render SPA thành HTML cho bot. Chi phí: $200/tháng. Vấn đề: bot thấy version cũ, render delay 5-10 giây, một số trang timeout. SEO team frustration level: 99/100.

Cuối cùng migrate sang Next.js SSG cho product pages. Organic traffic tăng 400% trong 3 tháng. CEO: "Tại sao không làm từ đầu?" Dev team: *nhìn nhau im lặng*

💡 Bài học: Nếu SEO quan trọng (e-commerce, blog, marketing), SSR/SSG là KHÔNG THƯƠNG LƯỢNG. SPA + prerendering là hack, không phải solution. Meta frameworks solve SEO by default — đó là lý do chúng tồn tại.

📖 Chuyện 3: Build Time 45 Phút cho 10,000 Trang

Blog platform với 10,000 bài viết, dùng Next.js SSG. Mỗi lần build: 45 phút. Sửa typo một bài → build lại TOÀN BỘ 10,000 trang. CI/CD pipeline chạy lâu hơn cả code review. Developer push code lúc 9h sáng, deploy lúc 10h trưa.

Thử ISR: giảm build time xuống 5 phút (chỉ build popular pages), còn lại on-demand. Nhưng rồi phát hiện: stale content serve cho 60 giây đầu, SEO crawler đôi khi index version cũ, cache invalidation phức tạp hơn tưởng tượng.

Cuối cùng chuyển sang Astro + Content Collections: build 10,000 trang markdown trong 90 giây. Không cần server runtime. Static files serve từ CDN. Team dev khóc vì hạnh phúc.

💡 Bài học: SSG không scale tuyến tính — 10x pages ≠ 10x build time (thường tệ hơn). Với content sites lớn, cân nhắc: Astro (build speed), ISR (incremental update), hoặc on-demand generation (build khi request). Đo build time là metric quan trọng — nó ảnh hưởng trực tiếp đến deploy velocity.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: SSR mọi thứ — kể cả dashboard behind auth

Dashboard chỉ authenticated users mới thấy → Google bot không index → SSR vô nghĩa. Tệ hơn, SSR dashboard phải handle auth trên server, tăng complexity. Dùng CSR cho authenticated apps, SSR/SSG cho public content.

❌ Sai Lầm #2: Dùng window / document trong server component

Server không có window, không có document, không có localStorage. Code dùng browser APIs trong SSR → crash. Luôn check typeof window !== 'undefined' hoặc dùng dynamic import với ssr: false.

❌ Sai Lầm #3: Fetch data trong useEffect thay vì server loader

Meta framework cho bạn server-side data fetching (getServerSideProps, loader, useFetch). Dùng nó! Fetch trong useEffect = waterfall (HTML → JS → fetch → render), trong khi server fetch = parallel (fetch + render cùng lúc).

❌ Sai Lầm #4: Không set cache headers cho SSR responses

SSR render mỗi request → server cost cao. Set Cache-Control headers hợp lý: s-maxage=60, stale-while-revalidate=300 cho CDN caching. Giảm server load 90% mà user vẫn thấy content fresh.

❌ Sai Lầm #5: Ignore streaming SSR

Traditional SSR: server render TOÀN BỘ page rồi gửi. Streaming SSR: gửi shell ngay, stream content khi ready. Next.js loading.js, React Suspense boundaries enable streaming. TTFB giảm dramatically — user thấy layout ngay, content fill dần.

❌ Sai Lầm #6: Deploy SSR app lên static hosting

SSR cần SERVER chạy Node.js. Netlify static hosting, GitHub Pages — KHÔNG chạy SSR được. Cần Vercel, AWS Lambda, hoặc self-hosted Node.js server. SSG mới deploy được lên static hosting. Hiểu rõ infrastructure requirement trước khi chọn rendering strategy.
