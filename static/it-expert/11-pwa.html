📜 Thân Thế & Lai Lịch

Lê Ứng Dụng bắt đầu sự nghiệp mobile bằng cách build native app — iOS bằng Objective-C, Android bằng Java. Mỗi feature viết 2 lần. Mỗi bug fix 2 lần. Mỗi release chờ App Store review 3-7 ngày. Ông kiệt sức. Rồi Service Workers ra đời, và ông thấy ánh sáng cuối đường hầm: một codebase, chạy everywhere, install từ browser, offline capable, push notifications. 'Đây là tương lai!' ông la lên. Nhưng tương lai đi kèm service worker cache bugs, iOS Safari limitations, và push notification complexities. Sau nhiều năm chiến đấu với service worker lifecycle, ông trở thành chuyên gia PWA — hiểu rõ từng browser quirk, từng caching strategy, từng platform limitation. Motto: 'PWA không thay thế native app — nó là lựa chọn thứ ba, và đôi khi là lựa chọn tốt nhất.'

🗝️ Bí Kíp

Bí Kíp #1: Service Worker Lifecycle — Hiểu để không phát điên

Service Worker có lifecycle riêng: Install → Activate → Fetch. Install: download và cache resources. Activate: clean up old caches. Fetch: intercept network requests, serve from cache hoặc network. QUAN TRỌNG: new service worker install nhưng KHÔNG activate cho đến khi tất cả tabs cũ đóng (waiting state). skipWaiting() force activate ngay — nhưng cẩn thận: old cached HTML + new service worker = potential mismatch. Dùng Workbox library — abstract away lifecycle complexity.

Bí Kíp #2: Caching Strategies

Cache First: Check cache → return if found → else fetch. Tốt cho static assets (CSS, JS, images). Network First: Fetch → fallback to cache if offline. Tốt cho API responses, dynamic content. Stale While Revalidate: Return cache immediately → fetch in background → update cache. Tốt cho content hay thay đổi nhưng cần fast response. Network Only: Analytics, non-cacheable. Cache Only: Pre-cached resources during install. Workbox makes implementing these strategies trivial.

Bí Kíp #3: Web Manifest — Từ website thành app

manifest.json: name, short_name, icons (multiple sizes), start_url, display: standalone (fullscreen app experience), theme_color, background_color. Splash screen auto-generated từ manifest. Icons: tối thiểu 192x192 + 512x512 PNG. maskable icon cho adaptive icon trên Android. Apple: cần <meta name='apple-mobile-web-app-capable'> + <link rel='apple-touch-icon'> vì Safari KHÔNG đọc manifest icons đầy đủ.

Bí Kíp #4: Push Notifications — Đừng spam

Push API + Notification API. Flow: request permission → get subscription → send to server → server push via VAPID. Timing: ĐỪNG request permission ngay khi user vào trang — 90% sẽ deny. Request SAU khi user engage (đọc 3 bài, add to cart, etc). Content: personal, actionable, timely. 'Đơn hàng #123 đang giao' = tốt. 'Ghé thăm app nhé!' = spam. iOS Safari: Push API support từ iOS 16.4 — cuối cùng! Nhưng user phải 'Add to Home Screen' trước.

Bí Kíp #5: Offline-first Design

Design cho offline đầu tiên, online là enhancement. IndexedDB cho offline data storage. Background Sync API: queue actions khi offline, sync khi online. Conflict resolution: last-write-wins (simple), merge (complex), CRDT (advanced). UI: indicate offline status rõ ràng, disable actions that REQUIRE network, cache và serve essential content. Test: Chrome DevTools > Application > Service Workers > Offline checkbox.

🌙 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Service Worker Cache Gone Wrong

Deploy new version, users vẫn thấy OLD version. Tại sao? Service worker cache served old HTML, old JS, old CSS — dù server đã có version mới. User phải close tất cả tabs → mở lại → chờ new SW activate → refresh. Một số users cached version cũ suốt 2 tuần vì 'không bao giờ close Chrome trên điện thoại'. Worse: một bug trong caching logic cache API responses vĩnh viễn. User thấy outdated product prices. Một user mua sản phẩm với giá cũ (thấp hơn) vì cache serve old price — legal issue. Fix: implement cache versioning, skipWaiting() + clients.claim() với update notification UI, set max-age cho API caches, never cache payment-related APIs.

💡 Bài học: Service worker cache = great power, great responsibility. Luôn: version caches, clean up old versions, set TTL cho API caches, show 'update available' UI, NEVER cache sensitive/transactional data. Test cache invalidation flow kỹ càng — cache bugs là những bugs khó reproduce nhất.

📖 Chuyện 2: Offline Sync Conflicts — Khi Hai Người Edit Cùng Lúc

Collaborative note-taking PWA. User A edit note offline. User B edit CÙNG note offline. Cả hai go online → sync → conflict. Last-write-wins: User A's changes biến mất. User A: 'Tôi vừa viết 2 trang, đâu hết rồi?!' Team implement merge strategy: operational transform cho text, field-level merge cho structured data. Vẫn có edge cases: User A xóa paragraph, User B edit paragraph đó → merge cái gì? Team cuối cùng add conflict resolution UI: hiển thị cả 2 versions, user chọn.

💡 Bài học: Offline sync conflict resolution là BÀI TOÁN KHÓ nhất trong distributed systems (tại client level). Simple apps: last-write-wins + conflict notification. Medium: field-level merge. Complex: CRDT (Conflict-free Replicated Data Types) — Yjs, Automerge. Design decision phải sớm — retrofit conflict resolution vào existing app = pain.

📖 Chuyện 3: Push Notification 3 Giờ Sáng — Thảm Họa Marketing

Marketing team dùng push notification API, schedule blast promotion cho 9h sáng. Bug trong timezone handling: server ở UTC, user ở UTC+7, nhưng code dùng server timezone. Kết quả: notification gửi lúc 2h sáng giờ địa phương cho 50,000 users. 2h sáng, 50,000 điện thoại kêu. Twitter explode với complaints. App rating từ 4.5 sao xuống 2.1 sao trong 12 giờ. Unsubscribe rate: 60%. Marketing budget bỏ ra 6 tháng build subscriber list — bay trong 1 đêm. Fix: timezone-aware scheduling, user timezone stored at subscription time, quiet hours setting (không gửi 22h-8h local time), double confirmation cho mass notifications.

💡 Bài học: Push notifications là PRIVILEGE, không phải RIGHT. Respect user: 1) Request permission at right moment, 2) Send relevant/personal content only, 3) Respect quiet hours, 4) Make unsubscribe easy, 5) Test timezone handling obsessively. One bad notification = thousands of unsubscribes.

💀 Sai Lầm Kinh Điển

❌ Sai Lầm #1: Cache everything forever

Cache without expiry = users stuck on old version. Version caches, set TTL, implement cache cleanup trong activate event. Đừng cache HTML forever — cache CSS/JS/images with content hash, HTML with short TTL.

❌ Sai Lầm #2: Request push permission on first visit

User chưa biết app làm gì → popup 'Allow notifications?' → 90% click Block. Block = PERMANENT (user phải vào browser settings để unblock). Request sau khi user engage và understand value.

❌ Sai Lầm #3: Ignore iOS Safari limitations

iOS Safari: no Push API until 16.4, no Background Sync, service worker killed after ~days inactive, limited cache storage. Test PWA trên iOS Safari riêng — đừng assume Android Chrome behavior applies.

❌ Sai Lầm #4: Không test offline scenarios

App 'offline capable' nhưng không ai test offline. Disconnect WiFi → app crash vì fetch error not handled. Test MỌII network condition: offline, slow 3G, flaky connection, switch online/offline rapidly.

❌ Sai Lầm #5: Service worker block page load

Heavy computation trong service worker fetch handler → page load slow. Service worker runs on SEPARATE thread nhưng fetch interception adds latency. Keep fetch handler light, dùng cache-first cho static assets.
