📜 Bí Kíp

Bí Kíp #1: Spark Architecture — Driver, Executor, Shuffle

Driver: chương trình chính, tạo execution plan. Executors: worker processes xử lý data. Shuffle: redistribute data giữa executors (tốn nhất!). Narrow transformations (map, filter) = không shuffle = nhanh. Wide transformations (groupBy, join) = shuffle = chậm. Giảm shuffle = giảm runtime.

Bí Kíp #2: Partition Strategy — Nghệ Thuật Cắt Bánh

Quá ít partitions = ít parallelism, có executor ngồi chơi. Quá nhiều partitions = overhead scheduling. Rule of thumb: 2-4 partitions per CPU core. Data skew: 1 partition có 90% data = 1 executor làm hết = chậm. Repartition bằng salted key để phân tán đều.

Bí Kíp #3: dbt — SQL Đã Đủ

dbt (data build tool): viết transformations bằng SQL, dbt lo compilation, dependency, testing. Models: SELECT statements biến thành tables/views. Refs: {{ ref('model_name') }} tạo dependency graph. Tests: unique, not_null, accepted_values, relationships. dbt giúp data engineers có software engineering practices: version control, CI/CD, documentation.

Bí Kíp #4: File Formats — Parquet Là Vua

CSV: human-readable, terrible cho performance. JSON: flexible, terrible cho analytics. Parquet: columnar, compressed, predicate pushdown, perfect cho analytics. 1TB CSV → 200GB Parquet (compression) → query chỉ đọc cột cần thiết (column pruning). ORC: tương tự Parquet, phổ biến trong Hive ecosystem.

Bí Kíp #5: Spark OOM — Hiểu Memory Management

Spark memory = Execution (shuffle, join, sort) + Storage (cached data) + User (UDFs). OOM thường do: data skew (1 partition quá lớn), cartesian join (vô tình), collect() large dataset về driver. Fix: tăng partitions, broadcast small tables, tránh collect(), tăng spark.sql.shuffle.partitions.

Bí Kíp #6: Incremental Models — Đừng Rebuild Mọi Thứ

dbt incremental model: chỉ process data mới từ lần chạy cuối. {{ config(materialized='incremental') }} + {% if is_incremental() %} WHERE updated_at > (SELECT max(updated_at) FROM {{ this }}) {% endif %}. Giảm runtime từ giờ xuống phút. Nhưng phải handle late-arriving data và dedup.

📖 Chuyện Kể Đêm Khuya

💥 Chuyện 1: Spark Job OOM Trên 1TB Data

Team cần JOIN bảng orders (500GB) với bảng products (2GB). Spark job chạy 20 executors, mỗi executor 8GB RAM. After shuffle: OOM. Tại sao? Default shuffle partitions = 200. 500GB / 200 = 2.5GB per partition. Mỗi executor xử lý vài partitions → memory boom.

Lão fix 3 bước: 1) Tăng spark.sql.shuffle.partitions từ 200 lên 2000. 2) Broadcast bảng products (2GB < broadcast threshold): broadcast(products_df) — eliminate shuffle hoàn toàn. 3) Cache intermediate results. Runtime giảm từ "crash after 2 hours" xuống 15 phút.

💡 Bài học: Broadcast join khi 1 bảng nhỏ (< 10GB). Tăng shuffle partitions cho large data. Monitor Spark UI: check shuffle read/write size, GC time, spill to disk. Hiểu execution plan bằng .explain().
🔥 Chuyện 2: Hadoop Cluster Fire — Yarn Queue Chaos

Hadoop cluster 100 nodes, shared giữa 5 teams. Không ai config YARN queue properly — mọi job submit vào default queue. Một ngày đẹp trời, team ML submit training job chiếm 80% cluster resources. 4 teams còn lại: tất cả jobs queued. ETL pipeline delay 8 tiếng. Daily reports không có data. Stakeholders nổi điên.

Lão dọn dẹp: tạo YARN capacity scheduler với 5 queues, mỗi queue có min/max resources. Team ML queue max 30%. ETL queue guaranteed 25%. Preemption enabled: nếu ETL queue cần resources mà ML không dùng hết → lấy lại. Hòa bình được lập lại.

💡 Bài học: Shared cluster PHẢI có resource management. YARN queues, Kubernetes resource quotas, Spark fair scheduler. Không governance = 1 team chiếm hết resources = mọi team khổ.
🧱 Chuyện 3: dbt Model Dependencies — 500 Models Spaghetti

Data team 8 người, dùng dbt, mỗi người thoải mái tạo models. 2 năm sau: 500 models, dependency graph nhìn như bản đồ metro Tokyo. dbt run mất 4 giờ vì dependencies chain dài 15 levels. Thay đổi 1 staging model → cascade rebuild 200 downstream models.

Lão áp dụng dbt best practices: tách thành layers (staging → intermediate → marts), naming conventions (stg_, int_, fct_, dim_), tags cho grouping, selectors cho partial runs. Refactor mất 2 tháng nhưng dbt run giảm xuống 30 phút.

💡 Bài học: dbt project cần architecture từ đầu. Medallion/layered approach: staging → intermediate → marts. Naming conventions. Documentation. Code review cho SQL models cũng quan trọng như application code.

⚠️ Những Sai Lầm Kinh Điển

❌ collect() large DataFrame về driver

df.collect() trên 10 triệu rows = OOM driver. Dùng .take(n), .show(n), hoặc write to file thay vì collect.

❌ UDF thay vì built-in functions

Python UDF trong Spark = serialize data → Python process → deserialize. 10-100x chậm hơn built-in Catalyst functions. Dùng pyspark.sql.functions built-in, tránh UDF khi có thể.

❌ Small files problem

1 triệu files 1MB thay vì 1000 files 1GB. Mỗi file = 1 Spark task = scheduling overhead. HDFS namenode OOM. Coalesce/repartition output files. Target file size: 128MB-1GB.

❌ Không partition output data

Write 1TB Parquet không partition = mọi query scan 1TB. Partition by date: query 1 ngày chỉ scan 3GB. Nhưng đừng partition by high-cardinality column (user_id) → triệu folders.

❌ dbt models quá complex

1 model SQL 500 dòng với 12 CTEs và 8 JOINs. Impossible to test, impossible to debug. Chia thành multiple models, mỗi model 1 transformation logic.
