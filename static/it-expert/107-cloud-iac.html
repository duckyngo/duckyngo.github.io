📚 Thân Thế & Lai Lịch

Cloud IaC Đại Sư sống trong thế giới vendor-specific IaC. CloudFormation cho AWS, ARM/Bicep cho Azure, Deployment Manager cho GCP. "Tại sao không dùng Terraform?" — "Vì CloudFormation tích hợp native, support mọi AWS feature ngay ngày ra mắt, và change sets cho safe deployments."

Chàng ta viết CloudFormation YAML cho 5 năm. 2000 dòng cho một VPC + ECS cluster. Rồi CDK ra đời: same infrastructure, 200 dòng TypeScript. "Tại sao tôi viết YAML suốt 5 năm?!" CDK generates CloudFormation behind the scenes — best of both worlds.

Bên Azure, chàng ta dùng ARM templates (JSON horror) rồi Bicep (human-readable Azure IaC). "Bicep là điều ARM templates nên là từ đầu," chàng ta nói.

📜 Bí Kíp

1. CloudFormation — AWS Native: YAML/JSON templates. Stack = deployment unit. Change Sets cho safe updates. Nested stacks cho modularity. Drift detection. Rollback on failure. StackSets cho multi-account deployment. Custom resources (Lambda-backed) cho missing features. Intrinsic functions: !Ref, !GetAtt, !Sub, Fn::ImportValue.
2. AWS CDK — Infrastructure as Code++: TypeScript, Python, Go, Java, C#. Higher-level constructs: L1 (CloudFormation), L2 (opinionated defaults), L3 (patterns). cdk synth generates CloudFormation. cdk diff cho change preview. Construct library cho reusable patterns. cdk watch cho hot-deploy during development.
3. Bicep — Azure's Better ARM: Domain-specific language for Azure. Transpiles to ARM JSON. Simpler syntax, type safety, modules. Bicep Registry cho shared modules. What-if deployment cho preview. Deployments scoped to resource group, subscription, or management group. VS Code extension excellent.
4. Stack Management: Environment separation: dev/staging/prod stacks. Parameter files per environment. Outputs cho cross-stack references. Import/export values. Stack policies cho production protection. Automatic rollback on failure.
5. CDK Constructs — Reusable Patterns: L3 constructs encapsulate complex patterns. Projen cho project management. CDK Pipelines cho self-mutating CI/CD. cdk-nag cho security/compliance checks. Construct Hub cho community constructs. Versioned, published as npm/pip packages.

📖 Chuyện Kể Đêm Khuya

🔄 Chuyện CloudFormation Rollback Stuck

CloudFormation stack update: add new RDS instance. Creation takes 15 minutes. During creation, Lambda function in same stack fails health check. CloudFormation initiates rollback. Rollback tries to delete RDS — takes 10 minutes. During deletion, another resource timeout. Rollback stuck in UPDATE_ROLLBACK_FAILED state.

Manual intervention required: skip failing resources (ContinueUpdateRollback with resources to skip). Fix underlying issue. Re-attempt update. Total time: 3 giờ cho what should be 20-minute deployment.

💡 Bài học: CloudFormation rollbacks can get stuck. DependsOn carefully designed. Separate stateful resources (RDS) from stateless (Lambda). Small, incremental changes. ContinueUpdateRollback cho stuck rollbacks. Stack policies prevent accidental deletion of critical resources.

🏗️ Chuyện CDK Synth Surprise

CDK code: const bucket = new s3.Bucket(this, 'MyBucket'). Developer expects 1 S3 bucket. cdk synth output: S3 bucket + bucket policy + CloudFormation custom resource cho auto-delete + Lambda function cho custom resource. Total: 4 CloudFormation resources từ 1 dòng CDK code. Developer confused: 'I only wanted a bucket!'

L2 constructs add 'sensible defaults' — sometimes more than expected. Solution: cdk diff ALWAYS before deploy. Understand L2 construct behavior. Use L1 constructs (CfnBucket) cho explicit control khi needed.

💡 Bài học: CDK L2 constructs create more resources than visible in code. ALWAYS cdk diff before deploy. Understand construct documentation. L1 constructs cho explicit control. cdk synth cho full CloudFormation review. Surprises = tech debt nếu không understood.

💪 Chuyện Bicep Module Saves Hours

Team managing 50 Azure App Services. ARM template per service: 500 lines JSON. Total: 25,000 lines JSON. Maintenance nightmare. Bug fix = update 50 files.

Đại Sư create Bicep module: appService.bicep (100 lines). Each service: 20-line Bicep file referencing module. Total: 1,100 lines. Bug fix = update 1 module. Deployment time unchanged. Maintenance time: -90%. Team happiness: +200%.

💡 Bài học: Modules/constructs cho DRY infrastructure code. Single source of truth cho common patterns. Pin module versions cho stability. Test modules independently. Shared module registry cho organization-wide reuse.

⚠️ Những Sai Lầm Kinh Điển

1. 2000-line single template: Monolith CloudFormation = blast radius everything. Split into nested stacks or separate stacks. Microservice-like infrastructure.
2. CDK without cdk diff: Deploy without reviewing generated CloudFormation. Surprises in production. ALWAYS review changes.
3. ARM templates khi Bicep available: JSON ARM templates painful to write and read. Bicep transpiles to ARM — better syntax, same result. Migrate.
4. No deletion protection: Production RDS in CloudFormation without DeletionPolicy: Retain. Stack delete = data loss. Set DeletionPolicy cho stateful resources.
5. Hardcoded AMI IDs: AMI IDs region-specific. Use SSM Parameter Store cho latest AMI. Or CDK/CFN constructs that auto-resolve.
