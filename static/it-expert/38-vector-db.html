Lão hay nói: "Tôi đã chờ 7 năm để thế giới hiểu vector similarity search quan trọng thế nào. Giờ thì mọi người biết — nhưng 90% trong số họ dùng sai."

📜 Bí Kíp

Bí Kíp #1: Embedding — Biến Text Thành Vector

Embedding model biến text → dense vector (array of floats). OpenAI text-embedding-3-small: 1536 dimensions. Sentence-transformers all-MiniLM: 384 dimensions. Cohere embed-v3: 1024 dimensions. Chọn model phù hợp: multilingual? domain-specific? Kích thước dimension ảnh hưởng storage và speed. Nhỏ hơn = nhanh hơn nhưng ít chính xác hơn.

Bí Kíp #2: HNSW vs IVF — Hai Trường Phái Index

HNSW (Hierarchical Navigable Small World): graph-based, recall cao, build index chậm, tốn RAM. Tốt cho < 10 triệu vectors. IVF (Inverted File Index): partition-based, build nhanh hơn, có thể on-disk. IVF-PQ thêm Product Quantization để nén vector, giảm memory 4-8x. Trade-off: recall vs speed vs memory.

Bí Kíp #3: Chunking Strategy — Nghệ Thuật Cắt Văn Bản

RAG pipeline: document → chunks → embeddings → vector DB. Chunk quá nhỏ = mất ngữ cảnh. Chunk quá lớn = embedding bị diluted. Sweet spot: 200-500 tokens với 50-100 token overlap. Thử semantic chunking (cắt theo ý nghĩa, không phải ký tự). Recursive text splitter tốt hơn fixed-size splitter.

Bí Kíp #4: Hybrid Search — Vector + Keyword = Power

Pure vector search yếu ở exact matches ("mã đơn hàng ABC123"). Pure keyword search yếu ở semantic ("cách xử lý khách hàng giận dữ" vs "customer complaint handling"). Hybrid search kết hợp cả hai: BM25 score + cosine similarity, weighted fusion. Weaviate và Qdrant hỗ trợ native hybrid search.

Bí Kíp #5: pgvector — Khi PostgreSQL Là Đủ

Dưới 1 triệu vectors? pgvector đủ rồi. Ưu điểm: không cần thêm infra mới, SQL quen thuộc, JOIN với relational data, ACID transactions. CREATE INDEX ON items USING hnsw (embedding vector_cosine_ops). Nhược: chậm hơn Milvus/Qdrant ở scale lớn. Nhưng simplicity > performance cho hầu hết use cases.

Bí Kíp #6: Metadata Filtering — Đừng Search Cả Biển

Filter trước, search sau. Ví dụ: search trong documents của department "Engineering" thay vì toàn công ty. Pre-filtering giảm search space → nhanh hơn và chính xác hơn. Tất cả vector DBs hỗ trợ metadata filtering: Pinecone namespaces, Weaviate where filters, Qdrant payload filtering.

Bí Kíp #7: Evaluation — Đo Lường Chất Lượng Search

Metrics: Recall@k, MRR (Mean Reciprocal Rank), nDCG. Tạo golden dataset: 100-200 queries với expected results. Test mỗi khi thay đổi: embedding model, chunking strategy, index parameters. Không đo = không biết search quality tốt hay xấu.

📖 Chuyện Kể Đêm Khuya

🤦 Chuyện 1: Sai Embedding Model, Hỏng Toàn Bộ Search

Startup xây RAG chatbot cho tài liệu tiếng Việt. Dev chọn embedding model all-MiniLM-L6-v2 — model tiếng Anh, 384 dimensions. Index 10,000 documents tiếng Việt. Search "chính sách bảo hành" → trả về document về "warranty policy"... viết bằng tiếng Anh. Vì embedding model không hiểu tiếng Việt, nó encode dựa trên character patterns thay vì semantic meaning.

Lão đổi sang multilingual model multilingual-e5-large. Search quality tăng vọt. Nhưng phải re-embed toàn bộ 10,000 documents — tốn 4 giờ compute và $50 API calls. Và phải rebuild index.

💡 Bài học: Embedding model phải phù hợp với ngôn ngữ và domain. Multilingual model cho multi-language data. Domain-specific fine-tuning cho specialized domains (medical, legal). Test search quality TRƯỚC KHI index toàn bộ data.
💥 Chuyện 2: Dimension Mismatch — Lỗi Ngớ Ngẩn Nhất Thế Giới

Team A embed documents bằng OpenAI text-embedding-ada-002 (1536 dims). Team B build search API dùng model text-embedding-3-small cũng 1536 dims nhưng... khác encoding space. Insert vectors từ model A, query bằng vectors từ model B. Kết quả: hoàn toàn ngẫu nhiên. Search "hóa đơn tháng 3" trả về recipe nấu phở.

Mất 2 tuần mới phát hiện vì: kết quả có vẻ "gần đúng" đôi khi (purely by chance), và team nghĩ là do data quality. Lão debug bằng cách check cosine similarity distribution — nếu mismatch, similarity scores sẽ cluster quanh 0 thay vì có clear separation.

💡 Bài học: Vectors PHẢI được tạo bởi CÙNG MỘT model cho cả indexing và querying. Store model name + version cùng vectors. Khi đổi model = re-embed TOÀN BỘ data. Không có shortcut.
💸 Chuyện 3: Pinecone Bill Shock — Khi AI Gặp Credit Card

Startup dùng Pinecone serverless cho RAG. Dev index 50 triệu vectors (mỗi chunk 1 document). Queries chạy tốt. Hóa đơn tháng đầu: $3,000. Tháng 2: $5,000. CEO hỏi: "Tại sao vector database đắt hơn cả AWS EC2?"

Điều tra: 50 triệu vectors quá nhiều. Nhiều chunks trùng lặp (cùng document được chunk nhiều lần do bug). Và queries không dùng metadata filter nên search toàn bộ 50M vectors mỗi lần. Lão giảm xuống 5 triệu vectors (deduplicate + better chunking), thêm namespace filtering. Bill giảm xuống $400.

💡 Bài học: Vector count = cost. Optimize chunking strategy, deduplicate, dùng metadata filtering. So sánh managed (Pinecone) vs self-hosted (Qdrant, Milvus) về cost. Với > 10M vectors, self-hosted thường rẻ hơn nhiều.

⚠️ Những Sai Lầm Kinh Điển

❌ Dùng vector DB khi keyword search là đủ

Search exact product SKU, order number, email address? Đây là exact match — Elasticsearch hoặc SQL LIKE/= giải quyết tốt hơn, nhanh hơn, rẻ hơn. Vector search cho semantic, không phải exact.

❌ Chunk quá lớn hoặc quá nhỏ

Chunk = 1 paragraph → mất ngữ cảnh. Chunk = toàn bộ document 50 trang → embedding diluted, không capture chi tiết. Sweet spot: 200-500 tokens với overlap.

❌ Không dùng metadata filtering

Search toàn bộ 10 triệu vectors khi chỉ cần search trong 1 department. Chậm và kém chính xác. Luôn filter trước bằng metadata (category, date range, source).

❌ Tin tưởng cosine similarity mù quáng

Similarity score 0.85 ≠ "85% giống nhau". Threshold phụ thuộc vào model, data, và use case. Calibrate threshold bằng golden dataset, không phải cảm giác.

❌ Bỏ qua reranking

Vector search retrieves top 100, nhưng order có thể không optimal. Cross-encoder reranker (Cohere Rerank, BGE-reranker) trên top results tăng quality đáng kể. Two-stage: fast retrieval → precise reranking.

❌ Không version control embeddings

Đổi embedding model nhưng không re-embed old data. Half vectors từ model cũ, half từ model mới. Search quality giảm thảm hại. Luôn re-embed toàn bộ khi đổi model.
