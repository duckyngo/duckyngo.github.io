📜 Bí Kíp Câu Hỏi

1

Prompt Structure — Framework CRISPE

Capacity (role): "You are a senior tax accountant." Request (task): "Analyze this invoice." Input (data): The actual data. Specifics (constraints): "Output in JSON, max 200 words." Personalization (tone): "Explain like I'm 5." Examples (few-shot): 2-3 input/output examples. Luôn explicit — LLM không đọc ý bạn.

2

Chain of Thought — Buộc LLM suy nghĩ

"Let's think step by step" tăng accuracy trên reasoning tasks 20-40%. Variations: (1) Zero-shot CoT: thêm "Think step by step." (2) Few-shot CoT: cho ví dụ có reasoning steps. (3) Self-consistency: generate nhiều CoT paths, majority vote. (4) Tree of Thought: explore multiple reasoning branches. Khi dùng: math, logic, multi-step problems. Khi KHÔNG dùng: simple extraction, translation — CoT chỉ tốn tokens.

3

RAG Architecture — Ground LLM trong facts

RAG pipeline: (1) Chunk documents (500-1000 tokens, overlap 100-200). (2) Embed chunks (sentence-transformers, OpenAI embeddings). (3) Store trong vector DB (Pinecone, Weaviate, Qdrant, pgvector). (4) Retrieve top-K relevant chunks (K=3-5). (5) Generate answer grounded trong retrieved context. Key insight: retrieval quality >> generation quality. Bad retrieval → LLM hallucinate dù prompt perfect.

4

Chunking Strategy — Chia nhỏ đúng cách

Chunk size matters enormously. Quá nhỏ: mất context. Quá lớn: noise dilutes signal. Best practices: (1) Semantic chunking: split at paragraph/section boundaries, không mid-sentence. (2) Recursive splitting: try section → paragraph → sentence. (3) Overlap: 10-20% overlap giữa chunks để không mất info ở boundaries. (4) Metadata: giữ source, page number, section title cùng với chunk. Test different chunk sizes trên your data — không có one-size-fits-all.

5

Advanced RAG — Beyond basic retrieval

Hybrid search: combine vector search + BM25 keyword search (0.7 semantic + 0.3 keyword). Re-ranking: retrieve 20 chunks, re-rank bằng cross-encoder, keep top 5. Query transformation: rewrite user query trước khi search (HyDE: generate hypothetical answer → embed answer → search). Multi-step retrieval: first retrieval → identify gaps → second retrieval. Parent document retrieval: embed small chunks, retrieve parent documents.

6

Evaluation — Đo lường quality

RAG evaluation 3 components: (1) Retrieval quality: precision@K, recall@K, MRR — retrieved đúng chunks không? (2) Generation quality: faithfulness (không hallucinate ngoài context), relevance, completeness. (3) End-to-end: answer correctness. Tools: RAGAS framework (automated metrics), LangSmith (tracing + eval). Luôn có human eval set bên cạnh automated metrics.

📖 Chuyện Kể Đêm Khuya

🎯 Chuyện 1: Một từ thay đổi mọi thứ

Team anh Hỏi Đáp build classification system. Prompt: "Classify this text as positive, negative, or neutral." Accuracy: 67%. Sau 2 tuần thử 200 variations, anh thay prompt thành: "Classify this text. Before answering, analyze the sentiment indicators step by step. Then output ONLY one of: positive, negative, neutral."

Accuracy: 91%. Sếp: "2 tuần chỉ để sửa câu hỏi?" Anh: "2 tuần để tìm ra CÂU HỎI ĐÚNG. Einstein mất 20 năm cho E=mc². Tôi mới mất 2 tuần."

💡 Bài học: Prompt engineering là iterative process. Mỗi từ matters. Specific > vague. Structured output instructions > open-ended. Step-by-step reasoning > direct answer. Và luôn test trên eval set, không chỉ thử 1-2 examples.
🔍 Chuyện 2: RAG retrieves tài liệu sai — nhưng LLM vẫn trả lời

RAG chatbot cho nội bộ công ty. Nhân viên hỏi: "Chính sách nghỉ phép 2025?" RAG retrieve: document về chính sách 2020 (chunk tương tự semantic). LLM confidently trả lời dựa trên chính sách CŨ. Nhân viên nghỉ phép theo thông tin sai, HR gọi.

Root cause: chunk metadata không có year, embeddings không distinguish 2020 vs 2025 vì content tương tự. Fix: thêm metadata filtering (year), hybrid search (keyword "2025"), và instruction "If the retrieved context doesn't mention 2025, say you don't have current information."

💡 Bài học: RAG garbage in → LLM garbage out (nhưng confident garbage). Retrieval quality là bottleneck #1. Luôn: (1) Metadata filtering, (2) Hybrid search, (3) Instruct LLM to acknowledge when context insufficient, (4) Test retrieval independently trước khi test end-to-end.
💰 Chuyện 3: Prompt quá dài tốn hết budget

Anh Hỏi Đáp build system prompt 3,000 tokens với đủ rules, examples, formatting. Mỗi API call: 3,000 (system) + 500 (user) + 1,000 (response) = 4,500 tokens. 100,000 requests/ngày × 4,500 tokens = 450M tokens/ngày. GPT-4: $4,500/ngày chỉ riêng system prompt.

Optimize: tách system prompt thành core (500 tokens) + dynamic parts (chỉ inject khi cần). Dùng few-shot examples chỉ khi task phức tạp. Kết quả: average 1,500 tokens/request. Bill giảm 3x.

💡 Bài học: System prompt tokens tính cho MỌI request. Optimize prompt length: (1) Chỉ include instructions cần thiết, (2) Dynamic injection based on query type, (3) Move examples sang few-shot chỉ khi cần. Mỗi token trong system prompt nhân với total requests = big cost.

⚠️ Những Sai Lầm Kinh Điển

❌ Prompt vague: "Summarize this" → kết quả inconsistent. "Summarize in 3 bullet points, each under 20 words, focusing on financial impact" → consistent.
❌ Không test trên eval set: Thử 2-3 examples, thấy OK → deploy. Production: 40% failure rate. Cần 50+ diverse test cases.
❌ RAG chunk size one-size-fits-all: FAQ cần chunks nhỏ (200 tokens). Legal documents cần chunks lớn (1000+). Test different sizes.
❌ Chỉ dùng vector search cho RAG: Semantic search miss exact matches. Hybrid (vector + keyword) luôn tốt hơn pure vector.
❌ Few-shot examples không representative: Examples bias output. Chọn diverse examples covering edge cases, không chỉ happy path.
❌ Tin LLM trả lời "I don't know" khi không biết: LLM thường hallucinate thay vì thừa nhận. Cần explicit instruction + retrieval grounding.
