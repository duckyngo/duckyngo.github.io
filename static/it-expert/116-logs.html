📚 Thân Thế & Lai Lịch

Nhật Ký Đại Sư nhớ thời grep log files trên production servers. tail -f /var/log/application.log | grep ERROR. Mỗi server SSH riêng. 20 servers = 20 terminal windows. "Distributed grep," chàng ta gọi.

ELK Stack (Elasticsearch + Logstash + Kibana) thay đổi cuộc đời: centralized logs, full-text search, dashboards. Nhưng Elasticsearch là memory monster — 3 nodes × 32GB RAM minimum. "Logs cost more than the application they monitor," chàng ta than.

Rồi Grafana Loki ra đời: "Like Prometheus, but for logs." Index labels only, not full text. 10x cheaper storage. Tích hợp Grafana. "Finally, logs that don't bankrupt us," chàng ta vui mừng.

📜 Bí Kíp

1. ELK Stack: Elasticsearch: full-text search, powerful query DSL. Logstash: log processing pipeline. Kibana: visualization. Filebeat: lightweight log shipper. Powerful nhưng resource-intensive. ILM (Index Lifecycle Management) cho retention. Snapshot cho backup.
2. Grafana Loki: Index labels only (not full text) = cheaper storage. LogQL query language (similar to PromQL). Tích hợp Grafana natively. Promtail/Alloy cho log collection. Multi-tenant support. S3/GCS cho long-term storage. Best cho: Prometheus+Grafana users muốn add logs.
3. Log Collection: Fluentd: plugin-based, Ruby. Fluent Bit: lightweight C version. Vector: Rust-based, fast, observable. Choose: Fluent Bit cho lightweight (containers), Vector cho performance, Fluentd cho plugin ecosystem.
4. Structured Logging: JSON format: {"level":"error","msg":"connection failed","service":"api","duration_ms":1500}. Parseable, searchable, filterable. ĐỪNG log unstructured text. Include: timestamp, level, service, trace_id, relevant context. ĐỪNG log: passwords, API keys, PII (compliance risk).
5. Retention & Cost: Hot storage (SSD): 7-14 days cho real-time analysis. Warm storage: 30-90 days cho investigation. Cold/archive: 90+ days cho compliance. Retention policies automated. Log volume reduction: log levels (don't DEBUG in production), sampling, aggregation.

📖 Chuyện Kể Đêm Khuya

💾 Chuyện Elasticsearch Disk Full

Elasticsearch cluster 3 nodes × 1TB. Application logs: 50GB/day. ILM policy: nope. 20 ngày sau: disk 100%. Elasticsearch goes read-only (to protect data). New logs cannot be indexed. Kibana shows old data only. Alerting based on logs: blind.

Emergency: delete old indices manually. Setup ILM: hot (7 days, SSD) → warm (30 days) → delete (>30 days). Index size management: rollover at 50GB. Disk monitoring alerts at 70%, 80%, 90%.

💡 Bài học: ILM policies from DAY ONE. Monitor disk usage aggressively. Elasticsearch read-only mode at ~95% disk = no new logs = blind monitoring. Size cluster for expected log volume + 30% headroom. Consider Loki if log search is more important than full-text indexing.

🔍 Chuyện Unstructured Logs Nightmare

Application logs: 'User john@company.com failed login at 2024-01-15 10:30:45 from IP 192.168.1.100.' Parsing: grok pattern = 20 minutes of regex crafting. Log format changes = parser breaks. 10 different log formats across 20 microservices. Logstash grok filters = 500 lines of regex hell.

Fix: mandate structured logging (JSON) across all services. Standard fields: timestamp, level, service, trace_id. No more parsing — just JSON decode. Search by any field. Filter, aggregate, correlate. Development time for logging library: 2 days. Saved parsing maintenance: infinite.

💡 Bài học: Structured logging (JSON) is NON-NEGOTIABLE for microservices. Standardize log format across services. Shared logging library cho consistency. No grok patterns needed. Machine-parseable from day one. Retrofit is painful — implement early.

🔐 Chuyện Password Trong Error Log

Application throws database connection error. Error message includes full connection string: 'Failed to connect to postgres://admin:P@ssw0rd123@db.prod:5432/production'. Log shipped to centralized ELK. Indexed, searchable. Anyone with Kibana access can search for 'postgres://' and find production database credentials.

Fix: log scrubbing (mask sensitive patterns). Application: never include credentials in error messages. Fluentd/Vector: regex filter to mask patterns before shipping. Audit logs for sensitive data regularly.

💡 Bài học: Logs WILL contain secrets accidentally. Defense in depth: (1) Application: never log credentials, (2) Log pipeline: scrub/mask sensitive patterns, (3) Access control: limit who can read logs, (4) Audit: regularly search logs for leaked secrets. PII in logs = GDPR/compliance violation.

⚠️ Những Sai Lầm Kinh Điển

1. Log everything at DEBUG: Log volume explodes. Storage costs explode. Signal-to-noise ratio: 0. Production: WARN/ERROR. Staging: INFO. DEBUG: only when debugging specific issue.
2. No structured logging: Unstructured text = parse hell. JSON structured logging from day one. Standard fields across services.
3. No retention policy: Logs forever = unlimited storage cost. Define retention by log type. Access logs: 30 days. Error logs: 90 days. Audit logs: 1 year.
4. Single log pipeline: Log pipeline down = no logs = blind. Redundant log shipping. Buffer/queue between collector and storage.
5. No correlation: Logs without trace_id. Cannot follow request across microservices. Distributed tracing correlation in logs essential.
