📚 Thân Thế & Lai Lịch

Pháp Sư là developer chuyển sang DevOps. HCL syntax của Terraform là rào cản: "Tại sao phải học một ngôn ngữ mới chỉ để tạo infrastructure? Tôi đã biết TypeScript!" Pulumi cho phép viết infrastructure bằng ngôn ngữ quen thuộc: TypeScript, Python, Go, C#, Java.

Ngày đầu tiên dùng Pulumi, chàng ta viết for loop để tạo 10 EC2 instances. "Trong Terraform phải dùng count hoặc for_each, trong Pulumi chỉ cần for loop bình thường!" Chàng ta phấn khích. Rồi chàng ta viết class inheritance cho infrastructure modules. Rồi generic types. Rồi async/await. Và chàng ta nhận ra: power of real programming language cũng đi kèm complexity of real programming language.

📜 Bí Kíp

1. Real Programming Languages: TypeScript/JavaScript, Python, Go, C#, Java, YAML. Full IDE support: autocomplete, type checking, refactoring. Testing: unit tests cho infrastructure logic. Debugging: breakpoints, step through. Conditionals, loops, functions — real programming constructs, không phải DSL workarounds.
2. State Management: Pulumi Cloud (default backend): managed state + secrets + history. Self-managed backends: S3, Azure Blob, GCS, local file. Stack = environment (dev, staging, prod). pulumi stack ls cho danh sách. Config per stack: pulumi config set key value. Secrets encrypted automatically.
3. Pulumi vs Terraform: Pulumi: real languages, better abstractions, steeper learning curve cho infra engineers. Terraform: HCL, larger ecosystem, more community modules, familiar cho ops teams. Both: state-based, plan/preview before apply, supports major clouds. Choose based on team skills — developers prefer Pulumi, ops prefer Terraform.
4. Component Resources: Custom resources encapsulate complex patterns. Reusable across projects. Type-safe interfaces. Versioned và published as packages. Example: VpcComponent encapsulates VPC + subnets + route tables + NAT Gateway. Higher-level abstractions cho team consumption.
5. Automation API: Programmatically drive Pulumi operations. Embed Pulumi trong larger applications. REST API triggers infrastructure changes. Platform engineering: self-service infrastructure. Custom CLIs cho infrastructure operations. CI/CD integration without CLI dependency.

📖 Chuyện Kể Đêm Khuya

🔄 Chuyện TypeScript IaC Loops

Developer viết Pulumi program với complex TypeScript: async/await cho API calls during infrastructure creation, Promise.all cho parallel resource creation, class hierarchies cho module system. Code elegant nhưng... pulumi preview mất 5 phút vì await external API calls. pulumi up fail khi API rate limited. Debugging = debugging distributed TypeScript application.

Đại Sư simplify: remove runtime API calls (use config instead), flatten class hierarchy, replace Promise.all with simple loop. Preview: 10 giây. Deploy: reliable. 'Infrastructure code should be boring,' Đại Sư advise.

💡 Bài học: Just because you CAN use advanced language features doesn't mean you SHOULD. Infrastructure code should be predictable, simple, debuggable. Avoid runtime API calls during preview/up. Keep abstractions shallow. Real language power for testing and type safety, not for complexity.

📊 Chuyện Pulumi Preview vs Actual

pulumi preview shows: 3 resources to create. pulumi up creates: 3 resources + triggers 5 dependent changes not shown in preview (cloud-side effects). Developer confused: 'Preview said 3 changes!' Reason: some cloud API side effects not predictable by Pulumi — like default security groups, auto-generated resources.

💡 Bài học: Preview is best-effort, not guarantee. Cloud providers create implicit resources. Always verify after apply. Drift detection (pulumi refresh) cho actual vs expected state. Integration tests cho infrastructure validation.

🔀 Chuyện Migration Terraform to Pulumi

Team decide migrate 500 Terraform resources to Pulumi. Estimated: 2 weeks. Actual: 3 months. Issues: state import complex, HCL → TypeScript translation non-trivial, team learning curve, CI/CD pipeline changes, Terraform modules → Pulumi components rewrite.

Alternative approach discovered too late: pulumi-terraform-bridge cho Terraform provider compatibility. coexistence strategy: new infra in Pulumi, existing in Terraform, migrate gradually.

💡 Bài học: Terraform → Pulumi migration is significant effort. Don't migrate everything at once. Coexistence strategy: new projects in Pulumi, existing in Terraform. pulumi import cho existing resources. Budget 3-6 months cho large migrations.

⚠️ Những Sai Lầm Kinh Điển

1. Over-engineering abstractions: 5 levels of class inheritance cho VPC. Infrastructure code needs simplicity, not design patterns exhibition.
2. Runtime dependencies in IaC: Calling external APIs during pulumi up. Fails if API down. Use config values instead of runtime lookups.
3. No preview before up: pulumi up --yes without review. ALWAYS preview. Especially for production stacks.
4. Mixing application and infrastructure code: Application logic in Pulumi program. Keep infrastructure code separate from application code. Clear boundaries.
5. Not using stack references: Hardcode outputs between stacks. StackReference cho cross-stack dependencies. Type-safe, maintainable.
