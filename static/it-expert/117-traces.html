📚 Thân Thế & Lai Lịch

Truy Vết Đại Sư nhớ thời monolith: 1 request → 1 server → 1 log file. Debug: grep error log. Done. Rồi microservices: 1 request → 30 services → 30 log files. Debug: grep 30 log files, correlate timestamps, guess which service failed. "I miss my monolith," chàng ta thở dài.

Distributed tracing thay đổi game: mỗi request có unique trace ID, propagated qua tất cả services. Jaeger/Zipkin visualize entire request journey. "Service A → Service B (50ms) → Service C (2000ms) → Database (1950ms)." Found bottleneck in 10 seconds thay vì 10 hours.

OpenTelemetry unify tracing, metrics, logs thành 1 standard. "One instrumentation to rule them all." Đại Sư adopt OTel và never look back.

📜 Bí Kíp

1. OpenTelemetry — The Standard: Vendor-neutral observability framework. APIs, SDKs, Collector. Auto-instrumentation cho popular frameworks. Traces, metrics, logs — unified. Collector: receive, process, export to any backend. OTLP protocol standard. Adopt OTel = future-proof instrumentation.
2. Trace Structure: Trace = collection of spans. Span = single operation (HTTP request, DB query, function call). Parent-child span relationships. Span attributes: service name, operation, status, custom tags. Span events: logs within span context. Baggage: key-value pairs propagated across services.
3. Sampling Strategies: Head-based: decide at trace start (percentage). Tail-based: decide after trace complete (based on outcome). Always sample: 100% traces (expensive). Rate-based: X traces per second. Best practice: 100% cho errors, sample cho success. Tail-based sampling catches interesting traces (slow, error)
4. Jaeger vs Tempo: Jaeger: self-hosted, Elasticsearch/Cassandra backend, mature. Tempo: Grafana project, object storage backend (cheap), no indexing required. Tempo: search by trace ID, TraceQL cho querying. Choose Tempo nếu already using Grafana stack. Jaeger cho standalone deployment.
5. Best Practices: Instrument all services — gaps in tracing = blind spots. Propagate trace context (W3C TraceContext standard). Add business context as span attributes. Trace database queries (slow queries visible). Set span status cho error detection. Service map từ traces cho dependency visualization.

📖 Chuyện Kể Đêm Khuya

🔍 Chuyện Trace Phát Hiện 30 Microservice Hops

API response time: 3 seconds. Team cannot identify bottleneck. Each service team: 'Our service is fast!' Đại Sư enable distributed tracing. Trace visualization shows: request goes through 30 services. No single service slow — but 30 × 100ms overhead = 3 seconds total. N+1 service call problem: Service A calls Service B 15 times sequentially instead of batch.

Fix: batch API calls (15 sequential → 1 batch). Cache frequently accessed data. Reduce service hops (combine related services). Result: 3 seconds → 400ms.

💡 Bài học: Distributed tracing reveals SYSTEMIC problems invisible to individual service monitoring. N+1 service call pattern common in microservices. Batch API calls, cache, reduce hops. Trace visualization: the most powerful debugging tool cho microservices.

👻 Chuyện Missing Spans

Trace shows: Service A → [gap] → Service C. Service B traces missing. Investigation: Service B not instrumented (new service, team forgot). Service B actually has 2-second latency — invisible without tracing. Users blame Service C (last visible span) while Service B is the real culprit.

Fix: mandatory tracing instrumentation cho ALL services. CI/CD check: verify OTel SDK configured. Service onboarding checklist includes tracing. 100% service coverage cho meaningful traces.

💡 Bài học: Incomplete tracing = misleading conclusions. ALL services must be instrumented. Gaps in traces = blame wrong service. Service onboarding checklist: logging, metrics, tracing. CI/CD enforcement of instrumentation. Context propagation verification tests.

📊 Chuyện Sampling Saves $50K

100% trace sampling cho all services. Trace storage: 500GB/day. Jaeger + Elasticsearch: 5 nodes × 32GB RAM. Monthly cost: $8,000. Review: 99% traces are successful, routine requests. Only 1% are interesting (errors, slow, anomalies).

Implement tail-based sampling: 100% cho errors, 100% cho slow (>1s), 1% cho normal. Trace storage: 15GB/day. Infrastructure reduced to 2 nodes. Monthly cost: $2,500. Savings: $66K/year.

💡 Bài học: 100% sampling is expensive and mostly unnecessary. Tail-based sampling: keep interesting traces, sample routine. 100% error traces, sample success. Cost savings 60-80% with smart sampling. OTel Collector supports tail-based sampling. Quality over quantity cho traces.

⚠️ Những Sai Lầm Kinh Điển

1. No tracing in production: Tracing only in staging. Production = where real problems happen. Enable tracing everywhere with appropriate sampling.
2. Trace ID not in logs: Logs and traces disconnected. Include trace_id in structured logs. Correlate logs ↔ traces ↔ metrics.
3. 100% sampling forever: Expensive, unnecessary. Start with head-based 10%, add tail-based cho errors/slow. Adjust based on needs.
4. No span attributes: Generic spans without context. Add: user_id, order_id, feature_flag — business context cho debugging.
5. Vendor lock-in instrumentation: Vendor-specific SDK. Use OpenTelemetry for vendor-neutral instrumentation. Switch backends without code changes.
