📜 Bí Kíp

Bí Kíp #1: Payment Processing — Luồng tiền không được phép sai

Flow chuẩn: Create Intent → Collect Payment Method → Authorize → Capture → Settlement. Idempotency key cho EVERY payment request — retry safe. Webhook handler phải idempotent (xử lý cùng event nhiều lần cho cùng kết quả). Stripe: PaymentIntent API là best practice. VNPay: hash HMAC-SHA512 verify return URL. Momo: RSA signature cho callback. LUÔN verify payment status server-side — đừng trust client-side redirect. Và log EVERYTHING — payment audit trail là legal requirement.

Bí Kíp #2: Fraud Detection — Khi máy học bắt kẻ gian

Rule-based first: velocity checks (5 transactions trong 1 phút = suspicious), geo-velocity (mua từ TPHCM rồi 5 phút sau từ Hà Nội), amount anomaly (thường mua 200K, đột nhiên 20M). ML models: Random Forest/XGBoost cho tabular data, graph neural networks cho network fraud. Feature engineering quan trọng hơn model choice: time since last transaction, device fingerprint, IP reputation. Balance precision vs recall: false positive = block khách hàng thật = mất revenue. False negative = cho fraud qua = mất tiền. Threshold tuning là nghệ thuật.

Bí Kíp #3: PCI DSS — Bảo vệ dữ liệu thẻ

12 requirements, 4 levels tuỳ transaction volume. Cách dễ nhất: ĐỪNG lưu card data — dùng tokenization (Stripe Elements, VNPay tokenization). Card data đi thẳng từ client đến payment processor, server bạn chỉ nhận token. Nếu buộc phải lưu: encrypt with HSM, segment network, quarterly vulnerability scan, annual penetration test. PCI DSS audit tốn $50K-200K/năm — tokenization miễn phí và giảm scope 90%.

Bí Kíp #4: Reconciliation — Mỗi đêm, mỗi đồng phải khớp

Three-way reconciliation: internal ledger ↔ payment gateway records ↔ bank statements. Mismatch types: timing (transaction today, settlement tomorrow), currency conversion (rate at time of transaction vs settlement), partial captures/refunds. Automated recon tool chạy nightly: match by transaction ID, flag mismatches, generate exception report. Manual review cho exceptions. Tolerance threshold: VND 0 — không có "close enough" trong tiền. Double-entry bookkeeping: mỗi debit có credit tương ứng. Trial balance phải zero.

Bí Kíp #5: Trading Systems — Microsecond matters

HFT (High-Frequency Trading): co-location (server cạnh exchange), kernel bypass (DPDK/RDMA), lock-free data structures, pre-compiled order templates. Latency budget: market data → strategy → order = <10μs. Non-HFT nhưng vẫn cần low latency: FIX protocol cho order execution, WebSocket cho real-time market data. Order matching engine: price-time priority. Risk checks PRE-trade: position limits, margin requirements, circuit breakers. Post-trade: P&L calculation, settlement (T+2 cho equities).

Bí Kíp #6: Regulatory Compliance — Luật chơi phải nắm

Việt Nam: NHNN (Ngân hàng Nhà nước) cấp phép trung gian thanh toán. Thông tư 39/2014 về thanh toán không dùng tiền mặt. KYC (Know Your Customer): CMND/CCCD verification, eKYC bằng NFC + face matching. AML (Anti-Money Laundering): transaction monitoring, STR (Suspicious Transaction Report) gửi NHNN. PSD2 (EU): Strong Customer Authentication (SCA), Open Banking APIs. International: GDPR cho EU customer data, SOX cho financial reporting. Compliance không phải checkbox — vi phạm = revoke license + criminal liability.

📖 Chuyện Kể Đêm Khuya

📖 Chuyện 1: Bug Double-Charge — 500 Khách Hàng Bị Trừ Tiền 2 Lần

Thứ 6 chiều, deploy payment callback handler mới. Test trên staging: OK. Deploy production 17:00. 17:15, alert: "Transaction volume anomaly — gấp đôi bình thường." Check logs: mỗi transaction thành công được xử lý 2 lần. Payment gateway gửi callback, handler process, trả 200 OK. Nhưng response tới gateway bị timeout (network glitch 500ms). Gateway retry callback. Handler process lần 2. Customer bị charge lần 2.

Root cause: handler không check idempotency. Cùng một transaction_id được xử lý 2 lần, tạo 2 records trong database, charge customer 2 lần. 500 transactions affected trong 15 phút trước khi rollback.

Aftermath: 500 refund requests. Manual processing vì refund system cũng không có bulk operation. Customer service làm overtime weekend. PR crisis trên Facebook. Và một lesson đắt giá: idempotency không phải nice-to-have trong payment systems.

💡 Bài học: EVERY payment handler phải idempotent. Check transaction_id trước khi process — nếu đã xử lý, return success mà không process lại. Database unique constraint trên transaction_id. Idempotency key cho mọi API call. Và test failure scenarios (timeout, retry, duplicate) chứ không chỉ happy path.

📖 Chuyện 2: Payment Callback Bị Mất — Tiền Đi Nhưng Không Về

E-commerce platform, khách hàng thanh toán qua VNPay. Flow: redirect đến VNPay → khách hàng nhập OTP → VNPay debit tiền → callback về server để update order status. Vấn đề: callback URL bị blocked bởi firewall mới deploy chiều hôm trước (security team thêm rule chặn external IPs mà không notify dev team).

Kết quả: VNPay debit tiền khách hàng thành công, nhưng callback về server bị firewall block → server không nhận được notification → order status vẫn "pending". Khách hàng: "Tôi đã trả tiền nhưng đơn hàng chưa được xác nhận!" 200 orders trong tình trạng limbo. Tiền đã trừ nhưng hàng chưa giao.

Fix ngắn hạn: mở firewall, chạy script reconciliation so khớp VNPay transactions với internal orders, update status manually. Fix dài hạn: implement polling mechanism — ngoài callback, cron job mỗi 5 phút query VNPay API check status cho pending orders. Belt AND suspenders approach cho payment.

💡 Bài học: ĐỪNG rely 100% vào callback/webhook cho payment confirmation. Implement cả push (webhook) VÀ pull (polling/query). Health check cho callback URL. Alert khi callback success rate giảm. Và LUÔN có reconciliation process — nó là safety net cuối cùng khi mọi thứ khác fail.

📖 Chuyện 3: Latency Trong Trading — 5 Milliseconds = $50,000

Trading firm deploy algo trading system. Strategy: arbitrage giữa 2 sàn — mua rẻ sàn A, bán đắt sàn B. Window: giá chênh lệch thường tồn tại 10-50ms trước khi market correct. System requirement: end-to-end latency < 5ms.

Version 1: Python + REST API. Latency: 200ms. Quá chậm — cơ hội mất trước khi order tới sàn. Version 2: Java + FIX protocol + WebSocket market data. Latency: 15ms. Gần nhưng vẫn miss 60% opportunities. Version 3: C++ + kernel bypass + co-located server + pre-compiled order templates. Latency: 3ms. Hit rate tăng lên 85%.

Nhưng vấn đề mới: competitor cũng co-locate, latency 2ms. Chênh lệch 1ms = họ luôn fill order trước, chúng tôi ăn slippage. Arms race. Cuối cùng nhận ra: competing on latency alone là game thua. Pivot sang strategy-based alpha — better prediction, wider opportunity window, less latency-sensitive.

💡 Bài học: Trong trading, latency = money. Nhưng competing purely on latency là arms race with diminishing returns. Better strategy > faster hardware. Nếu vẫn cần low latency: measure everything (network, serialization, processing), optimize bottleneck trước. Và production monitoring phải measure P99 latency, không chỉ average — one slow request = one missed trade.

⚠️ Những Sai Lầm Kinh Điển

❌ Sai Lầm #1: Dùng floating-point cho tiền

0.1 + 0.2 = 0.30000000000000004 trong IEEE 754. Dùng integer (store cents/đồng) hoặc decimal type. BigDecimal trong Java, decimal.Decimal trong Python, NUMERIC trong PostgreSQL. Sai 1 cent × 1 triệu transactions = $10,000 sai lệch.

❌ Sai Lầm #2: Không có idempotency trong payment API

Network timeout → client retry → double payment. EVERY payment endpoint cần idempotency key. Store key + result, return cached result cho duplicate requests. Stripe có built-in idempotency. Custom system phải tự implement. Không có ngoại lệ.

❌ Sai Lầm #3: Log credit card numbers

PCI DSS violation. Full PAN (Primary Account Number) KHÔNG được log, store, hoặc display. Mask: chỉ show last 4 digits (****1234). Tokenize ngay khi nhận. Audit log files định kỳ để phát hiện accidental PAN logging. Một log file chứa PAN = PCI DSS scope expansion = chi phí audit tăng 10x.

❌ Sai Lầm #4: Trust client-side amount

Client gửi {"amount": 0, "item": "iPhone 15"}. Server trust và charge 0đ. Hacker mua iPhone miễn phí. LUÔN calculate amount server-side từ cart items + prices trong database. Client amount chỉ dùng để verify, không để charge. Server-side validation cho EVERY financial transaction.

❌ Sai Lầm #5: Không có circuit breaker cho payment gateway

Payment gateway down → timeout 30s × 1000 concurrent requests = thread pool exhaustion → entire system down. Circuit breaker: after N failures, stop calling, return error immediately, retry after cooldown. Fallback: redirect to alternative gateway. Timeout: 5s max cho payment API call.

❌ Sai Lầm #6: Deploy payment changes on Friday

Friday deploy → bug → weekend → support team ít người → customer angry → Monday damage control. Payment system changes: deploy Tuesday-Wednesday. Feature flag rollout: 1% → 10% → 50% → 100%. Canary deployment mandatory. Và rollback plan phải tested, không chỉ documented.

❌ Sai Lầm #7: Bỏ qua timezone trong transaction timestamp

Server ở Singapore (UTC+8), database ở US (UTC-5), customer ở Việt Nam (UTC+7). Transaction lúc 23:59 ngày 31/12 theo customer, nhưng 01/01 theo server → reconciliation mismatch, tax reporting sai, statement sai ngày. LUÔN store timestamps dạng UTC, convert sang local timezone khi display.
