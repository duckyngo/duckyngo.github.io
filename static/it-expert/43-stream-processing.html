📜 Bí Kíp

Bí Kíp #1: Event Time vs Processing Time — Hai Đồng Hồ Khác Nhau

Event time: khi sự kiện xảy ra (timestamp trong data). Processing time: khi hệ thống xử lý event. Chúng KHÁC NHAU vì: network delay, queue backlog, retry. Event lúc 10:00:00 có thể được xử lý lúc 10:00:05. Dùng event time cho chính xác, processing time cho đơn giản.

Bí Kíp #2: Windowing — Cắt Dòng Chảy Thành Từng Đoạn

Tumbling window: [0-5min), [5-10min) — non-overlapping, mỗi event thuộc 1 window. Sliding window: [0-5min), [1-6min), [2-7min) — overlapping, event có thể thuộc nhiều windows. Session window: based on activity gap. Global window: tất cả events trong 1 window, trigger bằng custom logic.

Bí Kíp #3: Watermarks — "Tôi Tin Rằng Data Trước Thời Điểm Này Đã Đến Hết"

Watermark = estimated point in event time that all data up to that point has arrived. Flink auto-generates watermarks. Watermark quá aggressive (nhỏ delay) = miss late data. Watermark quá conservative (lớn delay) = kết quả chậm. Tune allowed lateness + side output cho late events.

Bí Kíp #4: Exactly-Once Processing — Khó Nhưng Cần

At-least-once: dễ, nhưng có duplicates. At-most-once: message có thể mất. Exactly-once: mỗi message xử lý đúng 1 lần. Flink: distributed snapshots (Chandy-Lamport). Kafka Streams: transaction + idempotent producer. Exactly-once end-to-end = pipeline tất cả components phải hỗ trợ.

Bí Kíp #5: Apache Flink — Vua Stream Processing

Stateful computation: Flink quản lý state (RocksDB backend cho large state). Savepoints: snapshot state để upgrade, migrate, scale. Event time processing native. SQL interface (Flink SQL) cho đơn giản hóa. Deployment: standalone, YARN, Kubernetes. Nhược: learning curve dốc, operational complexity.

Bí Kíp #6: Kafka Streams — Lightweight Stream Processing

Library (không phải cluster) — nhúng vào Java application. Dùng Kafka topics làm input/output. No separate cluster to manage. Tốt cho: microservice stream processing, simple transformations, exactly-once with Kafka. Nhưng: chỉ đọc từ Kafka, không flexible như Flink cho complex event processing.

Bí Kíp #7: Apache Beam — Write Once, Run Anywhere

Unified programming model: viết pipeline 1 lần, chạy trên Flink, Spark, Dataflow (Google Cloud). Tốt khi muốn vendor flexibility. Nhưng: abstraction layer = không tận dụng được features đặc thù của runtime. Jack of all trades, master of none.

📖 Chuyện Kể Đêm Khuya

📅 Chuyện 1: Late Data Đến Sau 3 Ngày

Hệ thống IoT, sensors gửi data qua mobile network. Bình thường latency 1-5 giây. Flink job aggregate sensor readings theo 1-minute tumbling window, watermark delay 10 giây.

Một ngày, mobile network outage 3 ngày ở vùng rural. Khi network phục hồi, 3 ngày data đổ ập vào. Events có event time 3 ngày trước nhưng arrive bây giờ. Watermark đã pass qua rất lâu → tất cả đều là "late data" → bị drop. 3 ngày data = mất.

Lão redesign: tách thành 2 streams. Main stream với reasonable watermark (30 giây). Late data stream với side output, ghi vào storage riêng. Batch job reconcile late data hàng ngày. Hybrid batch + stream approach.

💡 Bài học: Late data là thực tế, đặc biệt với mobile/IoT. Design cho worst case: side output cho late events, batch reconciliation. "Lambda architecture" không đẹp nhưng practical: real-time stream + batch correction.
💥 Chuyện 2: Backpressure Crash — Khi Downstream Không Kịp

Flink job đọc từ Kafka (100K events/s), transform, ghi vào Elasticsearch. Bình thường OK. Một ngày Elasticsearch cluster slow (disk issue). Flink ghi chậm → buffer đầy → backpressure lên Kafka consumer → consumer lag tăng → Kafka broker retention hết → DATA LOSS ở Kafka.

Chain reaction: Elasticsearch chậm → Flink backpressure → Kafka consumer lag → Kafka log retention expire → messages bị xóa trước khi consumed. Lão fix: tăng Kafka retention, thêm buffer topic trung gian, set Flink checkpoint interval hợp lý, alert trên consumer lag.

💡 Bài học: Backpressure propagation có thể gây data loss ở upstream. Monitor consumer lag religiously. Kafka retention phải > worst case recovery time. Buffer topics cho decoupling. Alert khi lag > threshold.
🔄 Chuyện 3: Exactly-Once — Lời Hứa Và Thực Tế

Team cần exactly-once cho payment processing stream. Flink + Kafka exactly-once = OK trong lý thuyết. Nhưng sink là external REST API (gọi bank). REST API không idempotent. Flink checkpoint → crash → restart → replay → gọi API lần 2 → chuyển tiền 2 lần!

Exactly-once trong Flink = exactly-once WITHIN Flink. External systems phải handle idempotency riêng. Lão thêm: unique transaction ID per event, check duplicate trước khi gọi API, idempotency key gửi kèm API call. "Exactly-once là end-to-end problem, không phải framework feature."

💡 Bài học: Exactly-once processing guarantee chỉ cover internal state. External side effects (API calls, emails, notifications) PHẢI được design idempotent. Unique ID + dedup check = insurance policy bắt buộc.

⚠️ Những Sai Lầm Kinh Điển

❌ Stream everything khi batch là đủ

Daily summary report không cần real-time. Monthly invoice không cần real-time. Stream processing phức tạp và tốn kém hơn batch. Dùng khi latency matters: fraud detection, alerting, real-time dashboards.

❌ Ignore backpressure

Consumer chậm hơn producer → buffer overflow → data loss hoặc OOM. Monitor consumer lag, implement backpressure handling, rate limiting, buffering.

❌ Stateful processing without checkpoint

Flink job không enable checkpointing, crash → mất state → kết quả sai. Luôn enable checkpointing, configure checkpoint interval (1-5 phút), save to durable storage (S3, HDFS).

❌ Processing time khi cần event time

"Đơn giản hơn" → nhưng kết quả khác nhau mỗi lần replay, window boundaries không deterministic. Dùng event time khi cần correctness.

❌ Unbounded state

State tăng mãi không clear: session map không có timeout, counter không có TTL. Flink state backend OOM. Set state TTL, clean up expired state.
