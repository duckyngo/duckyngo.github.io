Truyền Thuyết API Chiến Thần

🔴 Kẻ sinh ra từ những dòng JSON

Ngày xưa, ngày xưa... có một cậu bé sinh ra trong một thế giới mà mọi thứ đều được gửi qua HTTP. Cậu bé ấy chẳng biết đi, chỉ biết GET. Chẳng biết viết, chỉ biết POST. Muốn xóa gì, cậu dùng DELETE. Muốn sửa, cậu dùng PUT. Và khi lớn lên, cậu nhận ra mình có một sức mạnh phi thường — khả năng intercept mọi request và response trước khi chúng chạm tới đích.

Từ đó, người ta gọi cậu là API Chiến Thần — người nắm giữ 6 viên ngọc vô cực:

  • RESTful Principles — Ngọc Không Trạng Thái (Stateless)
  • HTTP Methods — Ngọc Ngũ Hành (GET, POST, PUT, PATCH, DELETE)
  • Interceptors — Ngọc Thời Gian (bắt request/response giữa đường)
  • Error Handling — Ngọc Không Gian (xử lý mọi status code)
  • Retry Logic — Ngọc Bất Tử (thất bại thì thử lại!)
  • Cancellation Tokens — Ngọc Hủy Diệt (hủy request khi không cần nữa)

💡 Chuyên môn sâu

API Chiến Thần thông thạo Axios 1.13 như một đầu bếp thông thạo con dao. Request/Response transformation? Dễ như ăn cháo. Từ việc thiết lập baseURL, cấu hình timeout, đến việc gắn Authorization: Bearer token tự động qua interceptors — tất cả đều được xử lý mượt mà như lụa. Và đừng quên: khi server trả về lỗi 401, Chiến Thần sẽ lặng lẽ refresh token và gửi lại request như chưa có gì xảy ra. Đẳng cấp là vĩnh cửu.

📚

Bí Kíp Axios Thất Truyền

01

Tạo Axios Instance riêng

Đừng bao giờ dùng axios global. Hãy tạo instance riêng với axios.create() để mỗi module có baseURL, headers, timeout riêng. Giống như mỗi võ sĩ cần thanh kiếm riêng vậy.

02

Interceptors là bảo bối

Request interceptor để gắn token. Response interceptor để bắt lỗi toàn cục. Đây là nơi tốt nhất để xử lý 401 và auto-refresh token. Nhớ: interceptor chạy theo thứ tự ngược — cái thêm sau chạy trước!

03

Retry với Exponential Backoff

Khi gặp lỗi 5xx hoặc network error, đừng bỏ cuộc. Hãy retry với delay tăng dần: 1s, 2s, 4s... Dùng axios-retry hoặc tự viết. Server cũng cần thời gian nghỉ ngơi chứ, đừng spam nó!

04

AbortController thay CancelToken

CancelToken đã bị deprecated từ Axios 0.22. Hãy dùng AbortController chuẩn của trình duyệt. Đặc biệt hữu ích khi user chuyển trang mà request cũ vẫn chưa xong — hủy nó đi cho sạch!

05

Transform data tại interceptor

Backend trả snake_case? Frontend cần camelCase? Đặt logic transform ở response interceptor. Một chỗ, một lần, không bao giờ lặp lại ở từng component. DRY là đạo!

06

Xử lý error tập trung

Đừng try-catch ở mỗi nơi gọi API. Hãy tạo một error handler chung trong response interceptor: hiện toast cho 4xx, redirect cho 401, log cho 5xx. Rõ ràng, gọn gàng, chuyên nghiệp.

Chuyện Đời Của Ông Nội API

👻

Cơn Ác Mộng Mang Tên CORS

Ngày xửa, ông nội còn trẻ, mới vào nghề code. Ông viết cái API đẹp như mơ, test bằng Postman thì mượt mà như lụa. Tự tin deploy lên, mở trình duyệt ra gọi... “Access-Control-Allow-Origin” đỏ lòm. Ông tưởng server hỏng, restart 15 lần, đổi 3 cái port, thậm chí còn tắt cả firewall.

Cuối cùng, sau 6 tiếng vật lộn, một thằng dev senior đi ngang nói: “Anh ơi, anh thiếu cái header Access-Control-Allow-Origin ở backend.” Ông nội suốt muốn bỏ nghề. Nhưng không, ông ở lại và trở thành... Chiến Thần. Từ đó, ông luôn dặn cháu: “CORS không phải bug frontend đâu con, nó là backend quên cấu hình!”

💡 Bài học: CORS là vấn đề của server, không phải client. Postman không có CORS vì nó không phải trình duyệt!
🔀

Vòng Lặp 401 Vô Tận — Khi Refresh Token Cũng Hết Hạn

Một ngày nọ, ông nội viết interceptor tự động refresh token khi gặp 401. Logic chạy ngon lắm: gặp 401 → gọi refresh → lấy token mới → gửi lại request. Tuyệt vời!

Nhưng ông quên một điều: cái API refresh token cũng có thể trả 401. Và interceptor lại bắt 401 đó, lại gọi refresh, lại 401, lại gọi... Tab trình duyệt crash sau 30 giây với 10,000 request. PM đứng đằng sau: “Sao server nó bị DDoS vậy anh?” Ông nội: “À... do anh tự DDoS chính mình.”

💡 Bài học: Luôn exclude endpoint refresh token khỏi interceptor, hoặc đặt giới hạn retry. Đừng để con rắn cắn đuôi mình!
😵

Quên Await — Data Đâu Rồi?!

Có lần, ông nội có một thằng cháu intern mới vào. Nó gọi API xong rồi console.log(data) ra được... Promise {<pending>}. Nó hỏi: “Ông ơi, sao API trả về cái Promise mà không trả data vậy ông?”

Ông nội nhìn code, thấy thiếu await trước axios.get(). Ông chỉ nói một câu: “Con ơi, gọi API mà không await, giống như gọi điện rồi cúp máy trước khi người ta trả lời vậy. Ai làm thế bao giờ?” Thằng cháu từ đó không bao giờ quên await nữa.

💡 Bài học: axios.get/post/put/delete trả về Promise. Luôn await hoặc .then(). Đừng để Promise nó trôi dạt!
🤯

Dùng Sai HTTP Method — GET Với Body

Một chàng trai trẻ trong team, tự tin như một con công, quyết định gửi GET request với body chứa 15 filters phức tạp. “GET nó cũng gửi body được mà, em test Postman OK rồi!”

Deploy lên production, load balancer nó strip hết body của GET request. API trả về toàn bộ database vì không có filter. 500GB data bay thẳng vào trình duyệt. Ông nội nghe tin, chỉ lắc đầu: “GET là để lấy, không phải để gửi. Muốn gửi data thì dùng POST. Đây là quy luật của trời đất, không ai thay đổi được!”

💡 Bài học: GET không nên có body (dù spec không cấm). Nếu cần gửi data phức tạp, dùng POST hoặc query params.

Lỗi Sai Kinh Điển Khi Làm REST API

💀

1. Không xử lý error response

Gọi API xong thì chỉ xử lý .then(), bỏ mặc .catch() như bỏ rơi người yêu cũ. Khi server trả 500, app im lặng chết như thiến định. User tưởng app hỏng, PM tưởng dev lười, dev thì... đang ngủ.

💣

2. Hardcode URL trong component

axios.get('http://localhost:3000/api/users') — Xin chúc mừng, bạn vừa tạo ra một quả bom hẹn giờ! Deploy lên production là nổ. Hãy dùng baseURL và environment variables như người văn minh.

🎭

3. Không set timeout

Không có timeout, request sẽ chờ mãi mãi như chờ người yêu cũ quay lại. Server đi ngủ 30 phút? User cũng chờ 30 phút. Set timeout 10-15 giây rồi hiện thông báo lỗi cho đàng hoàng!

🐦

4. Gửi token trong URL

/api/users?token=abc123 — Chúc mừng, token của bạn đã xuất hiện trong server log, browser history, và có thể cả Google nữa. Token thuộc về Authorization header, không phải query string!

⚡

5. Gọi API trong vòng lặp không kiểm soát

users.forEach(u => axios.delete(`/api/user/${u.id}`)) — 1000 users = 1000 request cùng lúc. Server: “Ai đang DDoS tôi vậy?” Hãy dùng Promise.all() với batch, hoặc gửi một request với danh sách IDs.

🙈

6. Đặt tên endpoint không RESTful

POST /api/getUserByIdAndDeleteIfInactive — Đây không phải REST, đây là tiểu thuyết. REST dùng danh từ (nouns), không phải động từ: GET /api/users/:id và DELETE /api/users/:id.

Vai Trò Trong Dự Án PBX

🔌 Axios Là Cầu Nối Giữa Vue.js và .NET Backend

Trong dự án PBX, mọi tương tác giữa giao diện người dùng (Vue.js + DevExtreme) và hệ thống backend (.NET) đều đi qua Axios. API Chiến Thần chịu trách nhiệm đảm bảo mọi request đều nhanh, đúng, và an toàn.

  • Quản lý phiên đăng nhập (Auth): Interceptor tự động gắn Bearer token vào mỗi request. Khi token hết hạn (401), tự động refresh và retry — người dùng không hề biết.
  • Gọi API quản lý cuộc gọi: Lấy danh sách extension, lịch sử cuộc gọi, trạng thái tổng đài — tất cả qua REST API với Axios instance chuyên dụng.
  • Xử lý lỗi tập trung: Mọi lỗi API (4xx, 5xx) đều được bắt ở response interceptor, hiện thông báo DevExtreme toast phù hợp. Không để user thấy trang trắng.
  • Cancel request khi chuyển trang: Dùng AbortController kết hợp với Vue Router navigation guard để hủy request cũ khi user chuyển route. Tiết kiệm tài nguyên, tránh memory leak.
  • Retry cho API quan trọng: Các API liên quan đến đồng bộ trạng thái điện thoại IP được cấu hình retry 3 lần với exponential backoff. Mất kết nối 1 giây không có nghĩa là mất cuộc gọi!
  • API Gateway Proxy: Axios gọi qua Node.js gateway (Agent #13) để đến backend .NET. BaseURL được cấu hình theo environment: dev, staging, production — không hardcode bao giờ.
  • Request/Response Logging: Mọi API call đều được log ở interceptor (chỉ ở dev mode) để debug nhanh. Production thì tắt để không lộ thông tin nhạy cảm.
