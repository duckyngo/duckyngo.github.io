🔮 Bí Kíp Trấn Phái

Setup Syntax là Chân Ái
Dùng defineStore('id', () => { ... }) thay vì options syntax. Tại sao? Vì nó giống hệt setup() trong component — dùng ref(), computed(), watch() thoải mái. TypeScript inference ngon lành, không cần khai báo type riêng. Ai đã quen Composition API thì chuyển sang setup store mượt như lụa.
storeToRefs() — Đừng Quên Anh Ấy
Khi destructure store, state và getters mất reactivity nếu không dùng storeToRefs(). Actions thì destructure bình thường. Đây là cái bẫy kinh điển mà 90% dev mới đều dính. Ta đã thấy cả team ngồi debug 3 tiếng chỉ vì quên cái này.
Store Composition — Store Gọi Store
Pinia cho phép store A import và sử dụng store B bên trong actions/getters. Nhưng cẩn thận! Chỉ gọi useOtherStore() bên trong action hoặc getter, không phải ở top-level. Và tuyệt đối tránh circular dependency — A gọi B gọi lại A thì hệ thống sẽ khóc.
Plugin Persistence — Nhớ Lâu Như Ông Ngoại
Dùng pinia-plugin-persistedstate để tự động sync state với localStorage/sessionStorage. Cấu hình chọn lọc field nào cần persist, serializer tùy chỉnh, và storage strategy. Đặc biệt hữu ích cho PBX: lưu user session, cài đặt giao diện, lịch sử cuộc gọi gần đây.
$patch() — Cập Nhật Hàng Loạt Như Gió
Thay vì gán từng property một, dùng store.$patch({ ... }) hoặc store.$patch((state) => { ... }) để cập nhật nhiều field cùng lúc. Pinia sẽ gom thành một lần update duy nhất, giảm re-render, devtools chỉ ghi nhận 1 mutation thay vì 10.
$subscribe() — Theo Dõi Mọi Biến Động
store.$subscribe((mutation, state) => { ... }) cho phép lắng nghe mọi thay đổi state. Kết hợp với { detached: true } để subscription tồn tại kể cả khi component unmount. Hoàn hảo cho logging, analytics, hoặc sync state giữa các tab.

Cuộc Đại Di Cư Từ Vuex Sang Pinia
Chuyện xảy ra vào mùa đông năm ngoái. Team có 47 cái Vuex modules, mỗi module có mutations, actions, getters, state — tổng cộng hơn 200 files. Sếp bảo: "Migrate sang Pinia đi, deadline tuần sau."

Anh lead nghe xong suýt ngất. Nhưng rồi khi bắt tay vào làm, mọi người mới nhận ra: mỗi Vuex module chuyển sang Pinia chỉ mất 15 phút. Xóa hết mutations (vì Pinia cho phép modify state trực tiếp), gộp actions và getters vào defineStore(), bỏ hết commit() và dispatch(). Code giảm 40%. TypeScript tự suy luận type. Devtools vẫn hoạt động ngon lành.

Cuối cùng, team xong trước deadline 3 ngày. Anh lead đãi cả team trà sữa. Từ đó không ai nhắc đến Vuex nữa, như một ký ức đau thương đã được chữa lành.
Bài học: Vuex mutations là thứ không cần thiết nhất trong lịch sử state management. Pinia đã chứng minh điều đó.
🐛
Vụ Án State Mutation Bí Ẩn Lúc 3 Giờ Sáng
Có một đêm, hệ thống PBX production báo lỗi: trạng thái cuộc gọi hiển thị "Đang kết nối" nhưng thực tế đã kết thúc từ 5 phút trước. User nhìn thấy mình đang gọi... nhưng bên kia đã cúp máy rồi.

Dev trực đêm mở Devtools lên, time-travel lại xem state thay đổi thế nào. Phát hiện ra: có 2 component cùng gọi callStore.updateStatus() nhưng với tham số ngược nhau — component A set "connected", component B set "ended", rồi component A lại set "connected"... Cứ thế đuổi nhau như Tom & Jerry.

Giải pháp? Thêm $subscribe() để log mọi mutation kèm stack trace, phát hiện component nào gọi sai. Rồi refactor lại để chỉ có một nguồn truth duy nhất cập nhật call status từ WebSocket event.
Bài học: State chỉ nên có một nguồn cập nhật. Nhiều nguồn = nhiều drama. Cuộc đời và code đều thế.
🔄
Vòng Xoáy Dependency — Store Gọi Store Gọi Store
Một ngày đẹp trời, dev junior viết useAuthStore() bên trong useCallStore(), rồi lại viết useCallStore() bên trong useAuthStore(). Lý do? Auth cần biết user có đang trong cuộc gọi không, và Call cần biết user đã đăng nhập chưa. Nghe thì hợp lý...

Cho đến khi app chạy lên và... treo. Console hiện lỗi maximum call stack exceeded. Hai store cứ gọi nhau mãi mãi như hai người yêu nhau nhưng không ai chịu nhắn trước.

Ông ngoại (tức là ta) bèn chỉ: "Con ơi, hãy tạo một useSharedStore() làm trung gian. Hoặc tốt hơn, chỉ gọi store kia bên trong action, không phải ở top-level. Lazy evaluation, hiểu không?"

Dev junior nghe xong, sửa trong 10 phút, chạy ngon lành. Từ đó cậu ta khắc lên bàn dòng chữ: "Không bao giờ import store ở top-level."
Bài học: Circular dependency là kẻ thù không đội trời chung của mọi kiến trúc. Hãy dùng lazy import hoặc mediator pattern.

💀 Sai Lầm Muôn Thuở

  • Destructure store mà quên storeToRefs()
    const { count } = useCounterStore() — count giờ chỉ là một số chết cứng, không reactive. UI sẽ không bao giờ cập nhật. Dev ngồi thắc mắc tại sao click nút mà số không tăng, rồi đổ thừa Vue bị bug.
    Sửa: const { count } = storeToRefs(useCounterStore())
  • Gọi useStore() ngoài setup() hoặc component
    Pinia store cần Pinia instance đã được install. Gọi useStore() trong file utils thường hoặc ở top-level module sẽ nhận lỗi "getActivePinia was called with no active Pinia". Đau đớn lắm.
    Sửa: Truyền pinia instance hoặc gọi trong setup/action context
  • Đặt store ID trùng nhau
    Hai store cùng ID 'main'? Chúc mừng, bạn vừa tạo ra bug âm thầm nhất lịch sử. Store sau sẽ ghi đè store trước, dữ liệu lẫn lộn, và bạn sẽ mất cả ngày tìm nguyên nhân.
    Sửa: Dùng naming convention rõ ràng — 'auth', 'call', 'ui', 'org'
  • Persist toàn bộ store vào localStorage
    Store có 50 fields, persist hết vào localStorage. User mở app lên, state cũ từ tuần trước load lên, conflict với API data mới. Kết quả: UI hiện thông tin sai, user confused, sếp gọi điện hỏi thăm sức khỏe.
    Sửa: Chỉ persist những gì cần thiết (settings, preferences), không persist data volatile
  • Quên $reset() khi user logout
    User A logout, user B login — nhưng store vẫn giữ data của user A. User B thấy cuộc gọi, contacts, settings của người khác. Đây không chỉ là bug, đây là bảo mật incident.
    Sửa: Gọi store.$reset() cho TẤT CẢ stores khi logout. Hoặc viết plugin reset toàn bộ.
  • Dùng store cho mọi thứ, kể cả local UI state
    "Modal đang mở hay đóng? Bỏ vào store!" — KHÔNG. Không phải mọi state đều cần global store. Modal state, form state, loading state riêng của một component thì dùng local ref() là đủ. Store chỉ dành cho shared state.
    Sửa: Hỏi "có component nào khác cần biết state này không?" — Không? Dùng ref().

📞 Pinia Trong Dự Án PBX

Hệ thống PBX tổng đài sử dụng Pinia 2.3 làm trung tâm quản lý toàn bộ state. Mỗi khía cạnh của hệ thống đều có store riêng, được thiết kế theo nguyên tắc single responsibility và composability. Dưới đây là kiến trúc store cốt lõi:

📞
useCallStore
Quản lý toàn bộ trạng thái cuộc gọi: incoming, outgoing, active, on-hold, transferring. Real-time sync qua WebSocket. State transition có validation chặt chẽ — không cho phép nhảy từ "idle" sang "on-hold" mà bỏ qua "active".
👤
useAuthStore
User session, token management, role-based permissions. Persist token vào sessionStorage với encryption. Auto-refresh token trước khi hết hạn. Khi logout, trigger $reset() cascade cho toàn bộ stores.
🏢
useOrgStore
Dữ liệu tổ chức: departments, extensions, ring groups, call queues, IVR menus. Cache thông minh với TTL — chỉ fetch lại khi data đã cũ hoặc khi nhận được invalidation event từ server.
🎨
useUIStore
Theme (dark/light), sidebar state, notification preferences, language setting. Persist vào localStorage để user không phải setup lại mỗi lần mở app. Getters computed để các component lấy đúng giá trị cần thiết.
📊
useRealtimeStore
Dashboard realtime: số cuộc gọi đang diễn ra, agent availability, queue wait times, SLA metrics. Subscribe qua WebSocket, $patch() bulk updates mỗi giây để tối ưu rendering performance.
📋
useHistoryStore
Lịch sử cuộc gọi, recordings, CDR (Call Detail Records). Pagination với infinite scroll, filter theo ngày/agent/direction. Local cache 50 records gần nhất để UI hiển thị nhanh khi chuyển tab.
// Ví dụ: useCallStore trong PBX project
export const useCallStore = defineStore('call', () => {
  const activeCall = ref<Call | null>(null)
  const callQueue = ref<Call[]>([])
  const isOnCall = computed(() => activeCall.value !== null)

  async function answerCall(callId: string) {
    const auth = useAuthStore() // lazy — trong action!
    await api.answer(callId, auth.token)
    activeCall.value = callQueue.value.find(c => c.id === callId)
  }

  return { activeCall, callQueue, isOnCall, answerCall }
})
