Thế Giới Container

Từ chỉ một lệnh chroot trên Unix đến hệ sinh thái Kubernetes thống trị cloud-native — hành trình 45 năm của công nghệ container hóa. Mọi thứ bạn cần biết, những hiểu nhầm phổ biến, và góc nhìn thực tế từ người vận hành homelab.

⌄

Container là gì thực sự?

Container là một đơn vị đóng gói phần mềm chuẩn hóa, bao gồm code, runtime, thư viện hệ thống và các thiết lập cần thiết để ứng dụng chạy được mọi nơi. Nếu bạn đã từng nghe câu "Trên máy em chạy được mà!" — thì container chính là lời giải cho vấn đề đó.

Hãy nghĩ đơn giản: VM là một căn hộ nguyên căn với hệ thống điện nước riêng, mái riêng, nền móng riêng. Còn Container là một phòng trong chung cư — dùng chung nền móng (kernel), chung hệ thống điện (OS), nhưng bên trong mỗi phòng được cách ly hoàn toàn, trang trí tùy ý.

Điểm mấu chốt

Container không phải là VM nhẹ. Chúng là hai khái niệm khác nhau về bản chất. VM ảo hóa phần cứng (hardware virtualization), container ảo hóa hệ điều hành (OS-level virtualization). VM có kernel riêng, container dùng chung kernel với máy host. Đây là sự khác biệt cơ bản nhất và cũng là nguồn gốc của mọi ưu-nhược điểm.

Dòng Thời Gian Container

45 năm tiến hóa — từ một system call đơn giản đến hệ sinh thái triệu đô

1979

chroot — Vụ Nổ Lớn

Unix V7 giới thiệu chroot() — cho phép thay đổi thư mục gốc của một process. Đây là hình thức "cách ly" sơ khai nhất: process nhìn thấy một filesystem giả lập, không thể thoát ra ngoài. Người ta gọi nó là "chroot jail" — nhà tù đầu tiên của thế giới container.

2000

FreeBSD Jails — Nhà Tù Thật Sự

FreeBSD 4.0 đưa Jails lên một tầm mới: không chỉ cách ly filesystem mà còn cách ly network, process tree, và users. Mỗi jail có IP riêng, hostname riêng. Đây là bước tiến quan trọng nhất trước khi Linux có namespaces.

2002-2008

Linux Namespaces & cgroups

Linux 2.4.19 (2002) giới thiệu mount namespace đầu tiên. Google phát triển cgroups (2006) để giới hạn tài nguyên. Đến 2008, cgroups được merge vào kernel. Hai công nghệ này — namespaces (cách ly) và cgroups (giới hạn) — chính là nền tảng của mọi container engine hiện đại.

2008

LXC — Container Đầu Tiên Trên Linux

LXC (Linux Containers) kết hợp namespaces + cgroups + chroot thành một giải pháp hoàn chỉnh. Lần đầu tiên người dùng có thể chạy một "máy ảo nhẹ" mà không cần hypervisor. Tuy nhiên, LXC còn khá thô sơ và khó dùng cho người mới.

2013

Docker — Cuộc Cách Mạng

Solomon Hykes giới thiệu Docker tại PyCon 2013 với một live demo 5 phút làm rung chuyển cả ngành. Docker không phát minh ra container — nó đại chúng hóa container bằng Dockerfile, image layer, Docker Hub, và CLI trực quan. Ban đầu dùng LXC bên dưới, sau chuyển sang libcontainer (nay là runc).

2014-2015

Kubernetes — Người Điều Phiối

Google mở mã nguồn Borg con — dự án Kubernetes (K8s). Phiên bản 1.0 ra mắt 07/2015. CNCF được thành lập. K8s nhanh chóng đánh bại Docker Swarm và Mesos để trở thành "hệ điều hành của cloud". Từ đây, container không chỉ là công cụ để đóng gói mà là nền tảng để vận hành quy mô lớn.

2015-nay

OCI, CRI & Thời Đại Hiện Đại

Open Container Initiative (OCI) chuẩn hóa image format và runtime spec. Kubernetes thêm CRI (Container Runtime Interface) cho phép dùng bất kỳ runtime nào. containerd và CRI-O nổi lên. Docker bị Kubernetes "loại" khỏi vị trí runtime mặc định (v1.24). Thời đại "Docker là tất cả" đã qua — giờ là hệ sinh thái mở, tiêu chuẩn hóa.

Linux Kernel

Chia sẻ với host

Linux Namespaces

Namespaces là cơ chế cách ly của container. Mỗi namespace tạo ra một "góc nhìn" riêng cho process:

PID — Process ID NET — Network MNT — Mount points UTS — Hostname USER — UID/GID IPC — Interprocess CGROUP — cgroup view TIME — Clock (Linux 5.6+)

Ví dụ: PID namespace làm cho process trong container thấy PID=1 của mình, dù trên host nó có thể là PID=45821. NET namespace cho mỗi container có network stack riêng với địa chỉ IP riêng.

cgroups v2 (Control Groups)

Nếu namespaces là "tường" thì cgroups là "hóa đơn điện nước" — giới hạn và giám sát tài nguyên mỗi container được dùng:

CPU
Giới hạn cores
RAM
Memory limit
IO
Disk bandwidth
PIDs
Số process

cgroups v2 (unified hierarchy) thay thế cgroups v1 từ kernel 5.x. Thay vì mỗi resource là một hierarchy riêng (v1), v2 gộp tất cả vào một cây duy nhất — đơn giản hơn, mạnh hơn, và hỗ trợ pressure stall information (PSI).

Overlay Filesystem

Đây là "phép thuật" giúp container image nhỏ và khởi động nhanh. OverlayFS xếp chồng nhiều layer lên nhau:

███ Container Layer (read-write)
███ App code layer (read-only)
███ Dependencies layer (read-only)
███ Base image layer (read-only)

Khi container ghi file, chỉ layer trên cùng bị thay đổi (copy-on-write). 100 container cùng image chỉ cần 1 bản base layer trên disk — tiết kiệm kẻ.

OCI Image & Runtime Spec

Open Container Initiative (OCI) định nghĩa 3 tiêu chuẩn mở:

Image Spec — định dạng đóng gói Runtime Spec — cách chạy container Distribution Spec — cách push/pull

Nhờ OCI, bạn có thể build image bằng Docker, chạy bằng Podman, deploy lên Kubernetes dùng containerd. Tất cả đều "nói chung một ngôn ngữ". Đây là lý do tại sao hệ sinh thái container phát triển mạnh — không bị vendor lock-in.

Tại sao cần hiểu những thứ này?

Nhiều người dùng Docker mà không hiểu namespaces hay cgroups — giống lái xe không biết động cơ hoạt động thế nào. Bình thường thì không sao, nhưng khi gặp sự cố (container bị OOM-kill, network không thông, quyền file sai) — thì hiểu nền tảng sẽ cứu bạn.

Image & Registry

Từ viết Dockerfile đúng cách đến vận hành private registry

Dockerfile Anti-Pattern vs Best Practice

❌ Sai (Bloated Image)

FROM ubuntu:latest
RUN apt-get update
RUN apt-get install -y python3
RUN apt-get install -y pip
RUN pip install flask
COPY . /app
CMD ["python3", "/app/app.py"]
# Kết quả: 800MB+ image
# Mỗi RUN = 1 layer dư thừa
# Không có .dockerignore
# Dùng :latest = không pin version

✅ Đúng (Multi-stage Build)

# Stage 1: Build
FROM python:3.12-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir \
    --target=/deps -r requirements.txt

# Stage 2: Runtime
FROM python:3.12-slim
COPY --from=builder /deps /usr/local
COPY . /app
USER 1000
CMD ["python3", "/app/app.py"]
# Kết quả: ~120MB image

Pin Version, Không Dùng :latest

FROM node:20.11-alpine thay vì FROM node:latest. Tag :latest thay đổi bất cứ lúc nào — build hôm nay khác build ngày mai. Trong production, hãy dùng digest: FROM node@sha256:abc123...

Multi-stage Build

Tách build environment và runtime environment. Build bằng image có compiler, rồi COPY artifact sang image nhỏ. Go binary từ 1GB+ xuống còn 10-20MB với FROM scratch.

.dockerignore Là Bắt Buộc

Giống .gitignore nhưng cho Docker build context. Không có nó, Docker gửi TOÀN BỘ thư mục vào daemon — bao gồm node_modules/, .git/, và cả file nhạy cảm nếu bạn không cẩn thận.

USER Không Phải Root

Mặc định container chạy bằng root. Thêm USER 1000 hoặc tạo user riêng. Đây là lớp bảo mật đơn giản nhưng rất hiệu quả — nếu container bị tấn công, attacker không có quyền root.

Hiểu Nhầm Phổ Biến

Những điều mọi người tưởng đúng nhưng thực ra sai bét

Hiểu nhầm

"Container bảo mật như VM"

Nhiều người nghĩ container cách ly tốt như VM. Thực tế, container chia sẻ kernel với host. Một kernel exploit có thể thoát khỏi container. Đó là lý do có gVisor (Google) và Kata Containers — chạy container trong micro-VM để có cả hai: nhẹ và an toàn.

Sự thật

Container cách ly ở mức process, VM cách ly ở mức hardware. Muốn chạy workload không tin tưởng? Dùng VM hoặc sandbox runtime.

Hiểu nhầm

"Docker = Container"

Docker phổ biến đến mức người ta dùng "Docker" và "container" thay thế nhau. Nhưng Docker chỉ là MỘT công cụ trong hệ sinh thái container. Bạn có thể chạy container mà không cần Docker: Podman, containerd, LXC, nerdctl...

Sự thật

Docker là platform. Container là công nghệ. Kubernetes đã loại Docker ra khỏi runtime mặc định từ v1.24 và thế giới vẫn chạy tốt.

Hiểu nhầm

"Container là stateless, không lưu được data"

Một hiểu nhầm rất phổ biến. Container CAN BE stateful với volumes. Docker volume, bind mount, và cả hệ thống CSI driver trên K8s cho phép gắn persistent storage. PostgreSQL, MySQL, Redis đều chạy trong container production được.

Sự thật

Container layer là ephemeral, nhưng volume là persistent. Biết dùng volume đúng cách là kỹ năng cơ bản.

Hiểu nhầm

"Cứ bỏ vào container là tự động nhanh hơn"

Container không làm code chạy nhanh hơn. Một ứng dụng chạy trên bare metal thường nhanh hơn trong container (dù chênh lệch rất nhỏ với Linux). Container giúp bạn deploy nhanh hơn, scale nhanh hơn, và quản lý dễ hơn — không phải chạy nhanh hơn.

Sự thật

Container tối ưu hóa workflow vận hành, không phải hiệu suất thực thi. Overhead của container trên Linux gần như bằng 0, nhưng trên macOS/Windows có overhead của VM ẩn bên dưới.

Hiểu nhầm

"Mọi thứ đều nên container hóa"

Không phải mọi workload đều phù hợp với container. Kernel module, GPU driver, NFS server, real-time system, hoặc ứng dụng cần truy cập phần cứng trực tiếp — những thứ này thường chạy trên VM hoặc bare metal tốt hơn.

Sự thật

Container là công cụ, không phải tôn giáo. Dùng đúng chỗ: microservices, web app, CI/CD, dev environment. Không ép mọi thứ vào container.

Hiểu nhầm

"Alpine luôn là base image tốt nhất"

Alpine nhỏ (5MB) nhưng dùng musl libc thay vì glibc. Điều này gây ra bug khó debug với nhiều ngôn ngữ (Python, Node, Go CGO). DNS resolution khác biệt, locale khác biệt, và performance khác biệt so với glibc.

Sự thật

Dùng -slim variant (debian-slim, ubuntu) cho hầu hết trường hợp. Alpine phù hợp cho Go static binary hoặc khi bạn hiểu rõ musl. distroless (Google) còn tốt hơn cho production.

Hiểu nhầm

"Không cần backup container vì có image"

Image chỉ chứa template. Data thực sự nằm trong volume, database bên trong, và config được mount vào. Xóa container mà quên backup volume = mất data vĩnh viễn. Đây là sai lầm rất thường gặp ở người mới.

Sự thật

Luôn backup volume data, lưu docker-compose.yml trong git, và có kế hoạch disaster recovery. Image có thể pull lại, data thì không.

Hiểu nhầm

"Container không cần monitoring"

"Nó tự restart mà, cần gì monitor." Sai hoàn toàn. Container có thể restart loop (CrashLoopBackOff), leak memory, tốn disk với log, hoặc chạy mà không phục vụ gì. Không có monitoring = mù quáng.

Sự thật

Dùng Prometheus + Grafana cho metrics, Loki cho logs, cAdvisor cho container resource usage. Thiết lập alerting cho CPU, RAM, restart count.
