Lịch sử Ảo hoá

Từ mainframe IBM đến Proxmox - hành trình hơn 50 năm của công nghệ ảo hoá.

1967

IBM CP/CMS

IBM phát minh khái niệm "virtual machine" đầu tiên trên mainframe System/360-67. Mỗi user có một "máy ảo" riêng - ý tưởng cách mạng cho thời đại đó. Đây là tổ tiên của mọi hypervisor ngày nay.

1998

VMware ra đời

VMware Workstation biến ảo hoá từ "đồ chơi mainframe" thành công cụ cho x86. Lần đầu tiên, bạn có thể chạy Windows trong Linux (và ngược lại) trên một chiếc PC thông thường. Ngành công nghiệp ảo hoá bắt đầu bùng nổ.

2003

Xen Hypervisor

Đại học Cambridge phát triển Xen - hypervisor mã nguồn mở Type-1 đầu tiên cho x86. Amazon Web Services sau này được xây dựng trên nền Xen. Citrix mua lại XenSource năm 2007 và thương mại hoá thành XenServer (nay là XCP-ng).

2006

KVM vào Linux Kernel

Avi Kivity (Qumranet, sau được Red Hat mua) viết KVM module. Được merge vào Linux kernel 2.6.20 năm 2007. Đây là bước ngoặt: Linux kernel chính thức trở thành một hypervisor. Không cần cài thêm gì - chỉ cần CPU hỗ trợ VT-x/AMD-V.

2008

Proxmox VE 1.0

Proxmox Server Solutions GmbH (Áo) phát hành phiên bản đầu tiên. Kết hợp KVM + OpenVZ trên nền Debian, với giao diện web quản lý. Mục tiêu: tạo ra một giải pháp ảo hoá enterprise-grade nhưng hoàn toàn mã nguồn mở và miễn phí.

2014

Chuyển sang LXC

Proxmox VE 3.4 thay thế OpenVZ bằng LXC - công nghệ container chính thống của Linux kernel. Đây là quyết định sáng suốt: LXC được maintain trong mainline kernel, trong khi OpenVZ luôn là patch ngoài. Hiệu năng và tính tương thích được cải thiện đáng kể.

2017

Ceph tích hợp sâu

Proxmox VE 5.0 mang Ceph storage vào tầm tay người dùng. Cài đặt và quản lý Ceph cluster trực tiếp từ web UI. Hyper-converged infrastructure không còn là đặc quyền của VMware vSAN hay Nutanix.

2020

Proxmox Backup Server

PBS ra đời - giải pháp backup chuyên dụng cho Proxmox. Incremental backup, deduplication, encryption, verify - tất cả miễn phí. Đây là câu trả lời cho Veeam Backup trong thế giới mã nguồn mở.

2023-2025

Kỷ nguyên hậu VMware

Broadcom mua lại VMware, thay đổi chính sách license khiến cộng đồng phẫn nộ. Hàng ngàn doanh nghiệp chuyển sang Proxmox. Proxmox VE 8.x trên Debian 12 Bookworm trở thành lựa chọn hàng đầu cho cả homelab lẫn production.

KVM / QEMU

Full Virtualization Engine

KVM là một kernel module biến Linux thành hypervisor Type-1. Nó tận dụng hardware virtualization extensions (Intel VT-x, AMD-V) để chạy guest OS ở tốc độ gần native. QEMU là user-space component phụ trách emulate các thiết bị phần cứng (disk, network, USB...) cho VM.

Khi bạn tạo một VM trong Proxmox, thực chất là khởi động một tiến trình QEMU với KVM acceleration. Mỗi vCPU là một thread, mỗi VM là một process - đơn giản và mạnh mẽ.

LXC Containers

OS-level Virtualization

LXC sử dụng namespaces và cgroups của Linux kernel để tạo ra các môi trường cách ly mà không cần hypervisor. Container chia sẻ kernel với host, nên không có overhead của hardware emulation. Khởi động trong vài giây, tiêu thụ RAM ít hơn nhiều so với VM.

Proxmox quản lý LXC qua công cụ pct. Bạn có thể mount thư mục host vào container, giới hạn CPU/RAM/IO, và thậm chí chạy Docker bên trong LXC (unprivileged container cần cấu hình thêm).

VirtIO

Paravirtualized I/O

Thay vì emulate một ổ cứng IDE hay card mạng Intel thật, VirtIO tạo ra một "giao thức" tối ưu giữa guest và host. Guest biết mình đang chạy trong VM và dùng driver chuyên biệt để giao tiếp trực tiếp với hypervisor. Kết quả: disk I/O nhanh hơn 2-5 lần, network throughput gần line-rate.

Trên Linux, VirtIO driver có sẵn trong kernel. Trên Windows, bạn cần cài VirtIO driver ISO (từ Fedora project). Đây là bước quan trọng nhất khi tạo Windows VM trên Proxmox.

virtio-blk virtio-net virtio-scsi virtio-gpu Cần driver

OVMF / UEFI

Firmware ảo hoá

OVMF (Open Virtual Machine Firmware) là implementation UEFI cho máy ảo, dựa trên EDK II của Tianocore. Khi bạn chọn BIOS type = OVMF trong Proxmox, VM sẽ boot qua UEFI thay vì SeaBIOS truyền thống.

OVMF cần thiết cho: Secure Boot, GPU passthrough (nhiều GPU đời mới chỉ có UEFI GOP, không có legacy VBIOS), boot từ ổ GPT > 2TB, và Windows 11 (yêu cầu TPM + Secure Boot). Proxmox tự động tạo EFI disk khi bạn chọn OVMF.

Cloud-Init

Tự động hoá VM

Cloud-Init là chuẩn công nghiệp để cấu hình VM tự động khi boot lần đầu. Thay vì cài đặt thủ công (chọn ngôn ngữ, tạo user, cấu hình mạng...), bạn chuẩn bị một template image + cloud-init config. VM sẽ tự nhận hostname, IP, SSH key, chạy script tuỳ ý.

Trong Proxmox, bạn tạo một cloud-init drive (IDE/SCSI), điền thông tin qua GUI hoặc API. Kết hợp với Terraform/Ansible, đây là nền tảng Infrastructure as Code cho homelab và production.

IaC ready SSH Keys Network Config User Data

Ignition

CoreOS / Flatcar Config

Ignition là phiên bản "cloud-init cho CoreOS" - được thiết kế cho Fedora CoreOS, Flatcar Linux, và các hệ điều hành immutable. Không giống cloud-init chạy mỗi boot, Ignition chỉ chạy một lần duy nhất trong lần khởi động đầu tiên (initramfs stage).

Nếu bạn chạy Kubernetes trên bare-metal hoặc VM, Ignition + Flatcar/FCOS là combo phổ biến. Proxmox hỗ trợ Ignition qua fw_cfg QEMU interface - bạn truyền config dưới dạng JSON transpiled từ Butane YAML.

Immutable OS Butane YAML First-boot only fw_cfg

Kinh nghiệm thực tế

Khi tạo VM mới trong Proxmox, luôn chọn: Machine type = q35 (hỗ trợ PCIe passthrough), BIOS = OVMF (nếu không cần legacy), SCSI controller = VirtIO SCSI single (io_thread=on), Network = VirtIO. Đây là combo cho hiệu năng tốt nhất. Với template, dùng cloud-init image (Ubuntu/Debian cloud image, hoặc Rocky Linux GenericCloud) để triển khai VM trong vài giây.

Storage: Nơi Dữ liệu Sống

ZFS, Ceph, LVM-thin, NFS, iSCSI - mỗi loại storage có điểm mạnh riêng. Chọn sai là đau đầu cả năm.

💾

ZFS

Copy-on-Write Filesystem

ZFS là filesystem/volume manager tiên tiến nhất thế giới. Được Sun Microsystems phát triển, nay là dự án OpenZFS. ZFS kết hợp RAID, LVM, và filesystem thành một thể thống nhất với tính năng data integrity chưa từng có.

Pools & VDEVs: ZFS pool gồm nhiều VDEV (mirror, RAIDZ1/2/3, stripe). Mỗi VDEV là một nhóm disk. Dữ liệu được phân phối qua các VDEV. Mất 1 VDEV = mất pool. Nên luôn dùng mirror hoặc RAIDZ.

Datasets: Giống "thư mục thông minh" - mỗi dataset có quota, compression, snapshot riêng. Proxmox tạo dataset cho mỗi VM/CT.

Snapshots: Tức thì, không tốn thêm dung lượng (CoW). Rollback trong tích tắc. Kết hợp zfs send | zfs recv để replicate sang host khác.

Scrub: Quét toàn bộ pool để phát hiện và tự sửa bit rot. Nên chạy monthly. Proxmox tự schedule scrub.

Ceph

Distributed Storage

Ceph là hệ thống lưu trữ phân tán - dữ liệu được chia nhỏ và phân phối qua nhiều node, không có single point of failure. Proxmox tích hợp Ceph sâu nhất trong các hypervisor mã nguồn mở.

CRUSH Map: Thuật toán phân phối dữ liệu thông minh - không cần central metadata server. Client tính toán vị trí dữ liệu từ CRUSH map, giảm bottleneck.

OSD: Mỗi ổ disk là một OSD daemon. Ceph tự replicate (hoặc erasure code) dữ liệu qua các OSD trên các node khác nhau.

RBD: Proxmox dùng RADOS Block Device làm storage cho VM/CT. Live migration hoạt động tức thì vì storage là shared.

LVM-Thin

Thin Provisioning

LVM-Thin là phiên bản nâng cấp của LVM truyền thống, hỗ trợ thin provisioning - cấp phát dung lượng theo nhu cầu thực tế thay vì cố định. Một thin pool 500GB có thể chứa 10 VM mỗi cái "claim" 100GB nhưng chỉ dùng 30GB.

LVM-Thin hỗ trợ snapshot nhanh (nhanh hơn LVM standard). Proxmox mặc định dùng LVM-Thin khi cài với local storage. Đơn giản, ổn định, nhưng không có data integrity checking như ZFS.

NFS / iSCSI / SMB

Network Storage

NFS: Giao thức chia sẻ file qua mạng, phổ biến nhất cho shared storage trong Proxmox cluster. Dễ setup, hỗ trợ live migration. Hiệu năng phụ thuộc vào network và NFS server.

iSCSI: Block-level storage qua TCP/IP. Hiệu năng cao hơn NFS cho random I/O. Thường dùng với SAN (TrueNAS, Synology, Dell PowerStore). Kết hợp LVM on iSCSI để có snapshot.

SMB/CIFS: Chủ yếu cho ISO images, container templates, VZDump backup. Không khuyến nghị cho VM disk do overhead.

NFS v3/v4 iSCSI SMB/CIFS Shared Storage Network dependent

Storage chọn như thế nào?

Single node homelab: ZFS mirror (2 ổ) hoặc RAIDZ1 (3 ổ). Đơn giản và an toàn nhất. Nhớ dành RAM cho ARC (quy tắc: 1GB RAM / 1TB storage). Cluster nhỏ (3 node): Ceph nếu có 10GbE, hoặc NFS từ NAS chuyên dụng. Production lớn: Ceph + NVMe OSD + dedicated network. Tránh: Chạy ZFS trên hardware RAID controller - ZFS cần truy cập trực tiếp disk (HBA mode / IT mode).

Clustering & High Availability

Biến nhiều server rời rạc thành một "siêu máy tính" thống nhất - sức mạnh thực sự của Proxmox.

🔗

Cluster & Quorum

Corosync + pmxcfs

Proxmox cluster dùng Corosync để giao tiếp giữa các node (multicast hoặc unicast). Mỗi node có một "vote" và cluster cần quorum (đa số > 50%) để hoạt động. Cluster 3 node chịu được mất 1 node. Cluster 5 node chịu được mất 2 node.

pmxcfs là filesystem đặc biệt của Proxmox - lưu toàn bộ cấu hình cluster trong SQLite database, replicate real-time qua Corosync. Thay đổi config trên node A tức thì xuất hiện trên node B.

Tại sao quorum quan trọng? Tránh "split-brain" - tình huống 2 nhóm node đều nghĩ mình là cluster chính và cùng chạy VM, gây corrupt dữ liệu. Quorum đảm bảo chỉ một nhóm được phép hoạt động.

High Availability (HA)

Tự động failover

HA trong Proxmox nghĩa là: khi một node chết, các VM/CT được đánh dấu HA sẽ tự động khởi động lại trên node khác. Không cần người can thiệp - downtime chỉ vài phút thay vì hàng giờ.

HA Manager theo dõi trạng thái từng resource. HA Groups cho phép chỉ định VM chỉ được chạy trên một nhóm node cụ thể (ví dụ: nhóm node có GPU). Fencing là cơ chế "giết" node lỗi trước khi khởi động VM trên node mới - tránh 2 node cùng chạy 1 VM.

Yêu cầu: Cluster ít nhất 3 node (cho quorum) + shared storage (Ceph, NFS, iSCSI). Local storage không hỗ trợ HA vì dữ liệu không di chuyển được.

Live Migration

Di chuyển VM không downtime

Live migration cho phép di chuyển VM đang chạy từ node A sang node B mà user không hề biết. RAM được copy iteratively (dirty pages), cuối cùng VM bị pause vài millisecond để sync lần cuối, rồi resume trên node đích.

Với shared storage: Chỉ copy RAM - nhanh, đơn giản. Với local storage: Proxmox hỗ trợ "storage migration" - copy cả disk qua mạng, chậm hơn nhưng vẫn hoạt động. Với Ceph RBD: Gần như tức thì vì disk nằm trên cluster storage.

Corosync Deep Dive

Cluster Communication

Corosync là lớp giao tiếp nền tảng của cluster. Nó dùng Totem Single Ring Protocol để đảm bảo tất cả node nhận tin nhắn theo cùng thứ tự (total ordering). Mỗi node có một Corosync ring (hoặc link trong Corosync 3).

Multi-link: Proxmox hỗ trợ nhiều network link cho Corosync (link0, link1). Nếu một link chết, cluster vẫn hoạt động qua link còn lại. Luôn cấu hình ít nhất 2 link cho production.

QDevice: Node bỏ phiếu bên ngoài cluster (chạy trên Raspberry Pi cũng được). Cho phép cluster 2 node vẫn có quorum khi mất 1 node. Giải pháp cho homelab không đủ 3 server.

Totem Protocol Multi-link QDevice UDP Unicast/Multicast

Bài học từ thực tế

Cluster 2 node là "anti-pattern" phổ biến nhất. Khi mất 1 node, node còn lại không có quorum và từ chối hoạt động - đúng thiết kế, nhưng gây hoảng loạn. Giải pháp: thêm QDevice (corosync-qdevice), hoặc chấp nhận rủi ro bằng cách set pvecm expected 1 khi node chết. Cluster 3 node luôn là lựa chọn tối ưu cho homelab.

PCIe Passthrough: Hardware trong VM

Gắn trực tiếp GPU, NIC, NVMe, USB controller vào VM - hiệu năng 100% native.

🧱

IOMMU

Input-Output Memory Management Unit

IOMMU (Intel VT-d / AMD-Vi) là nền tảng của PCIe passthrough. Nó tạo ra một "bản đồ bộ nhớ" riêng cho từng thiết bị PCIe, ngăn không cho thiết bị trong VM truy cập bộ nhớ của host hoặc VM khác.

IOMMU Groups: Các thiết bị PCIe được nhóm thành các IOMMU group. Bạn chỉ có thể passthrough nguyên một group, không thể tách lẻ. Một số mainboard có IOMMU grouping tệ (gộp nhiều thiết bị vào 1 group), cần ACS override patch (rủi ro bảo mật).

Kích hoạt: Thêm intel_iommu=on hoặc amd_iommu=on vào kernel command line (GRUB hoặc systemd-boot).

Intel VT-d AMD-Vi DMA Isolation IOMMU Groups
🔌

VFIO

Virtual Function I/O

VFIO là framework trong Linux kernel cho phép user-space program (QEMU) truy cập trực tiếp PCIe device một cách an toàn. Khi bạn "passthrough" một device, thực chất là bind nó vào vfio-pci driver thay vì driver gốc.

Quy trình: (1) Bật IOMMU, (2) Load vfio-pci module, (3) Unbind device khỏi driver gốc, (4) Bind vào vfio-pci, (5) Assign cho VM trong Proxmox UI. Host không thể dùng device nữa - nó thuộc về VM hoàn toàn.

vfio-pci Device unbind Full performance Host mất device
🎮

GPU Passthrough

Gaming / AI / Transcoding

GPU passthrough là use case phổ biến nhất. Gắn NVIDIA/AMD GPU vào VM để chơi game, train AI model, hoặc hardware transcoding (Plex/Jellyfin). VM nhận được 100% hiệu năng GPU - không khác gì cắm trực tiếp.

NVIDIA: Từng có "Code 43" block trên consumer GPU (GeForce). Nay đã bỏ từ driver 465+. Vẫn cần: OVMF BIOS, machine type q35, hidden state, vendor_id.

AMD: Thường dễ passthrough hơn NVIDIA. Nhưng có "reset bug" trên một số dòng (Navi, Polaris) - GPU không reset đúng khi VM restart, cần vendor-reset module.

Intel iGPU (GVT-g): Chia sẻ iGPU cho nhiều VM. Hỗ trợ có giới hạn, đang chuyển sang SR-IOV trên Intel Flex/Arc.

100% Performance NVIDIA GeForce AMD Radeon Reset bug AMD GVT-g / SR-IOV
🔄

SR-IOV

Single Root I/O Virtualization

SR-IOV cho phép một PCIe device vật lý tạo ra nhiều "Virtual Functions" (VF) - mỗi VF hoạt động như một device riêng biệt, có thể assign cho VM khác nhau. Một NIC 10GbE tạo ra 64 VF, mỗi VM có "NIC riêng" với near-native performance.

NIC SR-IOV: Intel X710, Mellanox ConnectX - phổ biến nhất. Bypass hoàn toàn software bridge, giảm latency và tăng throughput đáng kể so với virtio-net.

GPU SR-IOV: Intel Data Center GPU Flex (ATS-M), NVIDIA A-series (vGPU). Chia sẻ GPU cho nhiều VM mà không cần license đắt đỏ (Intel) hoặc cần license (NVIDIA).

Kinh nghiệm GPU Passthrough

Trước khi mua hardware cho GPU passthrough, kiểm tra: (1) CPU/Mainboard có hỗ trợ IOMMU không, (2) IOMMU groups có tách biệt không (dùng script find /sys/kernel/iommu_groups/), (3) GPU có nằm trong group riêng không. Server motherboard (Supermicro, ASRock Rack) thường có IOMMU grouping tốt hơn consumer board. Và nhớ: bạn cần 2 GPU (hoặc 1 GPU + 1 iGPU) nếu muốn host vẫn có display output.

Deduplication & Verify

Tiết kiệm dung lượng

PBS chunk dữ liệu thành các block cố định, hash mỗi block. Nếu 2 VM có cùng OS base image, PBS chỉ lưu 1 bản. Tỷ lệ dedup thường 2-5x cho môi trường đồng nhất.

Verify: PBS có thể kiểm tra integrity của mọi backup bất kỳ lúc nào. Không cần restore thử - chỉ cần chạy verify job. Backup bị corrupt sẽ được phát hiện trước khi bạn cần nó.

Encryption & Sync

Bảo mật dữ liệu

Client-side encryption: Dữ liệu được mã hoá trên PVE host trước khi gửi đến PBS. PBS admin không thể đọc dữ liệu backup. Dùng AES-256-GCM.

Remote sync: PBS có thể replicate datastore sang PBS khác qua internet (pull hoặc push). Kết hợp encryption, bạn có offsite backup an toàn mà không cần VPN phức tạp.

AES-256-GCM Client-side encrypt Remote sync Tape backup

Quy tắc 3-2-1

3 bản sao dữ liệu, trên 2 loại media khác nhau, với 1 bản ở offsite. PBS làm cho quy tắc này dễ thực hiện: (1) Dữ liệu gốc trên PVE, (2) Backup trên PBS local, (3) Sync sang PBS remote hoặc tape. Chi phí gần như bằng 0 so với các giải pháp thương mại như Veeam ($$$).

Hiểu Nhầm Phổ Biến

Những quan niệm sai lầm mà ngay cả người dùng lâu năm cũng mắc phải.

Sai

"Proxmox là Type-2 hypervisor vì chạy trên Linux"

Nhiều người nghĩ: Proxmox cài trên Debian = Linux chạy trước = Type-2 (như VirtualBox). Sai hoàn toàn.

Đúng

KVM là kernel module - Linux kernel chính nó trở thành hypervisor khi load KVM. Giống ESXi là "OS + hypervisor", Linux + KVM cũng là "OS + hypervisor". Nhiều tài liệu phân loại KVM là Type-1. VM chạy ở Ring -1 (VMX root), host OS ở Ring 0 - không có "host OS chạy trước rồi mới có hypervisor".

Sai

"Container LXC không an toàn, dễ bị escape"

Nhiều người đánh đồng LXC với Docker rồi kết luận container không an toàn bằng VM.

Đúng

LXC unprivileged container trong Proxmox cực kỳ an toàn. User namespace mapping đảm bảo root trong container = nobody trên host. Kết hợp AppArmor profile + seccomp filter, attack surface rất nhỏ. Tất nhiên, VM vẫn cách ly tốt hơn (separate kernel), nhưng LXC unprivileged đủ tốt cho hầu hết workload.

Sai

"Cần mua subscription để dùng Proxmox"

Rất nhiều người mới nghĩ pop-up "no valid subscription" nghĩa là Proxmox sẽ ngừng hoạt động hoặc bị giới hạn tính năng.

Đúng

Proxmox VE miễn phí 100% tính năng. Subscription chỉ cho bạn: (1) enterprise repo ổn định hơn, (2) support ticket. Dùng no-subscription repo hoàn toàn hợp pháp và đầy đủ chức năng. Pop-up chỉ là reminder - bạn có thể dismiss hoặc remove bằng script.

Sai

"ZFS cần ECC RAM, không có thì mất dữ liệu"

Một "myth" tồn tại hơn 10 năm trong cộng đồng: "Không có ECC thì ZFS sẽ ăn hết dữ liệu."

Đúng

ZFS hoạt động tốt trên non-ECC RAM. ECC tốt hơn cho MỌI filesystem (ext4, XFS, NTFS đều hưởng lợi). ZFS checksumming phát hiện corruption trên disk, nhưng không fix RAM error - đúng, nhưng ext4 cũng không! ZFS với non-ECC vẫn an toàn hơn ext4 với non-ECC. ECC là "nice to have", không phải "must have".

Sai

"Proxmox không phù hợp cho production"

Một số IT manager: "Proxmox chỉ là đồ chơi homelab, production phải dùng VMware."

Đúng

Proxmox được dùng bởi hàng ngàn doanh nghiệp, ISP, trường đại học, và cả chính phủ. Hetzner - một trong những hosting provider lớn nhất châu Âu - chạy Proxmox cho toàn bộ infrastructure. Với Ceph, HA, backup tích hợp, Proxmox đáp ứng mọi yêu cầu production. Vấn đề không phải "Proxmox có đủ tốt không" mà là "team bạn có đủ skill Linux không".

Sai

"Snapshot là backup, không cần PBS"

Nhiều người tạo ZFS/LVM snapshot rồi yên tâm: "Dữ liệu an toàn rồi."

Đúng

Snapshot nằm trên cùng storage với dữ liệu gốc. Disk chết = mất cả data lẫn snapshot. Snapshot cũng tích lũy dirty blocks, gây chậm I/O theo thời gian. PBS backup sang server/disk riêng mới là backup thực sự. Snapshot là "undo button" tiện lợi, không phải disaster recovery.

Sai

"GPU passthrough không ổn định, hay crash"

Nhiều bài forum report lỗi passthrough, khiến người mới sợ không dám thử.

Đúng

GPU passthrough hoạt động cực kỳ ổn định khi cấu hình đúng. 90% lỗi đến từ: IOMMU chưa bật, BIOS setting sai, dùng SeaBIOS thay vì OVMF, hoặc AMD reset bug (cần vendor-reset module). Một khi setup đúng, VM chạy GPU passthrough ổn định hàng tháng không restart. Vấn đề là tài liệu hướng dẫn thường outdated hoặc thiếu bước.

Sai

"Ceph quá phức tạp cho homelab"

"Ceph là enterprise-only, homelab 3 node chạy Ceph là phí phạm."

Đúng

Proxmox đã đơn giản hoá Ceph đến mức bạn có thể setup 3-node Ceph cluster trong 30 phút qua Web UI. Với 3 node, mỗi node 1 SSD OSD, bạn có shared storage với auto-replication, live migration, và HA. Chi phí: 3 con SSD. Tất nhiên cần 10GbE (hoặc ít nhất 2.5GbE bonded) để Ceph hoạt động tốt.

qm start 100 qm shutdown 100 qm stop 100 # force stop qm reboot 100 qm reset 100 # force reset
