Thế Giới DNS

Danh bạ điện thoại của Internet - Hệ thống phân giải tên miền mà không có nó, mọi thứ trên mạng chỉ là những con số vô nghĩa. Từ HOSTS.TXT đến DoH, từ "it's always DNS" đến kiến trúc phân tán lớn nhất hành tinh.

⌄

DNS là gì thực sự?

Domain Name System - Hệ thống phân giải tên miền. Nôm na là "danh bạ điện thoại" của Internet. Con người nhớ tên (google.com), máy tính hiểu số (142.250.199.46). DNS là cầu nối giữa hai thế giới đó. Mỗi lần bạn gõ một URL vào trình duyệt, DNS thầm lặng làm việc phía sau để biến cái tên đó thành địa chỉ IP mà máy tính có thể kết nối tới.

Insight quan trọng

DNS không chỉ là "tra cứu tên miền ra IP". Nó là cơ sở dữ liệu phân tán lớn nhất thế giới, chạy trên hàng triệu máy chủ, phục vụ hàng nghìn tỷ truy vấn mỗi ngày. DNS còn phục vụ email routing (MX records), xác thực (SPF/DKIM qua TXT records), service discovery (SRV records), và hàng tá mục đích khác. Nếu Internet là thành phố, DNS chính là hệ thống bảng chỉ đường mà không có nó, bạn chỉ có thể đi lạc.

Dòng Thời Gian DNS

Từ một file text duy nhất đến hệ thống phân tán phục vụ cả hành tinh

1960s - 1983

Kỷ nguyên HOSTS.TXT

Trước khi DNS tồn tại, mọi máy tính trên ARPANET dùng chung một file duy nhất: HOSTS.TXT, được quản lý bởi SRI-NIC. Mỗi khi có máy mới, ai đó phải gọi điện hoặc email để cập nhật file này, rồi mọi người FTP về. Với vài trăm máy thì ổn, nhưng khi mạng bắt đầu nổ...

1983 - 1984

DNS ra đời - RFC 882 & 883

Paul Mockapetris phát minh DNS. Ý tưởng thiên tài: thay vì một file tập trung, tạo hệ thống phân tán dạng cây (hierarchy). Mỗi tổ chức quản lý phần tên miền của mình. RFC 1034 và 1035 (1987) trở thành nền tảng DNS hiện đại.

1984 - 1990s

BIND thống trị

Berkeley Internet Name Domain (BIND), viết bởi sinh viên UC Berkeley, trở thành DNS server phổ biến nhất. Gần như mọi DNS server trên Internet đều chạy BIND. Phiên bản BIND 4, rồi BIND 8, rồi BIND 9 (viết lại từ đầu bởi ISC năm 2000).

1997 - 2005

DNSSEC & Cuộc chiến bảo mật

Nhận ra DNS không có cơ chế xác thực, DNSSEC được đề xuất. Nhưng triển khai cực kỳ phức tạp. Sự kiện DNS cache poisoning của Dan Kaminsky (2008) là hồi chuông cảnh tỉnh, đẩy nhanh việc áp dụng DNSSEC. Root zone được ký DNSSEC năm 2010.

2015 - 2020

DNS mã hóa: DoH & DoT

DNS truyền thống gửi plaintext qua UDP port 53 - ai cũng đọc được. DNS over TLS (DoT, port 853) và DNS over HTTPS (DoH, port 443) ra đời. Cloudflare 1.1.1.1 (2018) và Firefox bật DoH mặc định tạo làn sóng thay đổi lớn. Cuộc tranh cãi privacy vs. network control bùng nổ.

2020 - Nay

DNS hiện đại & Homelab Revolution

Pi-hole, AdGuard Home biến DNS thành vũ khí chống quảng cáo. CoreDNS trở thành DNS mặc định trong Kubernetes. Oblivious DoH (ODoH) thêm lớp ẩn danh. DNS trở thành công cụ bảo mật, riêng tư, và tùy biến trong tay mỗi người dùng.

Insight: TTL - Bí mật của tốc độ DNS

Time To Live (TTL) là thời gian mà kết quả DNS được cache. TTL = 3600 nghĩa là resolver sẽ nhớ câu trả lời trong 1 giờ mà không cần hỏi lại. TTL thấp (60s) giúp thay đổi nhanh nhưng tăng tải DNS server. TTL cao (86400s) giảm tải nhưng khi bạn đổi IP, phải đợi rất lâu. Mẹo: Trước khi migration, hạ TTL xuống 60-300s vài ngày trước, đổi IP, rồi tăng TTL lại sau.

Recursive Query

Client hỏi resolver: "Cho tôi IP của example.com" - resolver phải đi tìm bằng mọi cách và trả kết quả cuối cùng. Đây là kiểu query từ máy tính bạn đến DNS server.

Iterative Query

Resolver hỏi các server khác: "Bạn có biết example.com?" - server trả lời "không, nhưng hỏi server kia". Resolver tự đi hỏi tiếp. Đây là cách resolver nói chuyện với root/TLD/authoritative.

DNS Caching

Bộ nhớ đệm ở mọi tầng: Trình duyệt cache, OS cache, router cache, resolver cache. Nhờ cache, 90%+ truy vấn không cần đi đến root server. Đây là lý do DNS nhanh đến kinh ngạc dù kiến trúc phức tạp.

Negative Caching

Cache cả "không tìm thấy": Khi domain không tồn tại (NXDOMAIN), kết quả cũng được cache. Điều này giải thích tại sao sau khi tạo record mới, đôi khi phải đợi vài phút mới hoạt động - vì NXDOMAIN cũ vẫn nằm trong cache.

DNS Security

DNS được thiết kế năm 1983 - thời mà bảo mật chưa phải ưu tiên. Giờ chúng ta phải vá lại từng lỗ hổng

🔏

DNSSEC

Chữ ký số cho DNS

Thêm chữ ký cryptographic vào DNS response. Resolver có thể xác minh rằng câu trả lời DNS thực sự đến từ authoritative server, không bị giả mạo trên đường truyền. Dùng chuỗi ký tin cậy (chain of trust) từ root zone xuống.

Hạn chế: Không mã hóa nội dung (ai cũng vẫn thấy bạn hỏi gì), chỉ xác thực. Triển khai phức tạp, key rotation dễ gây sự cố. Nhiều domain lớn vẫn chưa bật DNSSEC.

Xác thực Phức tạp triển khai Chain of Trust Không mã hóa
🔒

DoH (DNS over HTTPS)

Mã hóa DNS qua HTTPS

Gói DNS query vào HTTPS request thông thường (port 443). ISP không thể thấy bạn đang truy vấn domain nào. Trông giống traffic HTTPS bình thường nên rất khó chặn. Firefox, Chrome, Windows 11 đều hỗ trợ.

Tranh cãi: Sysadmin ghét DoH vì bypass được DNS filter nội bộ. Enterprise phải chặn DoH hoặc triển khai DoH server riêng. Pi-hole trở nên vô dụng nếu client dùng DoH đến 1.1.1.1.

Privacy Port 443 Bypass DNS filter Browser native
🛡️

DoT (DNS over TLS)

Mã hóa DNS qua TLS

Mã hóa DNS query bằng TLS trên port 853 riêng. Sysadmin thích DoT hơn DoH vì dễ quản lý: block port 853 = block DoT. Android 9+ hỗ trợ "Private DNS" chính là DoT. Nhanh hơn DoH một chút vì không có HTTP overhead.

Privacy Dễ quản lý Port 853 Android native
☠️

DNS Cache Poisoning

Tấn công giả mạo cache

Kẻ tấn công gửi DNS response giả vào cache của resolver, khiến mọi người hỏi "bank.com" đều nhận được IP của trang giả. Dan Kaminsky (2008) phát hiện lỗ hổng cho phép poisoning dễ dàng, gây chấn động toàn ngành.

Phòng chống: DNSSEC, source port randomization, 0x20 encoding (randomize viết hoa/thường trong query), dùng DoH/DoT.

Nguy hiểm Kaminsky Attack DNSSEC phòng chống
💥

DNS Amplification Attack

DDoS qua DNS

Kẻ tấn công gửi DNS query nhỏ (60 bytes) với IP nguồn giả (của nạn nhân) đến open resolver. Resolver trả response lớn (4000+ bytes) về nạn nhân. Hệ số khuếch đại lên đến 70x. Đây là một trong những kiểu DDoS phổ biến nhất.

Phòng chống: Không chạy open resolver, Response Rate Limiting (RRL), BCP38 (ingress filtering chống IP spoofing).

DDoS 70x khuếch đại Open resolver RRL phòng chống
🕳️

DNS Tunneling

Đường hầm dữ liệu qua DNS

Mã hóa dữ liệu vào DNS query/response để bypass firewall. Vì DNS query hầu như luôn được cho phép, kẻ tấn công có thể exfiltrate dữ liệu hoặc tạo C2 channel. Tools: iodine, dnscat2. Rất chậm nhưng rất lén lút.

Phát hiện: Monitor query length bất thường, entropy cao trong subdomain, volume query đến một domain đột biến.

Data exfiltration Khó phát hiện Bypass firewall

DNSSEC vs DoH/DoT - Khác nhau hoàn toàn

DNSSEC = xác thực (đảm bảo câu trả lời DNS không bị giả mạo, nhưng ai cũng đọc được). DoH/DoT = mã hóa (ẩn nội dung query khỏi bên thứ ba, nhưng không đảm bảo câu trả lời đúng). Cần CẢ HAI để có DNS thực sự an toàn. Hiện tại, rất ít hệ thống triển khai đồng thời cả hai.

DNS Trong Homelab

Split-horizon DNS, local domain resolution, và những bí kíp mà dân homelab cần biết

Split-Horizon DNS (Split-Brain DNS)

Cùng một tên miền nhưng trả về IP khác nhau tùy thuộc vào người hỏi ở đâu. Khi bạn ở trong mạng nội bộ, app.company.com trả về IP private (192.168.1.10). Khi ở ngoài Internet, cùng tên đó trả về IP public. Rất phổ biến trong enterprise và homelab.

🏠

Từ nội bộ

app.home.lab → 192.168.1.10
Truy cập trực tiếp, nhanh

≠
🌍

Từ Internet

app.home.lab → 203.0.113.5
Qua reverse proxy, NAT

1

Dùng domain riêng cho homelab

Đừng dùng .local (dành cho mDNS), .home, hay .lan (không chính thức). Tốt nhất: mua domain thật (home.yourdomain.com) hoặc dùng domain đã dành riêng cho testing (.test). Nếu dùng domain không hợp lệ, một ngày nào đó ICANN có thể cấp cho ai đó và bạn sẽ gặp vấn đề.

2

Pi-hole + Unbound = Combo hoàn hảo

Pi-hole chặn ads, Unbound làm recursive resolver riêng. DNS query không cần gửi ra bên ngoài (trừ khi cache miss). Privacy tốt hơn, không phụ thuộc Cloudflare/Google. Setup: Pi-hole forward đến Unbound trên localhost:5353.

3

Wildcard DNS cho reverse proxy

Thay vì tạo A record cho mỗi service, dùng wildcard: *.home.lab → IP của reverse proxy (Traefik/Nginx). Proxy dựa vào Host header để route đến đúng container. Một record, vô số services.

4

Conditional forwarding

Router forward DNS đến Pi-hole. Pi-hole forward .home.lab đến DNS server nội bộ (ví dụ trên pfSense hoặc OPNsense), mọi thứ khác forward ra Unbound/Cloudflare. Mỗi tầng xử lý phần mình giỏi nhất.

5

HA cho DNS nội bộ

DNS down = mọi thứ down. Chạy ít nhất 2 instance Pi-hole/AdGuard. Dùng Keepalived + virtual IP, hoặc đơn giản hơn: DHCP phát 2 DNS server. Gravity Sync để đồng bộ blocklist giữa 2 Pi-hole.

6

SSL với local DNS

HTTPS cho service nội bộ? Dùng Let's Encrypt DNS-01 challenge: chứng minh bạn sở hữu domain bằng cách tạo TXT record, không cần expose port 80 ra Internet. Traefik + Cloudflare API = auto-renew wildcard cert cho *.home.lab.

Kiến trúc DNS homelab gợi ý

Thiết bị → Pi-hole/AdGuard (chặn ads, local DNS) → Unbound (recursive resolver, DNSSEC validation) → Root Servers. Cho nội bộ: Pi-hole có conditional forward cho zone .home.lab đến authoritative DNS (BIND/CoreDNS/dnsmasq trên router). Đơn giản, hiệu quả, private.

10 Hiểu Nhầm Về DNS

Những điều "ai cũng tin" nhưng thực ra sai - bao gồm cả meme huyền thoại

Hiểu nhầm #1

"It's always DNS" - DNS luôn là thủ phạm

Meme nổi tiếng nhất ngành IT. Mỗi khi có sự cố mạng, câu đầu tiên mọi người nói là "chắc do DNS". Và buồn cười ở chỗ... nó đúng khá thường xuyên. Nhưng đây vừa là meme, vừa là bias.

Thực tế

DNS liên quan đến mọi kết nối mạng, nên tất nhiên nó "dính" nhiều sự cố. Nhưng thường DNS chỉ là triệu chứng, không phải nguyên nhân gốc. Network down, firewall block, certificate expired - tất cả đều "trông giống DNS issue". Kiểm tra DNS trước là tốt, nhưng đừng dừng ở đó.

Hiểu nhầm #2

"Đổi DNS sang 8.8.8.8 là nhanh hơn"

Nhiều người tin rằng đổi DNS từ ISP sang Google/Cloudflare sẽ làm Internet nhanh hơn đáng kể. Thậm chí có video YouTube hướng dẫn "tăng tốc Internet bằng cách đổi DNS".

Thực tế

DNS chỉ ảnh hưởng đến thời gian phân giải tên miền (thường dưới 50ms, và được cache). Tốc độ download/upload phụ thuộc vào bandwidth ISP, không liên quan đến DNS. Đổi DNS tốt cho reliability và privacy, không phải tốc độ download.

Hiểu nhầm #3

"DNS chỉ có 13 root server"

Mọi tài liệu đều nói "có 13 root server" (A đến M). Nhiều người lo lắng: chỉ 13 máy phục vụ cả Internet? Nếu DDoS hết 13 cái thì Internet sập?

Thực tế

Có 13 root server identities (địa chỉ IP), nhưng đằng sau mỗi identity là hàng trăm server vật lý phân bố khắp thế giới nhờ anycast. Tổng cộng hơn 1,700 instance. DDoS 13 IP thực chất phải DDoS 1,700+ server - gần như không thể.

Hiểu nhầm #4

"DNS propagation mất 24-48 giờ"

Câu kinh điển từ mọi nhà cung cấp hosting: "DNS thay đổi sẽ mất 24-48 giờ để propagate". Nhiều người tin rằng có một quá trình "lan truyền" chậm chạp từ server này sang server khác.

Thực tế

Không có "propagation" theo nghĩa đen. Thứ bạn đang đợi là TTL cache hết hạn. Nếu TTL cũ là 86400 (24h), resolver đã cache kết quả cũ sẽ giữ nó tối đa 24h. Nếu bạn hạ TTL trước khi đổi, thay đổi có thể có hiệu lực trong vài phút.

Hiểu nhầm #5

"flush DNS trên máy tôi sẽ sửa mọi thứ"

Khi DNS "có vấn đề", phản xạ đầu tiên: chạy ipconfig /flushdns. Và nhiều người tin rằng thao tác này sẽ giải quyết mọi vấn đề DNS.

Thực tế

Flush DNS trên máy bạn chỉ xóa OS cache cục bộ. ISP resolver vẫn cache kết quả cũ. Trình duyệt có cache DNS riêng. Và upstream authoritative server có thể chưa cập nhật zone. Flush giúp thỉnh thoảng, nhưng không phải "thuốc chữa bách bệnh".

Hiểu nhầm #6

"DNS không quan trọng cho bảo mật"

Nhiều sysadmin focus vào firewall, antivirus, IDS mà quên mất DNS. DNS "chỉ là tra cứu tên miền" mà, có gì nguy hiểm?

Thực tế

DNS là attack vector cực kỳ lợi hại: cache poisoning redirect sang phishing, DNS tunneling để exfiltrate data, DGA (Domain Generation Algorithm) cho botnet C2, DNS amplification cho DDoS. Monitoring DNS là một trong những cách hiệu quả nhất phát hiện malware trong mạng.

Hiểu nhầm #7

"CNAME và A record giống nhau"

Cả hai đều "trỏ domain đến đâu đó", nên nhiều người dùng thay thế nhau tùy tiện. Đặc biệt phổ biến khi setup hosting hay CDN.

Thực tế

A record trỏ đến IP cố định. CNAME trỏ đến tên miền khác (rồi resolver đi tìm IP của tên đó). CNAME không dùng được ở zone apex (example.com), không thể song song với record khác cùng tên, và thêm một bước lookup. Dùng sai CNAME gây loop và sự cố khó debug.

Hiểu nhầm #8

"DoH giải quyết mọi vấn đề privacy"

DoH mã hóa DNS query, ISP không đọc được. Vậy là hoàn toàn ẩn danh, đúng không?

Thực tế

DoH ẩn nội dung DNS query khỏi ISP, nhưng: (1) ISP vẫn thấy IP bạn kết nối tới sau đó, (2) SNI trong TLS handshake lộ hostname (trừ khi dùng ECH), (3) DoH provider (Cloudflare/Google) vẫn thấy mọi query của bạn. DoH là một lớp trong nhiều lớp privacy cần thiết.

Hiểu nhầm #9

"DNS chỉ dùng UDP port 53"

Ai cũng biết DNS = UDP 53. Nên khi config firewall, chỉ mở UDP 53 và nghĩ là xong.

Thực tế

DNS dùng cả UDP và TCP port 53. TCP cho response lớn hơn 512 bytes (EDNS0 nâng lên 4096 nhưng vẫn cần TCP cho zone transfer AXFR/IXFR). DoT dùng TCP 853, DoH dùng TCP 443. Block TCP 53 = zone transfer fail, DNSSEC response bị cắt.

Hiểu nhầm #10

"TTL = 0 nghĩa là không cache"

Muốn thay đổi DNS có hiệu lực ngay? Set TTL = 0. Record sẽ không bao giờ bị cache, phải không?

Thực tế

RFC nói TTL = 0 nghĩa là "không nên cache", nhưng thực tế nhiều resolver bỏ qua TTL cực thấp và áp dụng minimum TTL riêng (thường 30-300s). Một số CDN/resolver coi TTL = 0 là 30s. Và TTL = 0 tạo tải khổng lồ lên authoritative server vì mọi query đều phải hỏi lại.

DNS Cheat Sheet

Các lệnh và kỹ thuật troubleshooting DNS mà mọi sysadmin cần thuộc lòng

dig - Tra cứu DNS cơ bản

Công cụ #1 cho DNS troubleshooting. Có trên mọi Linux/macOS.

dig example.com dig example.com A dig example.com AAAA dig example.com MX dig example.com TXT dig example.com ANY

dig - Chỉ định DNS server

Hỏi DNS server cụ thể thay vì dùng mặc định.

dig @8.8.8.8 example.com dig @1.1.1.1 example.com dig @192.168.1.1 home.lab

dig - Output ngắn gọn

Bỏ phần thừa, chỉ lấy kết quả cần thiết.

dig +short example.com dig +short example.com MX dig +noall +answer example.com

dig - Trace toàn bộ resolution

Xem từng bước DNS resolution từ root đến authoritative.

dig +trace example.com dig +trace +additional example.com

dig - Kiểm tra SOA & NS

Xem authoritative server và zone info.

dig example.com SOA dig example.com NS dig +nssearch example.com

dig - Reverse DNS lookup

Tìm hostname từ IP address.

dig -x 8.8.8.8 dig -x 192.168.1.1

nslookup - Quick lookup

Có sẵn trên Windows, Linux, macOS. Đơn giản hơn dig.

nslookup example.com nslookup example.com 8.8.8.8 nslookup -type=MX example.com nslookup -type=TXT example.com

host - Lookup nhanh

Output clean nhất. Tốt cho scripting.

host example.com host -t MX example.com host -t TXT example.com host 8.8.8.8

Flush DNS Cache

Xóa cache DNS cục bộ trên các OS.

# macOS sudo dscacheutil -flushcache sudo killall -HUP mDNSResponder # Windows ipconfig /flushdns # Linux (systemd-resolved) sudo systemd-resolve --flush-caches

Kiểm tra DNSSEC

Xác minh DNSSEC có hoạt động cho domain.

dig +dnssec example.com dig +short example.com DNSKEY dig +short example.com DS # Flag "ad" = Authenticated Data

Kiểm tra DNS over HTTPS

Test DoH bằng curl.

# Cloudflare DoH curl -s -H 'accept: application/dns-json' \ 'https://1.1.1.1/dns-query?name=example.com' # Google DoH curl -s \ 'https://dns.google/resolve?name=example.com'

Debug: DNS không phân giải

Checklist khi "DNS không hoạt động".

# 1. Kiểm tra resolver đang dùng cat /etc/resolv.conf # 2. Ping trực tiếp IP (bypass DNS) ping 1.1.1.1 # 3. Test DNS server cụ thể dig @1.1.1.1 example.com # 4. Trace toàn bộ chain dig +trace example.com # 5. Kiểm tra firewall sudo tcpdump -i any port 53

Zone Transfer test

Kiểm tra zone transfer (nên bị chặn trên production).

# Thử AXFR (full zone transfer) dig @ns1.example.com example.com AXFR # Nếu thành công = misconfigured! # Zone transfer nên bị restrict

Kiểm tra TTL còn lại

Xem record còn cache bao lâu nữa.

# TTL hiển thị trong cột thứ 2 dig example.com | grep -A1 "ANSWER" # Chạy lại sau vài giây, # TTL sẽ giảm = đang đọc cache

SPF/DKIM/DMARC check

Kiểm tra email authentication records.

# SPF dig +short example.com TXT | grep spf # DKIM dig +short selector._domainkey.example.com TXT # DMARC dig +short _dmarc.example.com TXT

Pro tip: dig vs nslookup

dig là công cụ chuyên nghiệp: output chi tiết, hỗ trợ mọi record type, trace, DNSSEC. nslookup đơn giản hơn, có trên mọi OS (kể cả Windows mặc định). Trong thực tế, hãy học dig - nó sẽ cho bạn thông tin cần thiết để debug bất kỳ vấn đề DNS nào. nslookup đã bị coi là deprecated trên nhiều hệ thống (dù vẫn hoạt động).
