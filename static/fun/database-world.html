Database là gì thực sự?

Database (cơ sở dữ liệu) - Không chỉ đơn giản là "nơi lưu dữ liệu". Hãy tưởng tượng bạn có một thư viện khổng lồ với hàng triệu cuốn sách. Nếu chỉ chất đống sách lên sàn, bạn sẽ mất cả đời để tìm một cuốn. Database chính là hệ thống tổ chức, lưu trữ, truy vấn và bảo vệ dữ liệu - giống như thủ thư siêu năng lực biết chính xác mọi cuốn sách nằm ở đâu.

Tại sao cần Database?

Bạn có thể lưu mọi thứ vào file CSV hoặc JSON. Nhưng khi có 10 triệu dòng dữ liệu, 100 người cùng đọc/ghi, và bạn cần đảm bảo không mất dữ liệu khi mất điện - đó là lúc bạn cần một database thực thụ. Database giải quyết bài toán mà file đơn thuần không thể: concurrent access, data integrity, crash recovery, và query optimization.

ACID - Bốn trụ cột của database đáng tin cậy

Mọi database quan hệ (relational) đều phải đảm bảo bốn tính chất này. Đây là lý do vì sao bạn tin tưởng ngân hàng không "mất" tiền của bạn.

A

Atomicity

Một giao dịch hoặc hoàn thành toàn bộ, hoặc không có gì xảy ra. Chuyển tiền: trừ A và cộng B phải cùng thành công hoặc cùng thất bại. Không có chuyện trừ A mà quên cộng B.

C

Consistency

Dữ liệu luôn ở trạng thái hợp lệ. Nếu rule nói "số dư không được âm", không giao dịch nào có thể vi phạm. Database từ chối mọi thao tác phá vỡ ràng buộc.

I

Isolation

Các giao dịch đồng thời không "thấy" nhau. Hai người cùng chuyển tiền từ một tài khoản sẽ không gây ra race condition. Mỗi giao dịch tưởng như mình là duy nhất.

D

Durability

Khi giao dịch đã commit, dữ liệu được lưu vĩnh viễn - kể cả khi server crash ngay lập tức. Nhờ WAL (Write-Ahead Log), database ghi log trước khi ghi data.

Dòng Thời Gian Database

60 năm phát triển - từ flat file đến distributed database chạy trên hàng ngàn node

1950s-1960s

Kỷ nguyên Flat File

Dữ liệu lưu trong file tuần tự trên băng từ. Muốn tìm một bản ghi phải đọc từ đầu đến cuối. Không có khái niệm "query". Thời kỳ đồ đá của lưu trữ dữ liệu - nhưng đây là nơi mọi thứ bắt đầu.

1960s

Hierarchical & Network DB

IBM IMS (1966) - database đầu tiên thực sự. Dữ liệu tổ chức dạng cây (hierarchical) hoặc đồ thị (network/CODASYL). Nhanh hơn flat file nhưng cứng nhắc - thay đổi cấu trúc = viết lại chương trình. Vẫn được dùng trong một số hệ thống ngân hàng đến tận ngày nay.

1970

Relational Model - Bước ngoặt lịch sử

Edgar F. Codd công bố bài báo huyền thoại "A Relational Model of Data for Large Shared Data Banks" tại IBM. Ý tưởng cách mạng: dữ liệu lưu trong bảng (table), truy vấn bằng logic toán học thay vì con trỏ. Ban đầu bị IBM phản đối vì "quá chậm" - nhưng lịch sử đã chứng minh Codd đúng.

1979-1995

SQL chinh phục thế giới

Oracle (1979), DB2 (1983), Sybase/SQL Server (1987), PostgreSQL (1996, tên cũ Postgres từ 1986), MySQL (1995). SQL trở thành ngôn ngữ chuẩn. Thời kỳ vàng của RDBMS khi mọi doanh nghiệp đều cần "relational database". Oracle trở thành gã khổng lồ.

2005-2012

NoSQL Boom

Google BigTable (2006), Amazon Dynamo (2007), MongoDB (2009), Cassandra (2008), Redis (2009). Web 2.0 với hàng tỷ users phá vỡ giới hạn của RDBMS truyền thống. "NoSQL" ban đầu nghĩa là "No SQL" nhưng sau được đổi thành "Not Only SQL". Schema-less, horizontal scaling, eventual consistency.

2012-nay

NewSQL & Distributed Era

CockroachDB, TiDB, YugabyteDB, Vitess, PlanetScale. Kết hợp ưu điểm của cả hai thế giới: ACID của SQL + khả năng scale của NoSQL. Database chạy phân tán trên hàng ngàn node, tự động sharding, global consistency. Spanner của Google chứng minh rằng "CAP theorem không phải bất khả thi".

Core Concepts - Xương Sống Của Mọi Database

Hiểu những khái niệm này, bạn sẽ debug nhanh hơn 10x và thiết kế hệ thống tốt hơn

Indexing - Tại sao query nhanh hay chậm

🌳 B-tree Index

Index mặc định của hầu hết database. Cấu trúc cây cân bằng, tìm kiếm O(log n). Tuyệt vời cho equality (=) và range queries (<, >, BETWEEN). Mỗi node chứa nhiều key, giảm disk I/O. PostgreSQL và MySQL đều dùng B-tree là mặc định.

DefaultRange QueryO(log n)

#️⃣ Hash Index

Chỉ hỗ trợ equality (=), không hỗ trợ range. O(1) cho lookup - nhanh hơn B-tree cho exact match. Redis bản chất là một hash table khổng lồ. Trong PostgreSQL, hash index ít được dùng vì B-tree đủ tốt và linh hoạt hơn.

O(1) LookupNo RangeEquality Only

🔤 GIN Index (Generalized Inverted)

Dành cho full-text search, JSONB, arrays trong PostgreSQL. "Inverted index" - thay vì "document → words", nó lưu "word → documents". Khi bạn search text trong PostgreSQL mà không muốn dùng Elasticsearch, GIN là câu trả lời.

Full-textJSONBArrays

🌐 GiST Index (Generalized Search Tree)

Index cho dữ liệu "không truyền thống": geometric, geographic (PostGIS), range types, text search. Khi bạn cần "tìm tất cả nhà hàng trong bán kính 5km", GiST index trên PostGIS xử lý việc đó cực nhanh.

Query Optimization & EXPLAIN ANALYZE

EXPLAIN ANALYZE - Bạn thân nhất của DBA
-- Query chậm: Full Table Scan (Sequential Scan)
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_email = 'alice@example.com';

  Seq Scan on orders  (cost=0.00..35811.00 rows=1 width=256)
    Filter: (customer_email = 'alice@example.com')
    Rows Removed by Filter: 1000000   ← Quét 1 triệu dòng để tìm 1!
    Planning Time: 0.1 ms
    Execution Time: 245.3 ms          ← Chậm!

-- Thêm index:
CREATE INDEX idx_orders_email ON orders(customer_email);

-- Query sau khi có index:
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_email = 'alice@example.com';

  Index Scan using idx_orders_email on orders  (cost=0.42..8.44 rows=1 width=256)
    Index Cond: (customer_email = 'alice@example.com')
    Planning Time: 0.1 ms
    Execution Time: 0.05 ms           ← Nhanh gấp 5000 lần!

Connection Pooling - Đừng mở 1000 connections

Mỗi connection đến PostgreSQL tốn ~10MB RAM và một process riêng. 100 connections = 1GB RAM chỉ cho overhead. Giải pháp: PgBouncer hoặc Pgpool-II - proxy đứng giữa app và database, tái sử dụng connections. 1000 app connections có thể chia sẻ 20 database connections thực sự. Đây là bước tối ưu đầu tiên mọi production setup cần làm.

🔄 MVCC (Multi-Version Concurrency Control)

PostgreSQL không lock row khi đọc. Thay vào đó, mỗi transaction thấy một "snapshot" riêng của dữ liệu. Readers không block writers, writers không block readers. Đây là lý do PostgreSQL xử lý concurrent workload tốt hơn MySQL (trước InnoDB).

📊 VACUUM & Bloat (PostgreSQL)

Do MVCC, PostgreSQL giữ lại old row versions (dead tuples). VACUUM dọn dẹp chúng. Nếu quên VACUUM, bảng "phình" ra (bloat), query chậm dần. autovacuum mặc định bật, nhưng cần tune cho bảng lớn. Đây là "hidden cost" mà người mới hay bỏ qua.

📝 WAL (Write-Ahead Log)

Trước khi ghi data vào disk, database ghi log trước. Nếu crash, đọc WAL để khôi phục. Đây là cách ACID property "Durability" được thực hiện. WAL cũng là nền tảng cho streaming replication - replica đọc WAL từ primary để đồng bộ.

Replication & High Availability

Database chạy trên một server là single point of failure - hãy nhân bản nó lên

Master-Slave (Primary-Replica)

Write vào Primary, đọc từ Replica. Đơn giản, phổ biến nhất. PostgreSQL streaming replication, MySQL replication đều theo mô hình này. Nhược điểm: Primary chết = phải failover thủ công hoặc tự động.

Master-Master (Multi-Primary)

Cả hai node đều nhận write. Conflict resolution phức tạp. Galera Cluster (MariaDB), MySQL Group Replication. Cẩn thận: auto-increment conflicts, split-brain. Chỉ dùng khi thực sự cần multi-region write.

Galera Cluster (Synchronous)

Mọi node đều là master, đồng bộ synchronous (wsrep). Virtually synchronous replication. Tối thiểu 3 nodes. MariaDB Galera Cluster rất phổ biến cho HA MySQL/MariaDB trong production.

Sharding & Partitioning

Khi một server không còn đủ sức chứa - hãy chia dữ liệu ra nhiều nơi

Vertical Partitioning

Chia bảng theo cột. Cột thường dùng (id, name) tách khỏi cột ít dùng (bio, avatar). Giúp giảm I/O vì đọc ít data hơn mỗi query. PostgreSQL hỗ trợ table inheritance cho vertical partitioning.

Horizontal Partitioning (Sharding)

Chia bảng theo hàng, mỗi phần trên một server riêng. Shard key quyết định dữ liệu đi đâu. Consistent hashing đảm bảo thêm/bớt shard không phải di chuyển toàn bộ data. Citus (PostgreSQL), Vitess (MySQL) hỗ trợ auto-sharding.

Consistent Hashing

Hash ring: mỗi key được map vào vòng tròn, đi theo chiều kim đồng hồ đến node gần nhất. Thêm/bớt node chỉ di chuyển 1/N dữ liệu thay vì toàn bộ. DynamoDB, Cassandra, Redis Cluster đều dùng.

Lời khuyên vàng: Đừng shard quá sớm!

Sharding thêm complexity khổng lồ: cross-shard queries, distributed transactions, operational burden. PostgreSQL trên một server mạnh (64 cores, 512GB RAM, NVMe SSD) có thể xử lý hàng trăm triệu rows thoải mái. Hãy tối ưu query, thêm index, dùng partitioning trước. Chỉ shard khi thực sự cần - và khi đó, dùng Citus hoặc Vitess thay vì tự xây.

Backup & Recovery

Không có backup = không có database. Đơn giản vậy thôi.

💾 Logical Backup (pg_dump / mysqldump)

Export dữ liệu thành SQL statements hoặc custom format. Dễ hiểu, portable, có thể restore vào version khác. Nhược điểm: chậm với database lớn (hàng trăm GB), lock tables khi dump (mysqldump --single-transaction giải quyết vấn đề này cho InnoDB).

PortableSQL formatSlow for large DB

📸 Physical Backup (pg_basebackup / Barman)

Copy toàn bộ data directory ở mức file system. Nhanh hơn nhiều so với logical backup. pg_basebackup cho PostgreSQL, Percona XtraBackup cho MySQL/MariaDB. Barman (Backup and Recovery Manager) là tool enterprise-grade cho PostgreSQL.

FastFile-levelSame version only

⏰ Point-in-Time Recovery (PITR)

Khôi phục database về BẤT KỲ thời điểm nào trong quá khứ. Dựa trên base backup + replay WAL logs đến thời điểm mong muốn. "Ối, ai đó chạy DELETE FROM users lúc 14:35!" → Restore về 14:34. Đây là lý do bạn phải archive WAL.

Any point in timeWAL-basedLife saver

📦 WAL Archiving & Shipping

PostgreSQL ghi mọi thay đổi vào WAL. Archive WAL ra nơi an toàn (S3, NFS) cho phép PITR và tạo standby servers. pgBackRest là tool tốt nhất hiện tại: parallel backup/restore, compression, encryption, S3/GCS/Azure support.

CAP Theorem & Distributed Databases

Định lý khiến mọi kiến trúc sư phần mềm mất ngủ

Định lý CAP (Eric Brewer, 2000)

Trong một hệ thống phân tán, bạn chỉ có thể đảm bảo 2 trong 3 tính chất sau cùng lúc. Đây không phải lựa chọn "pick 2" cứng nhắc, mà là trade-off liên tục mà bạn phải cân nhắc.

C
Consistency
A
Availability
P
Partition
Tolerance
C

Consistency

Mọi node trả về dữ liệu giống nhau tại cùng một thời điểm. Đọc sau khi ghi luôn thấy giá trị mới nhất. Ví dụ: PostgreSQL synchronous replication.

A

Availability

Mọi request đều nhận response (không timeout), kể cả khi một số node chết. Hệ thống luôn "sống". Ví dụ: Cassandra - vẫn trả lời dù mất vài node.

P

Partition Tolerance

Hệ thống vẫn hoạt động khi mạng giữa các node bị đứt. Trong thực tế, network partition LUÔN có thể xảy ra, nên P là bắt buộc → chọn C hoặc A.

Database CAP Consistency Model Giải thích
PostgreSQL (single) CA Strong Không phân tán → không cần P. ACID đầy đủ.
MongoDB (replica set) CP Strong (default) Khi primary chết, cluster bầu primary mới → không available trong vài giây.
Cassandra AP Eventual / Tunable Ưu tiên availability. Consistency tunable qua quorum (ONE, QUORUM, ALL).
Redis Cluster CP Strong Khi master chết, failover → unavailable tạm thời.
CockroachDB CP Serializable Strong consistency qua Raft consensus. Sacrifice availability khi partition.
DynamoDB AP Eventual (default) Eventually consistent reads mặc định. Strong reads tốn gấp đôi cost.

Sự thật về CAP: Không phải "chọn 2 trong 3"

CAP theorem thường bị hiểu sai. Trong thực tế, partition là sự kiện hiếm gặp. Hầu hết thời gian, hệ thống vừa consistent vừa available. CAP chỉ bắt bạn chọn khi partition thực sự xảy ra. Và "consistency" trong CAP (linearizability) khác với "consistency" trong ACID. Đừng nhầm lẫn!

Hiểu Nhầm Phổ Biến

Những myth sai lầm mà ngay cả developer kinh nghiệm cũng mắc phải

Sai

"NoSQL luôn nhanh hơn SQL"

Một trong những myth nguy hiểm nhất. NoSQL nhanh hơn cho một số workload cụ thể (key-value lookup, write-heavy). Nhưng PostgreSQL với đúng index có thể trả lời query trong <1ms, và cho phép JOIN phức tạp mà NoSQL phải fetch nhiều lần.

Thực tế

Tốc độ phụ thuộc vào data model, query pattern, indexing - không phải "SQL hay NoSQL". MongoDB không có magic nào làm nó nhanh hơn PostgreSQL cho mọi use case.

Sai

"MongoDB không cần schema"

MongoDB là "schema-flexible", KHÔNG phải "schema-less". Dữ liệu vẫn có cấu trúc - chỉ là database không enforce. Kết quả: app code phải validate, bugs khó tìm hơn, data inconsistency rất phổ biến.

Thực tế

Bạn vẫn cần thiết kế schema. MongoDB từ v3.6+ có JSON Schema Validation. Schema nên ở database level, không phải chỉ ở application level. "Schema-less" là marketing, không phải engineering practice.

Sai

"Index giải quyết mọi vấn đề performance"

Thêm index vào mọi cột? Mỗi index tốn disk space, RAM, và làm chậm INSERT/UPDATE vì phải cập nhật index. Một bảng với 20 indexes sẽ write cực chậm.

Thực tế

Index là trade-off: đọc nhanh hơn, ghi chậm hơn. Chỉ tạo index cho cột thực sự được query. Dùng EXPLAIN ANALYZE để xác nhận index có được dùng không. Composite index thường hiệu quả hơn nhiều single-column indexes.

Sai

"PostgreSQL chậm hơn MySQL"

Myth từ thời PostgreSQL 7.x (2000s). Ngày nay PostgreSQL thường nhanh hơn hoặc ngang MySQL cho hầu hết workload, đặc biệt khi cần complex queries, JSON, full-text search, concurrent writes.

Thực tế

PostgreSQL có query planner tiên tiến hơn, MVCC tốt hơn, và hỗ trợ parallel query từ v9.6. MySQL/MariaDB vẫn nhỉnh hơn cho simple read-heavy workloads (nhiều SELECT đơn giản). Cả hai đều "đủ nhanh" cho 99% use cases.

Sai

"Redis chỉ là cache"

Redis bắt đầu như cache, nhưng giờ là "data structure server" đa năng: pub/sub (real-time messaging), streams (event sourcing), sorted sets (leaderboards), Lua scripting (atomic operations), RedisJSON, RediSearch.

Thực tế

Redis là Swiss Army Knife của infrastructure. Session store, rate limiter, message queue, real-time analytics, distributed lock - tất cả trong một. Nhiều startup dùng Redis làm primary database (kết hợp AOF persistence).

Sai

"ORM thay thế việc học SQL"

Django ORM, SQLAlchemy, Prisma giúp viết code nhanh hơn. Nhưng khi ORM generate query tệ (N+1 problem, missing joins, full table scan), bạn không biết fix vì không hiểu SQL.

Thực tế

ORM là abstraction layer, không phải replacement. Mọi senior developer đều phải thành thạo raw SQL. Biết đọc EXPLAIN ANALYZE, viết CTE, dùng window functions - đây là kỹ năng không ORM nào thay thế được.

Sai

"Database trong Docker là production-ready"

Docker cho development thì tuyệt vời. Nhưng production database trong Docker có risks: volume mounting issues, I/O overhead, container restart mất data nếu quên volume, resource limits phức tạp.

Thực tế

Nhiều công ty lớn vẫn chạy database trên bare metal hoặc VM cho production. Docker OK nếu bạn hiểu rõ volume management, resource limits, và có backup strategy. Cho homelab thì Docker hoàn toàn ổn. Cho production business-critical, cần cân nhắc kỹ.

Sai

"Normalize mọi thứ / Denormalize mọi thứ"

Cực đoan nào cũng sai. Normalize quá mức tạo ra 50 bảng JOIN 10 tầng cho một query đơn giản. Denormalize quá mức tạo ra data inconsistency và storage waste.

Thực tế

Pragmatic normalization: normalize cho data integrity, denormalize có chọn lọc cho performance. PostgreSQL JSONB cho phép "semi-structured data trong relational world" - best of both worlds. Đo lường trước khi tối ưu.

Sai

"Elasticsearch có thể thay thế database chính"

Elasticsearch tuyệt vời cho search và analytics. Nhưng nó không có transactions, data loss possible khi node chết giữa chừng, update/delete chậm, và near-real-time (không phải real-time).

Thực tế

Elasticsearch là secondary index, không phải source of truth. Pattern đúng: ghi vào PostgreSQL (primary) → sync sang Elasticsearch (search). Mất Elasticsearch? Rebuild từ PostgreSQL. Mất PostgreSQL? Bạn gặp vấn đề lớn.

Sai

"Scale up đã hết đường, phải scale out"

Nhiều team vội vàng chuyển sang distributed database khi server hiện tại mới dùng 20% capacity. Chưa optimize query, chưa thêm index, chưa dùng connection pooling mà đã nghĩ đến sharding.

Thực tế

Một PostgreSQL server với NVMe SSD, 64GB RAM, proper tuning xử lý được workload mà 90% startups cần. Stack Overflow chạy trên 2 SQL Server instances. Optimize trước, scale sau. "Premature optimization is the root of all evil, but premature distribution is worse."

Cheat Sheet

Những lệnh và kiến thức thiết yếu để bắt đầu ngay

SQL Essentials

Tạo bảng & ràng buộc

CREATE TABLE users ( id SERIAL PRIMARY KEY, email VARCHAR(255) UNIQUE NOT NULL, name VARCHAR(100) NOT NULL, created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE orders ( id SERIAL PRIMARY KEY, user_id INT REFERENCES users(id), total DECIMAL(10,2) NOT NULL, status VARCHAR(20) DEFAULT 'pending' );

JOIN - Kết hợp bảng

-- INNER JOIN: chỉ lấy matching rows SELECT u.name, o.total FROM users u INNER JOIN orders o ON u.id = o.user_id; -- LEFT JOIN: tất cả users, kể cả chưa order SELECT u.name, COUNT(o.id) as order_count FROM users u LEFT JOIN orders o ON u.id = o.user_id GROUP BY u.name;

Window Functions

-- Rank users by total spending SELECT name, SUM(total) as spending, RANK() OVER (ORDER BY SUM(total) DESC) as rank FROM users u JOIN orders o ON u.id = o.user_id GROUP BY u.name; -- Running total SELECT date, amount, SUM(amount) OVER (ORDER BY date) as running_total FROM transactions;

CTE (Common Table Expression)

-- Đọc dễ hơn subquery WITH active_users AS ( SELECT user_id, COUNT(*) as orders FROM orders WHERE created_at > NOW() - INTERVAL '30 days' GROUP BY user_id HAVING COUNT(*) >= 3 ) SELECT u.name, a.orders FROM users u JOIN active_users a ON u.id = a.user_id ORDER BY a.orders DESC;

EXPLAIN ANALYZE

-- Xem query plan thực tế EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) SELECT * FROM orders WHERE status = 'pending' AND created_at > '2024-01-01'; -- Tìm: Seq Scan = BAD (cho bảng lớn) -- Index Scan = GOOD -- Rows Removed by Filter = quá nhiều?

Index hiệu quả

-- Composite index (thứ tự cột quan trọng!) CREATE INDEX idx_orders_status_date ON orders(status, created_at DESC); -- Partial index (chỉ index rows cần thiết) CREATE INDEX idx_orders_pending ON orders(created_at) WHERE status = 'pending'; -- Covering index (tránh table lookup) CREATE INDEX idx_users_email_name ON users(email) INCLUDE (name);

PostgreSQL Tuning (postgresql.conf)

# RAM = 16GB server shared_buffers = 4GB # 25% RAM effective_cache_size = 12GB # 75% RAM work_mem = 256MB # per-sort maintenance_work_mem = 1GB # VACUUM/INDEX wal_buffers = 64MB random_page_cost = 1.1 # SSD effective_io_concurrency = 200 # SSD max_connections = 200

MySQL/MariaDB Tuning (my.cnf)

# RAM = 16GB server [mysqld] innodb_buffer_pool_size = 12G # 75% RAM innodb_log_file_size = 2G innodb_flush_log_at_trx_commit = 1 innodb_file_per_table = 1 max_connections = 200 query_cache_type = 0 # disable in 8.0+ tmp_table_size = 256M max_heap_table_size = 256M

Checklist tối ưu

1. EXPLAIN ANALYZE mọi slow query

2. Thêm index cho WHERE/JOIN columns

3. Connection pooling (PgBouncer)

4. Tune shared_buffers/buffer_pool

5. Monitor với pg_stat_statements

6. VACUUM ANALYZE thường xuyên

7. Tránh SELECT * - chỉ lấy cột cần

8. Pagination: keyset thay vì OFFSET

9. Batch INSERT thay vì từng row

10. Read replica cho read-heavy workload
