Thế Giới Storage

Từ cuộn băng từ đến distributed storage - Hành trình lưu trữ dữ liệu của nhân loại. Mọi thứ bạn cần biết về RAID, filesystems, Ceph, ZFS, và tại sao mất dữ liệu luôn xảy ra vào lúc bạn ít ngờ nhất.

⌄

Storage là gì thực sự?

Storage - Không đơn giản chỉ là "chỗ để lưu file". Trong thế giới IT hạ tầng, storage là xương sống quyết định hiệu năng, độ tin cậy, và khả năng phục hồi của toàn bộ hệ thống. Mỗi byte dữ liệu bạn tạo ra đều cần một nơi trú ngụ an toàn, và cách bạn tổ chức "ngôi nhà" đó quyết định sống còn của doanh nghiệp.

Có ba mô hình lưu trữ cơ bản, mỗi loại phục vụ mục đích khác nhau. Hiểu sự khác biệt giữa chúng là bước đầu tiên để thiết kế hệ thống storage đúng đắn.

🧱

Block Storage

Dữ liệu chia thành block cố định. Như ổ cứng vật lý - raw, nhanh, low-level. Dùng cho VM disk, database.
VD: iSCSI, FC LUN, EBS

📂

File Storage

Dữ liệu tổ chức theo file/folder hierarchy. Quen thuộc nhất với con người. Chia sẻ qua mạng dễ dàng.
VD: NFS, SMB, CephFS

🪣

Object Storage

Dữ liệu lưu kèm metadata. Flat namespace, truy cập qua API (HTTP). Scale vô hạn, giá rẻ.
VD: S3, MinIO, Ceph RGW

Insight quan trọng

Sai lầm phổ biến nhất là cố dùng một loại storage cho mọi workload. Database cần block storage (IOPS cao, latency thấp). Chia sẻ file giữa các user cần file storage (permission, locking). Backup, media, log cần object storage (rẻ, scale tốt). Hiểu workload rồi mới chọn storage - không phải ngược lại.

Dòng Thời Gian Storage

70 năm tiến hóa - từ băng từ khổng lồ đến chip nhỏ hơn móng tay

1951 - 1960s

Kỷ nguyên Băng Từ (Magnetic Tape)

UNIVAC I dùng băng từ kim loại để lưu dữ liệu. Truy cập tuần tự, chậm kinh khủng, nhưng là bước đột phá so với thẻ đục lỗ. IBM 726 tape drive nặng gần 1 tấn, lưu được... 2MB. Ngày nay tape vẫn sống khỏe cho cold storage (LTO-9: 18TB/cuộn).

1956 - 1980s

HDD ra đời - IBM RAMAC 350

Ổ cứng đầu tiên: 50 đĩa từ 24 inch, nặng 1 tấn, chứa 3.75MB, giá thuê $3,200/tháng (tương đương $35,000 ngày nay). Lần đầu tiên con người có thể truy cập dữ liệu ngẫu nhiên (random access) thay vì đọc tuần tự. Cuộc cách mạng thực sự.

1988

RAID - Bài báo thay đổi mọi thứ

Patterson, Gibson, Katz tại UC Berkeley công bố paper "A Case for Redundant Arrays of Inexpensive Disks". Ý tưởng: dùng nhiều ổ rẻ thay vì một ổ đắt. Từ đây RAID trở thành nền tảng của mọi hệ thống storage enterprise.

2000s

SAN & NAS thương mại hóa

NetApp, EMC, Dell trở thành những ông lớn storage. Fibre Channel SAN cho enterprise, iSCSI cho mid-market. NAS (Network Attached Storage) đi vào mọi văn phòng. Synology, QNAP ra đời phục vụ SMB. Storage tách khỏi server, trở thành tầng riêng.

2006-2010

SSD thương mại & ZFS

SSD đầu tiên cho consumer ra đời. Sun Microsystems phát hành ZFS (2005) - filesystem cách mạng với copy-on-write, checksumming, pooled storage. Flash storage bắt đầu thay thế spinning disk cho workload đòi hỏi hiệu năng. Game đã thay đổi mãi mãi.

2010-2015

NVMe & Distributed Storage

NVMe protocol ra đời - nói chuyện trực tiếp với CPU qua PCIe, bỏ qua bottleneck SATA/SAS. Ceph, GlusterFS, HDFS trưởng thành. Software-Defined Storage (SDS) thay thế hardware SAN đắt đỏ. Dữ liệu không còn nằm trên một box - nó phân tán khắp nơi.

2020-nay

CXL, PLC NAND, và Tương lai

CXL (Compute Express Link) mở ra memory-semantic storage. PLC NAND (5 bit/cell) hứa hẹn SSD siêu rẻ. NVMe-oF (over Fabrics) cho phép truy cập NVMe qua mạng. Tiering tự động bằng AI. Ranh giới giữa RAM, storage, và network đang mờ dần.

RAID KHÔNG PHẢI BACKUP - Ghi nhớ bằng xương bằng máu

RAID bảo vệ bạn khỏi hỏng ổ cứng vật lý. RAID KHÔNG bảo vệ bạn khỏi: ransomware (mã hóa tất cả ổ cùng lúc), lỗi người dùng (xóa nhầm file - RAID xóa trên tất cả ổ), lỗi phần mềm (filesystem corruption lan trên cả array), thiên tai (cháy/lũ hủy cả server). RAID là availability, backup là recoverability. Bạn cần CẢ HAI. Luôn nhớ quy tắc 3-2-1: 3 bản sao, 2 loại media, 1 offsite.

Tại sao RAID-Z tốt hơn RAID truyền thống?

RAID 5/6 truyền thống có "write hole" - nếu mất điện giữa lúc ghi data và parity, dữ liệu sẽ không nhất quán. ZFS RAID-Z giải quyết triệt để bằng copy-on-write: không bao giờ ghi đè dữ liệu cũ, mà luôn ghi vào vị trí mới rồi cập nhật pointer. Thêm vào đó, ZFS tự checksum mọi block và phát hiện bit rot mà RAID truyền thống hoàn toàn mù tịt.

Filesystem - Bộ Não Của Storage

Filesystem quyết định cách dữ liệu được tổ chức, bảo vệ, và truy cập trên ổ đĩa

King of Data Integrity

Được Sun Microsystems tạo ra (2005), giờ sống dưới dạng OpenZFS. Filesystem + Volume Manager trong một. Copy-on-write, checksumming mọi block, compression, deduplication, snapshots, send/receive. "Filesystem cuối cùng bạn cần học."

Pro tip

ZFS yêu cầu 1GB RAM / 1TB storage là minimum. Nếu bật dedup, nhân lên x5. ARC (Adaptive Replacement Cache) của ZFS cực kỳ hiệu quả nhưng ngốn RAM. Đừng tiếc RAM cho ZFS - đó là đầu tư xứng đáng nhất.

Btrfs

Linux-native CoW Filesystem

B-tree filesystem, "Butter FS". Mục tiêu ban đầu là thay thế ext4 trên Linux. Copy-on-write, snapshots, subvolumes, compression, checksumming. Synology dùng Btrfs làm default. Tuy nhiên, RAID 5/6 của Btrfs vẫn chưa ổn định sau nhiều năm.

Chon filesystem nao?

Quy tac don gian: NAS/Proxmox/TrueNAS = ZFS (data integrity la so 1). Linux desktop/server thong thuong = ext4 (don gian, on dinh). RHEL/CentOS + file lon = XFS. Synology NAS = Btrfs (Synology da tune tot). Dung chon filesystem vi "nghe noi hay" - chon vi no phu hop workload cua ban.

Network Storage - Khi Dữ Liệu Bay Qua Dây Mạng

Các giao thức cho phép truy cập storage từ xa - mỗi cái sinh ra cho một mục đích

NFS (Network File System)

Vua chia sẻ file trên Linux/Unix. Sun Microsystems tạo ra (1984). NFSv3 phổ biến nhất, NFSv4 thêm security (Kerberos), stateful. Mount từ xa như ổ local. Đơn giản, nhanh, nhưng permission model phụ thuộc UID/GID. Dùng cho VM storage, home directories, shared data.

Linux nativeĐơn giảnUID/GID mappingFile-level

SMB/CIFS (Samba)

Ngôn ngữ của Windows. Microsoft protocol, Samba là implementation trên Linux. SMBv3 có encryption, multichannel, transparent failover. Dùng khi cần chia sẻ file với Windows clients. Active Directory integration tốt. Chậm hơn NFS trên Linux-to-Linux.

Windows nativeAD IntegrationOverhead cao hơn NFSFile-level

iSCSI

Block storage qua Ethernet. SCSI commands đóng gói trong TCP/IP. Tạo virtual disk (LUN) truy cập qua mạng như ổ local. Rẻ hơn Fibre Channel nhiều lần. Cần VLAN riêng, jumbo frames, multipath cho production. Dùng cho VM disks, database.

Fibre Channel (FC)

Premium block storage. Mạng chuyên dụng cho storage, tốc độ 8/16/32/64 Gbps. Latency cực thấp, throughput cực cao. Cần switch FC riêng ($$$), HBA card riêng. Dùng trong enterprise lớn, ngân hàng, bệnh viện. Đang bị NVMe-oF đe dọa.

Tương lai của network storage. NVMe commands chạy qua RDMA, TCP, hoặc FC. Latency gần bằng NVMe local. Biến remote NVMe SSD thành gần như local disk. Đang thay thế dần iSCSI và FC cho high-performance workload.

NAS hay SAN? Câu hỏi muôn thuở

NAS (Network Attached Storage) chia sẻ files qua NFS/SMB - đơn giản, rẻ, ai cũng dùng được. SAN (Storage Area Network) chia sẻ block devices qua iSCSI/FC - hiệu năng cao, cho VM/database. Trong homelab, NAS là đủ cho 90% nhu cầu. SAN (iSCSI) cần khi chạy Proxmox cluster với shared storage cho live migration.

Distributed Storage - Khi Một Máy Là Không Đủ

Phân tán dữ liệu qua nhiều node - scalability, fault tolerance, và sự phức tạp đáng sợ

🐙

Ceph

Unified Distributed Storage

Ceph là "Swiss Army knife" của distributed storage. Một cluster Ceph cung cấp đồng thời block (RBD), file (CephFS), và object (RGW) storage. Dựa trên RADOS - layer phân tán dữ liệu bên dưới. Proxmox tích hợp native. Không cần vendor lock-in.

RADOS - Reliable Autonomic Distributed Object Store. Nền tảng của mọi thứ trong Ceph. Mọi data (block, file, object) cuối cùng đều thành RADOS objects.
RBD - RADOS Block Device. Virtual disk cho VM. Proxmox dùng RBD cho VM storage. Thin provisioning, snapshots, live migration.

MinIO

S3-Compatible Object Storage

MinIO là cách đơn giản nhất để tự host S3-compatible object storage. Viết bằng Go, hiệu năng cực cao (325 GiB/s trên 32 node benchmark). Erasure coding tự động. API 100% tương thích S3. Deployment đơn giản: 1 binary, 1 command. Từ single-node đến distributed cluster.

Ceph hay không Ceph? Câu hỏi cho Homelab

Ceph mạnh nhưng minimum 3 node (khuyến nghị), mỗi node cần ổ riêng cho OSD. Nếu bạn có cụm Proxmox 3+ node, Ceph là lựa chọn tuyệt vời - live migration, HA, self-healing tự động. Nếu chỉ có 1-2 node, hãy dùng ZFS local + NFS share hoặc Proxmox Backup Server. Đừng ép Ceph vào nơi nó không thuộc về.

SSD Endurance - Tuổi Thọ Của Flash Storage

Mọi ô NAND đều có ngày chết - câu hỏi là khi nào, và bạn chuẩn bị thế nào

📊

TBW & DWPD - Hai chỉ số sống còn

Endurance Metrics

TBW (Terabytes Written) - Tổng dung lượng bạn có thể ghi vào SSD trước khi nó "hết tuổi thọ". Ví dụ: Samsung 870 EVO 1TB có TBW 600TB. Nghĩa là bạn có thể ghi 600TB dữ liệu vào ổ này.

DWPD (Drive Writes Per Day) - Số lần ghi đầy ổ mỗi ngày trong thời gian bảo hành (thường 5 năm). Enterprise SSD: 1-10 DWPD. Consumer: 0.3-0.6 DWPD.

Công thức: DWPD = TBW / (dung lượng x 365 x số năm bảo hành)

Thực tế về SSD endurance trong Homelab

Đừng quá lo lắng về TBW cho consumer workload. Một SSD 1TB TLC hiện đại (600TBW) chạy 24/7 với 50GB ghi/ngày sẽ sống ~33 năm trước khi hết TBW. Bạn sẽ thay ổ vì lý do khác (dung lượng, tốc độ) trước khi nó chết vì endurance. Tuy nhiên, với ZFS metadata devices, Ceph WAL/DB, hoặc database write-heavy, hãy dùng ổ enterprise (MLC/high-endurance TLC). Kiểm tra SMART thường xuyên: smartctl -a /dev/sda - tìm Media_Wearout_Indicator hoặc Percentage_Used.

SSD overprovisioning - Bí mật ít người biết

SSD luôn có dung lượng NAND thực nhiều hơn dung lượng quảng cáo. Phần "thừa" này dùng cho wear leveling, bad block replacement, và garbage collection. Enterprise SSD (800GB thay vì 1TB) có ~28% OP vs consumer (~7% OP). Bạn có thể tăng OP bằng cách không partition hết ổ - để trống 10-20% cuối ổ giúp SSD sống lâu hơn đáng kể.

Câu Chuyện Mất Dữ Liệu Kinh Điển

Học từ nỗi đau của người khác - rẻ hơn nhiều so với tự trải nghiệm

GitLab - "Chúng tôi đã xóa production database" (2017)

Một kỹ sư GitLab chạy rm -rf trên thư mục database của production server thay vì staging. 300GB dữ liệu PostgreSQL biến mất trong tích tắc. Điều đáng sợ hơn: trong 5 phương pháp backup họ có, không phương pháp nào hoạt động đúng. LVM snapshots không được cấu hình. pg_dump chạy nhưng file rỗng. Azure disk snapshots chỉ backup 24 tiếng một lần.

GitLab phải livestream quá trình khôi phục trên YouTube. Cuối cùng họ khôi phục được từ bản pg_dump 6 tiếng trước - mất 6 tiếng dữ liệu của hàng triệu user.

Bài học

Test backup thường xuyên. Backup không được test = không có backup. Và đừng bao giờ để một người có quyền rm -rf production database lúc 11 giờ đêm khi đang mệt.

OVH Strasbourg - Cháy data center (2021)

Đám cháy thiêu rụi hoàn toàn data center SBG2 của OVH (nhà cung cấp cloud lớn nhất châu Âu). 3.6 triệu website offline. Nhiều khách hàng mất dữ liệu vĩnh viễn vì họ tưởng "dữ liệu trên cloud = đã được backup". Sai. Cloud provider cung cấp hạ tầng, KHÔNG tự động backup dữ liệu của bạn.

Bài học

Cloud không phải backup. "Nhà cung cấp sẽ lo" là ảo tưởng nguy hiểm nhất. Luôn có backup offsite, ở một geographical location khác. Quy tắc 3-2-1 tồn tại là có lý do.

Toy Story 2 - Suýt mất cả bộ phim (1998)

Một ai đó chạy rm -rf / trên server chứa toàn bộ file animation Toy Story 2. 90% bộ phim bị xóa. Hệ thống backup cũng hỏng - không ai kiểm tra. May mắn thay, Galyn Susman (technical director) đang nghỉ thai sản và có một bản copy trên máy tính ở nhà - bản này đã cứu cả dự án trị giá hàng trăm triệu đô.

Bài học

Bản backup ngẫu nhiên trên máy tính cá nhân đã cứu Pixar. Nhưng đó là may mắn, không phải chiến lược. Hãy có chiến lược backup rõ ràng, test định kỳ, và có nhiều bản copy ở nhiều nơi.

Homelab classic - "RAID controller chết, mang ổ sang máy khác"

Một câu chuyện quen thuộc trong cộng đồng homelab: RAID controller hardware hỏng, người dùng cắm ổ cứng vào máy khác (controller khác) và bật lên. Controller mới không nhận array cũ, khởi tạo lại các ổ, xóa sạch metadata RAID. 20TB dữ liệu ảnh gia đình, video, tài liệu - biến mất.

Bài học

Hardware RAID = vendor lock-in cho dữ liệu. Đây là lý do cộng đồng homelab khuyến nghị software RAID (ZFS, mdadm). Ổ cứng ZFS mang sang bất kỳ máy nào có ZFS đều import được pool. Không phụ thuộc controller hardware cụ thể.

Silent data corruption - Kẻ giết người thầm lặng

Một sysadmin nhận ra ảnh trong kho lưu trữ 5 năm bắt đầu có pixel lạ. Không phải virus, không phải lỗi phần mềm. Đó là bit rot - các bit trên ổ cứng tự thay đổi theo thời gian do từ tính suy giảm. ext4 và XFS KHÔNG phát hiện được vì không checksum data blocks. Khi phát hiện, file đã bị corrupt từ lâu, backup cũng chứa bản corrupt.

Bài học

Dùng filesystem có data checksumming (ZFS, Btrfs) cho dữ liệu quan trọng. ZFS scrub định kỳ phát hiện và tự sửa bit rot. Chạy zpool scrub hàng tháng - đó là "khám sức khỏe" cho dữ liệu.

8 Hiểu Nhầm Chết Người Về Storage

Những điều "ai cũng nghĩ đúng" nhưng khiến bạn mất dữ liệu

Hiểu nhầm #1

"RAID là backup"

Đã nói ở trên nhưng phải nhắc lại vì quá nhiều người vẫn tin. RAID bảo vệ khỏi hỏng ổ vật lý. Ransomware mã hóa tất cả ổ trong RAID cùng lúc. rm -rf xóa trên tất cả ổ. Controller chết có thể kéo cả array đi theo.

Thực tế

RAID = uptime. Backup = recovery. Bạn cần CẢ HAI. RAID giúp server không downtime khi ổ chết. Backup giúp bạn lấy lại dữ liệu khi mọi thứ khác đều thất bại.

Hiểu nhầm #2

"SSD không bao giờ chết đột ngột"

HDD thường có dấu hiệu trước khi chết (tiếng kêu, chậm dần). SSD thì không - nó có thể hoạt động hoàn hảo rồi biến mất hoàn toàn trong nháy mắt khi controller chết. Firmware bug có thể brick cả ổ. Samsung 840 EVO có bug firmware làm đọc chậm dần theo thời gian.

Thực tế

Theo dõi SMART data (smartctl), đặc biệt Reallocated_Sector_Ct, Media_Wearout_Indicator. Nhưng đừng phụ thuộc 100% - luôn có backup bất kể loại ổ.

Hiểu nhầm #3

"Dữ liệu trên cloud thì an toàn"

Cloud provider cam kết độ bền (durability) của hạ tầng, KHÔNG cam kết backup dữ liệu của bạn. S3 có 99.999999999% durability cho objects, nhưng nếu bạn xóa nhầm bucket - nó sẽ vui vẻ xóa. OVH cháy rụi data center, khách hàng mất dữ liệu vĩnh viễn.

Thực tế

Cloud là hạ tầng, không phải chiến lược backup. Bạn vẫn cần backup offsite, versioning, immutable backups. "Somebody else's computer" không có nghĩa là "somebody else's responsibility".

Hiểu nhầm #4

"ZFS cần ECC RAM, không có thì không dùng"

Tin đồn bắt nguồn từ một bài post của một ZFS developer bị hiểu sai. ZFS không YÊU CẦU ECC RAM. Nó KHUYẾN NGHỊ ECC - giống như MỌI hệ thống lưu trữ dữ liệu quan trọng đều nên có ECC. ext4 trên non-ECC RAM cũng dễ corrupt dữ liệu.

Thực tế

ZFS trên non-ECC RAM vẫn tốt hơn ext4 trên non-ECC RAM nhờ checksumming. ECC RAM tốt cho mọi thứ, nhưng thiếu ECC không phải lý do để tránh ZFS.

Hiểu nhầm #5

"NVMe nhanh hơn SSD SATA mọi lúc"

NVMe có bandwidth lý thuyết cao hơn nhiều lần (7GB/s vs 550MB/s). Nhưng với workload đọc ngẫu nhiên file nhỏ (4K random read) - sự khác biệt nhỏ hơn nhiều so với bạn nghĩ. Database nhỏ, web server, VM boot - SATA SSD vẫn đủ nhanh.

Thực tế

NVMe tỏa sáng ở sequential read/write lớn và queue depth cao. Cho homelab thông thường, tiền chênh lệch tốt hơn nên đầu tư vào dung lượng (2x SATA SSD vs 1x NVMe cùng giá) hoặc backup.

Hiểu nhầm #6

"Ceph có thể chạy trên 2 node"

Về kỹ thuật, có thể. Về thực tế, ĐỪNG. Ceph cần quorum cho MON (monitors). 2 node = không có quorum khi 1 node chết = split-brain = data corruption tiềm tàng. Minimum 3 node cho production Ceph là quy tắc sắt.

Thực tế

Ceph minimum 3 node (hoặc 2 node + 1 arbiter cho mon). Nếu có ít hơn 3 node, hãy dùng ZFS replication hoặc DRBD thay vì cố ép Ceph.

Hiểu nhầm #7

"Ổ cứng enterprise luôn tốt hơn consumer"

Enterprise HDD (như WD Ultrastar) thiết kế cho 24/7, vibration tolerance, 5 năm bảo hành. Nhưng chúng cũng NÓNG hơn, ỒN hơn, và ĐẮT gấp 2-3 lần. Trong homelab yên tĩnh, không rung, consumer NAS drive (WD Red Plus, Seagate IronWolf) hoàn toàn đủ dùng.

Thực tế

Chọn dựa trên workload thực tế. Homelab 24/7 nhẹ nhàng = NAS drives. Write-intensive, vibration, rack server = enterprise. Tiền tiết kiệm được hãy mua thêm ổ để tăng redundancy.

Hiểu nhầm #8

"Defrag SSD giúp nó nhanh hơn"

SSD không có đầu đọc vật lý - random access nhanh như sequential. Defrag KHÔNG giúp SSD nhanh hơn. Tệ hơn, defrag ghi thêm dữ liệu không cần thiết, giảm tuổi thọ SSD. Windows 10+ đã biết không defrag SSD (nó gửi TRIM thay vì defrag).

Thực tế

SSD cần TRIM (báo controller biết block nào đã xóa), không cần defrag. Trên Linux: fstrim -av hoặc enable discard mount option. ZFS: autotrim=on.

Storage Cheat Sheet

Những lệnh và kiến thức cần nhớ sẵn trong đầu

ZFS - Quản lý Pool & Dataset

zpool create tank mirror sda sdb # Mirror zpool create tank raidz2 sd{a..f} # RAID-Z2 zpool status # Tình trạng pool zpool scrub tank # Kiểm tra integrity zfs create tank/data # Tạo dataset zfs set compression=lz4 tank # Bật nén zfs set recordsize=1M tank/media # Tune cho file lớn

ZFS - Snapshot & Send/Receive

# Tạo snapshot zfs snapshot tank/data@today # List snapshots zfs list -t snapshot # Rollback zfs rollback tank/data@today # Send snapshot sang máy khác (backup!) zfs send tank/data@today | \ ssh backup-server zfs recv backup/data # Incremental send zfs send -i @yesterday @today | \ ssh backup zfs recv backup/data

SMART - Kiểm tra sức khỏe ổ đĩa

# Xem SMART data smartctl -a /dev/sda # Chạy self-test smartctl -t short /dev/sda smartctl -t long /dev/sda # Các chỉ số quan trọng: # Reallocated_Sector_Ct > 0 = cảnh báo # Current_Pending_Sector > 0 = lo lắng # Media_Wearout_Indicator < 10 = thay SSD # Temperature > 55°C = cải thiện tản nhiệt

Ceph - Quản lý Cluster

# Tình trạng cluster ceph status ceph health detail # Xem OSD tree ceph osd tree # Tạo pool ceph osd pool create mypool 128 # Xem usage ceph df rados df # Performance ceph osd perf

iSCSI & NFS - Cấu hình nhanh

# NFS server (exports) echo "/data 10.0.0.0/24(rw,sync,\ no_subtree_check)" >> /etc/exports exportfs -ra # NFS client mount mount -t nfs server:/data /mnt/nfs # iSCSI discovery iscsiadm -m discovery -t st \ -p 10.0.0.1:3260 # iSCSI login iscsiadm -m node --login

Disk Performance Testing

# Sequential write fio --name=seqwrite --rw=write \ --bs=1M --size=1G --numjobs=1 # Random 4K read (IOPS test) fio --name=randread --rw=randread \ --bs=4k --size=1G --numjobs=4 \ --iodepth=32 # Quick dd test (thô) dd if=/dev/zero of=test bs=1M \ count=1024 oflag=direct # iostat monitoring iostat -xz 1

Lời khuyên cuối cùng

Storage là nền tảng thầm lặng nhất nhưng quan trọng nhất trong hạ tầng IT. Khi nó hoạt động tốt, không ai nhớ đến. Khi nó chết, tất cả đều chết theo. Hãy đầu tư thời gian hiểu sâu về storage - từ cách NAND flash hoạt động đến cách Ceph phân tán dữ liệu. Đừng tiết kiệm ở storage để rồi trả giá bằng dữ liệu. Và nhớ: dữ liệu không tồn tại nếu nó chỉ có một bản sao. Test backup của bạn ngay hôm nay - đừng đợi đến khi cần mới phát hiện nó không hoạt động.
